## Phase 2 — Save/restore lifecycle channel

New stdlib `lifecycleChannel` (id `'lifecycle'`, event/sparse) projects
the trio of platform completion events:

  - `platform.save_failed`      → `{ kind: 'save_failed',      message? }`
  - `platform.restore_failed`   → `{ kind: 'restore_failed',   message? }`
  - `platform.restore_completed` → `{ kind: 'restore_completed' }`

The channel reads `event.payload.error` (the actual field for platform
events; the legacy event-listener was reading `event.data.error` which
was always undefined — a quiet pre-existing bug, fixed by going through
the canonical channel path).

Browser-side, `createLifecycleChannelRenderer` is callback-driven (not
slot-driven) since its two side effects already exist as `BrowserClient`
private methods (`appendSystemMessage` for the system-message DOM
shape, `renderCombinedStatus` for status refresh on `restore_completed`).
`registerDefaultBrowserRenderers` accepts an optional `lifecycle`
options object — when omitted, the renderer is skipped (CLI / test
scenarios that don't need lifecycle UI).

The legacy `restoreData?.turnCount` branch in BrowserClient was dead
(`RestoreCompletedData` carries `slotName`/`timestamp`, no `turnCount`).
The lifecycle channel does not carry a turn count — the standard `turn`
channel renderer keeps `currentTurn` synced on every packet via
`always`-mode re-emission.

Removes lines 276–296 from the `engine.on('event')` listener.

**Tests**:
  - stdlib: 7 new lifecycle.produce tests (33 channel tests total).
  - platform-browser: 7 new lifecycle renderer tests (58 tests total).
  - channel-service: AC-13 / AC-14 / AC-16 unchanged GREEN.
  - engine: 198 passing / 7 skipped.

**ADR candidate A** (raised, not yet written): lifecycle channel lives

## Phase 3 — Audio Tier C migration

Migrates Dungeo's audio handler from the legacy `audio.*` event
vocabulary (routed through `BrowserClient`'s event listener and
forwarded to `AudioManager.handleAudioEvent`) to canonical `media.*`
events that flow through stdlib's ambient/music channels.

Story changes:

- `stories/dungeo/src/audio/audio-setup.ts`:
  - `registerAudioHandler` now emits `media.ambient.play`/`.stop` and
    `media.music.play` instead of `audio.ambient.play`/`.stop_all` and
    `audio.music.play`. Since `media.*` has no `stop_all`, the handler
    emits one `media.ambient.stop` per `DUNGEO_AMBIENT_CHANNEL_IDS`
    entry.
  - `audio.effect` was never handled by `AudioManager` (silently
    dropped today), so it is no longer emitted. Future audio-effect
    support is ADR-B candidate territory.
  - Adds `DUNGEO_AMBIENT_CHANNEL_IDS = ['environment']` — the
    exhaustive set of ambient channel ids the story uses, consumed by
    both the per-channel stop emission and the
    `registerStoryAmbientChannels` hook.
  - `createMediaEvent` helper builds plain `ISemanticEvent` rather
    than routing through `createTypedEvent` — the `media.*` vocabulary
    is not registered in `@sharpee/core`'s `EventDataRegistry` (only
    `audio.*` is, in `@sharpee/media`), and a story package shouldn't
    extend the core registry.

- `stories/dungeo/src/index.ts`:
  - `DungeoStory.registerChannels(registry)` hook (ADR-163) registers
    `createAmbientChannel('environment')` so the `ChannelService`
    projects `media.ambient.*` events onto the dynamic
    `ambient:environment` channel.

- `stories/dungeo/src/browser-entry.ts`:
  - After `client.connectEngine`, registers `createAmbientChannelRenderer`
    on the channel renderer for each `DUNGEO_AMBIENT_CHANNEL_IDS` entry,
    wiring channel-side payloads back into the shared `AudioManager`.

Platform changes:

- `packages/platform-browser/src/BrowserClient.ts`:
  - Removes the `audio.*` branch from the `engine.on('event')`
    listener (the third of four leftover responsibilities).
  - Adds public `getChannelRenderer()` and `getAudioManager()`
    accessors so stories can register custom renderers in the window
    between `connectEngine()` and `start()`. `getChannelRenderer`
    throws if called before `connectEngine`.

- `packages/channel-service/tests/ac-16-cleanup-grep-gate.test.ts`:
  - Allow-lists `stories/dungeo/src/audio/` — the dungeo audio handler
    now legitimately emits canonical `media.*` events. ADR-B
    candidate would relocate this translation to stdlib so stories
    stay in the consumer role.

After Phase 3, only the `handleStoryEvent` branch remained in the
legacy listener.

## Phase 4 — Retire `handleStoryEvent` + AC-17 gate

Story-defined channels replace the `handleStoryEvent` event-listener
bypass entirely. After this commit, the only engine subscriptions in
`BrowserClient` are `channel:manifest` and `channel:packet`. The
legacy `engine.on('event')` listener is deleted.

Story-side migration:

- `stories/dungeo/src/channels.ts` (new):
  - `rnameChannel` (id `'dungeo.rname'`, event/sparse) projects
    `dungeo.event.rname` to the room name string.
  - `objectsChannel` (id `'dungeo.objects'`, event/sparse) projects
    `dungeo.event.objects` via the `formatDungeoObjects` helper
    (moved from `browser-entry.ts`).
  - `registerStoryEventChannels(registry)` helper called from
    `DungeoStory.registerChannels`.

- `stories/dungeo/src/index.ts`:
  - `DungeoStory.registerChannels` now invokes both
    `registerStoryAmbientChannels` (Phase 3) and the new
    `registerStoryEventChannels`.

- `stories/dungeo/src/browser-entry.ts`:
  - Removes `handleStoryEvent` function and the inline `formatObjects`
    helper.
  - Removes `callbacks: { handleStoryEvent }` from the `BrowserClient`
    config.
  - Registers two new renderers — `dungeo.rname` and `dungeo.objects`
    — both calling `client.displayText` with the channel-projected
    string.

Platform-side retirement:

- `packages/platform-browser/src/BrowserClient.ts`:
  - Deletes `setupEngineHandlers()` and its call site. The previous
    body was the empty `engine.on('event')` listener after Phases 1–3
    removed beep, save/restore, and audio routing branches.
  - Drops the now-unused `import type { ISemanticEvent }`.
  - Folds the debug-mode duplicate `channel:manifest` listener into
    the single listener, gating the `console.log` on `debugChannels`
    inside the same callback. Required so AC-17's "exactly two
    `engine.on()` calls" gate counts cleanly regardless of debug mode.

- `packages/platform-browser/src/types.ts`:
  - Removes `handleStoryEvent?` from `BrowserClientCallbacks`.
  - Drops the now-unused `ISemanticEvent` import.
  - Header comment reframes `BrowserClientCallbacks` as reserved for
    future hooks; documents that channels are the sanctioned
    replacement for raw-event interception.

AC-17 grep gate (new):

- `packages/channel-service/tests/ac-17-two-listeners-only.test.ts`:
  - Static gate over `BrowserClient.ts` source. Three assertions:

## Key Decisions

- **Lifecycle channel goes in stdlib** (Phase 2). Plan default; user
  authorized via `proceed`. ADR-A candidate.
- **Story owns the `audio.*` → `media.*` translation in Phase 3**, not
  stdlib. Faster to ship; long-term home (ADR-B) may relocate the
  translation to stdlib so stories stay in the consumer role.
- **Story-defined channels replace `handleStoryEvent` outright** in
  Phase 4 — no deprecation period (per the project's
  no-backwards-compat policy). ADR-C candidate.
- **AC-17 is a static grep gate** over `BrowserClient.ts` source, not
  a runtime listener-count assertion. Mechanical and zero-cost; runs
  with the rest of the channel-service test suite.
