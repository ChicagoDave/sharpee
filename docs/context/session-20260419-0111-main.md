## Key Decisions

### 1. Server location: `tools/server/` not a workspace package
David chose to place the server at root-level `tools/server/` next to `tools/vscode-ext/`. Not a member of the pnpm workspace, not a standalone repo. Consumes Sharpee via `file:../../packages/core`. This keeps the server from polluting the platform package graph and makes it trivially deployable as a Docker artifact.

### 2. ConnectionManager owned externally, injected into both WsServer and RoomManager
Pulling `ConnectionManager` construction out of `WsServer` and passing the single instance into both `WsServer` and `RoomManager` avoids a circular dependency that would otherwise require a placeholder-then-patch pattern at startup.

### 3. Sandbox binary is injectable for testing
`SandboxProcess` accepts `{binary, args}` override. Production uses `deno`; tests use `node` + `tests/fixtures/stub-sandbox.mjs`. This keeps the full Phase 4 test suite runnable without Deno installed and makes CI straightforward.

### 4. Error envelope shared across HTTP and WebSocket
Same `{ code, detail }` shape via a Hono `HTTPException` subclass. Clients render one error UI regardless of transport. Error codes are namespaced (`token_invalid`, `room_not_found`, `captcha_failed`, etc.) and match the wire-protocol error frame definition in `src/wire/browser-server.ts`.

### 5. Behavior Statement discipline applied throughout
Every test file carries a header Behavior Statement (DOES / WHEN / BECAUSE / REJECTS WHEN) derived from the action's invariants. Every assertion traces back to a DOES or REJECTS WHEN line. No mock-only assertions; all tests assert on actual DB state, actual registry state, or actual socket message payloads.
