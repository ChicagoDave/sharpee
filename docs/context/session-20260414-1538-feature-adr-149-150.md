## Key Decisions

### 1. `w.all` instead of `w.entities`
`WorldModel` has a private `entities: Map<string, IFEntity>` field. Assigning a prototype getter named `entities` silently collides with this private field — the getter is shadowed and never fires. Renamed the entry point to `w.all` to avoid the collision. This is documented in ADR-150.

### 2. Dual Declaration Merge (IWorldModel + WorldModel)
Merging only `IWorldModel` fails at runtime when `WorldModel` is passed as `IWorldModel` — TypeScript structural type checking looks at the actual class, and if `WorldModel` does not also declare the properties, the build rejects assignments or the methods are missing at runtime. Both interfaces must be augmented.

### 3. `in()` Method Dropped from EntityQuery
ADR-150 originally proposed an `in(containerId)` method on EntityQuery. After review, spatial scoping is better handled through entry points (`w.contents()`, `w.allContents()`) rather than mid-chain filtering. The method was removed from the spec and implementation.

### 4. Atomic Region/Scene Creation Helpers (ADR-149)
`world.createRegion()` and `world.createScene()` replace multi-step entity creation where it was easy to forget to add the trait or set required fields. `world.assignRoom()` validates the region exists before assigning, preventing silent partial-state bugs.

### 5. `getRegionCrossings()` as Boundary Contract (ADR-149)
Hierarchy traversal logic (which regions were entered or exited) lives in world-model, not in the going action. The going action calls `world.getRegionCrossings(from, to)` and receives `{ entered: string[], exited: string[] }`. This keeps spatial logic encapsulated.

### 6. Dungeo Integration Import Strategy
`import '@sharpee/queries'` side-effect must be placed at the story entry point (`index.ts`), not in individual handler files. The bundle resolves the augmentation once at load time; per-file imports cause resolution failures in the bundled output.
