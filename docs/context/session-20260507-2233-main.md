## Phase 2 — Port Four Legacy Themes

### Decision: Per-theme kit pattern (B over A)

At session start, two interpretations of Phase 2 were presented:

- **A** — Single-pass selector remap of the shared rule set (1 commit, pragmatic minimum).
- **B** — Per-theme complete CSS kits, each scoped under `[data-theme="<id>"]` (matches the ADR-170 "complete component kit per theme" framing).

David chose **B**.

### Specificity trap surfaced and resolved

Legacy selectors used id-qualified rules (`#menu-bar`, `#status-line`, `#modal-overlay`) at specificity `1,0,0`. New per-theme `.sharpee-*` selectors under `[data-theme="X"]` produce specificity `0,2,0` — they lose. The two rule sets cannot coexist: a `[data-theme="X"] .sharpee-menu-bar` rule would be silently overridden by any surviving unscoped `#menu-bar` rule.

Resolution: delete all unscoped legacy rules from `infocom.css` in the same step as the `dos-classic` kit lands. This accepted that `modern-dark`, `retro-terminal`, and `paper` would be visually broken until their kits landed — consistent with Phase 1's intentional-breakage philosophy.

### dos-classic kit

Full `[data-theme="dos-classic"]` block added targeting all 22 `.sharpee-*` component classes, plus the dialog-internal utility classes preserved from the prior visual design: `.save-input-row`, `.save-slot`, `.saves-list-label`, `.no-saves-message`, `.startup-info`, `.startup-question`, `.command-echo`, `.system-message`, `.game-status`. Color palette retained at `:root, [data-theme="dos-classic"]` variable block.

Removed at the same time:
- All unscoped legacy `#menu-bar`, `.menu-button`, `.menu-dropdown`, `.modal-dialog`, `#modal-overlay` rules.
- Dead `--dos-*` legacy variable aliases.
- Unused `@keyframes blink` and the orphan `#command-input:focus { caret-color }` rule.

## Key Decisions

### 1. Per-theme kit pattern (B over A) — ADR-170 alignment

The plan's text says "complete component kit per theme," which means each theme is a self-contained `[data-theme]`-scoped block. The pragmatic alternative (single shared remap) would have worked visually but would have diverged from the ADR's structural intent. David confirmed B.

### 2. Specificity-driven legacy removal in the same step as dos-classic

Could not keep unscoped legacy rules as a fallback alongside new `[data-theme="X"] .sharpee-*` rules. The legacy id rules win on specificity. Deleted legacy unscoped rules atomically with the dos-classic kit. Accepted that other themes would be temporarily broken — same philosophy as Phase 1.

### 3. Single-commit Phase 2

`git blame` on a `[data-theme="paper"]` line already isolates any regression to the paper kit. Per-theme commits inside one file would have required interactive staging for marginal bisect benefit. David chose the simpler single commit.

### 4. Visual verification deferred to Phase 4

CSS rendering cannot be verified in jsdom or Node. The walkthrough chain is the JS-side gate. Per the plan, the manual real-browser smoke test is Phase 4's acceptance gate. No per-kit visual verification was attempted this session.
