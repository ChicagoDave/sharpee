### 5. Root Cause: deno-entry.ts Not Copied to dist/

Investigation path:
- `loadStory` spawns a Deno subprocess
- `sandbox-process.ts:62` resolves the entry path to `dist/sandbox/deno-entry.ts` (Deno reads `.ts` directly; no compilation)
- `tsconfig.json` has `exclude: ["src/sandbox/**"]` â€” sandbox TypeScript files are excluded from tsc compilation
- Other sandbox files (`sandbox-process.js`, `message-framing.js`, `sandbox-registry.js`) do get compiled because they are `import`ed by non-sandbox server code, so tsc sees them as reachable
- `deno-entry.ts` is never imported anywhere â€” it is only ever **spawned** at runtime via `child_process.spawn`. tsc never sees it, never emits it, never copies it to `dist/sandbox/`
- Result: `dist/sandbox/deno-entry.ts` does not exist inside the container; Deno subprocess fails immediately

### 6. Fix: One Line in Dockerfile Builder Stage

Added `RUN cp src/sandbox/deno-entry.ts dist/sandbox/deno-entry.ts` immediately after `RUN npx tsc` in the builder stage, with an explanatory comment:

```dockerfile
# deno-entry.ts is excluded from tsconfig (sandbox isolation boundary) and is
# never imported by server code, so tsc does not copy it to dist/. Deno reads
# .ts directly at runtime, so we ship the source file explicitly.
RUN cp src/sandbox/deno-entry.ts dist/sandbox/deno-entry.ts
```

No other files required changes.

### 2. deno-entry.ts Fix Location: Dockerfile, Not tsconfig

The fix could have been approached two ways:
- Remove `src/sandbox/**` from the `tsconfig.json` exclude list â€” but the exclude exists to enforce the sandbox isolation boundary. Removing it might cause tsc to try to compile Deno-specific code using Node typings, breaking the build.
- Copy the file explicitly in the Dockerfile after `tsc` runs.

The Dockerfile cp is the minimal, targeted fix. It preserves the tsconfig intent and makes the special-case explicit and documented.

### 3. Comment in Dockerfile Documents the "Why"

The explanatory comment above the cp line ensures that a future maintainer who sees a `.ts` file being manually copied understands it is not a mistake or oversight. This is the same rationale applied to all non-obvious infra choices in this project.
