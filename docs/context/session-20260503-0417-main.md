## Key Decisions

### 1. `world: unknown` in `ChannelProduceContext`

`ChannelProduceContext` in `@sharpee/if-domain` declares `world: unknown`
rather than `world: IWorldModel`. This breaks a potential dependency cycle:
`if-domain` → `world-model`. Consumers (`stdlib` channel closures, story
channel closures) cast `world` to the concrete type they know they'll receive
(`WorldModel` or `IWorldModel`) at call sites. The engine guarantees the
actual value is a valid `WorldModel` instance.

### 2. Closure-per-channel replaces rule-based routing

The rule-based model (`ChannelRule`, `addRule`, `applyMatchingRules`) is fully
deleted. Each `IOChannel` carries its own `produce` closure. There is no shared
router. This eliminates rule-conflict resolution, reduces coupling, and makes
capability gating per-channel rather than system-wide. The implementation gain:
the `ChannelService.build()` loop is ~30 lines vs. the former 200+ line
rule-matching engine.

### 3. Standard channels are NOT capability-gated; media channels are

`mainChannel`, `promptChannel`, `locationChannel`, `scoreChannel`, `turnChannel`,
and the info/notification channels carry no `gatedBy` field — they always
appear in the manifest and always produce. Only media channels (`image:*`,
`sound`, `music`, `animation`, etc.) are gated behind capability flags
(`images`, `audio`, `animation`). This decision keeps text-only clients simple:
their manifest already omits the media channels; no filter logic in the
produce closures needed for text channels.

### 4. Engine constructs ChannelService in `start()`, not in constructor

`Story.registerChannels?(registry)` must run before `new ChannelService(...)` so
the story can add custom channels to the registry. Since `start()` is where the
story initializes, constructing ChannelService in the engine constructor would
race the story. Deferring to `start()` — after `registerChannels?` — guarantees
the registry is fully populated before the capability-filtered manifest is built.

### 5. `DEFAULT_TEXT_CAPABILITIES` guards legacy callers

Legacy engine callers (CLI transcript tests, unit tests, Zifmia) do not pass
`capabilities` to `engine.start()`. The `DEFAULT_TEXT_CAPABILITIES` constant
(text only; no images, audio, or animation) is used when no argument is
supplied. This means legacy code keeps working and produces a text-only channel
manifest — no breaking change until R8.

### 6. ADR-165 Renderer lives in channel-service (Option A)

Rather than creating a new `@sharpee/renderer` package, the `Renderer` class
and `ChannelRenderer` interface live in `packages/channel-service/src/renderer/`.
The renderer code is small (~200 lines), depends only on the wire packet types
(already in if-domain, re-exported from channel-service), and adding a package
for it would increase workspace config overhead without architectural benefit.

### 7. Shadow-mode parallel run (R5-B) before cutover (R5-C)

Rather than replacing `text:output` in a single step, R5-B ran the new channel
renderer in a hidden shadow alongside the live `text:output` path. This allowed
verifying that `channel:packet` flows were correct (and unit-testing the shadow
renderer) without risking the live browser experience. R5-C then promoted the
shadow to the visible path and retired `text:output` in a single, well-tested
commit.

### 8. Auto-save migrated from `text:output` to `channel:packet`

The browser auto-save hook previously fired on `text:output`. A `text:output`
event is only emitted when there are text blocks (empty turns, meta commands,
and idle ticks produce no blocks). `channel:packet` fires on every completed
turn regardless. Moving auto-save to `channel:packet` ensures saves are not
skipped on turns that produce channel data but no text output.

### 9. Score+turn composite override for legacy host layouts

Some host pages render score and turn count in a single combined `<span>`
element rather than two separate elements. The `registerDefaultBrowserRenderers`
function detects this pattern and installs a composite override renderer for the
combined slot that writes both values in a single DOM update. This keeps legacy
host HTML working without modification.

### 10. `flattenContent` inlined in stdlib (not imported from channel-service)

stdlib's channel closures need `flattenContent` to normalize `TextContent`
arrays. Importing it from `@sharpee/channel-service` would introduce a
dependency cycle (channel-service depends on if-domain types; stdlib would then
depend on channel-service). Instead, `flattenContent` is re-defined in
`packages/stdlib/src/channels/utils/` — a small enough function to duplicate
safely. The canonical implementation in `channel-service/src/utils/flatten.ts`
remains for channel-service consumers (CLI renderer, Playwright tests).

### 11. AC-14 asserts on `<p class="main-entry">`, not generic paragraph

The Playwright test could have asserted on any paragraph content. Using the
CSS class `main-entry` (assigned by the `main` channel's DOM renderer, not by
the legacy `TextDisplay` path) makes the assertion load-bearing: it
distinguishes channel-driven output from any legacy fallback. A passing AC-14
therefore proves the R5-C cutover is live in the bundle under test, not just
that text appeared somewhere.

### 12. CLI bundle consumes `channel:packet` via flat `outputBuffer` (not a Renderer instance)

The plan sketched a full CLI-flavored `Renderer` class with terminal-output
`ChannelRenderer` implementations for each channel. The implemented approach is
lighter: `bundle-entry.js` subscribes to `channel:packet` directly, reads
`payload.main` from the turn packet, flattens via `flattenContent`, and pushes
to the existing `outputBuffer` accumulator. The `channel:manifest` listener is
a no-op placeholder for future REPL-status work. This avoids creating a new
`Renderer` instance just to route a single channel's content — the CLI
transcript test cares only about main-channel text, and the flat subscription
is sufficient for AC-13. The no-op manifest listener documents the hook point
without over-engineering.

### 13. `debugStatsChannel` payload is purely state-driven (no `turn` field)

The first-draft `DebugStatsPayload` included a `turn` field alongside
`inventoryCount`. Since `turn` changes every command, every packet was unique
and the sparse-emit policy (change-detection) never suppressed a repeat. Removing
`turn` from the payload makes the payload purely state-driven: the channel
produces only when inventory size changes. This is the semantically correct
design for a "debug stats" channel and the one that exercises the sparse-emit
contract meaningfully.

### 14. `stories/channel-service-test/` uses stdlib actions, not a custom `stat` verb

The plan proposed a custom `stat` verb to trigger `debugStatsChannel` reads.
The implementation drops the custom verb — natural stdlib actions (look, take,
drop) change inventory state, which the `debugStatsChannel` closure detects on
every turn. This exercises the same sparse-emit semantics with one fewer moving
part (no custom grammar extension, no custom action, no additional test surface
for the verb dispatch path). Documented in `playable-story.ts` header.

### 15. AC-16 allow-list design: 6 prefixes cover all legitimate owners

The grep gate could have been implemented with a single broad exclusion
(e.g., "all of stdlib/"). Instead, 6 precisely-scoped prefix allow-lists make
the allowed zone explicit: stdlib channels source, stdlib channel tests,
channel-service source, channel-service tests, the test story, and the
auto-generated API docs. Any new file that emits ADR-101 event strings outside
these 6 prefixes fails the gate immediately — no false passes from accidentally
wide exclusions. The allow-list is also the canonical documentation of which
packages are permitted to know about ADR-101 event types.

### 16. Legacy platform deletion (Option A) over archival

Both `packages/platforms/cli-en-us/` and `packages/platforms/browser-en-us/`
were confirmed dead paths after R5-C (browser) and R6 (CLI). Three options were
on the table: delete (Option A), archive to `.archived/` (Option B), retain
with a deprecation header (Option C). Option A chosen (user-authorized) because
no external consumers reference these packages (they were always internal
implementation), and retaining dead code creates maintenance surface. Cloak of
Darkness's `run.ts` was the only caller; rewriting it to delegate to the CLI
bundle restores playability without coupling to the deleted packages.
