## Key Decisions

### 1. Single-file, no-inheritance rule
Each version is one standalone .ts file with no imports between versions. Deliberate duplication is preferred over abstraction because the reader needs to read a single file and understand a complete working story. Carried-forward code gets a one-line back-reference comment, not a shared import.

### 2. tutorials/ as a first-class top-level directory
Rather than placing the tutorial under stories/, it gets its own tutorials/ directory. This makes the distinction between "games built with Sharpee" and "tutorial material about Sharpee" clear in the repo layout.

### 3. Kindergarten-level comments
Every import, trait, method call, and configuration property gets a plain-English explanation in the version where it is first introduced. This is intentionally verbose because the audience is new Sharpee authors, not developers maintaining the platform.

### 4. All transcript tests target the latest version
Assertions must be compatible with the current V10 story structure. As new versions are added, the transcript for earlier versions still targets that earlier version's `story:` field in the transcript header, so they remain independently testable.

### 5. build.sh resolver helpers instead of registration
Rather than requiring each tutorial to be "registered" in build.sh, `resolve_story_dir()` and `resolve_story_pkg()` check both stories/ and tutorials/ and return the correct path. This avoids build script maintenance overhead as more tutorials are added.
