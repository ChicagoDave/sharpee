- **Who can save**: Any Command Entrant (not Participant; not Co-Host-gated). Saves are room-scoped, not user-scoped.

## Key Decisions

### 1. Save/Restore Is Command-Entrant-Level, Not Role-Gated Above

Saves are a gameplay action, not an admin action. Any Command Entrant can save or restore. This keeps the mechanic accessible to all active players without requiring Co-Host elevation just to checkpoint progress.

### 2. Auto-Named Saves Only

No user-supplied save labels in MVP. Auto-naming (timestamp + room title fragment) eliminates a UX surface that adds implementation cost for negligible early-adopter value. Human-readable labels are a post-MVP enhancement.

### 3. Type-to-Confirm Delete

Room delete is irreversible (cascade-deletes all participants, saves, and event log). Requiring the user to type the exact room title makes the action deliberate and self-documenting. Modal-with-checkbox is insufficiently friction-laden for an operation of this finality.

### 4. Hono + `ws` + `better-sqlite3` + Raw SQL

Chosen over Express, Fastify, Socket.IO, Prisma/TypeORM. Each choice favors minimal abstraction at MVP scale. Hono is TypeScript-native with no overhead. Bare `ws` avoids Socket.IO's namespace/room model which duplicates Sharpee's own room concept. Synchronous SQLite driver fits single-process architecture without async plumbing. Raw SQL with repositories keeps queries transparent and auditable; Kysely is the named upgrade path.

### 5. ADR-152 Abandoned Rather Than Rewritten

The prior session left ADR-152's fate open. Decision: abandon with a documented rationale rather than rewrite or delete. The ADR's historical context (message-passing boundary, security analysis of `.sharpee` execution) has reference value; abandonment preserves that while removing it from normative status. ADR-153 supersedes it entirely.

### 6. ADR-153 Patched In Place (Not Iterated to a New File)

The `/adr-review` gaps were addressed by expanding the single ADR-153 document rather than creating a revision. This is appropriate for a Proposed ADR that has not yet been implemented â€” no migration concerns, no readers depending on the prior version.

### 7. Zifmia Is Explicitly Not a Dependency

ADR-153's package dependency table calls out that the multiuser server does not import Zifmia. The server runs the engine directly via `@sharpee/engine`. This is a conscious architectural boundary: Zifmia is a client-side bundle runtime, not a server-side library.
