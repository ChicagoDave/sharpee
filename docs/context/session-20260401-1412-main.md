### Grammar: "using" synonym for "with" in instrument patterns

Investigated whether `USING` was a supported preposition for instrument patterns (e.g., `OPEN SILVER DOOR USING BRASS KEY`). It was not.

Initial approach duplicated all 7 instrument pattern definitions with `using` variants, adding 9 new `.define()` calls. This was then refined by discovering that the `PatternToken` interface in `packages/if-domain/src/grammar/grammar-builder.ts` already supports alternates via `|` pipe syntax (e.g., `in|into|inside`). The duplication was reverted and `with|using` was substituted in each existing pattern instead — zero new rules, single-file change.

### ADR-138: Audio System Architecture

Created `docs/architecture/adrs/adr-138-audio-system.md` — a comprehensive audio system architecture for Sharpee.

**Three audio categories:**
- SFX — one-shot sound effects
- Music — single active track with crossfade transitions
- Ambient — layered named channels (multiple concurrent sounds)

**Core design elements:**
- **Procedural audio via named recipes** — clients synthesize sound from oscillators and noise sources, no audio files required for baseline support
- **Audio effects as hints** — reverb, lowpass, highpass, distortion, delay; client may ignore unsupported effects
- **AudioRegistry** — central registration system in each story. Stories register all audio (cues, variation pools, atmospheres) in one file. Actions, daemons, and handlers reference names only, never sound design details.
- **AudioCue pattern** — `audio.cue('skin.activate')` returns typed events; whether the cue resolves to a single file or a pool is transparent to the caller
- **Variation pools** — multiple source variants per logical sound with random selection plus pitch/volume jitter to prevent repetition fatigue
- **Volume ducking** — `DuckPriority` (0–3) on SFX/procedural events, configurable `DuckingConfig` (attack, release, target channels)
- **Fluent AtmosphereBuilder** — `.ambient().music().effect().build()` matching the grammar API style used elsewhere in the platform

**TypeScript type definitions** in proposed `@sharpee/media` package:
- 9 event interfaces, discriminated union, type guards
- `EventDataRegistry` declaration merging with `@sharpee/core` (no `as any` casts)
- `createTypedEvent()` for event construction

**Concrete Entropy story examples** in the ADR:
- Atmosphere handler on `if.event.actor_moved`
- Action `report` and `blocked` phases emitting audio cues
- Scheduler daemon for ambient cycling

**Capability negotiation and player preferences** — per-category volume, mute state, persisted to localStorage.

The design went through several iterations: started with abstract examples, made concrete with real Sharpee integration points; removed all `as any` casts in favor of declaration merging; moved sound design out of actions into the AudioRegistry with the AudioCue abstraction; added the fluent atmosphere builder; incorporated ducking and variation pools after professional audio assessment.

### ADR-139: Speech Accessibility Architecture

Created `docs/architecture/adrs/adr-139-speech-accessibility.md` — text-to-speech and voice input for blind and motor-impaired players.

**TTS design:**
- IF-aware speech from existing text block types — room names, descriptions, and dialogue receive different prosodic treatment automatically
- Zero story author burden: works without any author-side changes
- Screen reader coordination: suppresses `aria-live` regions when TTS is active to prevent double-reading

**STT design:**
- Voice command input via Web SpeechRecognition API
- Transcription submitted to the parser as normal text input — no special parser path needed
- Push-to-talk default

**Implementation order**: ADR-138 first (establishes AudioManager and AudioContext infrastructure), then ADR-139 builds on top of it.

## Key Decisions

### 1. Use pipe alternates instead of duplicating grammar rules

The `PatternToken` interface already supports `word1|word2` syntax for prepositions and other literal tokens. Using `with|using` instead of cloning each rule keeps the grammar compact and avoids maintenance debt from near-identical parallel rules.

### 2. AudioRegistry as the single source of sound design truth

Rather than letting sound design details bleed into actions and handlers (file paths, volume levels, variation counts), all sound definitions are registered centrally. Actions emit semantic cue names only. This allows redesigning all audio without touching game logic.

### 3. Procedural audio as a first-class option

Named recipes (oscillator + noise synthesis) mean the platform has a working audio experience with zero asset files. File-based audio is additive, not required. This lowers the barrier for story authors and clients that cannot bundle audio files.

### 4. Declaration merging for type safety

Rather than using `as any` or a loose `Record<string, unknown>` event data type, audio events extend `EventDataRegistry` via TypeScript declaration merging. This gives full type safety at call sites while remaining an opt-in package with no forced dependency on `@sharpee/core`.

### 5. ADR-138 before ADR-139

ADR-139 (speech) depends on the AudioManager and AudioContext established by ADR-138. Documenting the dependency order explicitly avoids implementation sequencing mistakes.
