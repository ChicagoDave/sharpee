## Key Decisions and Lessons

### 1. `@main` on `NSApplicationDelegate` does not work on macOS 26 / Xcode 26

This was the session's biggest time sink. The `@main` attribute on a class conforming to `NSApplicationDelegate` launches the process but `applicationDidFinishLaunching` never fires — no window appears. The symptom looks like the app launched successfully but is invisible.

**Fix**: use an explicit `main.swift`. A file literally named `main.swift` is the Swift module's top-level code entry point. It creates `NSApplication.shared`, assigns the delegate, and calls `app.run()`. This works on all macOS versions and requires no Swift macros or attributes.

**Lesson**: for new AppKit projects on macOS 26, start with `main.swift` and skip `@main` entirely.

### 2. Setting `contentViewController` shrinks the window to fitting size

Assigning `window.contentViewController = vc` causes AppKit to resize the window's content area to `vc.view.fittingSize`. For a view that has not yet been laid out, `fittingSize` is essentially the minimum constraints size — far smaller than 1400×900.

**Fix**: always call `window.setContentSize(...)` *after* the `contentViewController` assignment. The autosaved frame (from `setFrameAutosaveName`) restores the window position and size across launches, but the `setContentSize` call is needed on first launch before any autosave exists.

### 3. Divider autosave vs. initial layout

`NSSplitView` restores divider positions automatically using `autosaveName`. The risk is setting programmatic initial positions on every launch, which would stomp the user's saved positions. Guard by checking whether the UserDefaults key `"NSSplitView Subview Frames {autosaveName}"` exists before setting any positions — if it exists, the restored positions are already correct.

### 4. Window title and status bar text

Late-session preference: window title is "Sharpee", not "Sharpee IDE". Status bar shows "main · Sharpee 0.1.0" (the branch name will be dynamic in a later phase).

### Swift 6 concurrency issues encountered and resolved

Two concurrency issues arose during P1 work. Both will recur as more UI delegates and completion handlers are added.

**Issue 1 — `NSOpenPanel` completion closure sendability**

```
error: sending value of non-Sendable type 'NSOpenPanel' ... risks causing data races
```

The completion handler passed to `NSOpenPanel.begin(completionHandler:)` is a `@Sendable` closure. When `AppDelegate` was not explicitly main-actor isolated, the compiler could not guarantee the closure body ran on the main actor.

Fix: annotate `AppDelegate` with `@MainActor`. The implicit isolation propagates into all closures defined within the class.

**Issue 2 — Non-`@MainActor` protocol conformance from a main-actor type**

```
error: conformance of 'MainSplitViewController' to protocol 'ProjectTreeDelegate' crosses
into main actor-isolated code and can cause data races
```

`MainSplitViewController` inherits from `NSSplitViewController`, which is main-actor isolated in Xcode 26 / Swift 6. Conforming to a protocol that has no explicit actor annotation caused the compiler to reject the conformance.

Fix: annotate `ProjectTreeDelegate` itself with `@MainActor`. Because all conforming types are UI view controllers (inherently main-actor), this is correct and makes the declaration site explicit rather than implicit.

**Pattern for future work**: any new protocol intended as a delegate for an `NSViewController` or `NSWindowController` subclass should be declared `@MainActor` upfront. Any `AppKit` completion-handler API called from within `AppDelegate` or a view controller is safe because those types are already `@MainActor`; the annotation just makes it explicit.
