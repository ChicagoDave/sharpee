### Phase 3: Save/restore round-trip e2e test

Largest sub-effort; two real platform bugs found and fixed.

- `tools/server/e2e/helpers/statusline.ts` — `readStatusLine()` parses location/score/turns from rendered StatusLine component via `aria-label="Game status"`.
- `tools/server/e2e/tests/save-restore.spec.ts`.

**Bug 1 — race condition (test harness):** `readStatusLine` matched the pre-command value before the post-save `look` turn completed. Fixed by switching to `expect.poll` keyed on turn-counter bump.

**Bug 2 — plan assertion mismatch (design):** `restored.turns === preSave.turns` failed because the multi-user sandbox owns its own monotonic `turnCount` (per ADR-162); `RESTORE` calls `hooks.applyTurnCount(world)` which overwrites the restored world's scoring-capability turn count with the live sandbox value. Assertion was architecturally unrealistic. Pivoted to asserting on `location`: moved player north before save; verified location reverts to start room after restore.

**Bug 3 — stale .sharpee bundle (real platform bug):** Location stayed at "Forest Path" after restore even though the RESTORED marker appeared. Investigation traced through:
- Verified engine source had the `world.loadJSON()` fix from `bf9b9564`.
- Verified docker container had it (container rebuilt).
- Found `dungeo.host.js` bundle inside the running container still contained `state.spatialIndex.loadJSON(...)` — the pre-fix v1 engine code.
- Root cause: `dungeo.sharpee` bundle in `/data/stories/` was from 2026-04-22, before the engine fix. Server loads the story bundle at startup; the Deno host.js is embedded in that bundle.
- Fix: `./build.sh -s dungeo -b --no-version`, copied `dungeo.sharpee` to `tools/server/stories/`, cleared compiled `host.js`, restarted container.

**Bug 4 — engine.executeTurn('restore') vs engine.restore() (real platform bug):** Location STILL stayed at "Forest Path" after container restart with fresh bundle. Full investigation:
- `restoringAction.validate()` checks `sharedData.saves` — populated by `loadSaveData()` in CLI/REPL flow, but empty in multi-user sandbox because saves live in SQLite, not in capability data.
- `validate()` returns `no_saves` → action goes to `blocked()` → `report()` never runs → `RESTORE_REQUESTED` platform event never fires → `engine.loadSaveData()` never called.
- Sandbox was emitting a RESTORED frame anyway (from its own state snapshot), masking the failure.
- Found public `engine.restore()` method (line 1565 of engine source) that bypasses the verb pipeline: calls `onRestoreRequested()` hook directly → `loadSaveData()` → `world.loadJSON()`.
- Fix: replaced `engine.executeTurn('restore')` with `engine.restore()` in `tools/server/src/deno-entry.ts`.

All four specs green after both fixes.

## Key Decisions

### 1. engine.restore() is the correct multi-user restore path

`engine.executeTurn('restore')` routes through `restoringAction.validate()` which checks `sharedData.saves` — always empty in multi-user because saves live in SQLite, not capability data. The public `engine.restore()` method bypasses the verb pipeline and calls `onRestoreRequested()` directly. This was a pre-existing bug masked by the sandbox emitting a RESTORED frame regardless. Fix committed in `0ea2ba9f`.

### 2. Story bundle in dev pipeline must be rebuilt after engine changes

`dungeo.sharpee` embeds a compiled `host.js` that includes the engine. Dev pipeline was using a 6-day-old bundle from before the `bf9b9564` engine fix. Going forward: after any engine change that affects restore/save behavior, rebuild the story bundle and copy it to `tools/server/stories/` before running e2e tests.

### 3. Architectural pivot to stateless multi-user + channel I/O

The long-running-Deno architecture (ADR-153/156/162) is replaced by a stateless model where each command load → execute → serialize → return, matching the original fyrevm-server design. This eliminates the Deno subprocess boundary, the WS reconnect complexity, and the world-replication overhead. Details in `docs/brainstorm/stateless-multiuser/overview.md`.

### 4. No scope gatekeeping on architectural pivots

When user proposes a larger change, present the actual scope without prejudging feasibility. See `feedback_no_scope_gatekeeping.md` added to memory.

### 5. tools/server preserved on legacy branch, not deleted

e2e suite will be re-derived as a platform dependency once the stateless architecture is established. The Playwright harness patterns (ARIA selectors, identity helpers, statusline parser) are worth preserving as reference.
