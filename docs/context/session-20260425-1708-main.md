## Key Decisions

### 1. ADR-159 fully superseded by ADR-161 (not amended)

ADR-159 stays in the filesystem as historical context but its vocabulary (`identity_id`, `username`, `secret`), its UX (paste-to-reclaim), and its HTTP auth surface (routes accept bare `identity_id` without credential verification) no longer reflect the system. A new ADR (161) was written and accepted rather than patching 159 because the changes touch identity shape, format, auth model, gate semantics, and UX all at once — they represent a different design, not a refinement.

### 2. HTTP auth gap closes by switching all identity-bearing routes to `(handle, passcode)`

The shipped Phase 3 WS hello auths properly with credentials. The HTTP room routes do not. The fix is auth uniformity: every route that acts on behalf of an identity takes `(handle, passcode)` and verifies via the hash service. The `Id` field is never accepted in a request body (server-internal only). This is a breaking change to the room-creation and join-room wire contracts, handled in Phase B.

### 3. Revert Phase 4 client work as a Pre-Phase clean slate

Rather than surgically amending the nine locally-uncommitted Phase 4 files to match the new ADR-161 model, the plan recommends reverting all nine and rebuilding from the correct contracts. The divergence is wide enough (wrong field names, wrong UX flow, wrong gate placement, wrong modal structure) that amendment would be more confusing and error-prone than a clean rebuild. The server Phase 1–3 commit (`bf6fe52c`) is NOT reverted — the rename + format change lands on top as new commits.

### 4. `username` + `display_name` collapse into a single `Handle`

For a small IF community the two-field distinction is over-modeled. A single user-chosen `Handle` — unique, public, 3–12 alpha chars — is the only human-readable identity field visible to other users. This simplifies the DB schema, the wire protocol, and the UI simultaneously.

### 5. Cross-device recovery is file-based, not reclaim-modal-based

ADR-159 specified a paste-Handle-and-passcode modal for cross-device reclaim. The user-stated requirement is download/upload as a CSV file. The two flows have different failure modes (the upload decision matrix), different validation, and different UI surfaces. The reclaim route and modal are removed entirely; upload-identity replaces them.

### 6. Erase is user-driven and hard-delete

ADR-159 deferred deletion to a future admin script with soft-delete semantics. The requirement is that the user erases from the UI, the identity row is hard-deleted immediately at both client and server, and the freed Handle is immediately reclaimable. No soft-delete, no `deleted_at` column.
