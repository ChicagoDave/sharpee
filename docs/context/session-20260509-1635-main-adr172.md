## Completed

### Step 6.1 — `ActionContext.emitSound` (engine factory + interface)

Added `emitSound(sound: Omit<ISound, 'sourceEntity' | 'sourceLocation'>): void` to the `ActionContext` interface in `packages/stdlib/src/actions/enhanced-types.ts`. The engine factory in `packages/engine/src/action-context-factory.ts` receives an optional `soundBuffer?: ISound[]` parameter and provides a closure implementation that auto-fills `sourceEntity = player.id` and `sourceLocation = currentLocation.id`. The recursive implicit-take sub-context inherits the same buffer reference. `packages/stdlib/src/actions/enhanced-context.ts` (`InternalActionContext`) gets a no-op implementation — this factory is used only for capability-interceptor binding and stdlib unit tests; production flows through the engine factory.

Five new tests in `packages/engine/tests/sound/emit-sound.test.ts` assert on actual buffer state: auto-fill of sourceEntity/sourceLocation, content preservation, multi-emit ordering, no-buffer no-op, and unlocated-actor fallback.

### Step 6.2 — `SoundDispatcher` (per-turn fan-out)

New class `packages/engine/src/sound/dispatcher.ts`. Constructor accepts an injectable `propagate` function (defaults to the production function from `./propagation`). `dispatch(buffer, world, timestamp)` enumerates all `ListenerTrait` entities sorted by entity-id, fans out (each sound × each listener), skips null propagation results, and builds `sound.audibility.heard` semantic events.

Exports `AUDIBILITY_HEARD_EVENT_TYPE` as a string constant (`'sound.audibility.heard'`) to avoid an engine→stdlib import cycle. This must stay in sync with `SOUND_EVENT_TYPES.AUDIBILITY_HEARD` in stdlib — noted as a latent drift risk in the ADR implementation history.

Nine new tests in `packages/engine/tests/sound/dispatcher.test.ts` cover: empty buffer, no listeners, full fan-out, null-skipping, entity-id sort determinism, iteration order (outer buffer × inner listener), timestamp threading, event shape, and a smoke test against the real `propagate` function.

### Step 6.3 — Turn-cycle wiring

`packages/engine/src/command-executor.ts`: `execute()` threads the sound buffer to both `createActionContext` call sites (main action path and implicit-take inference path).

`packages/engine/src/game-engine.ts`: Two new private fields — `soundBuffer: ISound[] = []` and `soundDispatcher = new SoundDispatcher()`. In `executeTurn()`: buffer is cleared at turn start (`soundBuffer.length = 0`), passed through `commandExecutor.execute`, and after the plugin tick, `soundDispatcher.dispatch(buffer, world, turn)` runs. Resulting events are merged into `turnEvents`, the engine's `eventSource`, the `'event'` emitter, the `onEvent` callback, and `result.events`. In `restartGame()`: buffer is cleared alongside other per-turn state.

Four new tests in `packages/engine/tests/sound/engine-wiring.test.ts`: sound emission via real action → `result.events` contains the audibility event with `audibilityTier: 'full'`; quiet turn → no sound events; two-turn carry-over boundary (each turn produces exactly one event); engine `'event'` emitter receives the event.

During this step, GitHub issue #109 was filed: `Story.extendParser()` is declared on the `Story` interface and implemented by Dungeo (`registerAllGrammar(parser)` at line 807 of `stories/dungeo/src/index.ts`), but `GameEngine.setStory()` never calls it. All neighboring hooks (`getCustomActions`, `getCustomVocabulary`, `extendLanguage`, `onEngineReady`) are called. Dungeo grammar likely reaches the parser via a side-channel — that should be confirmed when the fix lands. Workaround in engine-wiring test: call `parser.addVerb` directly. Out of scope for ADR-172.

## Key Decisions

### 1. Sound buffer lives on `GameEngine`, not `GameContext`

Sounds do not survive turn boundaries and are not part of serializable game state. A private field on `GameEngine` — cleared at the start of every `executeTurn()` and again in `restartGame()` — is simpler and safer than threading it through the context object. `GameContext` carries player/world state that can round-trip through save/restore; the sound buffer must not.

### 2. Dispatch insertion point: after plugin tick, before text service

Placing `soundDispatcher.dispatch()` between the plugin tick and `textService.processTurn()` means the `sound.audibility.heard` events arrive in the same turn packet seen by text rendering and the `audibility` channel. It also leaves a natural seam for Phase 7 (browser audio rendering) and a future L2 extension where NPC plugin actions push to the buffer during their tick.

### 3. `emitSound` is required (not optional) on `ActionContext`

Making it required means callers never need to null-check. The engine factory always provides an implementation (no-op when no buffer is wired). Hand-built test mocks need only a one-liner `emitSound: () => {}`. No optional-chaining tax anywhere in stdlib.

### 4. Duplicated `AUDIBILITY_HEARD_EVENT_TYPE` constant

An engine→stdlib import for the event-type string would invert the dependency direction. The string is duplicated in `packages/engine/src/sound/dispatcher.ts` with a comment cross-referencing `SOUND_EVENT_TYPES.AUDIBILITY_HEARD`. The risk is latent string drift. Accepted as the lesser coupling cost for Phase 6; could be extracted to `@sharpee/if-domain` in a future cleanup.

### 5. Tapestry test uses a synthetic world, not Dungeo

Platform tests must not modify real stories (project rule, user preference). The integration test constructs a minimal three-room world inline. This also makes the test reproducible regardless of the current state of the Dungeo story.
