### Phase 3 — Per-Side Wall Adjective Resolution (parser-side)

**Goal**: The command validator must resolve `OAK WALL` to the wall whose current-player-side adjective is `oak`, and reject a modifier that doesn't match the player's side (e.g., `OAK WALL` from the library side where the wall presents as `brick`).

**Changes**:
- `packages/world-model/src/world/wall-creation.ts` — `createWall` now calls `wall.setMinimumScope(2, [roomA, roomB])` for both connecting rooms. This is required because walls have no spatial containment; without this the validator's scope filter excludes them entirely.
- `packages/stdlib/src/validation/command-validator.ts` — `getEntityAdjectives` returns `[wall.getSide(playerRoom).adjective]` for `WallEntity` instances based on the player's current location. `scoreEntities` drops a wall from candidacy when modifiers were specified but none match the current side's adjective (strict match; non-wall entities keep existing soft-match behavior). Imports `WallEntity` from `world-model`.

**Note on scope**: ADR-173 Implementation Modules originally documented this change as living in `@sharpee/parser-en-us`. The actual entity-resolution code is in `@sharpee/stdlib/src/validation/command-validator.ts`; `parser-en-us` is unchanged. The ADR's "parser-side" framing was imprecise.

**New file**: `packages/stdlib/tests/unit/validation/wall-resolution.test.ts` — 8 tests covering:
- AC-5: single wall, bare `WALL` resolves
- AC-6: multiple walls → `AMBIGUOUS_ENTITY`
- AC-7: per-side adjective match (`oak` from parlor, `brick` from library, same wall; non-matching modifier from wrong side returns no match)
- Cross-room scope: wall not bordering player room is not resolvable

**Extended file**: `packages/world-model/tests/unit/world/wall.test.ts` — 2 new tests for the `setMinimumScope` mutation (closes a mutation-verification YELLOW grade from Phase 1+2).

**Behavior Statements produced**:
- `WorldModel.createWall` (extended): adds `setMinimumScope(2, [roomA, roomB])` mutation; rationale: walls lack spatial containment so default scope rules don't include them.
- `CommandValidator.getEntityAdjectives` (extended): returns per-side adjective for walls; never throws.
- `CommandValidator.scoreEntities` (extended): drops walls from candidacy when modifiers were specified but none match the per-side adjective.

**Mutation audit**: `createWall` — existing mutations preserved; new mutations `setMinimumScope(roomA, 2)` and `setMinimumScope(roomB, 2)` directly asserted by 2 new tests. Mutation-verification: GREEN.

## Key Decisions

- **Scope of per-side adjective enforcement**: `scoreEntities` applies strict modifier matching for walls (mismatched modifier → drop from candidacy). Non-wall entities keep existing soft-match behavior. This is necessary because walls score +10 via entity-type match regardless of adjective; without the wall-specific `continue`, `OAK WALL` from the library would silently resolve to a wall whose library-side adjective is `brick`. ADR-173 §End-to-end scenario explicitly requires "no match" in that case.
- **Where Phase 3 landed**: ADR-173 Implementation Modules named `@sharpee/parser-en-us` as the location for adjective resolution. The actual code lives in `@sharpee/stdlib/src/validation/command-validator.ts`. The ADR framing was imprecise; `parser-en-us` is unchanged.
