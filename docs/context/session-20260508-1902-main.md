### Phase 3 — Per-Side Wall Adjective Resolution (parser-side)

**Goal**: The command validator must resolve `OAK WALL` to the wall whose current-player-side adjective is `oak`, and reject a modifier that doesn't match the player's side (e.g., `OAK WALL` from the library side where the wall presents as `brick`).

**Changes**:
- `packages/world-model/src/world/wall-creation.ts` — `createWall` now calls `wall.setMinimumScope(2, [roomA, roomB])` for both connecting rooms. This is required because walls have no spatial containment; without this the validator's scope filter excludes them entirely.
- `packages/stdlib/src/validation/command-validator.ts` — `getEntityAdjectives` returns `[wall.getSide(playerRoom).adjective]` for `WallEntity` instances based on the player's current location. `scoreEntities` drops a wall from candidacy when modifiers were specified but none match the current side's adjective (strict match; non-wall entities keep existing soft-match behavior). Imports `WallEntity` from `world-model`.

**Note on scope**: ADR-173 Implementation Modules originally documented this change as living in `@sharpee/parser-en-us`. The actual entity-resolution code is in `@sharpee/stdlib/src/validation/command-validator.ts`; `parser-en-us` is unchanged. The ADR's "parser-side" framing was imprecise.

**New file**: `packages/stdlib/tests/unit/validation/wall-resolution.test.ts` — 8 tests covering:
- AC-5: single wall, bare `WALL` resolves
- AC-6: multiple walls → `AMBIGUOUS_ENTITY`
- AC-7: per-side adjective match (`oak` from parlor, `brick` from library, same wall; non-matching modifier from wrong side returns no match)
- Cross-room scope: wall not bordering player room is not resolvable

**Extended file**: `packages/world-model/tests/unit/world/wall.test.ts` — 2 new tests for the `setMinimumScope` mutation (closes a mutation-verification YELLOW grade from Phase 1+2).

**Behavior Statements produced**:
- `WorldModel.createWall` (extended): adds `setMinimumScope(2, [roomA, roomB])` mutation; rationale: walls lack spatial containment so default scope rules don't include them.
- `CommandValidator.getEntityAdjectives` (extended): returns per-side adjective for walls; never throws.
- `CommandValidator.scoreEntities` (extended): drops walls from candidacy when modifiers were specified but none match the per-side adjective.

**Mutation audit**: `createWall` — existing mutations preserved; new mutations `setMinimumScope(roomA, 2)` and `setMinimumScope(roomB, 2)` directly asserted by 2 new tests. Mutation-verification: GREEN.

## Key Decisions

- **Scope of per-side adjective enforcement**: `scoreEntities` applies strict modifier matching for walls (mismatched modifier → drop from candidacy). Non-wall entities keep existing soft-match behavior. This is necessary because walls score +10 via entity-type match regardless of adjective; without the wall-specific `continue`, `OAK WALL` from the library would silently resolve to a wall whose library-side adjective is `brick`. ADR-173 §End-to-end scenario explicitly requires "no match" in that case.
- **Where Phase 3 landed**: ADR-173 Implementation Modules named `@sharpee/parser-en-us` as the location for adjective resolution. The actual code lives in `@sharpee/stdlib/src/validation/command-validator.ts`. The ADR framing was imprecise; `parser-en-us` is unchanged.

## Phase 3 — Propagation function (`@sharpee/engine`)

- New file: `packages/engine/src/sound/propagation.ts` — exports `propagate(sound, listenerId, world, timestamp): IAudibilityEvent | null` and the `clarityToTier(clarity): AudibilityTier` mapping. Implementation is decomposed into:
  - `getAcousticEdges(roomId, world)` — enumerates the room's outgoing acoustic edges from exits-with-doors (open=1, closed=4) and walls (effective cost).
  - `wallEffectiveAcousticCost(wall, world)` — base cost from `AcousticTrait` (defaulting to `default` tier when absent) + sum of `AcousticDampenerTrait.contribution` from current-room obstructors on both sides via Phase 5's `findTraitsOnObstructors`. Negative effective cost clamps to 0 (a hole/amplifier passes sound but doesn't create clarity beyond budget).
  - `findShortestAcousticPath(source, target, world)` — Dijkstra with previous-edge tracking. Path cost = sum of edge costs + 1 unit per intermediate room (source contributes 0, every subsequent leg adds the room-traversal cost).
  - Wall-edge tracking lets the resulting `AudibilityEvent.wallId` name the crossed wall when (and only when) the path crosses exactly one.
- Same-room degenerate path returns `full` audibility regardless of intervening boundaries.
- Soundproof walls are dropped from the edge graph (Infinity cost would never produce a finite path; explicit drop avoids feeding Infinity into Dijkstra).
- `packages/engine/src/sound/index.ts` (new barrel) + root engine index re-export.
- 27 tests in `packages/engine/tests/sound/propagation.test.ts`: clarity-to-tier table coverage (every band), same-room, open/closed door, all four wall acoustic tiers + wallId presence, dampener +/- contributions, cross-side aggregation, AC-7 lift on movement, AC-8 non-acoustic obstructor doesn't affect propagation, multi-hop intermediate-room cost, no-path / no-room rejection, AudibilityEvent shape sanity.

### Phase 3 surfaced two latent barrel bugs

The propagation tests initially failed with `ListenerTrait is not a constructor` even though the trait was defined and tested in-package. Root cause: `@sharpee/world-model`'s root `src/index.ts` enumerates each trait subdir individually rather than re-exporting `traits/` wholesale. Phase 5 of ADR-173 (`obstructor-protocol`) and Phase 2 of ADR-172 (`acoustic`, `listener`) had been added to `traits/index.ts` but NOT to the root `src/index.ts`. Cross-package imports through `@sharpee/world-model` resolved at TS compile time but yielded `undefined` at runtime via the package's `exports` field.

Fixed by:
- Adding `export * from './traits/acoustic'`, `'./traits/listener'`, `'./traits/obstructor-protocol'` to `src/index.ts`.
- Adding the three new traits to `TRAIT_IMPLEMENTATIONS` in `src/traits/implementations.ts` (the existing `Record<TraitType, ITraitConstructor>` requires every TraitType to have an entry).
- Rebuilding both `dist/` (CJS) and `dist-esm/` (ESM) — vitest in dependent packages resolves through the package's `exports` field which routes ESM consumers to `dist-esm/`.

Saved a memory note (`project_world_model_root_barrel.md`) so future trait additions don't trip on this pattern.

## Phase 5 — `audibility` channel + sound prose defaults

**Cross-ADR seam resolved during Phase 5 implementation.** ADR-172 §Channel routing originally reserved the `'sound'` channel id for spatial sound propagation. ADR-163 had already claimed that id for the media-cue channel (`media.sound.play` payloads, gated by the `sound` capability flag). The two payloads are different concepts: media-cue is "play this audio file at the client's audio bus"; ADR-172's is "this listener perceives this propagated sound (rendered as prose + optional audio)". David chose to rename ADR-172's channel to `audibility` rather than displace ADR-163's existing channel; ADR-172 §Channel routing was updated to reflect.

- `packages/stdlib/src/channels/registry.ts` — registers `SOUND_CHANNELS` alongside `STANDARD_CHANNELS` and `MEDIA_CHANNELS` at module-init.
- `packages/stdlib/src/channels/index.ts` — re-exports.
