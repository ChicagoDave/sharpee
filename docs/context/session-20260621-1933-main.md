### Published-CLI blocker — root-cause diagnosis (devkit MODULE_NOT_FOUND)
A Docker "naive reader" QA pass found `@sharpee/devkit@1.0.7` crashing on every command:
```
Error: Cannot find module '../../core/index.js'
```
The naive QA report's stated root cause was wrong. The real bug was in tsf's npm publish transform: `src/orchestrator/index.ts` unconditionally forced `imports: 'relative'` for all npm builds, rewriting `@sharpee/core` → `../../core/index.js`. That path only resolves when deps are hoisted flat under `node_modules/@sharpee/`; devkit's nested file layout (`standalone/`, `commands/`) breaks the walk-up on a global install.

### tsf fix — honor configured publish-import style
In `/Users/david/repos/tsf`:
- `src/types.ts`: added `publishImports` field to `TargetConfig`.
- `src/orchestrator/index.ts`: changed the post-compile transform so it threads the target's configured `imports` value (as `publishImports`) past the tsc-step preserve mask, then applies it with precedence: per-package `ts-forge.json` override → target `publishImports` → `relative` default for back-compat.
- Rebuilt tsf. Backward-compatible — other tsf consumers (orionwriter) are unaffected.
- Commit `821d1e6` pushed to tsf main.
