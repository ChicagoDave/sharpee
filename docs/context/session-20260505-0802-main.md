## ADR-169 — Web Audio API for AudioManager (Fade-In/Out and Cross-Fade)

### First draft

Wrote `docs/architecture/adrs/adr-169-web-audio-api-fades.md`.

**Decision**: adopt the Web Audio API graph:
`AudioContext` (lazy singleton) → `MediaElementAudioSourceNode` → `GainNode`
→ destination. Use `linearRampToValueAtTime` for all fades.

Three implementation options were evaluated:

| Option | Description | Outcome |
|--------|-------------|---------|
| A | JS-driven `setInterval` volume steps | Rejected — not sample-accurate; competes with the event loop |
| B | Web Audio API `linearRampToValueAtTime` | **Selected** — sample-accurate, future graph extensibility |
| C | `BufferSourceNode` (full file decode) | Rejected — too heavy for ambient loops and music |

Key design choices baked into the ADR:

- **Defaults**: ambient 1500 ms in / 800 ms out; music 2000 ms in / 1500 ms out.
- **Per-event override**: `fadeIn`/`fadeOut` payload fields (already
  declared in `@sharpee/media`'s `events.ts`) are now honored. The
  current `AudioManager` ignores them; ADR-169 begins consuming them.
- **SFX exempt by design**: one-shot effects don't fade.
- **Cross-fade is automatic**: concurrent `linearRamp` calls on
  independent `GainNode`s; no coordination layer needed.
- **`unlock()` becomes async**: `AudioContext.resume()` must complete
  before queued events replay. This fixes a pre-existing contract gap
  (callers were not waiting for the context to resume before issuing
  play commands).
- **`AudioContext` failure does not throw**: falls back to instant-gain
  mode (plain `<audio>` volume) so sound still plays without ramps
  rather than going silent.
- **`AudioManagerLike` interface unchanged**: ADR-167 boundary preserved;
  existing translation tests still pass without modification.
- **Channel renderer unchanged**: `packages/platform-browser/src/channels/audio.ts`
  already spreads `...(value as Record<string, unknown>)`, so `fadeIn`/`fadeOut`
  fields are carried verbatim. Only `AudioManager.ts` itself needs to
  change.
- **Re-introduces `AudioContext`**: Phase 1 of the prior session deleted
  `audioContext` because it only powered the beep. ADR-169 brings it
  back for a different purpose — the Web Audio graph.

## Key Decisions

- **Web Audio API over JS volume steps** (option B over A): sample-accurate
  timing cannot be achieved with `setInterval`; Web Audio graph is the
  correct primitive and opens the door to future graph-based effects
  (EQ, convolution reverb).
- **`BufferSourceNode` rejected** (option C): decoding full audio files
  into memory buffers is prohibitively heavy for ambient loops and music
  tracks.
- **SFX stays exempt**: one-shot effects are fire-and-forget; imposing
  fades would be noticeable and wrong for impact sounds.
- **`unlock()` becomes async**: corrects a pre-existing contract gap. The
  old synchronous `unlock()` could not guarantee `AudioContext.resume()`
  had completed before callers issued play commands through it.
- **Defaults baked in, payload override supported**: `fadeIn`/`fadeOut`
  in the event payload override defaults; no constructor-time
  configuration needed.
- **`AudioContext` failure falls back gracefully**: the `AudioManager`
  must never go silent on context failure — degrade to instant-gain, not
  no-audio.
- **`AudioManagerLike` interface unchanged**: preserves the ADR-167
  boundary; no downstream type changes cascade from this implementation.
