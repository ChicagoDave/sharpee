**Product shape and deployment model:**
- Open-source, self-hostable platform software (not proprietary SaaS)
- Mastodon / Matrix / GitLab-CE model
- `sharpee.net` is a reference instance, not the product
- Desktop-only scope (no mobile for now)
- Docker as canonical deployment artifact: single-container MVP (Node server + Deno binary + SQLite)
- Reference instance on David's Ubuntu server with Apache reverse proxy

**Motivation and use cases:**
- Origin: IF players playing together via Discord/Zoom is awkward — command-line play cannot be shared or turn-taken naturally
- Early motivating use case: IF beta-testing for comp authors
- Additional use cases identified: game-night, classroom, club, private-group, comp-organizer, demo-instance

**Session and identity model:**
- Roles: Primary Host (creator) → Host (delegated) → Command Entrant → Participant
- Join codes format: `XYZB-3F56` (no account required at tester side; operators may configure author accounts)
- Durable session token issued on first join, stored in browser `localStorage` scoped to room URL, enables reconnection without password/email/recovery

**Input mechanic:**
- Lock-on-typing: anyone with command authority can type freely, but once someone begins composing, input is locked for all others until that command is submitted

**UI layout:**
- Left pane: story output; Right pane: chat

**Execution model:**
- Server-side engine (not in-browser); browser is thin client sending commands up, receiving domain events back

**Security posture:**
- `.sharpee` files are executable JavaScript (esbuild bundles story TS into ESM modules — confirmed at `build.sh` line 753)
- Cannot be made safe by filtering (Caja, AST denylists, `vm` module all known-inadequate)
- Must be sandboxed by construction
- Leading candidate: Deno subprocess per room with `--allow-none`
- Alternatives noted: Cloudflare Workers, Deno Deploy, isolated-vm, Firecracker

**Filesystem posture:**
- Story-runtime has zero filesystem access
- Save/restore mediated by server through opaque-blob API

**Persistence architecture (SQLite, three layers):**
1. Relational state: rooms, participants, lifecycle
2. Append-only session event log: all commands, outputs, chat, role-changes, saves, joins/leaves
3. Opaque save blobs
- Drawing on David's prior FyreVM client/server experience where full recording proved valuable for beta-testing and review

**Save scoping:**
- Room-scoped saves; handles "test alone" (one participant) and "test with others" (shared room) without special-casing

**Room lifecycle:**
- States: created / active / idle
- Default idle recycle: 14 days
- Primary Host can pin a room to bypass recycle
- Primary Host can delete anytime (full control, even if rude to current participants)
- Join codes returned to pool on recycle

## Key Decisions

### 1. Self-hostable over SaaS

The multiuser platform must be open-source and self-hostable. David controls the reference instance but the product is the platform. This aligns with community values (IF community is skeptical of proprietary infrastructure) and reduces single-point-of-failure risk.

### 2. Deno subprocess for story sandbox

Story code is ESM JavaScript that must be treated as untrusted. Deno with `--allow-none` is the leading sandboxing candidate because it is a purpose-built JS runtime with explicit permission grants, avoiding the well-known inadequacy of Node's `vm` module. This replaces the microVM approach that ADR-152 had proposed.

### 3. Lock-on-typing for shared input

Rather than turn-taking (one person "holds the keyboard"), the lock-on-typing model lets anyone start typing and the first to do so locks out others until the command is submitted. This is more natural for small groups and avoids explicit hand-off ceremony.
