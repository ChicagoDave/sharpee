## Goal
Implement ADR-252 Phase 3: a bare Chord `.story` builds to a self-contained browser app with **no flag and no `package.json`**, all metadata from the compiled Story IR, and **one shared build core** across devkit (author) + repokit (in-repo). ## Key discovery before coding (surfaced to David, ruled "full pipeline Chord-aware")
The design's Step-1 baseline ("`./repokit build --browser fernhill` still builds") did **not** hold in committed state: `stories/fernhill/` has no `package.json`, so both paths failed on it (devkit `process.exit(1)`; repokit threw in `stampVersions` + `buildStory`). fernhill's browser build only ever worked via a **scaffolded** `package.json` (the g3 proof's `runInitCommand`) — the exact wrong-metadata bug ADR-252 D2 fixes. Consequence: making the repokit `--browser` path run end-to-end required making the `runBuild` **pipeline** Chord-aware, beyond the design's enumerated files. David chose the full-pipeline interpretation (ADR-derived from D1/D2).

## What shipped (all 6 design steps; uncommitted)
- **Shared core** — new `packages/devkit/src/standalone/browser-core.ts`: `buildBrowser(storyFile, env, opts)` owns the whole Chord browser build; injected `BrowserBuildEnv` (stylesDir/templatesDir/esbuildCwd/engineVersion/mirror) is the only caller difference. Theme helpers + IR readers (`readBrowserMeta`/`readClientConfig`/`KNOWN_HEADER_KEYS`) live here once; exported from `@sharpee/devkit`; repokit gained the workspace dep.
- **devkit** (`build-browser.ts`, `cli.ts`): `.story` file OR dir-with-`.story` (no `src/index.ts`) → Chord core, browser default, no `--browser`; hybrid `.story`+`src/index.ts` → error; TS path unchanged.
- **repokit** (`browser.ts`, `build.ts`): `chordStoryFile()` detects Chord → delegates to the core (in-repo env + website mirror); `stampVersions` skips the package.json version read + `runBuild` skips `buildStory` for a Chord story. dungeo (TS) untouched.
- **Generated entry (D4)**: `chord-browser-entry.ts.template` tokenized; build fills from the header, init/init-browser scaffold fills concrete defaults. fernhill keeps its hand-written entry.
- **Migration (D6)**: dropped `--browser` from the Chord scaffold hint + devkit USAGE. Repokit/dungeo `--browser` left (out of scope).

## Key result — byte-identical parity (stronger than the AC)
The esbuild `--alias` was the **only** divergence between the two callers and is **redundant in-repo** (node-modules resolution with `--conditions=require` resolves platform-browser identically). Dropped it → author env and in-repo env produce **BYTE-IDENTICAL** output (even game.js), beating the "identical-modulo-stamp" AC. This deviates from phase3-design.md's `esbuildAlias` seam (recorded in plan.md).
