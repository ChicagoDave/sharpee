## Key Decisions

### 1. `load-story` then `start` two-step handshake
The protocol separates story loading (eval'ing story code and validating `window.SharpeeStory`) from game start (constructing the engine and running the opening look). This allows the parent to pre-validate story code without committing to a game session, and cleanly separates initialization phases.

### 2. Parent-owned save persistence
The bridge does not use `localStorage` directly. All save data is serialized and posted to the parent as `sharpee:save-data`; restore is triggered by the parent sending `sharpee:restore` with the serialized state back. This keeps the runtime stateless with respect to persistence and lets Lantern IDE own save slot management.

### 3. `window.Sharpee` as named exports, not a single default object
Individual named exports (`window.Sharpee.GameEngine`, `window.Sharpee.WorldModel`, etc.) match the existing module surface and allow eval'd story code to destructure exactly what it needs without knowing a wrapper object shape.

### 4. esbuild IIFE bundle, not tsc
The real runtime artifact is produced by esbuild in `build.sh`, not by `tsc`. The `package.json` build script does `tsc --noEmit` only (type-check). This is consistent with how Zifmia is built and avoids shipping CommonJS from a browser-target package.

### 5. `sharpee:` prefix on all protocol messages
All postMessage types are prefixed with `sharpee:` to avoid collisions with other postMessage senders (analytics, browser extensions, other iframes). The type guards check this prefix before dispatching.
