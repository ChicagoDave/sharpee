## Key Finding вЂ” BLOCKER
- **Published `@sharpee/devkit@1.0.7` CLI crashes on every command** (incl. bare
  `sharpee`) with `MODULE_NOT_FOUND`. `cli.js` requires `./standalone/ifid` at load time,
  so the bad require throws before arg parsing вЂ” `init`/`build`/`introspect`/`ifid` all
  dead. **The book is un-followable from В§1.4 onward with the published package.**
- **Root cause**: three shipped CLI files use monorepo-relative requires never rewritten
  to package-name imports at publish. `../../core/index.js` works in the source tree but
  in the flat published package climbs to a nonexistent `node_modules/@sharpee/core`; the
  bundled copy is at `devkit/node_modules/@sharpee/core`. Fix = require by package name.
- **Exact sites** (fix in TS source under `packages/sharpee/src/cli/`):
  - `standalone/ifid.js:6` вЂ” `../../core/index.js` в†’ `@sharpee/core`
  - `standalone/build.js:204` & `:205` вЂ” `../../transcript-tester/index.js` в†’ `@sharpee/transcript-tester`
  - `commands/introspect.js:68` вЂ” `../../bootstrap/index.js` в†’ `@sharpee/bootstrap`
- **Scan**: walked all 20 shipped JS files, resolved every executed `require()`/`import()`
  вЂ” 46 relative loads, **exactly 4 broken** (the sites above). No other broken imports.
  (A looser first pass flagged 10 `../packages/.../dist/index` refs in `repo.js`; those are
  string data in `BUNDLE_ALIASES`/`BUNDLE_DTS`, not real imports вЂ” false positives.)
