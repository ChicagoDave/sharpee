### 3. Save/restore platform-bug briefing (`docs/work/save-restore/2026-04-28-platform-save-restore-bug.md`)

User reported the diskette restore "restarts the game" — clicking restore from Troll Room landed the player at West of House (the opening location). Investigation traced the symptom to a much bigger problem.

**Finding: `packages/engine/src/save-restore-service.ts` is partially implemented.** `createSaveData()` only captures `spatialIndex` (entity traits + entity-by-location), `eventSource`, `turnHistory`, `parserState`, `pluginStates`. `loadSaveData()` only restores `spatialIndex` via `deserializeSpatialIndex` — line 362 has the original author's comment admitting it is partial: `"Note: Full deserialization would need to clear and recreate the world. For now, this restores entity traits and locations"`.

**Lost on every save+restore round-trip, on every platform** (CLI, platform-browser, Zifmia, multi-user):
- ScoreLedger (ADR-129) — score reverts to 0
- Capabilities — including `scoring.moves`, `scoring.deaths`, dungeo's death-penalty state machine, melee state
- World state values — `dungeo.player.deaths`, `dungeo.game_over`, `cure.ticks`
- Relationships
- ID counters

**What survives**: entity traits, entity locations, event source, turn history, parser state, plugin states. Enough to look correct at a glance — entities are where they were, openable.isOpen is preserved — but the "feel" of the saved game (score, progress flags) is silently reset.

**Why no test caught it.** The three existing save/restore tests (`game-engine.test.ts:225`, `:237`, `:259`; `integration.test.ts:56`) all grade YELLOW per CLAUDE.md rule 12: they assert API-surface fields (`metadata.turnCount`, `history.length`, return-shape) that come from `metadata` and `eventSource` — paths the broken `spatialIndex` code never touches. They pass even when the body of `loadSaveData` is a no-op. Walkthroughs use `--chain` (in-memory continuation, no save-format round-trip), so they don't catch it either.

**Why walkthroughs work despite this**: `packages/transcript-tester/src/fast-cli.ts:580` only re-creates the engine between transcripts when `--chain` is *not* set. Chain mode reuses the same live engine instance — never round-trips through save/restore.

**Fix path identified**: swap `serializeSpatialIndex` / `deserializeSpatialIndex` for `world.toJSON()` / `world.loadJSON()`. The world serializer already covers everything (proven by ADR-162 wire flow). One file (`packages/engine/src/save-restore-service.ts`), ~150 lines touched. Save version bumps `1.0.0 → 2.0.0`. Backward-compat recommendation: hard-break (alpha tier; existing v1 saves are already corrupted, even loading them via the old reader gives a degraded restore).

**Documented for review on the Mac.** Briefing includes: full evidence list, before/after state survival table, RED test list (six new tests should fail against current code, two should pass for regression), step-by-step fix plan, save-blob size impact (~10–20× growth, acceptable for dungeo per ADR-162 bandwidth baseline), rollout (rebuild CLI bundle + browser dist + docker image), and five open questions for chicagodave.

## Key Decisions

### 1. `node` export condition for world-model

Rather than fix `dist-esm`'s bare directory imports (large source-side refactor) or bundle the server with esbuild (workaround), add a Node-specific `node: ./dist/index.js` condition that routes Node ESM through the working CJS dist via cjs-module-lexer interop. Bundlers (Vite/Rollup) don't match `node` and continue to use the `import: ./dist-esm/index.js` condition. Single-line change in `packages/world-model/package.json`. Verified locally with `node -e "import('@sharpee/world-model')"`.

This unblocks the multi-user server but does *not* fix the underlying `dist-esm` build. If a future consumer needs real Node-compatible ESM (e.g., a Node-side script that wants tree-shaking), the dist-esm output still needs the `module: NodeNext` + explicit `.js` extension treatment. Filed mentally as future work; not blocking.

### 2. Sandbox-owned turn counter (no engine API change)

`GameEngine.sessionMoves` and `GameEngine.context.currentTurn` are both private. Adding a public accessor would be a packages/ change. Instead, the sandbox's `deno-entry.ts` keeps its own `turnCount` (incremented on each COMMAND frame, mirrored into the scoring capability before each world snapshot). Engine and sandbox counts will diverge on edge cases (RESTORE doesn't sync, sandbox doesn't see meta-only commands the engine increments on), but for the multi-user UI's "Turns:" display it's a clean, contained fix.

### 3. Drop `/maxScore` from StatusLine; show only `Score: N`

User direction. Matches platform-browser exactly. Zifmia conditionally shows `N/MAX` when `maxScore > 0`; we picked the simpler form. Max is still exposed via the in-game `score` command (which reads `world.getMaxScore()` directly) — same Infocom convention as the original mainframe Zork.

### 4. Two-commit split for the deploy work

`c258199e` (build/deploy plumbing) and `630bbe6b` (StatusLine + sandbox counter) are separable themes. Splitting keeps `git log` readable and lets a future revert target one without the other.

### 5. Hard-break recommended for save format v2

Detailed in the save-restore briefing. Alpha tier, dungeo is the only real player base, v1 saves are already corrupted in-place. Writing a v1 reader gets us "restore that doesn't actually restore" — worse UX than "this save is from before the upgrade." Final call deferred to chicagodave.
