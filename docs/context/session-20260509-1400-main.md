## What Was Built

### New File: `tools/ide/SharpeeIDE/Editor/TabBarView.swift`

A custom `NSView` that renders a horizontal strip of tab cells above the editor's `NSTextView`.

Key design decisions:

**No trailing constraint on the inner `NSStackView`.** The stack sits flush left and grows rightward at the sum of each cell's intrinsic width. Cells do not stretch to fill the bar — this matches VS Code / Xcode tab behavior where tabs are naturally sized and the bar can scroll (scrolling is deferred to a later step).

**Single 1pt bottom border (`Theme.border`)** separates the bar from the editor surface. The bar's background is `Theme.projectBackground`; active cells switch to `Theme.editorBackground` so the active tab visually "lifts" into the editor.

**Active vs inactive cells:**
- Active: `editorBackground` / `foreground` / `.medium` weight
- Inactive: `projectBackground` / `foregroundDim` / `.regular` weight

**Cell structure:** each `TabCellView` contains a `NSTextField` label (left-padded 12pt), a 16pt × 16pt `NSButton` close button (right-side), and a 1pt `NSView` separator on the trailing edge. The label's trailing constraint anchors to the close button's leading edge, so the label truncates (`.byTruncatingMiddle`) if the cell is narrow.

**Hit-test routing:** `NSButton` has a larger hit area than the close button's visual bounds, so it intercepts clicks before `mouseDown(with:)` on the enclosing `NSView`. This means `mouseDown` on `TabCellView` only fires when the user clicks the tab body — clean separation between select and close without needing gesture recognizer gymnastics.

**Public interface:** `setTabs(_:activeIndex:)` tears down and rebuilds all cells from scratch. `onSelect: ((Int) -> Void)?` and `onClose: ((Int) -> Void)?` are closures the controller wires up.

### Modified: `tools/ide/SharpeeIDE/Editor/EditorViewController.swift`

Replaced the single `currentDocument: Document?` property with `documents: [Document]` and `activeIndex: Int?`.

**Layout change:** `loadView()` now stacks `tabBar` above `scrollView` using Auto Layout. The tab bar is hidden (along with the scroll view) when no documents are open; the placeholder is shown instead.

**`openDocument(at:)`:** checks `documents.firstIndex(where: { $0.url == url })` first. If found, focuses that tab (`switchTo(index:)`). If not, loads via `Document.load(from:)`, appends, and activates the new tail index. File-read errors surface via `NSAlert(error:)`.

**`closeDocument(at:)`:** removes from the array, then resolves the new `activeIndex`:
- Empty after removal → `nil`
- Closed tab was active → `min(index, documents.count - 1)` (next tab, or previous if at end)
- Closed tab was before the active one → decrements `activeIndex` by 1

After updating `activeIndex`, calls `loadActiveDocumentIntoTextView()` and `refreshUI()`.

**`switchTo(index:)`:** persists the text view's current string to the outgoing document (`persistTextViewToActiveDocument()`), updates `activeIndex`, loads the new document's content, then refreshes the UI.

**`isSwappingContent` flag:** set to `true` inside `loadActiveDocumentIntoTextView()` for the duration of the programmatic string replacement. The `NSTextViewDelegate.textDidChange(_:)` handler checks this flag and short-circuits if it is set. Without this guard, each programmatic content swap would echo back into the active document's `content` property redundantly — and more importantly could interfere with undo history.

**Live edit sync (`NSTextViewDelegate`):** `EditorViewController` conforms to `NSTextViewDelegate`. On each keystroke, `textDidChange` copies `textView.string` into `activeDocument.content`. This is how edits survive tab switches: the model is always current as of the last keystroke.

**`closeAllDocuments()`:** added as a future hook for project-close / folder-switch scenarios.

## Key Design Decision: Single Shared NSTextView

The tab switcher does NOT create one `NSTextView` per document. It uses a single shared text view and swaps its string content on tab switch.

**Why this approach:**
- Simpler view hierarchy — no hidden text views to manage
- NSTextView is heavyweight; creating many of them for a typical IDE session (dozens of open files) would be wasteful
- Consistent with how many AppKit code editors work at the foundation level

**Tradeoff accepted:** per-tab undo history is not preserved across switches. `undoManager?.removeAllActions()` is called on each load, giving each "session" in the text view a clean undo stack, but switching away loses the undo chain for the prior file. This is explicitly deferred to step 1.5 or later.

**Tradeoff accepted:** cursor position and scroll offset are not preserved across tab switches. The text view scrolls to the top on each switch. Deferred to step 1.5.

## Disambiguation Algorithm

TypeScript projects use `index.ts` as the canonical barrel file name. Opening `packages/world-model/src/traits/index.ts` and `packages/world-model/src/actions/index.ts` simultaneously would produce two tabs both labeled `index.ts` — unusable.

`EditorViewController.makeDisplayTitles(for:)` is a static helper that disambiguates:

1. Compute display titles as file names alone.
2. Group indices by title — find all collisions.
3. For each collision group, walk up the path component tree (depth 2, 3, …) until the suffix strings within the group are all distinct.
4. Replace titles for that group with the minimum-depth suffix.

**Worked examples:**

| URLs | Depth 2 suffixes | Distinct? | Result |
|---|---|---|---|
| `.../traits/index.ts` | `traits/index.ts` | yes (with next) | `traits/index.ts` |
| `.../actions/index.ts` | `actions/index.ts` | yes (with prev) | `actions/index.ts` |

If `traits` and `actions` lived inside directories with the same name (hypothetically `.../x/traits/index.ts` vs `.../x/actions/index.ts`), depth 2 still disambiguates because `traits` ≠ `actions`.

If two paths were literally identical (which the filesystem prevents), the algorithm falls back to full absolute paths.

Files with unique names are never touched — a single `README.md` in the tab bar always renders as `README.md`.

### Close Confirmation for Dirty Tabs

After step 1.5 was working, the standard macOS "unsaved changes" sheet was added to `EditorViewController.closeDocument(at:)`.

**Flow:**

1. `persistTextViewToActiveDocument()` is called first to ensure `Document.content` is current if the tab being closed is the active one.
2. If `documents[index].isDirty` is false → `performClose(at:)` immediately (unchanged behavior).
3. If dirty → `promptCloseDirty(at:)`.

**`promptCloseDirty(at:)`** presents an `NSAlert` as a sheet using macOS-conventional button order:

| Button | Position | Key Equivalent | Action |
|---|---|---|---|
| Save | Rightmost / default | Return | Save then close |
| Cancel | Middle | Esc | No-op, tab stays open |
| Don't Save | Leftmost | ⌘D | Close immediately, edits lost |

**`handleCloseAlert(response:index:)`**:
- Save → `doc.save()`. On success: `performClose(at:)`. On failure: modal error alert, tab remains open dirty.
- Don't Save → `performClose(at:)` immediately.
- Cancel → no-op.

**`performClose(at:)`** is the extracted original removal logic (array splice + index resolution), unchanged.

- `var activeDocumentIndex: Int?` — alias for `activeIndex`.
- `var onStateChanged: (() -> Void)?` — callback fired from `switchTo(index:)`, `performClose(at:)`, and `closeAllDocuments()`. (`openDocument(at:)` routes through `switchTo`, so it gets exactly one notification per logical state change.)

**`MainSplitViewController`** (private, inside `MainWindow.swift`):
- Tracks `currentProject: Project?`.
- In `viewDidLoad`, wires `editorViewController.onStateChanged = { [weak self] in self?.persistSession() }`.
- `loadProject(_:expandedFolderURLs:)` updates `currentProject`, calls `setProject` with the expansion list, then `persistSession()`.
- `persistSession()` composes `SessionState` from `currentProject`, `editor.openDocumentURLs`, `editor.activeDocumentIndex`, and `projectTree.expandedFolderURLs`, then writes via `SessionStateStore.save`.

Forwarding methods added: `MainWindowController.loadProject(_:expandedFolderURLs:)`, `openDocument(at:)`, `switchToDocument(at:)` — and matching passthrough on `RootViewController`.

**`AppDelegate.restoreSession(into:)`** runs at the end of `applicationDidFinishLaunching`:
1. Reads `SessionStateStore.load()`.
2. If `projectURL` is non-nil and still exists on disk, loads the project and sets the window title.
3. For each URL in `openDocumentURLs` still on disk, calls `openDocument(at:)`.
4. After all documents are open, activates the saved `activeIndex` (clamped to surviving count).
5. Missing project folder → silent no-op (saved entry preserved for next launch).
6. Missing individual files → silently skipped.

### Design Notes

| Decision | Rationale |
|---|---|
| Dirty dot rebuilt on every `setTabs` | Cell is stateless; no in-place show/hide needed; simpler than toggling subview visibility |
| `false → true` guard in `textDidChange` | Avoids full tab-strip rebuild on every keystroke once already dirty |
| `UserDefaults` + JSON-encoded `SessionState` | Lightweight, no separate file to manage; appropriate for IDE workspace state that is not user data |
| Programmatic-expansion guard (`isApplyingProgrammaticExpansion`) | Prevents restoration-time write storm and state inconsistency where `persistSession` reads partial expansion mid-restore |
| Forward-compatible `Codable` decoder | Iteration-2 `init(from:)` defaults `expandedFolderURLs` so iteration-1 `UserDefaults` entries load cleanly |
| `onStateChanged` callback fires once per logical operation | `openDocument(at:)` routes through `switchTo`, avoiding double-fire on open + activate |
