## Completed

### Fixed Browser Build Website Copy Path
The build script's `[website]` step was copying processed browser output to `website/public/games/dungeo/` but the Astro play page iframe serves from `/web/dungeo/`. The stale files at `website/public/web/dungeo/` contained unprocessed `{{TITLE}}` template placeholders, causing a blank iframe on the live site.

Fixed `build.sh` line 840 to copy to `website/public/web/${STORY_NAME}` instead of `website/public/games/${STORY_NAME}`.

### Independent Story Versioning
The build script was overwriting the story's `package.json` version with the platform version (`SHARPEE_VERSION`) on every build. This tied story releases to platform releases, which is incorrect — a story can hit 1.0.0 while the platform remains at a lower version, and vice versa.

Changed `build.sh` to read the story version from the story's own `package.json` instead of overwriting it.

### Dungeon 1.0.0 Release
Set `stories/dungeo/package.json` version to `1.0.0`. This is the first production release of the Dungeo story.

### Website News Items

- Dungeon 1.0.0 release announcement
- Sharpee platform updates
- Walkthrough Viewer page launch

### Astro Website Rebuilds
Rebuilt the Astro website multiple times to pick up the corrected browser client files and the new news items. Final output landed in `website/dist/`.

## Key Decisions

### 1. Copy Target: `web/` Not `games/`
The iframe src in the Astro play page was always `/web/dungeo/` — the `games/` path was a latent bug introduced at some point in the build script. Fixing the copy target was the minimal correct change; no Astro template changes were needed.

### 2. Story Version Is Authoritative in Its Own package.json
Rather than having the build script stamp the story version from `SHARPEE_VERSION`, the build now reads `stories/${STORY_NAME}/package.json` and uses that value. Story authors own their version numbers independently of the platform release cycle.
