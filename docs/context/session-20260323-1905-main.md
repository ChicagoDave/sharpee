### Parser Bug Fix: `.instrument()` Slot Missing from Grammar Patterns
Root cause: All `define('verb :target with :tool')` patterns assigned the tool/key slot to `extras` instead of `command.instrument`, where stdlib actions (unlocking, locking, attacking, etc.) actually look for it. The commands silently parsed but the tool was invisible to actions.

Fixed 8 grammar patterns in `packages/parser-en-us/src/grammar.ts`:
- unlock :target with :key
- open :target with :key (locked door variant)
- cut :target with :tool
- attack :target with :weapon (4 verb aliases: attack, fight, hit, kill)
- dig :target with :tool
- take :item from :container

Also tightened a false-positive assertion in `tiny-room-puzzle.transcript`.

## Key Decisions

### 1. esbuild kept external in node_bridge.js
The bridge uses esbuild programmatically to compile `.ts` story paths. Bundling esbuild's native binary into node_bridge.js is not feasible. Decision: keep esbuild as a peer dependency. Host applications (Lantern, Zifmia desktop) must install esbuild alongside node_bridge.js. This is documented in build.sh help text and the ADR.

### 2. Atomic turn flushing (blocks → events → status)
Engine emits `text:output` and domain events separately and potentially interleaved. Bridge accumulates all events in an array during `executeTurn()`, then flushes the complete `blocks`, `events`, and `status` messages after the call resolves. This matches the ADR-135 spec and prevents partial-turn output races.

### 3. Console redirection to stderr in bridge-entry
stdout is exclusively used for newline-delimited JSON protocol messages. Any `console.log` calls inside engine or story code are redirected to stderr so they do not corrupt the protocol stream. Native host apps can choose to suppress or log stderr separately.

### 4. Grating wiring via LockableTrait.keyId (not attributes)
The skeleton key's `unlocksId` attribute was a no-op. The correct wiring is `LockableTrait.keyId` on the lockable entity, not a property on the key. The key is simply an item; the lockable object declares which key fits it.

### 5. Grammar .instrument() is the correct slot type for "with" phrases
The stdlib actions `unlocking`, `locking`, `attacking`, `searching`, `digging` all read `context.command.instrument` (not `context.command.extras`). The grammar `.instrument()` slot type sets this field. All `verb :target with :tool` patterns must use `.instrument()` for the second slot.
