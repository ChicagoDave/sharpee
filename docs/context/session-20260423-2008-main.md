## Completed

### Wire/Repository Type Split

Identified and resolved a Rule 6 cohesion violation in `tools/server/src/repositories/types.ts`.

The file mixed two unrelated concerns:
- Wire primitives (`Tier`, `TextBlock`, `DomainEvent`) — needed by both client and server builds
- Server-internal persistence types (`Save` with `blob: Buffer`) — Node-only

The Dockerfile was copying `repositories/types.ts` into the client build so the client could access the wire primitives, but tsc pulled in the `Save` interface with its `Buffer` type. The client tsconfig deliberately excludes `@types/node`, so the build failed.

**Refactor performed:**

1. Created `tools/server/src/wire/primitives.ts` (new file) — authoritative source for `Tier`, `TextBlock`, and `DomainEvent`. Header documents the module's invariant: no Node-only types may appear here, and explains the Docker copy boundary.

2. Updated `tools/server/src/wire/browser-server.ts` — import switched from `../repositories/types.js` to `./primitives.js`.

3. Updated `tools/server/src/wire/server-sandbox.ts` — import switched from `../repositories/types.js` to `./primitives.js`.

4. Thinned `tools/server/src/repositories/types.ts` — removed the three primitive definitions, added `import type + export type` re-export from `../wire/primitives.js` so the 7 existing server-internal importers (ws/handlers/*, ws/room-snapshot.ts, saves/save-service.ts) keep their existing paths without churn.

5. Updated `tools/server/Dockerfile` — dropped the extra `COPY tools/server/src/repositories/types.ts` line; rewrote the comment to reflect that `src/wire/` is now self-contained and is the correct boundary for the client copy chain.

**Verification:**
- Grep confirmed: zero imports from `src/wire/` reach outside the directory.
- Grep confirmed: zero `Buffer` or `node:` references anywhere under `src/wire/`.
- Docker build ran end-to-end: client build clean, server tsc clean, 325/325 server tests passing, image tagged `sharpee-server:latest`. Exit code 0.

## Key Decisions

### 1. Extract Wire Primitives into `src/wire/primitives.ts` (Option A)

Three options were considered:

- **A (chosen)**: Extract wire-shared primitives into `src/wire/primitives.ts`. Client Docker chain stops at `wire/` — no Node types leak.
- **B (rejected)**: Add `@types/node` to the client tsconfig. Quick, but permanently widens the client's type surface and hides future Node-type leaks.
- **C (rejected)**: Strip the `Save` interface in the Dockerfile with `sed`. Fragile, violates the "no sed to modify files" project convention, and is opaque to future maintainers.

Option A is the correct fix per Rule 6 (cohesion) and Rule 7 (boundaries). It also satisfies Rule 7a after the fact: the wire module's header now states the "no Node types" promise that was being silently violated.

### 2. Re-export from `repositories/types.ts` Rather Than Touch 7 Server-Internal Files

Rather than updating all 7 files that import primitives via `repositories/types.ts`, the now-thinned file re-exports them from `../wire/primitives.js`. This keeps the blast radius small and makes the refactor a one-commit, low-risk cleanup.

### 3. Wire Types Stay Server-Scoped, Not Moved to `@sharpee/core`

The alternative of moving `Tier`/`TextBlock`/`DomainEvent` to a shared package was considered and rejected. These types are server-scoped wire protocol for the multiuser feature — they are not general domain types appropriate for the core package.
