## Key Decisions

### 1. `null` Is a Valid Emission; `undefined` Means Skip

Phase 1 used `null || undefined` as the drop sentinel. ADR-163 §7 requires a "hide = `null` payload" semantic: a channel can actively publish a `null` value to signal that a client should hide or clear a resource (e.g., stop ambient audio with `{ id: 'rain' }` → `null`). Phase 2 narrowed the convention:

- Producer drops on `undefined` only
- `null` propagates through to the wire packet unchanged

`extractScore` in `platform-rules.ts` was updated simultaneously to return `undefined` (not `null`) for unparseable scores, so it does not accidentally publish a null score to the wire.

Any future extractor that returns `null` will produce a wire emission — this is intentional and load-bearing for hide/stop semantics.

### 2. `clear` Channel Registered Ungated

ADR-163 §6 lists `clear` as capability-gated but does not name a specific capability flag. Gating it without a named flag would require inventing a capability (`"clearable"`?) that no client declaration currently includes. Gating it to an existing flag (e.g., `images`) would prevent text-only clients from ever truncating `main`.

Decision: `clear` is registered without a capability gate. Every client can receive a `clear` packet. This is documented as an intentional deviation in `media-channels.ts`. If ADR-163 is updated to name a specific capability flag for `clear`, the channel definition should be updated accordingly.

### 3. Function-Valued Channel Resolvers for Dynamic Ids

ADR-163 §7 specifies `image:<layer>` and `ambient:<id>` as dynamic channel ids — the layer and id are carried in the event payload, not predetermined. Rather than generating per-layer rules at registration time or adding a generic-channel demux layer, `emit.channel` now accepts a function:

```typescript
emit: { channel: (input) => 'image:' + input.event.data.layer }
```

One rule covers the entire event class. The story registers the concrete channel ids it uses (`image:background`, `image:main`, etc.) via `registerMediaChannels()`. Routing to an unregistered id silently drops (rule is inert unless the channel exists).

### 4. Fixture Story Is Workspace-Registered but Not Built

The fixture story's `package.json` sets `main: ./src/index.ts`. Vitest's esbuild plugin handles TS transformation at test time. This avoids a separate build step and keeps the fixture light. The fixture must not be depended on for production code (only test devDeps). If a future phase needs the fixture in a production code path, add a build step then.

### 5. Reference Renderer: `clear` Fires After Channel Routing in the Same Packet

In `applyTurnPacket`, channel routing runs first (populating channel buffers), then `clear` packets are processed. A turn that both emits to `main` and sends a `clear` in the same packet will accumulate the `main` content and then immediately drop it — the buffer is empty after the packet. This matches real-renderer behavior (the client receives both the content and the clear instruction; the clear wins for that turn's final state).

### 6. Deterministic Story Init Proves §14 Repaint-Sequence Assumption

AC-12 resets the channel-service producer between the live-capture phase and the replay phase. Identical `storyInit()` call produces an identical CMGT manifest. Replay snapshot at turn 10 equals the live snapshot. This is the controlled proof of re-emission identity: packet capture + replay is sufficient to restore renderer state without engine replay.
