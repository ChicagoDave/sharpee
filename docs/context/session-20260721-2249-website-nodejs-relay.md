**Approach**: Investigated the live DNS/hosting state before touching anything, got two decisions from the user via AskUserQuestion (domain scope, deploy-flow style), then validated the build and the Node relay locally before writing the systemd/Apache/deploy-script artifacts, following the existing devarch.ai pattern already running on this box. **Go-live addendum (403 fix)**: After the user ran `--setup`, repointed DNS, and ran certbot, `https://sharpee.net` returned **403 Forbidden**. Diagnosis: the Node app was healthy (`localhost:3017 → 200`) and DNS/SSL were correct, but certbot's `--apache` installer generated the 443 vhost (`sharpee.net-le-ssl.conf`) with a stale static `DocumentRoot` (`/home/dave/repos/sharpee/website/dist` — the *old* `sharpee` repo path, which no longer exists) instead of the reverse proxy. Non-existent docroot + `Options -Indexes` = 403. Fix: authored the canonical proxy 443 vhost as `website/deploy/sharpee.net-le-ssl.conf` (mirrors the port-80 vhost and devarch.ai) for the user to copy over certbot's version and reload Apache. Documented the gotcha in both the vhost header and the README so a future certbot run doesn't silently reintroduce it.
