### 1. npm Regression Test — Combat Message Rendering Bug Fix

Ran the full npm-test suite (46 transcript tests against published @sharpee packages). 45 passed; 1 failed: test 35-combat — "attack dummy" produced no combat text, only daemon status text.

Root cause: In the attacking action's report phase (`packages/stdlib/src/actions/standard/attacking/attacking.ts`, line 441), combat message IDs like `combat.attack.missed` contain dots. The code was incorrectly treating all dotted IDs as "fully-qualified story keys" and skipping the `if.action.attacking.` prefix. Since the language provider stores messages with that prefix, the lookup failed silently and produced no output.

Fix: Narrowed the dot-check condition from `resolvedMessageId.includes('.')` to `customMessage && customMessage.includes('.')`. Standard combat message IDs now always receive the action-ID prefix. Story interceptor custom messages (e.g., `dungeo.melee.hero_attack`) can still use fully-qualified keys by passing a customMessage string.

Verification: All 1122 stdlib unit tests pass. Patched the published package in a temp directory and confirmed the combat transcript passes. Published as `@sharpee/stdlib@0.9.110`. Full npm regression suite: 46/46 passing.

### 2. tsf Build Tool — --package and --packageList Flags (tsf repo)

## Key Decisions

### 1. Narrow the Dot-Check to customMessage Only

The original intent of the dot-bypass was to allow story interceptors to provide fully-qualified message IDs (e.g., `dungeo.melee.hero_attack`) without the stdlib prepending the action prefix. The bug was that the check operated on `resolvedMessageId`, which also includes resolved standard combat IDs like `combat.attack.missed`. The fix correctly scopes the bypass to the `customMessage` parameter only, preserving the story interceptor pattern while restoring standard combat message resolution.

### 2. tsf as the Sole Publish Path

Committed to always publishing @sharpee packages through tsf rather than npm directly. The workspace:* dependency handling in tsf is the safeguard against version leakage. The new --package flag makes single-package operations as convenient as bare npm publish.

### 3. TheAlderman Design Decisions Made for Technical Coverage, Not Artistic Merit

All design choices (room layout, NPC personalities, weapon selection) were made to maximize coverage of the character package ADR surface area. The story is intended as a technical demonstration and developer reference, not a finished game. The author-checklist.md is the handoff document for any human who wants to turn it into a real game.

### 4. 216-Combination Randomization for Replay Value

Rather than a fixed solution, the game randomizes killer/weapon/location at runtime. This exercises the conditional branching in ConversationBuilder (NPC responses differ based on whether they are the killer) and demonstrates how story content can vary without branching all room descriptions.
