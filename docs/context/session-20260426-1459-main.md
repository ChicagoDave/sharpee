## Key Decisions

### 1. Two-step credential resolution mirrors WS hello
`findHashByHandle(handle)` ŌåÆ `{id, passcode_hash}`, then `verify`, then `findById(auth.id)` to get the canonical-case handle for the join session event. Defensive 500 `internal_error` guards the rare race where the row hard-deletes between the two lookups ŌĆö same defensive pattern already present in `hello.ts`.

### 2. Display name is server-resolved from `identity.handle`, never from the request body
This is a single-source-of-truth invariant. Clients cannot spoof a display name by sending a different string in the request body; the canonical handle comes from the identity row read after credential verification.

### 3. Stub hash format consolidated to `stub:<passcode>`
Both `test-app.ts` and `test-server.ts` now store `stub:<passcode>` so that `createStubHashService.verify` can actually compare the submitted passcode against the persisted hash in tests. The previous `'test-stub-hash'` literal was unverifiable and worked only because the old routes skipped verification entirely.

### 4. Client modal breakage deferred to Phase D, intentionally
Server-side auth gap closure is the security-critical work. UX wiring of the modals to the identity store is Phase D's responsibility. The client typecheck failure (`CreateRoomModal.tsx` / `PasscodeModal.tsx` using `display_name`) is documented as the expected interim state until Phase D lands.
