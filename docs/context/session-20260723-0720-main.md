### Key Decisions

1. **`buildPlaygroundBundle` is a sibling of `buildBrowser`, not a parameterization.** `buildBrowser()` gates the build on a specific `.story` compiling and derives all metadata from that story's `IRMeta`; the playground has no story at build time — nothing to gate on, no IR to source from. A parallel, simpler build path reusing only the CSS/theme-wiring helpers avoids twisting `buildBrowser()`'s contract to fit a case it wasn't shaped for.
2. **`/playground` is a new route; `/play` stays the Fernhill demo.** Retiring `/play` in favor of a generic playground would have dropped the just-shipped, asset-heavy Fernhill embed (audio/images that can't be example-picker fixtures) — this closes the plan-review TENSION recorded against Phase 1 in the plan file without regressing the earlier fix.
3. **AC-7 satisfied via declarative-content isolation, not strict origin isolation.** Chord content is declarative and cannot `eval` JS; a TS-hatch attempt is refused at compile time. Combined with `sandbox="allow-scripts allow-same-origin"` (needed for the runtime's `localStorage`), David judged this sufficient for Phase 1 and deferred stronger cross-frame isolation.
4. **Playground bundle output is gitignored; built on deploy instead of committed.** Reverses the plan's own Phase 2 open question (which had leaned toward committing the version-pinned bundle for deploy durability) — `deploy.sh` now runs `./repokit build --playground` as a guarded step so a clean deploy regenerates it, keeping ~1.2 MB+ of generated `game.js`/`game.js.map` out of git.
