### 5c — Story library admin routes (247/247 tests passing, +29 from 218)

**Platform change first** (David approved 2026-05-12): the bundle's `meta.json` had no top-level `id` field (the storyId only lived inside `story.js`'s runtime `config.id`). Added `id` to the meta.json convention in three platform files plus the zifmia fixture builder:
- `packages/sharpee/src/cli/build.ts` — adds `id: pkg.name` to the meta object
- `scripts/generate-meta.js` — adds `id: storyName`
- `build.sh` (line ~763) — adds `id: '${STORY_NAME}'`
- `tools/zifmia/tests/fixtures/build-bundle.ts` — adds `id: spec.config.id` to the meta JSON

Existing in-the-wild `.sharpee` bundles without `id` will now fail Phase 5c validation as `bad_structure`. This is consistent with the no-backcompat policy.

**Routes** in new `src/server/admin-stories.ts`, all gated by `[authMiddleware, adminMiddleware]`:
- **`POST /admin/stories`** — raw `application/octet-stream` body (50 MB cap via per-route `bodyLimit` and a scoped content-type parser). Pure `validateStoryBundle(bytes)` walks structure → missing_ifid → unsupported_format and returns `{ok:true, meta:{id, version, ifid, title, formatVersion}} | {ok:false, detail}`. On success: `installStoryBundle` (existing single-tx adapter method writing both `story_library` row and `story_bundles` blob) + `story.install` audit. 409 `invalid_bundle/version_already_installed` if `(id, version)` already present (pre-checked via `getStoryLibraryEntry` so the FK violation never surfaces as 500). Returns 201.
- **`PUT /admin/stories/:id`** — same body shape; additionally requires `bundle.meta.id === :id` (else 422 `story_id_mismatch`) and at least one existing row for `:id` (else 404 `story_not_found`). On success: install + `story.upgrade` audit (detail includes `previousVersions` array). Returns 200.
- **`DELETE /admin/stories/:id`** — 404 if no rows exist for that storyId; otherwise calls `removeStory` (UPDATE `active=0` across all versions) + `story.remove` audit (detail lists `versionsAffected`). Blobs preserved so pinned rooms keep loading. Returns 204.
- **`GET /admin/stories`** — admin list view returns `{stories: StoryLibraryEntry[]}` from unfiltered `listStories()` (includes inactive).

**Bundle protocol** (decided 2026-05-12 in /AskUserQuestion): server extracts everything from `meta.json`; no separate metadata channel. The `bad_signature` error code is reserved in the doc + type union but not emitted in v1 (signing deferred to v2).

### 5d — Admin room kill + identity passcode reset (260/260 tests passing, +13 from 247)

- **`src/server/admin-rooms.ts`** (new): `DELETE /admin/rooms/:id`. Auth + admin gated. 404 if room missing; otherwise reads `room.closedAt` to compute `wasAlreadyClosed`, calls existing idempotent `closeRoom` adapter method (which is `UPDATE WHERE closed_at IS NULL`), emits `room.kill` audit row with detail `{roomId, title, wasAlreadyClosed}`. 204 response. Soft-close preserves chat history, save_blobs, and audit rows for post-incident review.
- **`src/server/admin-identities.ts`** (new): `POST /admin/identities/:id/passcode_reset`. Generates a 16-char URL-safe random passcode using a 56-char ambiguity-free alphabet (`A-HJ-NP-Za-km-z2-9` — no `0/O`, `1/l/I` confusion). Modulo-bias-rejected sampling. ~92 bits of entropy. Hashes via existing `hashPasscode` (scrypt). Persists via `updateIdentityPasscode`. Invalidates every live session via `deleteSessionsForIdentity` (Phase 5a method). Emits `identity.passcode_reset` audit with `{identityId, handle}` ONLY — **plaintext is never written to the audit log or any persistent storage**. Returns 200 `{passcode, identityId, handle}` so the admin can communicate the new passcode out-of-band. The plaintext lives exactly once: in this single response body.

Both routes wired into the server bootstrap.
