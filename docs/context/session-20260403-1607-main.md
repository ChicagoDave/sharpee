## Key Decisions

### 1. Character Model Over Dialogue Model
Conversations are a projection of character internal state rather than a separate dialogue system. The author models the character; the system derives what can be said.

### 2. Constraint Solver, Not Generator
Author writes all prose responses and tags each with state predicates. The system evaluates current character state at runtime and selects the best-matching response. Authors retain full control of text; the platform manages selection logic.

### 3. Words Over Numbers
Authoring vocabulary uses TypeScript string literal union types (`'angry' | 'neutral' | 'warm'`) rather than numeric scales. Provides compile-time safety, natural reading, and avoids the decimal-tuning problem found in Versu and other simulation-heavy systems.

### 4. Five-Dimensional Cognitive Profile
Replacing a flat MentalState enum with five independent dimensions allows modeling complex states (perceptually accurate but belief-rigid, or lucid but fragmented) without combinatorial explosion. Each dimension is independently authored.

### 5. ADR Split: Character Model vs. Conversation System
The original combined ADR was split into ADR-141 and ADR-142. The two concerns have different lifecycles, different consumers, and different open questions. A character can have a full model without ever being talked to.

### 6. Explicit Author Priority for Response Selection
"Most specific wins" was replaced with "first match in declared order wins." Authors declare responses in priority order; the system selects the first whose constraints are satisfied. This is predictable, auditable, and familiar to authors who know Inform's rule ordering.

### 7. Directions as a Vocabulary
Hardcoded direction strings are the wrong abstraction — different story settings use different navigation metaphors. Making directions a vocabulary owned by the world-model registry, surfaced through a change-listener architecture, allows runtime vocabulary switching with automatic grammar propagation.

### 8. Change Listener Pattern for Grammar Propagation
Rather than requiring the engine to manually push vocabulary changes to the parser, the registry uses a `onChange()` listener. The engine registers one listener in `setStory()`, keeping the coupling minimal and the propagation automatic.

### 9. Clue Tutorial as Proof-of-Concept
Rather than testing the conversation system with a trivial example, the Clue tutorial exercises the full design: randomized state, multiple NPCs with distinct profiles, constraint selection under uncertainty, unreliable witnesses. If it works for Clue, it works generally.
