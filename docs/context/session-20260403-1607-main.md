### Design Exploration: Character Model Approach
- Key insight: conversations are a projection of character state — what a character knows, cares about, how they feel about the player, their mood, threat level, and cognitive state
- Architecture is a constraint solver, not a generator — author writes all responses, tags with conditions, system selects based on current character state
- Word-based vocabulary using TypeScript string literal union types for natural authoring with compile-time safety (no decimals, no magic numbers)
- Two-layer architecture: code-heavy API underneath, fluent builder on top

## Key Decisions

### 1. Character Model Over Dialogue Model
Conversations are a projection of character internal state rather than a separate dialogue system. The author models the character; the system derives what can be said. This eliminates the need to handcraft every conversation branch explicitly.

### 2. Constraint Solver, Not Generator
Author writes all prose responses and tags each with state predicates. The system evaluates current character state at runtime and selects the best-matching response. Authors retain full control of text; the platform manages selection logic.

### 3. Words Over Numbers
Authoring vocabulary uses TypeScript string literal union types (`'angry' | 'neutral' | 'warm'` rather than `disposition: number`). Provides compile-time safety, natural reading, and avoids the decimal-tuning problem found in Versu and other simulation-heavy systems.

### 4. Five-Dimensional Cognitive Profile
Replacing a flat MentalState enum with five independent dimensions allows modeling complex states (a character can be perceptually accurate but belief-rigid, or lucid but fragmented) without combinatorial explosion. Each dimension is independently authored.

### 5. Anti-Corruption Layer
The character model is internal state; the language layer owns all output. A translation boundary between them ensures character model vocabulary never leaks into message IDs or prose, maintaining the language-layer separation principle already established in the platform.

### 6. New Dedicated Package for Character Builders
The fluent authoring builder does not go into @sharpee/helpers. It is complex enough to warrant its own package, keeps the character model dependency graph clean, and signals to authors that it is a first-class authoring tool.

### 7. Clue Tutorial as Proof-of-Concept
Rather than testing the system with a trivial example, the Clue tutorial exercises the full design: randomized state, multiple NPCs with distinct profiles, constraint selection under uncertainty, unreliable witnesses. If it works for Clue, it works generally.

### 8. ADR Split: Character Model vs. Conversation System
The original combined ADR was split into ADR-141 (character model — who NPCs are) and ADR-142 (conversation system — how talking works). The two concerns have different lifecycles, different consumers, and different open questions. A character can have a full model without ever being talked to; a conversation system must be able to target that model as its input.

### 9. Explicit Author Priority for Response Selection
"Most specific wins" was replaced with "first match in declared order wins." Authors declare responses in priority order; the system selects the first one whose constraints are satisfied. This is predictable, auditable, and familiar to authors who have used Inform's rule ordering.
