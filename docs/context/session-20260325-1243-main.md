### run.sh isolation strategy

1. `mktemp -d` creates temp folder outside repo
2. Copies package.json, tsconfig.json, src/, tests/ into it
3. `npm install --no-fund --no-audit` from public registry
4. `npx tsc` compiles the story
5. Loops over each transcript file, runs via `npx transcript-test`, collects pass/fail
6. Reports summary; exits 1 if any failures
7. `trap cleanup EXIT` ensures temp folder is removed even on failure

## Key Decisions

### 1. Individual package imports rather than umbrella-only
Story imports from `@sharpee/engine`, `@sharpee/world-model`, etc. individually rather than only from `@sharpee/sharpee`. This is more realistic for a consumer project and verifies that sub-packages are properly published and their peer dependencies resolve correctly.

### 2. CommonJS output in tsconfig
The transcript-tester's story-loader uses `require()`. If the story compiles to ESM, `require()` will fail. CJS output is required for compatibility.

### 3. Each transcript is independent (no --chain)
Failures are isolated to the specific feature being tested. A broken container test does not cascade into the lockable test. The loop collects all failures before reporting, so the full picture is visible in one run.

### 4. Temp folder pattern (mktemp outside repo)
Running in a temp folder outside the monorepo guarantees no workspace symlinks or hoisted node_modules bleed through. This confirms the packages work as a true consumer would experience them — no special Node resolution shortcuts.
