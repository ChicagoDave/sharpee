## Key Decisions

### 1. WeasyPrint as PDF engine
CSS-driven, shares stylesheet with web output; no LaTeX dependency. Chosen over Prince/Typst after confirming WeasyPrint works with the CSS float/flex model already written for the HTML target.

### 2. One spine, two depths (reading-tracks model #1)
Single chapter ordering carries both Author and Programmer perspectives; `::: under-the-hood` divs are the depth separator. Avoids a separate Programmer track that would require maintaining parallel chapter files.

### 3. Two-step PDF build
Direct pandoc → PDF via weasyprint subprocess fails on base64 data URIs. Workaround: pandoc emits a standalone HTML at the repo root (so relative `assets/` paths resolve), then weasyprint reads that file. Committed into `scripts/build-book.sh`.
