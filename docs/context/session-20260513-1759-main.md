Respects `CERTBOT_EMAIL` env var for the ACME account.

## Key Decisions

### 1. Pin Dockerfile to a multi-arch OCI index digest (one pin, both arches)
`docker buildx imagetools inspect` returned a `vnd.oci.image.index.v1+json` for `node:20-bookworm-slim`. Pinning to that index digest (rather than a per-arch image digest) means the same `FROM` line resolves correctly under `--platform linux/amd64` and `--platform linux/arm64` during multi-arch buildx. The deps and runtime stages share one digest so the runtime layer's libc / OpenSSL / system Node identical to the layer the native modules were compiled against — moving the pins independently would silently produce a mismatch.

### 2. Post-push label verification uses `docker pull --platform linux/amd64` + `docker inspect`, not `buildx imagetools --format`
First draft used a nested Go template (`{{ index (index .Image "linux/amd64").Config.Labels ... }}`) against `buildx imagetools inspect`. That works but is brittle across buildx versions. Simpler approach: pull the amd64-specific image, then `docker inspect --format '{{ index .Config.Labels "org.sharpee.story-runtime-baseline" }}'`. Same result, far less to break.

### 3. Pre-release versions skip floating-tag advancement
The compute-tags helper short-circuits on `1.0.0-rc1`-shaped versions: emits only the immutable `:X.Y.Z-pre` tag, no API call, no `:latest` movement. Production deployments never accidentally pull a pre-release via a floating tag.

### 4. Certbot via `--webroot`, never `--apache`
`--apache` rewrites the vhost. Our script owns that vhost — it has the WebSocket upgrade rules and the proxy carve-out. `--webroot` writes the ACME challenge file under `/var/www/html/.well-known/` (which the port-80 vhost already serves) and is otherwise transparent.

### 5. `setup-zifmia-apache.sh` is two-phase: write port-80 vhost first, then certbot, then rewrite with port-443
You cannot write an `<VirtualHost *:443>` block referencing `/etc/letsencrypt/live/zifmia.sharpee.net/fullchain.pem` until that cert exists, and you cannot get a cert until something is serving HTTP-01 on port 80. Writing the port-80 vhost first, fetching the cert, then rewriting with the full 80+443 config is the only ordering that keeps `apachectl configtest` clean throughout.
