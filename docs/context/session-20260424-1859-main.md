### AC-4R.2: Docker Build — Three Cascading Blockers Resolved

The path to a working container required three independent fixes in sequence:

**Blocker 1 — `.dockerignore` too narrow.** The file re-included only `packages/core` and
blocked root workspace files, inflating the context to ~2 GB. Widened the re-include to
`packages/**`; explicitly re-included `package.json`, `pnpm-workspace.yaml`,
`pnpm-lock.yaml`; added `**/dist-cjs` and `**/dist-npm` prune rules. Context shrank to
manageable size.

**Blocker 2 — `pnpm deploy` broke in pnpm v10.** `ERR_PNPM_DEPLOY_NONINJECTED_WORKSPACE`
fired during the Docker build. Added `--legacy` flag to the `pnpm deploy --prod` invocation
in the Dockerfile.

**Blocker 3 — `files` field in `package.json` excluded `dist-esm`.** All 24 `@sharpee`
packages had `"files": ["dist"]`, but the server's `exports.import` condition points at
`dist-esm/`. `pnpm deploy` respects `files`, so the ESM tree was omitted from the deployed
node_modules. Added `"dist-esm"` to the `files` array for the 15 packages the server
actually requires: `core`, `engine`, `world-model`, `stdlib`, `parser-en-us`, `lang-en-us`,
`if-domain`, `if-services`, `event-processor`, `plugins`, `plugin-npc`, `plugin-scheduler`,
`plugin-state-machine`, `text-blocks`, `text-service`. Confirmed `build.sh` and `ts-forge`
are unaffected (neither reads the `files` field).

### AC-4R.2 Gate Met — `[undefined]` Rendering Fix

First live run after a successful Docker build showed five `[undefined]` lines on the
initial `look`. Root cause: `blockText()` in `Transcript.tsx` assumed the stub sandbox's
`{kind:'para', text:'…'}` shape. The real engine emits `{key:'game.banner', content:['…']}`.

Fix in `Transcript.tsx`: handle `content: string[]` alongside `text: string`; fall back on
`block.key` before `block.kind` when neither text-bearing field is present. After this
change the dungeon opening scene rendered correctly. **AC-4R.2 is met.**

### Eager Initial Look on First Connect (Platform Parity)

User directive: opening scene should appear without the user typing, matching
`platform-browser` and Zifmia behavior. Implemented `RoomManager.ensureInitialLook(room_id)`:
- In-memory `initialLookDone` Set to avoid repeat submissions within a process lifetime
- Persistent check via `sessionEvents.listForRoom(kinds:['output'])` — if the room has any
  prior output, it has already been greeted
- If cold, submits a synthetic `{actor_id:'system', text:'look'}` command

Wired into `hello.ts` after `connections.register` so every connecting player immediately
sees the scene if none was previously delivered.

### Transcript Backlog on Welcome (Refresh Rehydrates History)

User noticed that a browser refresh showed an empty transcript even though the game sandbox
and session_events table were intact. The welcome message was not sending prior history.

**Wire:** added `TranscriptBacklogEntry` type and optional `transcript_backlog` field on
the welcome message (`browser-server.ts`).

**Server:** `buildRoomSnapshot` in `room-snapshot.ts` queries

**Client:** `roomReducer.ts` initializes `state.transcript = msg.transcript_backlog ?? []`

that do not carry it (build #12 failure; tests passed unchanged after the `?? []` fallback).

User reported apparent story restart on refresh. Copied the SQLite DB and confirmed 26

### SCORE / HELP / ABOUT: Meta-Command Text Capture

User reported SCORE returning blank output. Tested HELP, ABOUT, SCORE, INVENTORY, and LOOK
through the live sandbox:
- INVENTORY and LOOK returned text (they go through the standard turn path)
- HELP, ABOUT, and SCORE returned `text_blocks: []`

Root cause: `GameEngine.executeMetaCommand` takes a separate code path. It calls
`processMetaEvents(events)` which emits text blocks via the `text:output` engine event —
the returned `MetaCommandResult` has no `blocks` field. `platform-browser` works because it
subscribes to that event. `deno-entry.ts` only read `result.blocks`.

Fix in `deno-entry.ts`: subscribe to `engine.on('text:output')`, accumulate into
`capturedBlocks`, reset the array before each `executeTurn` call, use `capturedBlocks` as
the fallback when `result.blocks` is empty or missing.

Verified against the live bundle: `SCORE` now returns
`"You have scored 0 out of 616, earning you the rank of Novice."` correctly.

## Key Decisions

### 1. `files` field is the correct long-term fix for `pnpm deploy`
Rather than restructuring exports or symlinking, the minimal correct fix was to declare
`dist-esm` in `files`. This is the standard pnpm v10 behavior; it is not a workaround. The
`dist-cjs` / `dist-npm` directories were intentionally left out (the container needs neither).

### 2. Eager `player_command` echo over waiting for the output frame
Broadcasting the command frame before the engine turn completes gives the multi-user room
immediate acknowledgment ("your command was received") and a clean separation between who
typed what and what the engine said. The alternative — tagging output frames with the actor
— would have required modifying the engine's event shape.

### 3. `actor_id === 'system'` suppresses command echo
The initial-look command must be invisible to players. Using `actor_id = 'system'` as the
sentinel is consistent with the existing wire convention for system-generated events and
avoids a separate boolean flag.

### 4. `transcript_backlog` optional on wire — never a hard requirement
Making the field optional rather than required keeps existing test fixtures valid without
modification. The client handles absence gracefully (`?? []`). This is the correct default
for any additive field on the welcome message.

### 5. RAF chaser for auto-scroll
`useLayoutEffect` fires synchronously after DOM mutation but before paint. When two frames
(echo + output) arrive back-to-back, the second frame triggers a second layout that
`useLayoutEffect` may miss. The `requestAnimationFrame` chaser fires after paint, ensuring
the scroll target is always the final rendered state.

### 6. `deno-entry.ts` must mirror `platform-browser`'s event subscription
`platform-browser` registers a `text:output` listener for the engine's meta-command path.
`deno-entry.ts` initially did not. The fix aligns the Deno entry with the platform-browser
pattern so both paths are symmetric. This is the correct long-term shape; any future engine
meta path that emits via `text:output` will be captured automatically.
