### Task 4 — `DialogManager.ts` Full Rewrite + Native `<dialog>`

This was the most significant logic change in Phase 1.

**Native `<dialog>` API adopted:**

- All show paths call `dialog.showModal()`.
- All close paths call `dialog.close(returnValue)` with named constants `RESULT_CONFIRM` or `RESULT_CANCEL`.
- A single `close` event listener per dialog is the resolution point. Buttons set `dialog.returnValue` and call `dialog.close()`; ESC and backdrop-click close with the cancel value; every path resolves the pending promise from the close handler. No divergent resolution paths.
- Backdrop click: `dialog.addEventListener('click', e => { if (e.target === dialog) dialog.close(RESULT_CANCEL) })`. Native `<dialog>` surfaces the backdrop click as the dialog element itself as `e.target`.
- ESC: handled by the browser natively — no document-level `keydown` listener anymore.

**Visibility cleanup:**

- `noSavesMessage` and `restoreSlotsListEl` visibility toggling switched from `.modal-hidden` CSS class to native `el.hidden = true | false`.

**Interface cleanup:**

- `DialogElements.modalOverlay` removed from `types.ts`.
- `DOMElements.modalOverlay` removed from `types.ts`.
- `BrowserClient.ts:156` no longer passes `modalOverlay` through.
- `stories/dungeo/src/browser-entry.ts` no longer queries `#modal-overlay`.
- Dialog field types tightened from `HTMLElement | null` to `HTMLDialogElement | null` with explicit cast at the `getElementById` call site.

### Task 5 — `templates/browser/base.css` New File

New file (~250 lines, structural-only). Contents:

- **CSS reset**: box-sizing, margin/padding zero, button defaults, list-style reset for `<menu>` / `<ul>` / `<ol>`.
- **`.sharpee-window`**: flex column shell, `100dvh` with `100vh` mobile fallback.
- **Menu structural rules**: bar is flex row; bar item is `position: relative`; dropdown is hidden by default; `--open` on the item reveals the dropdown; nested submenus open on hover/focus-within.
- **Status bar / prose pane / input bar layout**: flex grow and shrink ratios.
- **`.sharpee-prose-overlay`**: `position: absolute; inset: 0; pointer-events: none` — the precise spec the ADR commits to.
- **Native `<dialog>` shaping**: `padding: 0; border: 0; min/max sizing; flex column`. `.sharpee-dialog:not([open]) { display: none }` to keep non-open dialogs out of layout flow (native `<dialog>` is `display: none` by default but `display: block` when open; the override ensures ungrouped dialogs stay invisible when closed).

- **Deliberately excluded:** colors, backgrounds, fonts, hover effects, decoration borders, scrollbars — all theme territory. `infocom.css` is untouched pending Phase 2.

## Key Decisions

### 1. 22-class `.sharpee-*` contract is the stable platform commitment

Per ADR-170. Once theme authors target this vocabulary, breaking it cascades. The ADR review was the gate, not just a formality. Three non-blocking observations noted but not turned into ADR amendments — they were clarifiable during implementation.

### 2. No abbreviations rule applied throughout the vocabulary

Per `feedback_no_abbreviations`: `title-bar` not `titlebar`, `menu-bar` not `menubar`, `prose-pane` not `prose`. Applied to every class name before HTML was written. This is a one-way commitment — renaming after Phase 2 lands breaks authored theme CSS.

### 3. All ids preserved verbatim in the new HTML

The JS-platform → DOM contract uses ids for individual controls and classes for generic affordances. Zero id changes in HTML; new `.sharpee-*` classes added alongside. This was the conservative choice: id stability means `BrowserClient.initialize()` and the story entry point require only the removal of `#modal-overlay`, not a broader renegotiation.

### 4. Native `<dialog>` migration is one-way

`modalOverlay` field deleted from `types.ts` and all consumers updated atomically. The `DialogManager` rewrite is complete and the old `.modal-hidden` / `#modal-overlay` pattern is gone. Reverting would require re-adding the field and re-implementing the overlay div, not just reverting a single file.

### 5. `<dialog>` elements placed as siblings of `.sharpee-window`, not nested inside it

Native `<dialog>` with `showModal()` uses the top layer; nesting it inside the game window div adds no benefit and could interfere with stacking context. Placed as siblings in `<body>`.

### 6. Baseline screenshots skipped; live `git checkout` is the comparison method for Phase 2

The `git checkout` approach gives a live A/B comparison that degrades with the same screen/font/DPI as the current working state — more reliable than a stale PNG captured on a different day. If a single ambiguous state arises in Phase 2, capture that specific state then.

### 7. Phase 1 visual breakage is intentional and acceptable

The acceptance criteria for Phase 1 is JS wiring correctness + build passing + walkthrough chain, not visual parity. Visual parity is Phase 2's gate.
