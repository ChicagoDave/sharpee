**Round 1 — initial render** — Window appeared as a narrow centered strip (~half viewport). **Round 2 — misinterpretation** — David said "full browser width and a larger font." **Round 3 — correct fix** — David clarified: same centered/standard width as the other four themes, with a larger font. Restored the 1 px window border, removed the `max-width: none` override (so `base.css`'s `80ch` rule applies again), bumped body to 20 px and chrome to 16 px. Title bar height increased from 22 to 26 px to accommodate the taller chrome font.

**Round 4 — one more bump** — David asked for one more increase. Body bumped from 20 to **24 px** (a clean 2× of FindersKeepers's 12 px native bitmap size — intended to render crisp on integer-scaled screens). Chrome stayed at 16 px. The contract established here is explicit: `base.css`'s `80ch` rule defines window width structurally; each theme's visual width is a side-effect of its body font's glyph metrics, not a per-theme override. System 6 tunes `--theme-font-size` upward to make 80ch land at a comparable visual band.

## Cross-Theme Variable-Scoping Fix

### Symptom

After Phase 3 landed, David noticed that the "paper" and "system 6" entries in the theme submenu rendered with wrong colors compared to the other three entries — dark text on a dark dropdown background (or pale text on a cream background) depending on the active theme.

### Root cause

Theme radio menu items are marked up as `<li class="sharpee-menu-option" data-theme="paper">Paper</li>` so the JS theme switcher can read which theme each entry refers to. The variable definition blocks were written as `[data-theme="paper"] { --theme-bg: ...; --theme-text: ...; }`, which targets ANY element carrying that attribute — including the `<li>` itself. When the kit's `.sharpee-menu-option { color: var(--theme-text); }` rule fired on the `<li>`, it resolved `--theme-text` from the locally-defined block rather than from the inherited active-theme variables on `<html>`.

The bug was latent in all five themes from Phase 2 onward. It was only visually striking for paper (`#1a1a1a` text) and system-6 (`#000000` text) because those two dark-on-light themes' `--theme-text` values have low contrast against dark-theme dropdown backgrounds. The other three themes' text colors happened to remain tolerable in the same situation.

### Fix

Changed all five variable definition blocks from `[data-theme="X"] { ... }` to `:root[data-theme="X"] { ... }`. Now only the `<html>` element (the actual theme driver) receives the variable definitions, and inner elements carrying a `data-theme` attribute for JS-selection purposes do not inherit or override them.

The descendant-combinator rules below the variable block — `[data-theme="X"] .sharpee-menu-option`, `[data-theme="X"] .sharpee-output-area`, etc. — were not changed. They already fire only when `<html>` has the matching theme attribute, and the fix does not affect that part of the cascade.

Specificity note: `:root[data-theme="X"]` is `(0, 2, 0)`, higher than bare `:root` `(0, 1, 0)`, so the override pattern `:root` defaults → `:root[data-theme="X"]` overrides continues to cascade correctly.

This fix is the final edit of the session and is **not yet verified in a real browser build** — it was landed immediately before `/finalize`. The change is five CSS selector renames; no behavioral risk to engine paths.

## Key Decisions

### 1. Bundled webfonts (Option B over Option A)

Fallback chain `Charcoal → Geneva → system-ui` would have zero licensing surface but would not reproduce the System 6 aesthetic on non-Mac systems. David chose to bundle ChicagoFLF + FindersKeepers from `sakofchit/system.css`. Provenance acknowledged as freeware/workalike distributed under MIT wrapper, not OFL/SIL. ADR-worthy if the project ever formalizes a font-asset policy, but not a new ADR this session.

### 2. Append to shared `infocom.css` (Option A over Option B)

Same decision pattern as Phase 2. Per-file split (ADR-171) is Phase 5 work. Pre-empting it in Phase 3 mixes scope boundaries. `infocom.css` is now 1,982 lines; splitting it becomes more compelling but remains a Phase 5 concern.

### 3. Window width via `base.css`'s `80ch` rule, tuned via font-size

The "looks full-width" experience is a consequence of each theme's body font's glyph metrics, not a per-theme `max-width` override. The correct lever is `--theme-font-size`. System 6 uses 24 px body to land 80ch at ~960 px, matching the other themes' visual band. This is the deliberate contract: structural width is a single shared rule; visual width is a font choice.

### 4. `:root[data-theme="X"]` variable scoping as the structural fix

The `[data-theme]` attribute is used for two distinct purposes: (1) as the theme driver on `<html>` and (2) as a JS-readable selector on inner `<li>` elements. Using an unqualified `[data-theme="X"]` attribute selector for variable definitions conflated these two roles. Restricting variable definitions to `:root[data-theme="X"]` resolves the conflict structurally. This pattern applies to any future use of a "marker attribute" on inner elements that shares vocabulary with a root-level driver.
