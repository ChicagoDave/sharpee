## Completed

### Ch 26 §26.5 book fix (commit 14cabe83)
The book example set `--sharpee-bg/-fg/-accent` custom properties. The platform components consume the `--theme-*` namespace; `--sharpee-*` properties appear zero times in platform CSS. Fixed §26.5 to use `--theme-bg/-text/-accent`, matching §26.3's dos-classic example and `tutorials/familyzoo/browser/familyzoo.css`. Noted in `docs/book/testing/EXECUTION-LOG.md`. Also captured the important clarification: the `sharpee-` namespace is for component *classes* (`.sharpee-window`, etc.) — the author class-override surface — not CSS custom properties.

### Root cause discovery (QA rerun after fix)
Even with the corrected `--theme-*` names, the variables-only theme was still inert. Investigation confirmed: there are 349 `var(--theme-*)` consumers in `styles.css`, every one scoped inside a `[data-theme="X"]` block; `base.css` has zero un-scoped consumers. So a variables-only author theme cannot skin anything — it can only be seen by the engine rules that belong to the same `[data-theme]` block. The `--theme-*` variables were designed to enable a single shared consumer layer, but that layer was never built. Instead, five hand-copied themes (~360 lines each, ~1988 lines total) each re-implement all component rules under their own selector.

### ADR-188 written and ACCEPTED
`docs/architecture/adrs/adr-188-themes-as-plugins.md` — status ACCEPTED, extends ADR-170. Decisions R1–R7 all resolved in-session. Key design: extract the shared component rules into a single un-scoped engine layer in `@sharpee/platform-browser`; ship a `:root` default token set (white-on-blue, id `classic`); themes become `[data-theme="<id>"] { --theme-*: … }` variable blocks (+ optional flourishes) delivered as `@sharpee/theme-<name>` npm packages with a `sharpee.theme` manifest. The design-token cascade (`:root` = always-on default; `[data-theme]` overrides via source-order; themes load after engine CSS) provides automatic graceful degradation — no fallback code. 16 `--theme-*` tokens defined as the published contract. Nine ACs and a Playwright real-path test suite specified.

## Key Decisions

### 1. Engine home: @sharpee/platform-browser (ADR-188 R3)
Three candidates — devkit, a dedicated `@sharpee/theme-engine` package, or `platform-browser`. Platform-browser won: the engine is runtime presentation (platform-browser's domain), it must be a *published* contract for theme packages to target (R1), and it keeps devkit author-only (ADR-187). Dedicated package rejected as over-factored unless zifmia later needs a platform-browser-independent base.

### 2. Design-token / :root-default cascade (ADR-188 R4)
`:root` holds default tokens (always on, zero FOUC); theme = `[data-theme]` token overrides; engine consumes tokens once, un-scoped; theme CSS loads after engine/`:root` CSS (equal specificity → source order). Robustness is automatic from the model; no fallback code needed. David delegated the CSS-architecture judgment.

### 3. Themes are npm packages with explicit opt-in (ADR-188 R1, R6)
`@sharpee/theme-<name>` — one CSS file + `package.json` `sharpee.theme` manifest `{ id, name, css }`. Author lists desired packages in config; no `node_modules` scanning. Resolves the "hostile to add a theme" consequence of the current architecture while giving themes proper versioning and shareability.

### 4. Both browser builds rewired independently (ADR-188 AC-4, plan Phases 2+4)
Post-ADR-187 there are two independent browser-build code paths. ADR-188 requires both to source CSS from platform-browser and wire listed theme packages. They duplicate the wiring logic (ADR-187 R1: separate codebases). This is the cross-ADR seam the first adr-review round missed.
