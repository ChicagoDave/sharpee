## Architectural Pivot â€” Component-Based Theming

David reframed the question: "can we adapt our themes to the same pattern as system.css?" This shifted the work from "add system.css as a special case" to "adopt the structural pattern that makes system.css possible, then port all themes to that pattern."

The diagnosis of the current model: today's four themes are CSS variable swaps over a fixed DOM structure. They share identical markup assumptions; only palette and font change. A structurally distinct kit like system.css cannot be expressed inside this model without either mangling the DOM or writing a wrapper that fights the kit's natural patterns.

The insight that landed: system.css forces the question, but the right answer is a component-based theming contract. Define a vocabulary of ~20 named component classes (`sharpee-window`, `sharpee-menubar`, `sharpee-prose`, etc.). The HTML declares these classes. Each theme is a full CSS file that styles them. The platform ships four ported legacy kits and one new System 6 kit. Future story-specific themes are drop-in CSS files targeting the same vocabulary.

This is consistent with the `feedback_web_client_author_customizable` direction (authors override per-story), the ADR-163 channels-as-universal-UI-surface principle, and the 2026-04-29 blog post on author-enabled client architecture.

## Key Decisions

### 1. Platform-browser is the target, not Zifmia

Zifmia is a sibling React client that may be shelved. All theme work targets `templates/browser/` + `packages/platform-browser/`.

### 2. Component-based theming over CSS-variable-swap extension

Adding system.css as a `[data-theme]` variable block would require either DOM restructuring for that theme alone (inconsistent model) or a wrapper that fights system.css's natural patterns. The cleaner path is a shared component-class contract that all themes, including the four legacy ones, target.

### 3. Legacy themes are ported, not redesigned

Phase 2 goal is visual parity, not visual evolution. CRT scanlines, ASCII borders, and other enhancements are explicitly out of scope until the structural migration is stable and David decides where they fit.

### 4. System 6 kit will target `.sharpee-*` selectors directly

Rather than vendoring `system.css` and adapting around its assumptions, write fresh CSS that achieves the System 6 aesthetic while targeting the Sharpee vocabulary. This avoids upstream coupling and gives full control over which system.css patterns to adopt.

### 5. The first plan is superseded but preserved

`docs/work/system-css-theme/plan-20260507-system-css-integration.md` is left in place as a reference artifact. The active plan is `docs/work/theme-architecture/plan-20260507-component-themes.md`.

## ADR-169 Implementation

### Behavior Statements Produced

Before writing code, Behavior Statements were produced for each side-effect function:

- `unlock()` â€” constructs AudioContext lazily, awaits resume() if suspended, replays pendingEvents, falls back to instant-gain mode on construction or resume failure.
- `handleAudioEvent('audio.ambient.play' / 'audio.music.play')` â€” starts per-stream graph (`MediaElementAudioSourceNode â†’ GainNode â†’ destination`), fades in new stream, cross-fades outgoing stream using the outgoing stream's stored `fadeOut` value.
- `handleAudioEvent('audio.ambient.stop' / 'audio.music.stop')` â€” fades out active stream using event payload `fadeOut` (preferred) then stream's stored `fadeOut` then default. Stop-during-fade-in handled via ramp-mid-fade pattern (cancelScheduledValues + setValueAtTime + linearRampToValueAtTime).
- `dispose()` â€” disconnects all graphs, closes AudioContext, clears all stream maps.

### Files Changed

**`packages/platform-browser/src/audio/AudioManager.ts`** â€” full rewrite, 134 lines â†’ ~330 lines.

Key additions:
- `ActiveStream` interface (`el`, `source`, `gain`, `fadeOut`, `targetVolume`, `teardownTimer`).
- Per-stream Web Audio graph: `MediaElementAudioSourceNode â†’ GainNode â†’ audioContext.destination`. Element volume fixed at 1.0 to avoid multiplying with GainNode.
- `linearRampToValueAtTime`-based fade-in and fade-out.
- Ramp-mid-fade pattern for stop-during-fade-in: `cancelScheduledValues â†’ setValueAtTime â†’ linearRampToValueAtTime`.
- `outgoingStreams: Set<ActiveStream>` tracks fading-out streams between play and final teardown.
- Async `unlock()` with lazy AudioContext construction, `suspend` resume, and `pendingEvents` replay.
- Instant-gain fallback mode when AudioContext construction or resume fails â€” audio plays via bare `el.volume`, fades skipped, no silent failure.
- Constants: `DEFAULT_FADE_IN_AMBIENT = 1500`, `DEFAULT_FADE_OUT_AMBIENT = 800`, `DEFAULT_FADE_IN_MUSIC = 2000`, `DEFAULT_FADE_OUT_MUSIC = 1500`.
- `validateFadeMs()`: rejects negative, NaN, non-finite, Infinity â†’ returns default. Honors `0` as instant (no ramp).
- Fresh `new Audio(src)` per play (Web Audio one-element-one-context binding rule).
- Per-channel `Map<string, ActiveStream>` for ambients; cross-channel ambients are independent.
- SFX path unchanged â€” bare `Audio.play()`, no graph.

**`packages/platform-browser/src/BrowserClient.ts`** â€” line 567, single-line change: `await this.audioManager.unlock()` (was sync). Comment updated to reference ADR-169 timing.

### Key Implementation Decisions

1. **Outgoing-stream fadeOut comes from the outgoing stream's stored value.** When `audio.ambient.play` arrives for a channel that has an active stream, the fade-out duration applied to the outgoing stream is the *outgoing* stream's stored `fadeOut` (set when it started). The new event's `fadeOut` is stored on the *new* stream for its eventual stop or replacement. Matches ADR-169: "the outgoing stream's fade-out, defaulting to ambient 800ms."

2. **Stop-event payload `fadeOut` overrides stream's stored value.** On stop events, prefer event payload `fadeOut`, fall back to stream's stored `fadeOut`, fall back to default. Lets a story request a fast cut on a specific stop event.

3. **Element volume always 1.0 in graph mode.** `MediaElementAudioSourceNode` does not disconnect from the underlying element's volume â€” they multiply. Setting `el.volume = 1.0` and routing all volume control through the GainNode prevents accidental compounding.

4. **Fresh `new Audio(src)` per play.** A `MediaElementAudioSourceNode` cannot be created twice from the same element. Mandated by ADR-169; implemented accordingly.

5. **Cross-channel ambients are independent.** Each ambient channel gets its own `ActiveStream` and its own graph. Multiple ambients can play concurrently. Cross-fade applies only to same-channel source replacement.
