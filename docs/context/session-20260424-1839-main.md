### Mid-session sub-fix: `text` field in override

Realized mid-Phase-3 that the old `event-processor` entity-handler override path copied `messageId`, `text`, AND `params` (processor.ts:308-322). Without a `text` field on the override shape, melee's pre-rendered combat strings would have been silently dropped in the migration. Added `text?: string` to `InterceptorReportResult.override`; helper updated to copy it onto the primary event's `data.text`.

### Critical `this`-binding regression and hardening

After completing Phases 1–3 with helper taking an `eventFactory` callback, walkthroughs failed at `> put guidebook in receptacle` with:

```
✗ Cannot read properties of undefined (reading 'createEventInternal')
```

Root cause: `EnhancedActionContext` (`packages/stdlib/src/actions/enhanced-context.ts:294-296`) has `event` as an unbound class method. Passing `context.event` as a callback strips `this`. The engine's closure-based context (`packages/engine/src/action-context-factory.ts`) uses an arrow function, so unit tests through that path were unaffected. The wt-15 cascade was caused by the bind crash on put triggering a drop sequence failure in wt-04, leaving the guidebook never placed, and wt-15 finding nothing to take.

**Hardening approach chosen (both options per user):**
1. Refactored helper signature to `context: InterceptorEventContext` (structural type). Call sites pass `context` directly — no callback to strip. Removed all `.bind(context)` from the 9 callsites.
2. New regression test `packages/stdlib/tests/unit/capabilities/interceptor-context-binding.test.ts` — 3 tests using a REAL `EnhancedActionContext` via `createActionContext`. Covers emit-only, override-only, combined. If a future refactor reintroduces a callback parameter, these tests fail with the same error class users saw in wt-15.

## Key Decisions

### 1. Override semantics must be explicit in the return type
The old `CapabilityEffect[]` return type of `postReport` carried no signal about override intent. The implicit contract — one `game.message` effect means "replace the primary" — was not preserved when ISSUE-068 migrated the rug handler to `CapabilityEffect[]` in the stdlib report path. The fix makes `override` vs. `emit` a named, documented distinction.

### 2. `InterceptorEventContext` object over callback parameter
Using a structural context object instead of an `eventFactory` callback eliminates the `this`-binding failure class at the helper boundary. The call site cannot accidentally strip `this` from an unbound class method when there is no callback extraction.

### 3. `text` field added to override shape
The old event-processor path copied three fields on override: `messageId`, `text`, and `params`. A `text`-only override (no `messageId`) is used by the melee interceptor for pre-rendered combat strings. Omitting `text` from the new shape would have silently dropped these strings. The field is optional; non-melee overrides simply omit it.

### 4. Regression transcript uses negative assertion
`[OK: not contains "doesn't move far"]` is stronger evidence of the fix than a positive assertion alone. The bug was that BOTH the override message AND the primary appeared; the negative assertion directly proves suppression.

### 5. ADR-157 written before Phase 5 (remaining audit)
The re-audit of other ISSUE-068 migrations (troll given/thrown behaviors) is deferred to a follow-up. Writing ADR-157 now ensures the audit lesson is documented and linked from CLAUDE.md even while Phase 5 is outstanding.
