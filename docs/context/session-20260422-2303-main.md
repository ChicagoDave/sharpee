## Key Decisions

### 1. Flat Promote/Demote Menu Items (Plan 02 Phase 3 Deviation)

Plan specified submenu items ("Promote → Co-Host", "Demote → Participant"). Implemented as flat top-level items instead. Rationale: the authority matrix only has one valid promotion target (Co-Host) and one valid demotion target (Participant) from any given role transition, so a submenu adds navigation overhead for zero disambiguation value. Deviation documented in-session.

### 2. Tab State in Local `useState` vs. Reducer (Plan 04 Phase 1 Deviation)

Plan 04 Phase 3 (unread counts + cross-boundary reducer UI action) requires tab state in the reducer. Rather than do a partial lift in Phase 1, tab state was kept in `Room`'s local `useState` with `openedDmPeers`. The cross-boundary lift is the explicit entry state for Phase 3. Deviation documented in-session; Phase 3 will resolve it.

### 3. Server Addenda Pattern

Three server changes were needed mid-plan and handled as in-session addenda rather than pre-planned phases: `welcome.chat_backlog`, `PATCH /api/rooms/:id` rename-room, and `presence.grace_deadline`. Each addendum was implemented, tested (+1 to +10 tests), and integrated before the corresponding client phase proceeded. This kept the wire contract ahead of the client implementation in all cases.

### 4. Save Name and Creator Columns Skipped

The wire type (`RoomSnapshot.saves`) carries no `actor_id` field and the `save` command accepts no name parameter. Two plan features (save-name prompt, save-creator column in `SaveList`) were skipped as out-of-scope given the wire contract. Documented as scope deviation in Plan 03 Phase 1 notes.

### 5. Container/Presentational Split for `Room`

`Room` default export is the container (owns WebSocket + reducer); `RoomView` is the named presentational export (accepts state as props). This pattern, established in Phase 7, was maintained through all subsequent phases and is what makes the 20-test `Room.test.tsx` suite possible without live WebSocket connections.
