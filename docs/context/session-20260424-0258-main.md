## Key Decisions

### 1. Option (b) â€” Rewrite Rather Than Delete

Per user choice. Preserves observable-WS coverage of the happy path under the `SHARPEE_REAL_SANDBOX=1` gate rather than losing it entirely. The tradeoff: tests are skipped in local dev without the env var set, but they exist and can be run in any environment with Deno present.

### 2. Hybrid Approach â€” Inline Fake Doubles for Rejection-Path Unit Tests

Per user choice over all-delete. Defense-in-depth rejection coverage for conditions the real sandbox cannot induce (sandbox timeout, save ID mismatch, corrupt blob, crash-before-READY, etc.) is preserved. The key distinction between the old `sandboxOverride` (SUT substitution, prohibited) and the new inline doubles (unit-test DI seam via `createSandboxRegistry(factory)`, permitted) was explicitly confirmed before implementing.

### 3. Rename `stub-sandbox.mjs` Rather Than Delete

Per user choice. The file is a collaborator subprocess for a wrapper test, not a substitute for the system under test. Renaming to `sandbox-process-child.mjs` documents the intent in the file name and removes any ambiguity for future readers. Deleting it would have removed legitimate test infrastructure.

### 4. `createSandboxRegistry(factory)` Retained as DI Seam

The No-Stub-Under-Test principle prohibits substituting the SUT. `sandboxOverride` allowed the real `RoomManager` and `SaveService` to spawn a stub instead of a real Deno process â€” that was SUT substitution. `createSandboxRegistry(factory)` is a different seam: it is the DI entry point for constructing the registry with a custom factory, which is standard unit-test isolation for the layer above the spawn point. This distinction is what allows the hybrid path to be architecturally sound.

### 5. SIGKILL Timing Fix for Crash Test

For the `turn-execution` crash test, the SIGKILL must be delivered after `inflightTurnId` is populated but before the OUTPUT frame arrives. The registry sets `entry.status='ready'` and then emits `'ready'` on `entry.process`. Attaching a `once('ready', ...)` listener fires synchronously inside that emission, guaranteeing the kill arrives after the status flip and before the microtask that resets `inflightTurnId`. Polling for status (the first approach) produced a race; the synchronous listener eliminated it.
