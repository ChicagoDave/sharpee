### devkit version-stamp fix — `@sharpee/devkit` bumped 1.0.6 → 1.0.7

- Fix: extracted `packages/devkit/src/standalone/version-stamp.ts` (`stampVersion` helper); `init-browser` now SEEDs a stub `src/version.ts` at scaffold time so it exists before any tsc run; `sharpee build` re-stamps it before compilation; `build-browser` still refreshes it on each browser build.

### tsf `type:module` publish defect (fix in `/Users/david/repos/tsf`)
- Symptom: built `game.js` threw `ReferenceError: Can't find variable: exports` in the IDE Play pane (WebKit) at the `@sharpee/platform-browser` import.
- Root cause: `tsf/src/sync/package-json.ts` correctly strips `manifest.module` when publishing CJS, but never touched `manifest.type`; packages with `"type": "module"` in source were published with CJS files still declaring `type: module`, making `exports` undefined in a no-CommonJS runtime.
- Fix: `delete manifest.type` added beside `delete manifest.module` in the tsf repo; tsf rebuilt. Not committed to the sharpee repo — lives in `/Users/david/repos/tsf` and needs its own commit there.
- Diagnostic note: local builds did not reproduce because local `platform-browser` has a proper dual `dist/`+`dist-esm/`; only a build against the published package in a WebKit/no-exports sandbox reproduced it.
