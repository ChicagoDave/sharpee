## Key Decisions

### 1. Author Override is a First-Class Platform Capability (Decision 6, ADR-163)

Original framing assumed "platform-shipped UI only for v1." David explicitly rejected this. The design intent, reaching back to fyrevm-server (2010 era), has always been that rendering belongs to the author. Platform ships defaults that work out of the box; any author can override any surface for their story. This settled the question rather than deferring it.

Implication: the wire must be data-only. Anything the wire encodes as presentation is something the author can never override.

### 2. Canonical Extension Pattern: Named Slots + Author Renderer + Engine Emissions

The correct architectural shape for author-controlled UX: story defines channels with structured JSON payloads (named data slots), ships its own HTML/CSS renderer, and the engine emits to those channels via story-specific verbs and event handlers. Platform routes; author renders. The wire shape is identical to the platform default. Exemplar: `stories/thealderman/docs/detective-sheet.jsx`.

### 3. Per-Channel Emit Policy Instead of Global Invariant

Initial draft required all channels to emit every turn. Refined to per-channel policy: `emit: 'always' | 'sparse'`. Standard channels always emit so clients can render predictably without tracking delta state. Story channels default to sparse with author opt-in to `'always'` for ephemeral UX elements (timers, gauges) that need every-turn refresh even when unchanged.

### 4. Triggers Are One-Way (No onComplete)

ADR-164 trigger model was originally bidirectional (engine fires trigger, client completes animation, client sends onComplete). David clarified triggers are advisory only: "the trigger is one way — we're just alerting the UX to maybe an animation or cut scene or something." No synchronization callback. Engine continues on its own timeline.

### 5. ADR-101 Was Dead Code

The audit confirmed ADR-101 (Proposed, Jan 2026) was never implemented. No references to its core primitives (`media.image.`, `ClientCapabilities`) exist anywhere in the codebase. This made superseding it straightforward — no migration needed, just a clean replacement.

### 6. text-service Retired as Wire Producer

ADR-164 explicitly retires text-service from its wire-producing role. text-service can remain as an internal content-assembly convenience if useful, but it no longer owns the surface between engine and client. Channel packets are the wire.
