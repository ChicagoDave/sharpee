## Key Decisions

### 1. Scoring in Execute Phase vs. Event Handlers
Feed-goat and feed-rabbit scoring is awarded inside the action's `execute` phase rather than via an event handler, because the feed action is story-owned and the execute phase is the cleanest place to call `world.awardScore()`. Room-visit and item-collection scoring uses chain handlers on `if.event.actor_moved` and `if.event.item_taken` to demonstrate both approaches in the same version.

### 2. Victory Daemon Instead of Score-Check Event Handler
The win condition uses a turn daemon that checks `world.getScore() >= 75` rather than a one-shot event handler on a specific action. This matches the dungeo victory-handler pattern and correctly handles any combination of scoring actions reaching the threshold.

### 3. `skipNextTick` Causes an Off-by-One
A fuse declared with `turns: 10` fires at tick ~11 due to `skipNextTick`. This is documented as ISSUE-056 (undocumented API behavior) and noted in the transcript comment. The transcript and implementation are aligned; the behavior is not a bug — the API just lacks documentation.

### 4. `[SKIP]` Does Not Execute — It Skips Entirely
Discovered this session (commit f80df965) that `[SKIP]` skips execution of the command entirely, rather than executing without asserting. This is correct by-design for DO-UNTIL loops, but ISSUE-054 was filed for an "execute but don't assert" assertion syntax that would be useful for advancing game state without caring about output.

### 5. `[OK]` Alone Fails When Output Exists
`[OK]` with no argument fails if any output is produced. The correct pattern for "I don't care about the output" is `[OK: contains ""]` or simply not asserting. Transcripts were updated to use specific `[OK: contains "X"]` assertions to be explicit.
