### Part 2 — React Error #310 (hooks ordering) in Room.tsx

User tested `play.sharpee.net`. Landing worked, room creation worked, but entering the room triggered React error #310 ("Rendered more hooks than during the previous render") and a blank `<div id="root">`.

Root cause: classic Rules-of-Hooks violation in `RoomView` in `tools/server/client/src/pages/Room.tsx`. Five hooks (`useState(toasts)`, two `useCallback`s for dismiss/push, `useRef(prevPhIdRef)`, `useEffect` for PH-transition toasts) were declared AFTER two conditional early returns (`if (state.closed)`, `if (!state.hydrated)`). First render (unhydrated) fired 8 hooks; second render (hydrated) fired 13 → crash.

Why the existing `Room.test.tsx` missed this: it only renders `RoomView` with a pre-built `state.hydrated: true` fixture, never exercising the unhydrated → hydrated transition.

Fix: hoisted all 5 misplaced hooks to the top of `RoomView`, above both early returns. Rebuilt Docker image (cache hit — fast), retagged, recreated container. Health green in 6s.

Status: fix applied and deployed; NOT yet committed.

### Part 7 — Remediation-1: Acceptance Gate Test (RED by Design)

Wrote `tools/server/tests/sandbox/deno-engine-integration.test.ts` (new, 198 lines). Zero stubs — imports only `spawnSandbox` from production code and wire types. Points `story_file` at `tools/server/stories/dungeo.sharpee`.

Five assertions designed to refuse stub values:
1. `INIT → READY`: `story_metadata.title !== '(stub)'`, `version !== '0.0.0'`
2. `COMMAND "look" → OUTPUT`: `text_blocks.length > 0`, serialized length > 200 chars, `events.length > 0`
3. `SAVE → SAVED`: `blob_b64.length > 500` (stub's hardcoded blob is 12 characters)
4. `SAVE then RESTORE`: observable `look` output byte-identical pre-save and post-restore across an intervening state-mutating command
5. `SHUTDOWN → EXITED` with `reason === 'shutdown'`

Result: 5/5 RED with `spawn deno ENOENT` — host machine does not have Deno installed natively (Deno exists only inside the Docker container). This is the correct signal: integration is unverifiable in this environment, no silent stub fallback.

Side-discovery (filed, not fixed): `SandboxProcess` never listens for `child.on('error')`, so a missing Deno binary crashes the server with an uncaught exception instead of emitting a `crash` event. Belongs in remediation-3.

### Part 9 — Deno Permissions Posture: Option B

Evaluated three permission strategies (A: stream bundle over stdin, B: narrow `--allow-read` on bundle + story, C: hybrid). User reframed the ADR-153 language — "no filesystem access" means no access to anything important (the server's DB, other rooms' saves, other stories, `/etc`) — not "zero bytes ever flow through a file descriptor." Under that reading, **Option B is the ADR's intent.** The container is the outer boundary; Deno's narrow `--allow-read=<host_bundle>,<story_file>` is the inner boundary against rogue story code.

User chose **Option B**.

### Part 10 — Story Compilation Strategy: Option D1

Pulled apart `dungeo.sharpee` and found that `story.js` uses bare-specifier imports (`from "@sharpee/stdlib"`, `from "@sharpee/world-model"`, etc.) because the `.sharpee` build uses `--external:@sharpee/*`. These resolve fine in Vite-hosted Zifmia, but in a Deno sandbox running an esbuilt `host.js`, there is no module named `@sharpee/world-model` — esbuild inlines everything as private scope. A naive "load host bundle, blob-URL import story.js" plan would fail at runtime on the first bare-specifier resolution.

Options enumerated:
- **D1 (chosen)**: server-side pre-bundle at install time — when a `.sharpee` file drops into `/data/stories/`, the server esbuilds `host+story` into a single self-contained ESM cached at `/data/stories-compiled/<slug>.host.js`. Re-bundle on mtime change. Operator drops file, users play — compile is transparent.
- D2: pre-bundle at Docker-build time only (operator-added stories need manual step)
- D3: re-bundle every room spawn (wasteful)
- D4: import map in deno.json (complex re-export surface)
- D5: change `.sharpee` format (breaks vision)

User chose **Option D1**. Vision intact.
