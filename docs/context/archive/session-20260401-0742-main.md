## Completed

### Fix: GamePrompt not showing in browser client command line

The GamePrompt system (ADR-137) was emitting a prompt block in `text:output`, but the main browser client (`packages/platform-browser`) was not handling it. The prompt text appeared as narrative text in the game output instead of updating the command line prompt element.

**Root cause**: `BrowserClient.setupEngineHandlers()` in `packages/platform-browser/src/BrowserClient.ts` passed all blocks (including the prompt block) directly to `renderToString()` without first extracting the prompt block. The newer `packages/platforms/browser-en-us/` platform had the correct implementation, but the main `platform-browser` package was never updated to match.

**Initial fix attempt**: Set `commandInput.placeholder` with the prompt text. This showed the prompt as background/hint text but did not replace the hardcoded `>` character in the `<span class="prompt">` element, so the prompt was not visible during normal input.

**Final fix** (2 files changed):
1. `packages/platform-browser/src/managers/InputManager.ts` — Added `setPrompt(text: string)` method that finds the `.prompt` span sibling of the command input and updates its `textContent` directly.
2. `packages/platform-browser/src/BrowserClient.ts` — Updated the `text:output` handler to extract the prompt block, pass its text to `inputManager.setPrompt()`, and filter it out before passing the remaining blocks to `renderToString()` as narrative text.

Built with `./build.sh -s dungeo -c browser`, copied to website with `copy-to-website.sh`, and rebuilt the Astro site. Deployed and verified working in the live browser client.

## Key Decisions

### 1. Update `.prompt` span textContent rather than `placeholder`

The `<span class="prompt">` element is the visual prompt glyph rendered beside the input field. Updating `placeholder` only affects the hint text visible when the field is empty — it does not change the displayed prompt character. Updating `textContent` on the span directly is the correct approach and matches what the `browser-en-us` platform implementation already does.

### 2. Filter prompt block before rendering

Rather than special-casing the prompt block inside `renderToString()`, the handler in `BrowserClient.ts` filters it out before the render call. This keeps `renderToString()` purely concerned with narrative text and avoids leaking prompt semantics into the render path.
