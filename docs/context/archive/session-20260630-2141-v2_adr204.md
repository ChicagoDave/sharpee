Fix the ADR-199 `regularPluralVerb` `-se`/`-ze` mis-strip bug surfaced during ADR-203, via a

## Status: COMPLETE — ADR-204 authored, reviewed, implemented; merged to main

## What happened

- **ADR-204 authored + reviewed.** `docs/architecture/adrs/adr-204-verb-pluralization-se-stem-heuristic.md`.
  `/adr-review` scored 11/11 READY. At acceptance the sibling `-ies` edge (die/lie/tie) was
  **folded in** (reviewer decision), and the atomic-ordering + Session-wording tightenings applied.
- **The bug:** `regularPluralVerb`'s `/(?:s|x|z|ch|sh)es$/` rule stripped `-es` from single
  `-se`/`-ze` stems (use→uses→"us", refuse→refuses→"refus", collapse→collapses→"collaps"). Latent
  in the shipped catalog only because ADR-203 had dodged the two live cases.
- **Implementation (branch `v2_adr204`, one commit):**
  - `english-assembler.ts`: regex → `/(?:ss|zz|x|ch|sh)es$/` (doubled-sibilant discrimination); a
    single `s`/`z` before `es` now strips only `-s`. Added the `-ie` closed set
    (`dies/lies/ties/vies` → die/lie/tie/vie) to `IRREGULAR_VERBS` (`conjugateVerb` consults it
    before the regex). Both are heuristic-with-escape-hatch: rare single-`s` stems (focus/bus/gas)
    add to the table on demand.
  - Reverted the two ADR-203 dodges now that they render correctly: `npc.unconscious`
    "falls"→"collapses"; conversation `refuse` "declines"→"refuses".
  - `verb-agreement.test.ts`: new ADR-204 describe block — `-se`/`-ze` correctness, genuine `-es`
    regression, unaffected classes, and the `-ie` closed set.
  - Amendment note added to ADR-199.

## Key decisions

- **Heuristic + closed-set escape hatch** (symmetric): open classes (`-se`/`-ze` verbs, `-y` verbs)
  ride the regex; the irreducibly-ambiguous closed sets (`-ie` bases; single-`s` bases) live in
  `IRREGULAR_VERBS`. The `uses`/`buses` and `carries`/`dies` surface collisions can't be resolved
  by rule — optimize for the common case, table the rare exceptions.
