## Key Decisions

### 1. ADR-163/164 Split Along Platform/Downstream Boundary

The original combined ADR conflated universal wire format (platform concern) with multi-user fan-out (server concern). The split makes ADR-163 implementable without any server infrastructure; ADR-164 inherits and extends it. This boundary prevents server-layer concepts from leaking into the platform API.

### 2. Single-Bundle as the Default Deployment Model

ADR-163 was reframed to lead with the single-bundle case (CLI, zifmia, platform-browser). Multi-user is documented as the transport exception that uses the same wire, not the motivating scenario. This ensures the platform API is shaped by the common case, not the complex case.

### 3. Re-Emission Identity as the Load-Bearing Platform Guarantee

Rather than adding a replay flag or a content-addressed packet ID, the platform guarantees that re-emitting the same turn with the same world state produces byte-for-byte identical channel packets. This is the foundation of all three repaint policies and removes the need for any special repaint mode in the producer.

### 4. `onDestroy` Hook Symmetric with `onCmgt`

The cross-ADR review surfaced that ADR-165 had `onCmgt` (renderer mount) but no teardown. Adding `onDestroy` (optional) ensures renderers can clean up DOM listeners, timers, or external resources. This is a small addition but closes a real resource lifecycle gap.

### 5. ADR-165 Scope: Renderer Owns Consumer-Side, Not Platform

The decision to create ADR-165 as a separate ADR (rather than folding renderer concerns into ADR-163 or ADR-164) reflects that the renderer layer is client-side and story-customizable. Platform and server are unconcerned with how a consumer displays a channel packet.
