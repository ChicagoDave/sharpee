## Key Decisions

### 1. Inline ScoringRead Rather Than Importing stdlib

The client boundary is `@sharpee/world-model` only. Rather than adding `@sharpee/stdlib` as a runtime dep to pull in its `ScoringCapability` interface, a minimal `ScoringRead { score: number; maxScore: number; turnCount: number }` interface was inlined in `StatusLine.tsx`. This is a deliberate narrow import â€” the client reads scoring capability data off the mirror via `getCapability('scoring')` and casts to `ScoringRead`. If the scoring capability shape changes in stdlib, the inline type becomes the change-surface (compile error on cast), which is the desired behavior. Adding stdlib to the client's dep graph would pull in action registration, grammar, and other server-side concerns.

### 2. Segment-Level Graceful Degradation in StatusLine

Three potential missing-data conditions (player entity, scoring capability, containing room) each degrade only the segment they feed. The component never throws. This keeps the status bar useful during loading and for stories that do not implement scoring. The behavior contract is: null mirror â†’ "â€¦" entire bar; partial mirror â†’ "â€”" per missing segment.

### 3. .test-d.ts Suffix for Compile-time Type Tests

Placing the mutation-rejection directives in a `.test-d.ts` file keeps Vitest from executing it as a test suite while still including it under the `tsc --noEmit` path through the `src/**/*` tsconfig include. This is the lightest mechanism available â€” no special Vitest config, no extra tsc invocation, no package-level type-test infrastructure.

### 4. SHARPEE_BANDWIDTH Gate for AC-8

The bandwidth baseline test drives a full real-sandbox run and prints detailed size stats. It is expensive (~15s) and deterministic only against a stable dungeo bundle. Gating on `SHARPEE_BANDWIDTH=1` keeps CI fast while making the baseline reproducible on demand. The test file and the measurement document together form the durable artifact; re-running with the env var re-measures against whatever bundle is current.

### 5. RoomTrait Required for getContainingRoom()

`WorldModel.getContainingRoom(id)` walks the spatial-index chain looking for an entity that has `RoomTrait`, not just entity-type 'room'. All StatusLine fixture builders and any future renderer tests that query room names must call `room.add(new RoomTrait())` on entities they intend to be found by this walk. Discovered during Phase E test development; noted here so future test authors do not repeat the investigation.
