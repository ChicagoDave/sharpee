## Key Decisions

### 1. Build Context = Repo Root, Not `tools/server/`
Dockerfile reaches into `../../packages/core` (file: deps), so build must be invoked from repo root:
```
docker build -f tools/server/Dockerfile -t sharpee-server:local .
```
Or via compose, where `context: ../..` is already set:
```
cd tools/server && docker compose build
```
`.dockerignore` placed at repo root accordingly.

### 2. Fail-Fast Tests in Builder Stage
Per user decision: `RUN npm test` runs in the node-builder stage before TypeScript compilation. A failing unit/integration test fails the image build. Trade-off accepted: slower builds in exchange for never shipping an image whose tests don't pass.

### 3. CAPTCHA Bypass via Env Passthrough, Not Compose Default
`CAPTCHA_BYPASS: "${CAPTCHA_BYPASS:-}"` — empty by default, set only when smoke test exports it. Operators in production never see the bypass enabled by accident.

### 4. Healthcheck Tool = curl (Not Embedded HTTP Probe)
Kept `curl` in the runtime image because compose healthchecks need a CLI HTTP client. Native Node probe would add complexity without benefit.
