### Browser Client Build Fix
- `@sharpee/platform-browser` declares `dist-esm` in its `package.json` exports under the `import` condition
- No ESM build step exists for this package (no `tsconfig.esm.json`)
- esbuild resolves the `import` condition before `require`, causing it to look for `dist-esm/index.js` which does not exist
- Fix: added `--alias:@sharpee/platform-browser=<repo-root>/packages/platform-browser/dist/index.js` to the browser IIFE esbuild command in `build.sh` (line 820)
- Browser client now builds successfully: 1.2M bundle

## Key Decisions

### 1. Accept Pulls Without Re-Planning
Both incoming platform pulls (f856111f and 9de3f020) were non-breaking and self-contained. Rather than triggering a re-plan, they were absorbed; the first pull's ADR-140 helpers became the basis for the tutorial content, and the second pull's `.addTrait()` additions required no tutorial rework.

### 2. Alias CJS Dist for Browser Bundle
Used an esbuild `--alias` flag pointing to the CJS `dist/index.js` rather than creating a full ESM build for `platform-browser`. The IIFE browser bundle does not need ESM, and the build already uses `--conditions=require`, making this alias consistent with the rest of the build pipeline. Creating a full ESM build would require a new `tsconfig.esm.json` and changes to the build script for a single edge case.
