### ADR-148: Concealment Action (Three Rounds to 14/14)

Wrote a full ADR for HIDE/DUCK BEHIND/UNDER/ON/INSIDE. This ADR fills a gap that ADR-144 (PlayerPresence) exposed: the `concealed` PlayerPresence state existed in the type system but no platform mechanism could put the player into that state.

The ADR went through three rounds of `/adr-review`:
- Round 1: 6/14 â€” missing state storage decision, unclear reveals mechanism, preposition delivery unspecified, no NPC detection design
- Round 2: 7/14 â€” improved state storage, but visibility integration was a parallel system (anti-pattern), reveals mechanism underspecified
- Round 3: 14/14 â€” all criteria met after integrating into the existing VisibilityBehavior pipeline and specifying the hook allowlist

Key design decisions made through the review process:

**State storage**: `ConcealedStateTrait` as a dynamic trait added to the player at hide-time and removed at reveal-time. Presence/absence of the trait IS the state â€” no boolean flag needed.

**Visibility integration**: `ConcealedStateTrait` registers the `if.scope.visible` capability and hooks into the existing `VisibilityBehavior.canSee()` pipeline. The concealment check runs inside the existing visibility resolution chain rather than as a parallel system.

**Reveals mechanism**: Pre-action engine hook (`if.hook.before_action`) with an allowlist of silent actions (LOOK, EXAMINE, INVENTORY, WAIT, HIDE). Everything not on the allowlist breaks concealment before the action executes.

**Preposition delivery**: `withDefaultSemantics({ position })` per grammar pattern â€” same pattern used by the `going` action for direction semantics. Each grammar variant (HIDE BEHIND, HIDE UNDER, HIDE INSIDE, HIDE ON) carries its preposition through the parse without a separate slot.

**NPC detection**: Story-level capability dispatch override on `if.scope.visible` â€” stories register a behavior on the concealment target entity that checks NPC awareness. Platform does not bake in any NPC awareness rules.

ADR-148 includes: behavior statements, end-to-end test, 5 state transition tests, 4 negative tests, save/restore test, and 10 explicit acceptance criteria.

## Key Decisions

### 1. ADR-143 WITHDRAWN Rather Than Repaired

The review found the implementation violated the language layer separation principle (English strings in world-model) and was missing most of its designed functionality. Repairing it would have required a full re-implementation anyway. The decision was to withdraw cleanly and document the right path for a future re-implementation rather than carry dead/broken code.

### 2. `directions()` on ActionGrammarBuilder Retained

The `directions()` method on `ActionGrammarBuilder` (from ADR-087, predating ADR-143) was explicitly kept. It is the correct API for mapping compass directions to `if.action.going` in grammar. ADR-143's additions were layered on top of this and were the parts removed.

### 3. Pre-existing ADR-082 Vocabulary Infrastructure Left In Place

`DirectionVocabulary` in `if-domain` (part of ADR-082) was correctly identified as separate from ADR-143's additions and left untouched. The audit distinguished what ADR-143 added from what already existed.

### 4. Story-First for John's Use Case

Instead of rushing a platform re-implementation, the correct short-term answer is a story-level grammar extension. This is consistent with the project's architecture â€” stories extend grammar for story-specific command patterns without requiring platform changes.

### 5. ConcealedStateTrait as Dynamic Trait (ADR-148)

Presence/absence of the trait is the state. This is cleaner than a boolean flag on a persistent trait because: the type system enforces the state (you can't be "a little concealed"), the trait is self-describing when present (carries position and target), and removal is atomic and side-effect-free.

### 6. Visibility Hook Rather Than Parallel System (ADR-148)

Early drafts of ADR-148 proposed a parallel visibility resolution path for concealment. The third review round identified this as an anti-pattern: it would create two code paths for the same concept. The correct design integrates `ConcealedStateTrait` into the existing `VisibilityBehavior.canSee()` pipeline via the `if.scope.visible` capability, which the engine already queries.

### 7. Before-Action Hook at Engine Layer (ADR-148)

The reveals mechanism must fire before `action.validate()` so the action sees the correct unconcealed state. Placing the hook between `createActionContext()` and `action.validate()` in `command-executor.ts` is the minimal, correct injection point. Story-level event handlers would fire too late (after the action completes).

### 8. Parser Bug Fix â€” semantics Merged Into extras

`withDefaultSemantics({ position })` values were being silently dropped during grammar match conversion. The fix merges `match.semantics` into the `extras` field in `convertGrammarMatch`. This is a systemic fix: any future grammar pattern that uses `withDefaultSemantics()` will now correctly propagate its values to the action context without additional plumbing.

### 9. Concealment-Test as Dedicated Test Story

Rather than adding concealment tests to Dungeo (which would mix concerns and require Dungeo rooms to have concealment targets), a small dedicated story was created. This keeps regression isolation clean: concealment-test proves the feature, Dungeo proves the integration with a full game.
