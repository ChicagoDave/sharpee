### Root Cause Analysis
Identified two root causes shared across 37 golden test files, plus individual issues in 7 non-golden files.

**Root Cause 1 — Broken `executeAction` helper (3 variants, 2 broken)**:
- Variant B: Called `action.report(ctx, result)` for the failure path instead of `action.blocked(ctx, result)`
- Variant C: Hand-rolled a fake `action.error` event object instead of calling `blocked()` at all
- Fix: Replaced all local helpers with the shared `executeWithValidation` from test-utils

**Root Cause 2 — Wrong event type assertions**:
- All 37 golden files asserted on `action.success` and `action.error` event types
- Actual actions emit domain events: `if.event.taken`, `if.event.take_blocked`, `if.event.opened`, `if.event.open_blocked`, etc.
- Fix: Updated all assertions to match actual domain event types per the mapping in plan.md

## Key Decisions

### 1. Single shared `executeWithValidation` in test-utils
Rather than patching each broken helper in place, the decision was made to standardize all 37 golden files on the single `executeWithValidation` export from test-utils. This prevents the same drift from recurring as new golden tests are added.

### 2. Domain event types are the ground truth for test assertions
Stdlib actions never emit `action.success` or `action.error`. These were artifacts from an earlier design that the test files were never updated away from. Going forward, all action tests assert on the domain events defined in each action's `*-events.ts` file.

### 3. `combat-service.test.ts` deleted rather than fixed
The file was a stale duplicate — the canonical combat tests live in `packages/extensions/basic-combat/`. Fixing it in stdlib would have created two competing test suites for the same logic.

### 4. ScopeLevel is a numeric enum, not a string enum
`scope-integration.test.ts` was passing string literals for ScopeLevel values. The fix used the actual numeric constants. This reflects a compile-time-invisible API difference that only surfaced at runtime.
