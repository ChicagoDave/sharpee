## Key Decisions

### 1. Custom Event Types for Chain Handlers
`game.message` events returned from `world.chainEvent()` handlers are consumed as overrides by the event processor. Using them caused output to be swallowed. The fix: emit custom domain events (`zoo.event.goats_react`, `zoo.event.penny_pressed`) and handle them separately rather than returning `game.message` from the chain handler.

### 2. Use `itemId` Not `item` for Entity Matching in Drop Handlers
`DroppedEventData.item` contains the item NAME (display string), not the entity ID. Entity matching in drop event handlers must use `event.data.itemId` to correctly identify the dropped entity.

### 3. Capability Registry Allows One Behavior Per Trait+Capability Pair
`createCapabilityDispatchAction` is not exported from the `@sharpee/stdlib` public API. The workaround: write a manual dispatch action that calls `findTraitWithCapability()` and `getBehaviorForCapability()` directly, then dispatches the four-phase behavior inline. Multi-entity variation is handled inside the single behavior by reading `trait.animalKind` rather than registering multiple behaviors.

### 4. `BehaviorRegistrationOptions` Uses `validateBinding` Not `condition`
The registration options type field is `validateBinding`, not `condition`. This was discovered at compile time; the correct field name was confirmed from the type definitions.
