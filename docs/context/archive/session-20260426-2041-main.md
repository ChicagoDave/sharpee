## Key Decisions

### 1. SerializedWorldModel Distinct from ISaveData

ADR-162 Decision 5 (original) incorrectly conflated the wire snapshot shape
with `ISaveData`. After code inspection, `ISaveData` is confirmed to be a
strict superset. The corrected decision:
`type SerializedWorldModel = ReturnType<WorldModel['toJSON']>` is
a `string` holding only the world serialization, not the full save
envelope. The two share the same underlying serializer for the world subset,
but the wire frame for live snapshots carries only the world — not engine
state, event history, or scheduler state.

### 2. STATUS_REQUEST/STATUS for Welcome Staleness (ADR-162 Decision 6)

Rather than allowing the welcome to serve a stale or absent mirror, the
server always issues STATUS_REQUEST and awaits STATUS before completing the
handshake. Timeout (2000ms) results in a hard close (`4500`), not a
degraded welcome. Rationale: a partially-initialized client mirror creates
subtle renderer bugs that are harder to debug than a clean connection
failure.

### 3. Optional-then-Tighten Rollout for Wire Fields

Each newly-introduced required field starts optional in the type-only phase
and tightens to required in the same commit that introduces its producer.
This keeps every intermediate phase typecheck-clean without requiring empty
string stubs. The pattern applies to all three field introductions in
ADR-162 (`Output.world`, `Restored.world`, `RoomSnapshot.world`).

### 4. Test-Server Mirror Wrap for STATUS_REQUEST

When `realSandbox: false`, the test helper substitutes a `RoomManager`
whose `getWorldSnapshot`/`requestStatusSnapshot` return `'{}'` immediately.
Production code is untouched. Placeholder bundles used in unit tests cannot
satisfy STATUS_REQUEST and would cause all welcome-path unit tests to block
on the 2000ms timeout.
