### Phase 2: ISSUE-057 — Multi-Word Alias Resolution

**Root cause**: The command validator (`command-validator.ts`) searched entities by `ref.head` (last token only, e.g., "babies" from "bush babies") instead of `ref.text` (full phrase). The parser's slot consumer already built multi-word phrases correctly — the validator was discarding them.

**Fix**: Maximal munch algorithm in `resolveEntity()`. Try full `ref.text` first; fall back to `ref.head` only when no full-text match is found. A new `findCandidatesByTerm()` helper centralizes term lookup. Scoring tiers in `scoreEntities()` were extended to prefer full-text matches over head-only matches.

This approach is inspired by compiler tokenization (longest match wins) and preserves Sharpee's alias model without adopting an Inform 6 bag-of-words strategy.

Also fixed ISSUE-061 (multi-word names in story grammar) as the same root cause applied.

### Phase 2: NPC Inventory Scope Fix

The ISSUE-057 fix exposed a scope bug: the thief's stiletto (aliases: `['knife']`) was treated as REACHABLE by the scope resolver, causing disambiguation conflicts with the player's knife on commands like "attack thief with knife".

**Root cause**: `canReach()` in `scope-resolver.ts` defaulted to `return true` for any visible entity not in a blocking container. NPC-held items had no special handling — they were visible and reachable by default.

**Fix**: NPC inventory is now treated as a "closed transparent container" — items are visible but not reachable unless the holder has `OpenInventoryTrait`. The new trait is opt-in for cases where NPC inventory should be reachable (a horse with saddlebags the player can use, a dead NPC whose items are lootable, etc.).

This is a platform rule, not a story-specific workaround.

### 5. Maximal Munch Over Bag-of-Words
For multi-word alias resolution, a maximal munch approach (try longest match first) was chosen over Inform 6's bag-of-words model. This preserves Sharpee's alias model and avoids false positives from partial token matches.

### 6. NPC Inventory Defaults to Visible-Only
Rather than a story-specific workaround, NPC-held items are now platform-level VISIBLE but not REACHABLE by default. This is a sensible simulation rule: you can see a thief's knife but cannot grab it during a fight. `OpenInventoryTrait` provides opt-in reachability for scenarios where it makes sense (dead NPCs, merchant stalls, etc.).

### 7. OpenInventoryTrait for Opt-In Reachability
New trait added to world-model rather than an ad-hoc flag. Follows the existing trait pattern and is registered in `implementations.ts` for consistent discovery.
