2. Standard channel vocabulary moves out of `channel-service` into `stdlib` (mirrors the
   `lang-en-us` / `parser-en-us` pattern for language-layer concerns)
3. Channels are first-class `IOChannel` instances; construction is identical whether platform or
   author creates them — no "standard" vs "custom" class split

## Key Decisions

### 1. Closure-Per-Channel Replaces Rule-Based Routing

The rule-based model (`registerChannel` + predicate + extractor function triples) required stdlib
and engine to emit every piece of channel data as named block types that rules could match against.
Channels that aggregate data from multiple stdlib events (e.g., score + turns + location on a
single status-bar packet) had no natural home for the aggregation logic. The closure model puts the
aggregation inside the channel definition itself, using the `ChannelContext` passed to `build(ctx)`
as the data source. No cross-cutting emitter changes required.

### 2. Package Shape: if-domain / channel-service / stdlib / engine

The four-package shape (§14) aligns with the existing pattern: `lang-en-us` owns text, `parser-en-us`
owns grammar, `stdlib` owns action logic. Channel definitions are a language-layer concern and
belong in `stdlib`. Types are domain contracts and belong in `if-domain`. The runtime class
(`ChannelService`) belongs in its own thin package. Engine composes all three via constructor
injection.

### 3. Eight-Phase Master Plan (R1–R8)

The prior 4-phase plan assumed the rule-based architecture would survive Phase 4. It will not.
R1–R2 are infrastructure phases that must complete before any platform client work begins. This
serializes the early work but prevents a second mid-flight pivot.
