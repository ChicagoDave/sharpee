## Key Decisions

### 1. SavesRepository.create() requires save_id as input
The repository originally generated its own save_id (UUID). This would desync from the sandbox-issued correlation token the caller already holds. Changed the signature to require save_id as a parameter — the caller is the authoritative owner of that token. Caught during implementation, before any tests ran against the broken behavior.

### 2. Four role events on succession, not three
The plan specified three events (promote new PH, promote new co_host, nominate new successor). A fourth event — demote old PH — was added because without it the old PH's participant row retains tier=primary_host. A reconnecting old PH would receive a stale welcome snapshot showing them still as primary host. The explicit demote event keeps the DB row truth.

### 3. Auto-nomination placed in hello.ts, not presence.ts
The plan suggested presence.ts as the home for auto-nomination. Presence.ts handles disconnect-only lifecycle events; first-join logic belongs in the hello handler, which already owns the successful handshake path. No behavioral change — just the right file.

### 4. role_change.actor_id made nullable
Succession is system-initiated (triggered by a timer, not a user command). Carrying a non-null actor_id would be fabricated. The wire type was widened to `string | null`; clients that display "X promoted Y" must now handle null actor gracefully (display as a system message).

### 5. E2E teardown bug: server.close() hung
During Phase 6 E2E tests, `server.close()` hung because WebSocket clients were not explicitly closed before the afterEach hook ran. Fixed by adding a `track()` helper that registers each client; afterEach closes all tracked clients before closing the server. The pattern is reusable for Phase 8+ E2E tests.

### 6. Discipline slip on Phase 6 execute
Wrote `save()` and `restore()` in a single edit when only `save` was in the active sub-task. Caught and acknowledged. Did not recur in Phase 7.
