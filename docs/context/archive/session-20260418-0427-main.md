## Completed

### User Activities Flow — Create-Room

Walked the create-room flow end to end and closed all major decisions:

- **Story selection**: Stories preloaded on server by operator; no upload UI in MVP; room-creation form lists available stories as a dropdown.
- **Who can create a room**: Fully open — no auth required. Gated only by CAPTCHA.
- **Architectural consequence**: No auth system in MVP. Everything rides on join-code + durable session token. This was the largest single structural decision of the session.
- **Create-room form fields**: Story (required dropdown), Room title (optional with generated fallback), Display name (required), CAPTCHA.

### User Activities Flow — Joining

Walked the join flow and closed all decisions:

- **Join code handoff**: Both URL (`https://{instance}/r/XYZB-3F56`) and raw code displayed, each with its own copy button, always visible in the room header.
- **Default role on join**: Every joiner is a Participant — no per-room toggle, no config, no ceremony.
- **Participant capabilities**: Can see the story, see chat, send chat freely (no unmute ceremony). Cannot enter story commands. Promotion requests are informal — just ask in chat.

### User Activities Flow — Playing (Lock-on-Typing)

Walked the play flow and closed all lock-on-typing implementation details:

- **Lock trigger**: First keystroke (not focus; not first non-whitespace character).
- **Lock release**: On submission, on empty input (backspace to empty), on 60-second AFK timeout, or on Co-Host force-release.
- **Lock visibility / live preview**: Everyone in the room (all tiers) sees keystrokes in real time — rendered as `Alice is typing: > take swo▮`. This is the primary differentiator over screen-share + Discord.
- **Implementation notes**: Debounced keystroke broadcast, deltas only, not persisted (only submitted commands are logged).
- **Command attribution in story pane**: None. Story pane renders as classic single-player IF. Attribution lives in the session event log for out-of-band review. Cleanly separates presentation from audit.

### Role Terminology Rename

"Host" renamed to "Co-Host" throughout the document to disambiguate from Primary Host and to match videoconference-model intuition (Zoom/Meet use "Co-Host" for this tier).

### Promotion and Demotion Flow

- **Promotion**: Strict one-level-down. Primary Host can promote to Co-Host only. Co-Host can promote to Command Entrant only. Co-Host tier is Primary-Host-guarded.
- **Demotion**: Primary-Host-only at all tiers. Co-Hosts cannot reverse their own grants. Intentional friction — forces discussion, discourages Co-Host ping-pong, preserves Primary Host as ultimate arbiter.

### New MVP Requirement: Private Messages

The demotion rule (Co-Hosts can't demote) only works if Co-Hosts have a private channel to reach the Primary Host. This surfaced DMs as an MVP requirement:

- **DM scope**: Minimal axis only — Primary Host ↔ Co-Host. No Co-Host ↔ Co-Host DMs. No DMs involving Command Entrants or Participants.
- **DM persistence**: All DMs logged to the session event log (kind=dm); cascade-delete with room.
- **Recording-transparency notice**: All participants shown a clear notice on join — "This session is recorded. Everything in this room is logged: every command, chat message, DM, role change. Logs persist for the lifetime of the room. Be on good behavior — this includes Hosts and Co-Hosts. Your DMs are not exempt." Persistent REC indicator in room UI.
- **UI**: Channel tabs for Primary Host (Room + one tab per DM'd Co-Host) and Co-Hosts (Room + Primary Host tab only). No tabs for Command Entrants or Participants.

### Disconnect and Reconnect Flow

- **Seat-holding**: Held indefinitely for Participant, Command Entrant, and Co-Host (durable token promise holds forever). Primary Host is special: 5-minute grace period before auto-transfer.
- **Mandatory successor nomination**: Primary Host must designate a specific Co-Host as successor. The platform enforces this.
- **Auto-nomination**: First joiner after Primary Host is auto-designated successor (passive notice; PH can change any time).
- **On PH disappearance** (5-minute idle or session-resumed without PH present): designated successor becomes Primary Host.
- **Cascading refill**: When the designated Co-Host vacates, the earliest-joined still-present Participant is auto-elevated to Co-Host successor. Chain continues indefinitely.
- **Invariant**: As long as ≥2 people are in the room, it always has exactly 1 Primary Host and ≥1 Co-Host successor.
- **Original Primary Host on return**: Rejoins as Participant (fell out of the succession chain).

### Moderation: Mute-not-Kick

Decided the platform has no kick mechanism in MVP:

- **Mute effect**: Disables chat-send. Muted user keeps presence, keeps role, keeps gameplay ability, sees everything.
- **Muted UX**: "You've been muted" on their input; mute icon next to name in participant list (visible to all).
- **Mute authority**: Primary Host can mute anyone below. Co-Host can mute Command Entrant or Participant (not other Co-Hosts). Flat unmute authority (same tiers can unmute).
- **Persistence**: Mute survives disconnect/reconnect; clears on unmute or room recycle.
- **All mute actions logged.**
- **Rationale**: Avoids token/IP/ban rabbit holes entirely. Social visibility of the mute icon is part of the deterrent ("cruel and effective"). Room delete remains as the nuclear escape hatch.

## Key Decisions

### 1. No Auth System in MVP

Open room creation (CAPTCHA only) means no user accounts at the tester side. Everything rides on join-code + durable session token. This was the largest structural decision — it simplifies the platform substantially and defers auth complexity to post-MVP.

### 2. Demotion Is Primary-Host-Only

Co-Hosts cannot reverse their own grants. This forces discussion before demotion, which directly surfaced the need for private messages between Co-Host and Primary Host. The constraint and its consequence arrived together.

### 3. Cascading Succession Invariant

The room always maintains exactly 1 Primary Host and ≥1 Co-Host successor as long as ≥2 people are present. Succession is automatic (designated Co-Host → earliest Participant) with no admin intervention required. Original Primary Host rejoins as Participant on return.

### 4. Live-Preview Keystroke Sharing

Every room occupant (all tiers) sees keystrokes in real time during lock. This is the primary differentiator over the existing screen-share + Discord workaround. Implementation is debounced deltas, not persisted, not attributed in the story pane.

### 5. Mute Replaces Kick

No kick mechanism in MVP. Mute is visible to all (social deterrent), survives reconnect, and sidesteps the IP/token/ban complexity entirely. Room delete is the escape hatch for truly unmanageable situations.

### 6. Recording Transparency Notice

All participants see a clear recording disclosure on join. The notice explicitly covers DMs and applies equally to Hosts and Co-Hosts. Persistent REC indicator in UI throughout the session.

### 7. "Co-Host" Terminology

Replaces earlier "Host" label. Removes ambiguity between Primary Host and delegated hosts. Aligns with videoconference conventions (Zoom, Google Meet).
