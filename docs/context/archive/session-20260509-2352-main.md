## Key Decisions

### 1. Phase 7a handler ships now, ports during ADR-174 Phase 1
The audibility handler lives in text-service today. ADR-174 Phase 1 migrates it to prose-pipeline. No revert, no flag — the port is mechanical and the tests travel with it.

### 2. text-service complete removal replaces the 2026-05-03 R8 disposition
The prior disposition retained text-service block-production indefinitely as a pragmatic interim. User did not endorse that interim as design intent. ADR-174 supersedes R8: text-service is removed in three phases (prose-pipeline introduction → consumer migration → package deletion).

### 3. Decoration mechanism: bracket-form span+class, sharpee- namespace reserved
ADR-091's BBCode approach is superseded. The bracket syntax `[em:word]` generates `<span class="sharpee-em">word</span>`. Platform-defined types always get the `sharpee-` prefix. Author-defined classes pass through bare. This mirrors HTML+CSS — no new learning surface for web authors.

### 4. Prose pipeline lives inside engine, not as a new workspace package
Weighed against a separate `@sharpee/prose-pipeline` package. User chose engine-internal: prose production is engine-coupled, it avoids a new registration point in the 6-step new-package checklist, and the boundary is enforced by directory structure rather than package manifest.

### 5. IDecoration shape change and CORE_DECORATION_TYPES removal
The old `{ type, content }` shape exposed decoration type as a string key that could be any arbitrary value. The new `{ className, content }` shape makes the CSS intent explicit and removes a layer of indirection. The `CORE_DECORATION_TYPES` constants object is removed; callers use the string literals from the locked vocabulary.
