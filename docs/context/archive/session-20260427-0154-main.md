### 3. Save Button Did Nothing — saveService Never Wired

User reported: typing `save` in-game did nothing; diskette button modal Save click also did nothing.

**Root cause:** `tools/server/src/saves/save-service.ts` exports `createSaveService(...)`. All required deps are constructed in `index.ts`, but `createSaveService` was never imported and never called. The WS server's `deps.saveService` was `undefined`. When the client sent `{kind:'save'}`, the handler returned `{kind:'error', code:'not_ready', detail:'save service not wired'}`. Zero client-side references to `not_ready` meant the error was silently dropped.

**Fix:** Three edits to `tools/server/src/index.ts`: import `createSaveService`, instantiate it after `roomManager`, pass it to `createWsServer`. Container rebuilt; healthy.

### 4. Latent Bug Research: In-Game Save Bridge

Typing `save` in the REPL still does nothing after wiring saveService. Traced the full pipeline:

`engine.executeTurn('save')` → stdlib saving action emits `platform.save_requested` → engine's `PlatformOperationHandler` calls `onSaveRequested` → hook in `deno-entry.ts:126` captures save data into `capturedSaveData` → COMMAND handler returns OUTPUT and **never forwards `capturedSaveData`**. The wire protocol has no frame for "engine wants to save."

**Recommended bridge (Design B, not implemented this session):** Add `AUTO_SAVE_REQUESTED` wire frame from sandbox → server with `{turn_id, blob_b64}`. Server adds `SaveService.persistCaptured(...)` that skips the IPC round-trip. Estimated ~80 LOC plus tests. Blocked on two user decisions: tier gate (CE+ for in-game save?) and save-name source (parsed command extras vs auto-name).

### 5. Latent Bug Research: Restore Always Picks Most-Recent Save

User reported: selecting an older save in the UI always restores the most recent. Pipeline traced end-to-end as correct (SaveList passes the right `save_id`, client forwards verbatim, WS handler invokes `saveService.restore`, SaveService loads the right blob, sandbox decodes and stashes in `pendingRestoreData`). Three hypotheses to discriminate: (A) `SaveRestoreService.load()` doesn't fully replace the in-memory WorldModel, (B) all save blobs are byte-identical, (C) a follow-up clobbers the loaded state. Needs a SQL diagnostic on the saves table to discriminate — not resolved this session.

### 6. Latent Gap Research: Status Line Never Implemented

User reported: status line (room name / score / moves) was never implemented in the multiuser client.

Investigated Zifmia's status line (`packages/zifmia/src/components/status/StatusLine.tsx`): it queries the world model directly via `extractCurrentRoom(world, roomId)` and capability reads. The Zifmia idiom is renderer holds a reference to the live `WorldModel`.

**The multiuser stack is the only Sharpee surface that breaks this contract.** Engine + world model live behind the Deno sandbox boundary; the server is a wire bridge; the client is two hops away. Status line is the surface that revealed the gap. The same problem will hurt every future renderer feature (map, inventory, scope-aware UI, disambiguation, etc.).

### 7. Architectural Insight: World Model as Wire Payload

Two clarifying observations surfaced:

1. The world model **is** the data store. It has `toJSON()`/`fromJSON()`. The engine's `ISaveData` blob — already round-tripping for save/restore — is the world model serialized. Not trapped; just JSON.
2. Renderers need **read access only**. The sandbox is the sole writer. Server and client mirrors only need queries. No coordination problem, no eventual consistency.

Architecture diagram written to `/home/dave/public_html/sharpee-world-replication.html` (three-layer: sandbox authoritative, server read-only mirror, client read-only mirror; vertical arrows: COMMAND down, OUTPUT carrying `text_blocks + events + world` up).

### 8. ADR-162 Written

`docs/architecture/adrs/adr-162-world-model-replication.md` — Status: PROPOSAL.

Five sub-decisions:
1. Authoritative writer unchanged (sandbox keeps full ADR-153 isolation).

5. Snapshot shape = `ISaveData` shape (one serializer, one hydrator).

## Key Decisions

### 1. Single Build Entry Point in Dockerfile

`pnpm run build` (npm script) rather than `pnpm exec tsc` directly, so asset-copy steps defined in the script are not bypassed by Docker. Consistent with the Dockerfile's existing precedent for manual asset copies.

### 2. dist-esm Output for @sharpee/media (Platform Edit)

`@sharpee/media` needed `dist-esm/` to survive the `.dockerignore` `**/dist` pruning that all other packages already handle. The edit to `packages/media` was approved by the user as the correct fix (not a workaround), given that all other sharpee packages already have this output.

### 3. World Model Replication Rather Than Field-by-Field Projection (ADR-162)

Rather than projecting individual fields over the wire (e.g., a `statusLine` API call), the snapshot of the full serialized world model rides the existing `OUTPUT` frame. Server and client hydrate `WorldModel.fromJSON`. Renderers query their local mirror using the same `WorldModel` read API that Zifmia already uses. This eliminates the impedance mismatch between the sandbox-side world model and all future renderer features.

### 4. ReadOnlyWorldModel Lives in tools/server Initially

No platform edit on speculation. The `ReadOnlyWorldModel` type starts in `tools/server/src/wire/world-mirror.ts`. Promotion to `@sharpee/world-model` deferred until a second consumer surfaces in a different package.
