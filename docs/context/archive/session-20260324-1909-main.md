## Completed

### Bug Fix: V16 Petting Action Score Not Awarded
- Root cause: capability dispatch routes `execute` to the behavior, not the action. The `awardScore()` call for petting (5 pts) was in the action's `execute` phase, which is never called when capability dispatch is active.
- The behavior's `execute` was a no-op, so the petting mechanics worked (correct messages appeared) but no score was ever awarded.
- Fix: moved `awardScore()` from the action's `execute` into `PettableBehavior.execute` in `tutorials/familyzoo/src/v16.ts`.
- Dead code in the action's `execute` phase was removed.

### Bug Fix: False Positive Transcript Assertions in V16
- The V16 transcript used `[OK: contains "75"]` to verify a perfect score. The score output format is "You have scored X out of a possible 75", so "75" appears in the string regardless of the actual score — making the assertion always pass.
- Similarly, `[OK: contains "0"]` was fragile as the initial score check.
- Fix: changed assertions to `[OK: contains "scored 0"]` and `[OK: contains "perfect score"]` to actually verify score values.

## Key Decisions

### 1. Capability Dispatch Routing Clarification
When an action uses capability dispatch (trait declares capability, behavior registered via `registerCapabilityBehavior`), the engine routes `execute` and `report` to the behavior — not to the action's `execute`/`report` phases. Any side effects (including `awardScore()`) that must happen on successful execution must live in the behavior's `execute`, not the action's. This is now documented by the corrected code as a teaching example in V14 and V16.

### 2. Transcript Assertion Specificity
Assertions like `[OK: contains "75"]` are fragile when the checked string also appears as a label (e.g., "out of a possible 75"). Always assert on the full distinctive phrase (e.g., "scored 75" or "perfect score") rather than a substring that appears in multiple contexts.
