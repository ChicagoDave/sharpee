## Key Decisions

### 1. `closeIdentitySockets` operates on a participant_id snapshot
The close handler races the cleanup loop to mutate the `byParticipant` and `byIdentity` indexes. The implementation resolves sockets from the identity index snapshot first, then iterates `ws.close()` calls. Each individual socket's `close` event handler completes the per-socket index cleanup.

### 2. Erase order: verify â†’ close sockets â†’ delete row
Closing before deletion ensures that any socket close handler's downstream DB reads (presence broadcast, session-event append) see the identity row still present. Deleting first would create a window where those reads fail on a missing FK.

### 3. ON DELETE CASCADE on participants.identity_id
Chosen over manual cascade inside the erase route, to match the existing CASCADE pattern on `rooms â†’ participants / session_events / saves`. This preserves the architectural invariant that rooms are the deletion root. Rooms with no surviving participants are handled by the PH-grace eventual-close recycler that was already in place.

### 4. Single rate-limit bucket across all three identity routes
One limiter instance instantiated once in `app.ts`, shared across `POST /api/identities`, `POST /api/identities/upload`, and `POST /api/identities/erase`. Three separate limiters would give an attacker 30 requests/minute instead of 10.

### 5. AppDeps.connections required, not optional
The pre-existing optional field meant `index.ts` passed it but `test-server.ts` did not. Both callers are now uniform; the type system enforces it.
