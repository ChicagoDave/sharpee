## Key Decisions

### 1. Phase 3 Design Pivot — Smell Catch Before Implementation

The original Phase 3 framing was wrong: it proposed lifting per-browser UI state into the shared-state projection layer. The correct split keeps `dmReadCursors` in `RoomState` (it is derived from server messages) and `activeTabId`/`openedDmPeers` in component-local state (they are per-browser UI facts).

This was caught before a single line of wrong code was written. The catch led directly to the boundary-check gate — the smell was the signal, the gate is the remediation.

### 2. Dual-Switch Exhaustiveness Pattern

Rather than merging `ServerMsg` and `UiAction` into one switch, the reducer uses two independent switches. The ServerMsg branch retains its existing `default: never` exhaustiveness guard unchanged. The UiAction branch has its own `satisfies never`. Neither guard is weakened by the widening of the input type.

This pattern means "the wire protocol is fully handled" remains a compile-time check. If a new ServerMsg variant ships, TypeScript still forces a case. If a new `ui:*` action is added, TypeScript forces a case there too.

### 3. markDmRead Instead of Raw dispatch Exposure

The `useWebSocket` hook exposes a narrowly-named `markDmRead(peerId, upToEventId)` method rather than passing the raw `dispatch` function to consumers. This enforces the boundary discipline at the API surface: components cannot accidentally dispatch arbitrary reducer actions.

### 4. Server Snapshot is Now Viewer-Aware

`room-snapshot.ts` accepts a `SnapshotViewer` parameter and filters `dm_threads` by tier. Participants receive `{}` regardless of whether DM rows reference them. This is defense in depth — the dm-handler axis check on the receive side is the primary control; the snapshot filter is the secondary control. Both layers enforce the same rule independently.

### 5. Seed Cursors to Max on Welcome

When the client receives a welcome with `dm_threads`, cursors jump to each thread's max `event_id`. This means every rehydrated message reads as already-seen and tab badges start at 0 after a reload. The alternative (showing all backlog as unread) was explicitly rejected in the acceptance criteria — a reconnect is not a notification event.

### 6. Boundary-Check Gate as DEVARCH-CANDIDATE

The hook and CLAUDE.md rule are tagged `[DEVARCH-CANDIDATE]` to flag them as candidates for promotion into the DevArch methodology. They are functional in this repo today but not yet abstracted into the DevArch tooling layer.
