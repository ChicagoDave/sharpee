## Completed

### GDT MessageId Fix
Root cause: `tryProcessDomainEventMessage()` in text-service.ts intercepts events carrying a `messageId` before they reach `handleGenericEvent()`. GDT events carried pre-rendered text in a `message` field, but the text service only looked up the `messageId` (which was not registered) and emitted an error placeholder.

Fix: Updated `gdt-action.ts` and `gdt-command-action.ts` to use `GDTEventTypes` constants as messageIds with a `params` object carrying the text, aligning with the convention used throughout stdlib actions.

### ADR-137: Input Modes
Designed and wrote the ADR covering the full InputMode concept — alternate input modes where the entire input→processing→output pipeline behaves differently (different parser, prompt, turn semantics). Examples: GDT, conversation mode, combat mode.

Key design decisions captured in the ADR:
- `InputMode` as a first-class concept with `id`, `prompt`, `advancesTurn`, `handleInput`
- `GamePrompt` type as a property of the mode, delivered as a `PROMPT` text block (FyreVM channel primitive)
- Mode stack (push/pop) for nesting
- Engine IS the standard mode — no extraction needed at this stage
- Immediate scope: GDT only. Full interface deferred until second use case.

### Text Service Fallback for Inline Text
Fixed `tryProcessDomainEventMessage()` to fall back to inline `message`/`text` fields when a messageId does not resolve or resolves to an empty placeholder. This fixes both GDT output and melee combat output — melee registers empty placeholder messages and provides actual text via `game.message` events.

### Melee Combat Double-Prefix Fix
The attacking action was prepending `if.action.attacking.` to any messageId provided by the melee interceptor via `customMessage`. When the value was already a fully-qualified story key (`dungeo.melee.hero_attack`), this produced a broken lookup key.

Fix: Keys containing a dot are treated as fully-qualified and pass through unchanged. Keys without dots get the action-namespace prefix. Same convention as capability dispatch.

### Transcript Test Updates

## Key Decisions

### 1. GamePrompt is a property of InputMode, not standalone
The prompt changes when the mode changes. Coupling them avoids the need to keep two pieces of state in sync and makes the relationship explicit in the ADR and code.

### 2. Engine IS the standard mode
Rather than extracting the standard pipeline behind an InputMode interface immediately, the engine routes to registered `InputModeHandler` instances only when an alternate mode is active. This keeps the change minimal and defers the larger refactor until a second use case justifies it. Documented as known debt in ADR-137.

### 3. Text service falls back to inline text
`tryProcessDomainEventMessage()` now tries inline `message`/`text` fields when the messageId lookup fails or returns empty. This handles GDT and melee combat, and is the correct general behavior: events intended for programmatic consumption carry a messageId but no inline text; events carrying pre-rendered inline text should surface that text rather than silently dropping output.

### 4. Dot-in-key means fully qualified (attacking action)
Keys with dots pass through unchanged in the attacking action's messageId construction. Same convention as capability dispatch. Avoids double-prefixing from melee interceptor-provided story keys.
