### #5 Ch.20 "NPCs never act" — DIAGNOSED + FIXED  ← main work
**Method (per David): build outside the repo with the published npm packages and run it.**
Built Ch.20 standalone against `@sharpee/*@1.0.8`; drove turns headlessly (engine.setStory
→ engine.start → executeTurn, subscribing to `text:output`/`event`).

Findings:
- NPCs are wired correctly and DO act: the NpcPlugin registers, **Sam patrols** (moves
  Main Path → Petting Zoo → Main Path), the parrot's behavior fires (squawk events appear
  in the raw event stream). So NOT book wiring, NOT a plugin-tick bug.
- The narration never RENDERS — squawk present in raw `event`, absent from `text:output`.
- First (wrong) theory: prose-pipeline drops NPC events → filed #149 as a pipeline gap.
- **David's correction:** Dungeo has several NPCs and its regression is clean → the
  pipeline renders NPC events fine. Re-investigated.
- **Real root cause:** `@sharpee/lang-en-us` registers `npc.speaks`/`npc.shouts`/movement/
  observation/specific-emote ids, but NOT the generic `npc.speech`/`npc.emote` that
  `NpcBehavior` `speak`/`emote` emit when a custom behavior passes arbitrary `data.text`
  (what the Family Zoo `parrotBehavior` does). Dungeo renders because it uses
  story-specific registered ids (`RobotMessages`/`TrollMessages`). The tutorial's generic
  ids resolved to nothing → silent NPCs.
- **Proof:** unregistered → 0 squawks rendered over 25 turns; registered as `"{text}"` →
  11 rendered.
