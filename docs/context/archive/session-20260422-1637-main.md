### Behavior Statements

**`CreateRoomModal`**:
- **DOES**: on submit, calls `createRoomFn({story_slug, title, display_name, captcha_token})`; on success writes `sharpee.token.<room_id> = token` in localStorage and invokes `onCreated(response)`. On `ApiError` with known codes (`missing_field`/`invalid_title`/`unknown_story`) surfaces an inline field-level error; on unknown codes or generic `Error`, surfaces a form-level alert.
- **WHEN**: form Submit button pressed.
- **BECAUSE**: the modal is the Primary Host entry point; token persistence is a Decision-5 requirement.
- **REJECTS WHEN**: client-side: empty/whitespace title, empty display name, empty story slug. Server-side: any non-2xx response.

**`buildClientConfigScript` (server)**:
- **DOES**: returns a `<script>` tag that assigns `window.__SHARPEE_CONFIG__ = {captcha: {provider, siteKey}}` with JSON-escaped `<` to block `</script>` break-outs.
- **WHEN**: server builds the Hono app.
- **BECAUSE**: CAPTCHA provider + public site key need to reach the browser before the SPA boots so the widget can render the right challenge.
- **REJECTS WHEN**: never — function always returns a string. Secrets are never interpolated by construction.

### Deviation — jsdom → happy-dom (with polyfill)

Node 25 now ships an experimental built-in `localStorage` (requires `--localstorage-file=...` to function). This silently shadowed both jsdom's and happy-dom's Storage polyfills in Vitest, producing `window.localStorage.clear is not a function`. Fix: swapped test env to `happy-dom` and installed an in-memory `Storage` instance in `tests/setup.ts` that satisfies the Storage interface. The `swallows storage exceptions` token test spies on `window.localStorage` directly (not `Storage.prototype`) since the polyfill doesn't live on the built-in Storage class.
