## Key Decisions

### 1. Apache instead of Caddy
The host's existing apache2 configuration made Caddy impractical. All required proxy and WebSocket modules were already loaded, so the switch was low-friction. The vhost pattern (ProxyPass + ProxyPassReverse + upgrade headers) is standard apache2 reverse proxy configuration.

### 2. Loopback bind before firewall hardening
Rather than enabling ufw first (which would have required careful rule ordering given Docker's iptables manipulation), the safer immediate fix was binding the container to `127.0.0.1`. This eliminates the Docker-bypasses-ufw attack surface without touching iptables or ufw state. ufw hardening can proceed independently.

### 3. X-Forwarded-Proto header flaw noted but not blocking
Certbot's auto-generated SSL vhost inherits the `RequestHeader set X-Forwarded-Proto "http"` line from the HTTP vhost. This is incorrect for a TLS terminator â€” it should be `"https"`. However, Sharpee does not currently read this header, so the misconfiguration is harmless and was left for a follow-up fix rather than holding up deployment.
