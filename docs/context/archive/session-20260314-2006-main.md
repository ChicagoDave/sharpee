### Research — Community Thread Analysis
Fetched and analyzed five intfiction.org threads to map the actual pain landscape:
- "Why is writing IF still drawing diagrams instead of writing?" — prose-first vs. diagram-first debate; community split on whether the problem is tools or discipline

- "Tools for writing and narrative design" — gap analysis; existing tools assume you already know IF conventions
- Competitive landscape — LLM+IF projects; nothing currently attempts silent prose-to-world inference

### Target Persona Definition
Based on Jeff Nyman's post about working with published novelists, screenwriters, and J.J. Abrams during a formal IF workshop. Key finding: experienced writers found IF tools to be "less like a medium and more like a cage." Persona is a professional prose author, not a programmer or hobbyist game designer — someone whose natural unit of work is a scene, not a room object.

### Design Principles (8 established)
1. Scene-first, world-later — no upfront structural commitment
2. Verification through play, not code review — the test is whether it runs, not whether the diagram is right
3. Progressive disclosure of mechanics — IF vocabulary surfaces only when the author writes something that implies it
4. Speed-to-playable under 60 seconds — from prose to running game without any setup steps
5. Brainstorm-to-playable without tool switching — the writing surface IS the authoring surface
6. Mystery Box compatible — partial, ambiguous, or unresolved worlds are legal; the tool doesn't demand completeness
7. Silent observation, seam-based inference — the tool watches what the author writes, infers structure, never interrupts
8. Prose is the source of truth — code is a derived artifact, not the canonical representation

### Key Design Decisions Resolved
- **Scope**: Parser IF only (not choice-based); Sharpee is the target engine
- **Audience**: Solo authors, not teams; no multi-user collaboration in v1
- **Format**: Prose as source of truth; Sharpee code is generated, not hand-edited
- **Versioning**: Passive and automatic; no explicit save/commit workflow for the author
- **Input format**: Inferred from prose; no template, no form, no required syntax

### Hostile Critique Written
`docs/madness.md` — "The Case Against Lantern" — 10 reasons the design cannot work:
1. Prose is ambiguous; inference will be wrong constantly
2. Authors won't trust inferred structure
3. 60-second playable is a lie for non-trivial scenes
4. Sharpee code generation requires full semantic understanding
5. NLP for IF is an unsolved research problem
6. Authors will start editing the generated code and break the loop
7. Mystery Box compatibility conflicts with engine completeness requirements
8. Silent observation fails when the author needs to ask a question
9. No authoring tool has succeeded by hiding its model from the user
10. This is a research project disguised as a product

Conclusion of the critique: none of the objections are impossible, only hard. The 4-step R&D sequence was identified from the critique.

### R&D Sequence Identified (from hostile critique)
1. Problem 1: Single-scene inference — can Lantern correctly parse one scene's entities, exits, and objects from prose alone?
2. Problem 2: Trust through play — does generated output match author intent well enough that the author continues rather than abandoning?
3. Problem 3: Cross-scene consistency — can inferred structure remain coherent across 10+ scenes?
4. Problem 4: Seam detection — can Lantern identify the boundaries between scenes in longer prose passages?

### Guardrails Established
Lantern's core constraint: **Lantern never writes prose, dialogue, puzzle designs, or descriptions.** Every word the player reads comes from the author. The LLM's role is structural inference and code generation only. This is a hard product boundary, not a soft preference.

### Naming Decision
"Lantern" chosen over alternatives. Rationale:
- Deep IF heritage: brass lantern is the canonical IF object
- Metaphor of illumination — the tool sheds light on structure the author already wrote, it doesn't generate the light
- Functional pair with Sharpee: engine + lantern
- Simple, memorable, not jargon

## Key Decisions

### 1. Prose as source of truth, code as artifact
The central architectural inversion from all prior IF authoring tools. No existing tool (Twine, Inform, Ink, Quest) treats prose as primary — all require the author to work in a tool-specific notation. Lantern's entire claim to novelty rests on this inversion. Consequences: code generation must be reliable enough that the author never needs to edit it; inference failures surface as playtest failures, not as error dialogs.

### 2. This replaces the template-based authoring concept
A prior design direction envisioned a template system — structured forms the author fills in. The session concluded that templates are still a cage: "fill in this box" is structurally the same problem as "drag this room onto the map." Lantern's design says: there is no template, there is no form. You write. The tool figures out what you wrote.

### 3. Silent observation over conversational assistance
The tool does not ask questions during writing. It watches. Questions break creative flow; inference cards in a side panel don't. The author can dismiss, correct, or ignore inference — but is never interrupted. This is a deliberate UX constraint, not a technical default.

### 4. Problem 1 is the validity gate
Before any UX or product work proceeds, single-scene inference must be demonstrated to work at an acceptable accuracy threshold (TBD, but roughly "author accepts inferred structure more often than they correct it"). If Problem 1 fails, the product concept fails. All other work is downstream of this result.

## Challenges Encountered

### Community skepticism about LLM involvement
Multiple intfiction threads reflect deep suspicion of AI in the authoring workflow, primarily around loss of authorial control and homogenization of output. The guardrail (Lantern never writes prose) directly addresses this, but community trust will require demonstrated artifacts, not argument.

### Ambiguity about what "inference" means in practice
The research phase clarified that "parse a scene" is actually a cluster of distinct problems: entity recognition, exit detection, implied containment, tense and POV inference, disambiguation of metaphor vs. literal description. Problem 1 needs a precise test harness before any inference code is written.

### Scope discipline
The design process repeatedly pulled toward features (collaborative mode, multi-author, Twine export, choice-based support). Each was explicitly deferred. The session ended with a narrower scope than it started with, which is the correct direction.
