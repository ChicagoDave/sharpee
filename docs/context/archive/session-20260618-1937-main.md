### Root-caused and fixed `WorkspaceRoot.find()` infinite loop
`URL.deletingLastPathComponent()` on a trailing-slash directory URL at filesystem root produces a growing `/../` path instead of converging, so the `parent.path == current.path` guard never fired. The loop allocated ever-longer strings until macOS swapped to disk. This was also a production hang: `AppDelegate` calls `WorkspaceRoot.find` on every project open, so any folder outside a Sharpee workspace would hang the shipping IDE.

### Migrated workspace marker to devkit monorepo signature
Replaced the `build.sh`-presence check (deleted in ADR-180 cutover) with detection of both `pnpm-workspace.yaml` AND `packages/core/` in the same directory, mirroring `devkit/src/repo.ts findMonorepoRoot`. Added explicit `current.path == "/"` termination. Rewrote `WorkspaceRoot.swift` and `WorkspaceRootTests.swift` (8 cases: walk-to-root, two-marker AND condition, regression for the infinite-loop bug). Updated stale `currentRepoRoot` doc comment in `AppDelegate.swift`.

## Key Decisions

### 1. Workspace marker = devkit monorepo signature
Chose `pnpm-workspace.yaml + packages/core/` (both required) rather than inventing a new `.sharpee-workspace` marker, keeping the IDE's detection aligned with `devkit/src/repo.ts` — a single source of truth for "is this a Sharpee monorepo?"

### 2. Theme removed from IDE Build Settings
ADR-180 has no theme build flag; theme is a runtime/browser-client concern (ThemeManager / `data-theme` attribute, or `theme.css` in standalone projects). No `--theme` argument on `./sharpee build`.

### 3. IDE spawns `./sharpee` CLI, not linked libraries
BuildRunner will spawn the `./sharpee` wrapper as a subprocess rather than linking `@sharpee/devkit` or `@sharpee/bootstrap` into the Swift app. Keeps the IDE decoupled from JS package internals.
