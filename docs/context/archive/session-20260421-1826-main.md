## Key Decisions

### 1. LockManager + AfkTimer constructed inside createWsServer (default), overridable via WsDeps
Mirrors the `ConnectionManager` pattern already in place. Tests override via `WsDeps` to inject real instances they can inspect; production code uses the defaults. Keeps wiring self-contained and avoids singleton state.

### 2. Socket-close lock release (unplanned but required for correctness)
A holder whose WebSocket drops silently would freeze the room until the AFK timer fires — potentially minutes. Releasing on close brings the guaranteed unlock latency down to zero and makes the AFK timer a last-resort fallback rather than the primary recovery path. Documented in the `server.ts` close handler.

### 3. Phase 4 E2E tests updated to reflect Phase 5 contract
The turn-execution tests now acquire the lock before submitting a command, matching the stricter Phase 5 contract. Preserving the Phase 4 paths would have required either mocking the lock guard or special-casing the no-lock path; updating the tests is cleaner and more representative of real usage.

### 4. No race-case unit test for force-release
Simulating "holder releases between `getState()` check and `forceRelease()` call" requires mocking `LockManager`, which tests the mock rather than real behavior. The race-handling code is a two-line guard and is covered by inspection. All other authority and state paths are tested directly.
