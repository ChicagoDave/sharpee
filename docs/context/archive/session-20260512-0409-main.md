## Key Decisions

### 1. The zifmia name is reserved
`tools/zifmia/` will be reclaimed for the corrected ADR-177 build when implementation begins. `tools/shite/` is the parts-bin. The name boundary is intentional so future git blame is unambiguous.

### 2. Submitter is just another WS subscriber
`POST /command` returns `{turnId}` acknowledgment only. The `TurnPacket` (channel-io per ADR-163) is broadcast over WS to every browser in the room including the submitter. No HTTP polling for engine output; no asymmetric response-as-output pattern. Symmetric broadcast.

### 3. Transport split (HTTP carries process; WS carries chat + turn broadcast)
HTTP: identity claim/erase, room CRUD + governance (create/join/rename/pin/delete/promote/demote/mute/force-release/nominate-successor), saves, stories, `POST /command` (ack only), `GET /state` (hydration on entry/reconnect: `cmgt + transcript_backlog + roster + recording_notice`). WebSocket: `chat:send/chat:message`, `lock:acquire/release/state`, `turn` frame (full `TurnPacket`), `role_change`, `presence`, `room_restored`.

### 4. Identity is handle-only (ADR-161 amended)
No passcode, no Bearer token. Threat model: small IF community. Anyone typing a handle claims it.

### 5. No `/admin/*` HTTP surface
`is_admin` is a database column set via CLI. The admin upload endpoint designed in Phase 5/6 is removed. Stories come from directory scan.
