## Key Decisions

### 1. Popover rather than stacked Modal for IdentityPickerButton

IdentityPickerButton wraps IdentityPanel in a popover (click-outside + Esc dismissal) rather than a Modal. This keeps EraseIdentityModal as the only stacking Modal layer when erase is triggered. Avoids z-index arithmetic and the double-Esc behavior that would occur if a parent Modal and a child Modal both captured the key.

### 2. No explicit onErased callback — storage event drives teardown

On a successful erase, the modal calls `clearStoredIdentity()`. The App's existing `subscribeToIdentityChanges` listener fires, identity state goes to null, IdentityPanel and the modal unmount naturally. One source of truth (the identity store), no prop-drilling a callback through IdentityPickerButton → IdentityPanel → EraseIdentityModal.

### 3. localStorage retained on server erase error

When the server returns `bad_passcode` or `unknown_handle` during an erase attempt, the modal surfaces a reload alert but does not clear localStorage. A stuck or mismatched identity may still be useful for diagnosis or upload-as-claim to another instance. Clearing it silently would make the error harder to investigate.

### 4. Wire-type extraction was a Phase F prerequisite

Adding `participants: RoomParticipantSummary[]` to a duplicated `RoomSummary` type would have doubled the drift surface in the same session that was supposed to reduce it. Per memory rule on co-located wire-type sharing, the shared `wire/http-api.ts` file was extracted first and the F2 shape change was made once, in one place.

### 5. Confirmation gate is case-sensitive

The handle typed into the EraseIdentityModal must match the handle exactly as displayed (the canonical-case string stored in the identity). Server-side uniqueness is case-folded, but the confirmation gate matches what is shown in the UI. Consistent with the GitHub "type the repository name" pattern.

### 6. json_group_array for room-list query

A single SQL round-trip returns participants as a JSON array string; parsing happens in JS with a deterministic alphabetical sort. Alternative (N+1 queries per room) rejected for obvious performance reasons. Alternative (separate participants batch query) would require a join-and-group in JS anyway; the aggregation is cleaner in SQL.
