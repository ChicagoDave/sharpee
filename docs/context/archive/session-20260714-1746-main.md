### ADR-214 interview + review COMPLETE (still DRAFT, pending accept)
- Ran `/devarch:adr-interview` — all **6 open questions resolved** and folded in:
  - OQ1: capability-dispatch verbs → `on <verb> it` on the entity is the seam
    (dispatch action falls back to the entity's on-clause). No new construct.
  - OQ2: extensions → top-level `use <extension>` enables a shipped extension +
    admits its published vocabulary into the catalog only when used. Combat
    reachable via `use combat`. Depth details → ADR-215.
  - OQ3: browser emits → give `emit` a data payload (full channel access
    generically) + typed sugar (`play sound`/`play music`/`show image`). ADR-216.
  - OQ4: timer controls → **full** scheduler parity, every control (cancel,
    inline `in N turns`, `on every N turns`, data-driven delays, reschedule,
    pause/resume, entity-bound cleanup, priority). ADR-217.
  - OQ6: pure-IR → trusted platform extensions ship with the runtime; `use`
    keeps a story pure-IR + browser-runnable (refines ADR-210 AC-4).
  - OQ5: order → foundations first (catalog adjectives → door loading → dispatch
    fallback → timer controls → ADR-215 extensions/combat → ADR-216 emit/media).
- **Accepted boundary folded in** (David): parity is full *access*, not full
  *implementation* — big/complex extensions built in Sharpee+TS and *exposed* to
  Chord via `use`; a gap = reachable neither as core Chord nor a platform
  extension. Saved to memory ([[sharpee-chord-parity-goal]]), incl. the "never
  scope down / until-needed framing rejected" correction.
- Ran `/devarch:adr-review` — READY (umbrella ADR); applied all fixes: stale

  AC-3 additive-only, AC-4 governance), explicit ADR-210 AC-4 refinement note,
  two merged-paragraph nits.
- **ADR-214 is DRAFT, pending David's accept** (has no Open Questions section;
  ready to flip DRAFT → ACCEPTED on his word).
