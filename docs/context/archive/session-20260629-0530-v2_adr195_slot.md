## Key Decisions

### 1. Room descriptions migrate to the phrase pipeline

The legacy `room.ts`/`getMessage` path was the architectural anomaly — ADR-192's direction is that all prose flows through `renderViaPhrase`. Migrating the description body (not the full room-name block) onto the phrase pipeline was the correct fix, not adding a slot-blind workaround in the legacy handler.

### 2. Detail clauses use an ADR-193-style state-clause registry, not ADR-090 capability dispatch

ADR-090's `CapabilityBehavior` is a four-phase mutation interface. Prose injection at read time has no mutation phase. The `ClauseContributor` shape in `state-clauses.ts` mirrors ADR-193's `StateAdjective` hook exactly — a pure `(entity) => Phrase[]` read — and is the correct interface for this use.

### 3. `detailWhenOn`/`detailWhenLit` are additive fields, not replacements for `onDescription`/`litDescription`

The `description` getter on `IFEntity` swaps to `onDescription` / `litDescription` when the trait is active. Registering state clauses under those fields would have caused the clause text to be emitted *twice* (once from the getter, once from the slot). The dedicated `detailWhen*` fields are ignored by the getter and consumed only by the clause registry, preserving the base description and appending detail without duplication.

### 4. `params.__slots__` staging seam for target-aware slot contribution from actions

Actions know their target at report time; the global `SlotContributor` runs with empty params and cannot know the target. Staging a `__slots__` map in `params` lets the action pre-populate the turn slot store immediately before `renderViaPhrase` is called, without requiring a new API surface or routing the examine target through the global contributor.
