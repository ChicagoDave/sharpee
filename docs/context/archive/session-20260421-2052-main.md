## Key Decisions

### 1. N-6 Redesigned: Boot-time Validation Instead of Spawn-at-Create
The plan specified spawning a sandbox at `create-room` time to validate the story slug. Rejected due to async atomicity complexity, cold-start latency, and DoS surface. `StoryHealth` validates all stories at boot and caches results. Create-room does a cheap sync lookup — instant rejection, no orphan-sandbox cleanup branch.

### 2. `session_events.append` Signature Kept Additive
An earlier iteration changed the return type to `{event_id, ts}` for chat/dm timestamp alignment. Reverted when grep showed 5 test fakes would need updating. Final design: optional `ts?` input parameter — zero caller churn, same alignment benefit.

### 3. `closeRoom` Added to ConnectionManager
Delete-room and recycle-sweeper both needed to close all sockets for a room. Added a single `closeRoom(room_id, code, reason)` method. Thirteen test fake files needed a one-line stub — mechanical but tracked.

### 4. Sweeper Disabled in Tests via 32-bit setTimeout Cap
`Number.MAX_SAFE_INTEGER` triggered Node's `TimeoutOverflowWarning` and silently clamped to 1 ms. Replaced with `2_147_483_647` (32-bit max) which Node accepts cleanly and keeps the sweeper inert during tests.

### 5. Muted Moderators Retain Role Authority
Per Decision 9, being muted does not strip role authority. Mute/unmute handlers explicitly allow muted PH/co_host to still moderate. `promote`/`demote` still block muted senders — a contradiction noted as a one-line follow-up, out of scope for this session.

### 6. Crash-Listener Stacking Bug Fixed
Discovered during Phase 11 implementation: `room-manager.ts` was calling `sandbox.once('crash', ...)` inside `submitCommand`, so each command invocation added a new listener. A single sandbox crash would fire all stacked listeners and broadcast duplicate crash messages. Fixed with a `crashAttached: Set<string>` guard.
