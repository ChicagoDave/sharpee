### 2. Native-Node ESM Extensionless-Import Fix (`build.sh`)
Diagnosed a Node ESM resolution failure: `dist-esm/` files emitted by `tsc -p tsconfig.esm.json` used extensionless relative imports (e.g., `from './types'`), which native Node ESM rejects. Vitest/vite-node tolerated this during tests but the production code path exposed it immediately.

Root cause: `tsf` already had `esmExtensions: true` on the `esm` target in `ts-forge.config.json` for every package, but `build.sh` was invoking `pnpm exec tsc -p tsconfig.esm.json` directly, bypassing `tsf`. Fix: changed the two per-package ESM build calls in `build.sh` (lines ~568 and ~783) to invoke `tsf build --condition esm --filter '$pkg'` instead. Future `./build.sh` runs durably emit valid Node ESM across the workspace.

This resolves the open question from the prior session ("how did the prior Deno implementation do this?"): the prior `tools/server/` used esbuild to bundle `@sharpee/*` inline at install time, so Node/Deno never resolved them. The current design keeps `@sharpee/*` as runtime peers (preserving the drop-in story design intent David wanted); the fix makes the `dist-esm/` artifacts correct rather than changing the resolution strategy.

## Key Decisions

### 1. ESM Fix via `tsf` Rather Than Bundling
Chose to fix `dist-esm/` at build time rather than switch to esbuild bundling of `@sharpee/*`. This preserves the drop-in story design intent: story bundles reference `@sharpee/*` as runtime peers, not inlined deps. The `tsf build --condition esm` path was already configured and just needed `build.sh` to call it.

### 2. Class Collision Rename (`sharpee-room-layout`)
Renaming the grid class rather than namespacing the prose-pane styles is the correct fix: the `sharpee-room` class name is part of the engine's decoration protocol (ADR-174) and cannot be changed there. The UI layout class was the local artifact.

### 3. Backlog Hydration Map-Based Pairing
Replaced the `flatMap` approach with explicit `Map<turnId, {command, output}>` pairing. This is strictly correct: a command event without a matching output (e.g., a turn that produced no prose) is excluded rather than emitted with null fields that would confuse the renderer.

### 4. DM Panel Intentionally Unmounted
`dm-panel.ts` component code is present and correct. It is not mounted in `room.ts` pending a UI placement decision. This is deliberate deferral, not a bug.
