### Task #6: afk-timer.ts (104 lines) + 6 GREEN tests
- `tools/server/src/ws/afk-timer.ts` — new module
- **Deviation from plan**: single global sweep timer that iterates `LockManager.listHeld()`, instead of one `setInterval` per room. Rationale: `LockManager` already tracks all rooms centrally; per-room timers would need lifecycle wiring on room create/delete with no behavioral benefit. Deviation is noted in the file header.
- API: `start()` / `stop()` / `tick()` — `tick()` is publicly callable so tests can drive sweeps deterministically without patching real timers
- On each tick, for each held room: if `locks.isAfk(room)` → `locks.forceRelease`, append `role(force_release, participant_id=null /* actor=system */)`, broadcast `lock_state { holder_id: null }`
- Honors injectable `afkTimeoutMs` (passed through to `LockManager.isAfk`) and `intervalMs` (setInterval interval); `start()` is idempotent (second call is a no-op)
- `handle.unref()` so the timer does not keep the Node event loop alive at shutdown

- `tools/server/tests/ws/afk-timer.test.ts` — 6 tests, all green, all GREEN-graded:
  - No holder → tick is a no-op (0 broadcasts, 0 event rows)
  - Below threshold (AFK_TIMEOUT_MS − 1ms) → skipped, holder unchanged
  - At/beyond threshold → released, `holder_id` flips to null, `lock_state` broadcast, session event row with `target_participant_id = prior`
  - Multiple rooms: the AFK one is released, the still-active one is untouched
  - Custom `afkTimeoutMs` boundary test (499ms vs 500ms)
  - `start()` idempotency: 3 ticks in 25ms with a 5ms interval produce exactly one release transition (not two)
- All assertions verify actual state mutations on `LockManager.getState()`, captured broadcast payloads, or captured `session_events.append` payloads — never return values alone or "didn't throw."

## Key Decisions

### 1. Extend `role.op` enum vs. new EventKind for force_release
Option 1 chosen. Reasoning: `force_release` is an authority-gated action consistent with the other role ops (promote / demote / mute / unmute / nominate); the audit trail shape is identical (actor + target). Zero schema migration.

### 2. Single global AFK sweep vs. per-room timers
Deviated from the plan wording ("per-room setInterval"). A global sweep over `LockManager.listHeld()` is simpler and equivalent — it processes exactly the same set of rooms per tick but needs no explicit start/stop on each room lifecycle event. Documented in the module header.

### 3. Defer handler-level tests for release-lock
`release-lock.ts` is a thin delegate: verify participant + sanity checks, call `locks.release`, conditionally broadcast. The state mutation (`locks.release`) is already covered by `lock-manager.test.ts`. Broadcast wiring will be covered by Task #9's two-user lock-race integration test. This matches the approach last session took for `draft-delta.ts` — no dedicated unit test, coverage via integration.
