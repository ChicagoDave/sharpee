## Key Decisions

### 1. Worker = semaphore slot, not Node Worker thread

The ADR-175 ┬¦OQ-2 phrasing "Worker pool sizing" is about per-container concurrent turn capacity, not OS-thread parallelism. JS is single-threaded; spawning Worker threads would also require the engine + world-model to be serialized across thread boundaries, which they aren't designed for. A counting semaphore over `executeTurnStatelessly` gets us the only property that matters: bounded simultaneous CPU + SQLite WAL load per container. The Postgres adapter (Phase 7) gets horizontal scaling via multiple containers, which is the right axis.

### 2. Lease acquisition stays inside the pool slot

If `acquireRoomLease` were outside `pool.run(...)`, two requests to the same room would both wait on the lease, neither inside a pool slot ŌĆö fine ŌĆö but the moment they were admitted to the pool they'd compete for the lease there. Putting the lease inside the slot means a request that wins its lease keeps the slot for the full turn cycle, and a request waiting on a lease is *not* holding a slot another room could be using. The pool measures executor-attempts, not lease-waiters.

### 3. Unbounded FIFO queue (no 503 reject)

Bounded queues introduce a new error code (`turn_queue_full`) and a tuning knob (`ZIFMIA_QUEUE_DEPTH`). v1 doesn't need either: client-side latency under load is the natural backpressure signal, and the existing per-request HTTP timeout already bounds the worst case. If queue depth becomes a real concern operationally, the `pool.queued` getter is the metric to alert on ŌĆö adding admission control later is straightforward.

### 4. Invalid env ŌåÆ warn + fallback, never throw

`ZIFMIA_WORKER_COUNT=foo` shouldn't crash the server at boot. We log a single stderr warning and fall back to `cpus - 1`. The capacity contract (Ōēź 1) is preserved on every path including 1-CPU machines.

### 5. Concurrent perf number is informational, not a gate

The 8-room concurrent p95 (~120 ms on M4) exceeds 100 ms, but that's an aggregate load number, not an AC-14 violation. AC-14 is the per-request budget ŌĆö single-room p95 was 16 ms HTTP (3c.iii). Documented this distinction prominently in `perf-baseline.md` so the next person running the harness doesn't read FAIL on the gate and think AC-14 regressed.

### Key decisions (4a)

#### A. Hard cutover, no back-compat reader

`decodeEnvelope` rejects payloads that don't have `envelopeVersion: 1`. Per the standing project policy ("server admin owns lifecycle ŌĆö schema/wire/credential changes are one-shot cutovers"), there is no reader for pre-4a raw `ISaveData` blobs. Operators cut over by replaying or discarding old saves. This is documented in the module header and surfaces as a `Error: unsupported envelopeVersion undefined` if a stale blob is read.

#### B. Transcript window is per-blob, not per-turn

Each save_blob carries its own window. The executor reads the prior window from the *previous* save_blob, appends the new entry, truncates, writes the new window. So every blob is self-describing ŌĆö Phase 4c (restore) and Phase 4d (compaction) can read any blob and recover the player-facing transcript without chaining backwards through rows. This costs roughly 1 KB / entry ├Ś 1000 entries = ~1 MB transcript per blob in the worst case (gzipped), which is acceptable next to the world snapshot's typical size.

#### C. World `turnCount` is absolute; transcript is a sliding window

The engine's `metadata.turnCount` keeps counting past 1000. The transcript window's first entry holds an absolute turn number (e.g. turn 501 after 1500 turns at window 1000). Clients render the transcript using the entry's own `turn` field; score / move-counter UI uses `metadata.turnCount`. The two were always separate concepts; the test makes this explicit (asserts turnCount=12 alongside transcript turns [8..12]).

#### D. `submitter` is required on the executor input

Previously the executor was driven by `(adapter, roomId, command)` and was agnostic to who sent the turn. Adding `submitter` makes the executor responsible for attributing the transcript entry. Alternative: the route could synthesize and persist transcript entries itself. Rejected ŌĆö the route then needs the prior transcript too, duplicating envelope decode work the executor already does, and AC-13 (engine-throw ŌåÆ no transcript entry) becomes harder to enforce uniformly.

#### E. Test helper `exec()` over individual call updates

Rather than adding `submitter: ...` to 9 individual executor call sites, the test file introduces a single `exec(input)` helper that fills the stub. This made the migration mechanical and keeps test bodies terse. The helper's existence is documented; it's not load-bearing for what each test asserts.

### Key decisions (4c)

#### A. Destructive rollback semantics, locked in by test

The truncation is a one-way operation: once forward history is gone, it's gone. The 9-test suite has explicit assertions for what stays (`[1,2]` after truncating from turn 4) and what's removed (future named saves). Any future "preserve branches" change would have to consciously delete or update these assertions ŌĆö making the destructive semantics structurally enforced, not just documentary.

#### B. Restore does NOT instantiate the engine

Earlier drafts had restore mirror the turn executor (build engine, restore, snapshot). But there's no command to run, the engine's reported `turnCount` after `restore()` equals the loaded value (so we'd collide with the existing save_blob row), and the engine work is redundant ŌĆö `executeTurnStatelessly` will do it on the next `/command`. Skipping the engine for restore cuts cost and removes a class of error paths.

#### C. Envelope-decode probe before truncation

If the target's envelope can't decode (somehow corrupted on disk), bailing out *before* the truncation means the room stays loadable from the previous save. If we truncated first and then noticed, we'd have destroyed the working forward history to enter an unreachable state. The probe is cheap (gunzip + JSON.parse) and saves the room.

#### D. `room:restored` as a distinct WS kind, not a `turn:broadcast`

Two reasons. First, `turn:broadcast` carries blocks/events for the turn that ran; a restore has no command output, so an empty `turn:broadcast` would be a wire lie. Second, restore is a different mental event for the client ŌĆö "the room rewound" reads differently than "someone took a turn." Dedicated kind makes the client handler explicit. Submitter-exclusion is the same rule (`identity !== submitter.id`) so the fan-out helper is structurally similar to the command route's.

#### E. `truncateRoomHistory` lives in the adapter contract

The temptation was to express truncation as two adapter calls (`listSaveBlobTurns` + `deleteSaveBlob` in a loop) in the route. Rejected ŌĆö the FK ordering and atomicity are storage concerns, not route concerns; expressing them inline would force the Postgres adapter (Phase 7) to re-prove the property via a different pathway and would block tests from verifying the atomic behavior. The adapter method is the right granularity.

## Phase 4d ŌĆö Compaction job (off-by-default GC)

### Design taken

- **Compaction rule**: keep the latest save_blob (active state ŌĆö `executeTurnStatelessly` reads it on every turn) and any save_blob that a `named_saves` row references. Delete the rest ŌĆö intermediate per-turn snapshots are unreachable (restore goes through named saves; `/state` reads the latest envelope's transcript window).
- **Atomicity at SQL level, not via lease**: the adapter's `compactRoomSaveBlobs` runs one `DELETE` with a `MAX(turn)` subquery inside a transaction. SQLite serializes it against `appendSaveBlob`, so the compactor never picks a "latest" that a turn invalidates mid-statement. No lease acquired ŌĆö compaction is maintenance and must not block turn execution.
- **Off by default**: operators opt in via `ZIFMIA_COMPACTION_ENABLED=true|1|yes`. Default tick interval is 5 minutes (`ZIFMIA_COMPACTION_INTERVAL_MS` override; invalid ŌåÆ fallback with one stderr warn).
- **Manual sweep always available**: even when periodic compaction is disabled, `handle.compaction.sweepNow()` runs a one-shot GC. Useful for ops drills and tests.

### Key decisions (4d)

#### A. No lease acquisition for compaction

Compaction is maintenance. If it held the per-room lease, every concurrent turn for that room would wait for the sweep to complete. With single-statement SQL atomicity, the lease buys nothing here ŌĆö SQLite already serializes the DELETE against any in-flight INSERT, and the `MAX(turn)` subquery inside the same statement guarantees we never delete a row that just became the latest. Documented in the adapter contract so the Postgres adapter (Phase 7) follows the same pattern (advisory lock for the per-room turn lifecycle; compaction relies on row-level locking instead).

#### B. Overlap-skip rather than queue

If a sweep takes longer than the interval (e.g., 1000 active rooms ├Ś slow disk), the next interval tick is skipped ŌĆö we don't queue a backlog of overlapping sweeps. The trade-off: under sustained slow-disk pressure, compaction effectively backs off and runs sequentially. The alternative (queue) would magnify any latency problem into a thundering herd. The test wraps the adapter with a 25 ms delay and proves overlapping ticks are dropped.

#### C. `sweepNow` always available

Disabled-mode still exposes `sweepNow`. Reasoning: operators may want to run a one-shot GC after a manual compaction-mode flip, or after a backfill that produced lots of intermediate save_blobs. Tests use it heavily ŌĆö exercising the GC logic without a timer is much cleaner than waiting on a fake clock.

#### D. Test seam via injected `scheduler`

The worker accepts an optional `scheduler: {setInterval, clearInterval}`. Tests pass a fake that records the handler and exposes a synchronous `tick()`. Production uses Node's built-in functions. This lets the test verify the worker's *scheduling* behavior (registers a handler, calls it on tick, respects overlap-skip) without depending on real-time advancement.

#### E. Env-parsing has its own functions

`parseCompactionEnabled` and `parseCompactionInterval` are exported separately so the perf and admin tooling (Phase 5) can reuse them without re-implementing the truthy-string rules. Same pattern as `parseWorkerCount` from Phase 3e.

### Key decisions (4e)

#### A. File-backed close+reopen ŌēĪ SIGKILL for our persistence layer

I considered spawning the real CLI (`dist/cli.js`) as a subprocess and SIGKILL-ing it mid-turn. Decided against for v1: the writes our app makes are single-statement SQL transactions in WAL mode, which SQLite commits durably the moment `COMMIT` returns. Whether the process then exits via `process.exit(0)` or `kill -9` does not change the on-disk state ŌĆö and SQLite's WAL recovery on the next open is a well-tested external invariant we don't own. The in-process test exercises the production read/write paths; subprocess testing would add fixture complexity (CLI spawn, port discovery, kill timing) without strengthening the assertions. If we ever ship a feature that holds significant in-memory state past a transaction boundary (e.g. write-back caches), revisit.

#### B. Per-room lease is in-memory by design ŌĆö no recovery needed

The SQLite adapter's `acquireRoomLease` is a Promise queue in JS memory. When the process dies, the lease state evaporates. A fresh process therefore starts with no lease held ŌĆö exactly what we want, since any turn the old process was mid-flight on did not commit. The integration test asserts this by running a healthy-room `/command` immediately after reopen and observing it gets turn 3 with no hang.

#### C. Compaction disabled during integration tests

The post-commit test asserts `listSaveBlobTurns` equals specific arrays. With the default-disabled compaction worker, that's safe ŌĆö but I want the assertion to remain stable even if someone later changes the default. Passing `compaction: { enabled: false }` explicitly in the bootstrap is one extra line that prevents a class of future-flake.

#### D. Pre-existing flake in saves.test, not introduced this phase

The "newest first" ordering in `saves.test.ts` was timing-dependent on `Date.now()` not ticking between two adjacent `INSERT`s. Under the lighter pre-integration test load, the millisecond clock always advanced; with the new 700 ms integration tests running first in the suite, ordering ties became reachable. Fix: 5 ms `setTimeout` between the two saves. Recorded for the recurrence-detector ŌĆö this is the second clock-tie I've fixed this phase (the other was a microtask-ordering assumption in Phase 3e worker-pool tests). If a third appears, worth a one-time audit of "tests that depend on `Date.now()` strictly increasing."
