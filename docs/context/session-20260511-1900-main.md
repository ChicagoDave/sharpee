## Key Decisions

### 1. Lock scope is per-connection, not per-identity

The typing-lock UX fires on `input:focus` events from a specific browser tab. Closing that tab must release the lock without affecting the user's other open tabs. Keying on `ClientConnection` rather than `identityId` makes this fall out naturally ‚Äî `releaseAllFor(conn)` on close does exactly what the UX needs.

### 2. Subscribe requires the room to exist

Initially 3d.i had no existence check on `room:subscribe`, but the chat and lock handlers both need the room to exist and producing handler-internal errors for ghost-room subscriptions was confusing. Rejecting at subscribe time with `room_not_found` produces a single, predictable failure point. Tests seed real rooms via `adapter.createRoom`.

### 3. Identity-level presence dedup

Two tabs from one identity show as one participant in `presence:roster` and emit only one `presence:joined` / one `presence:left` (transitions are 0‚Üí1 and N‚Üí0 connections per identity per room). Tested explicitly with a second-tab connect and a subsequent chat assertion to confirm no spurious presence frames are delivered.

### 4. Canonical-echo for chat

The sender's own WS connection receives the same `chat:message` as every other subscriber. Clients render the server's authoritative copy (with server-assigned `id` and `ts`) rather than optimistic local state. This eliminates an entire class of "my local copy disagrees with the persisted version" bugs without any client-side reconciliation logic.

### 5. HTTP fan-out excludes submitter by identity, not by connection

`command_echo` and `turn:broadcast` skip every connection whose `identityId` matches the submitter. Reason: the submitter may have multiple tabs subscribed via WS; if tab2 is subscribed while the HTTP POST comes from tab1, tab2 would otherwise receive a redundant `turn:broadcast` derivable from the HTTP response. Lock-state broadcast is exempted from the exclusion ‚Äî all subscribers, including the submitter, need to know their own lock was released.

### 6. AC-13 lock release on engine throw

The engine-throw path skips `command_echo` and `turn:broadcast` (no turn occurred), but the lock-release-and-broadcast step is unconditional. Without it, observer clients remain stuck with read-only inputs indefinitely. Verified with the real crashing-story fixture exercised through `executeTurnStatelessly` ‚Äî no stub or fake of the turn executor.
