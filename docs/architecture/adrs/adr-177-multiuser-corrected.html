The picture

Two browsers, one room, one engine. The process is HTTP. WebSocket is the live-broadcast rail. The submitter is just another subscriber — every browser in the room (including the one that typed) receives the same TurnPacket over the same WS broadcast.

Turn submission — sequence

A submits a command. HTTP carries the process; WS carries the render. The submitter receives the result over the same WS broadcast as every other participant.

Transport split

HTTP the process

  • POST /api/identities · claim · 409 on collision
  • POST /api/identities/erase · hard-delete + WS close 4007
  • GET /api/rooms · public lobby list (unauth OK)
  • POST /api/rooms · create as PH
  • POST /api/rooms/:id/join
  • POST /api/rooms/:id/{rename,pin,delete,nominate-successor,promote,demote,mute,force-release}
  • GET /api/rooms/:id/state?handle= · hydration (CMGT + transcript + roster + recording notice)
  • POST /api/rooms/:id/command → 200 {turnId} only
  • GET/POST /api/rooms/:id/saves · POST /restore
  • GET /api/stories · directory-scanned
  • GET /api/code/:join_code · resolve shareable

WebSocket the live rail

Client → Server (3 kinds):

  • hello { roomId, handle } · required first frame
  • chat:send { roomId, text }
  • lock:acquire / lock:release

Server → Client (7 kinds):

  • hello:ack { participantId, tier, lockHolder }
  • chat:message · broadcast incl. sender
  • lock:state { holder, expiresAt }
  • turn { roomId, turnId, submitter, packet } · the channel-io TurnPacket
  • role_change { participantId, tier, actorId? }
  • presence { participantId, connected, graceDeadline? }
  • room_restored { atSaveId, byHandle } · client re-fetches /state

Close codes: 4001 unknown_handle · 4003 not_in_room · 4004 room_not_found · 4005 hello_required · 4006 hello_timeout · 4007 identity_erased

Decisions in brief

1. Engine in-process

One turn = one HTTP submission. Load save blob → executeTurn → capture channel packets → persist new blob + session_event → broadcast over WS. No subprocess. No engine state between turns.

2. Channel-io is the shape

TurnPacket per ADR-163. CmgtPacket rides initial GET /state. The TurnPacket never rides an HTTP body outside the transcript_backlog field.

3. WS = chat + command-line sharing

Three C→S frames, seven S→C frames. Nothing else. New WS frame ⇒ new ADR amending this one.

4. HTTP is the process

Every state-changing request carries handle in-band (ADR-161 amended). Response is a small ack; render arrives over WS.

5. Identity is the handle

{id, handle, is_admin, created_at}. No passcode. No bearer token. No sessions table. Anyone who claims a handle is that identity. Threat model: small IF community.

6. Persistence (greenfield)

SQLite. v1 schema is taken on first boot. No migration from tools/shite/ or tools/server/. Future changes follow no-backcompat policy.

7. Stories by directory scan

StoryScanner + StoryHealth.validate() on boot and SIGHUP. No upload route. No story_library table. Operator drops files into stories/.

8. Recording notice

Static room-metadata constant. Carried in GET /state. Client renders once on first join, plus a persistent indicator.

9. Admin is CLI-only

is_admin column is set via sharpee-admin against the SQLite file. No /admin/* HTTP routes in v1.

What changed vs. tools/shite/ (the parts bin)

Discarded from tools/shite/

  • Admin upload of stories (POST /admin/stories + story_library + story_bundles)
  • /admin/* route family (column kept; surface gone)
  • Flat-room multi-tenant model (any-identified-user creates a room against any story)
  • Bearer session tokens, /identity/register|login|me, sessions table

Restored from tools/server/ @ 899653cf

  • Rooms as the primitive (not accounts)
  • Join code (Crockford-32, per-room)
  • PH + four-tier hierarchy + cascading succession
  • Pin / delete / mute / DMs / 14-day recycle
  • session_events append-only log
  • Story installation by directory scan
  • Recording notice (now in /state, not per-welcome)

Discarded from tools/server/

  • Deno subprocess per room (replaced by ADR-164 in-process)
  • Server ↔ sandbox wire (Init/Ready/Command/Output/…)
  • World-Model Replication (ADR-162) — replaced by channel-io
  • recording_notice as a per-welcome string (now static room metadata)

Constrains future sessions

  • New state change ⇒ new HTTP route with handle in-band. Never a new WS frame.
  • TurnPacket rides WS, not HTTP (except backlog).
  • Identity is {id, handle}. Sessions/bearer/passcode return only via a new ADR.
  • Stories live on disk. No upload endpoint.
  • is_admin has no HTTP surface in v1.

Prior art — where ADR-177 sits on the curve

Multi-user / shared-IF systems have been built before. Sketching the shape of each clarifies what ADR-177 takes from them and what it leaves on the table.

SystemShapeProperty worth noting
ifMUD + integrated interp (perlMud, Club Floyd)Stateful daemon hosting chat + identity + presence + room and an embedded IF interpreterSingleton game — one game at a time across the whole MUD. Works because the product is bounded; doesn't generalize to many concurrent rooms each running a different story.
ADR-153 Deno-subprocess-per-roomMany concurrent rooms; each room's engine in its own Deno subprocessStatefulness in the subprocess was the failure mode — crash = state loss. ADR-164 retired this; ADR-177 inherits the lesson.
Parchment / iplayif.com / LectroteEngine entirely in the browser, single-userThe "engine in client" extreme. Putting the engine in-browser makes shared rooms a separate infrastructure problem — effectively "build a MUD on top."
Twitch Plays / chat-driven streamsOne game, many viewers, chat → input pipelineAlmost exactly ADR-177's shape minus per-room. Validates "observers + driver + chat over shared game" as a social pattern.
MUD frameworks w/ web clients (Evennia, Ranvier)Long-running stateful server, web UI as a thin terminalArchitecturally indistinguishable from ifMUD — the browser is just a transport reskin.
Cloud gaming (GeForce Now, Stadia)Engine + scene graph + frame buffer server-side, video streamedThe extreme-stateful ceiling. ADR-177's TurnPacket is the text-IF analogue of a frame, with per-turn durability instead of per-frame.

What ADR-177 takes from each

  • From ifMUD — chat + identity + presence + room as the substrate model. Implemented as the WS layer (§3).
  • From ADR-153 (Deno) — the failure mode taught the design. Statefulness in a per-room process is the bug; per-turn durable storage is the fix.
  • From Parchment — nothing directly (engine is server-side here), but Parchment's renderer-in-browser pattern is what ADR-165 carries forward: the browser renders, it doesn't interpret.
  • From Twitch Plays — validation that "one engine, many observers, chat layered over it" works socially. ADR-177 just gives each group its own room and rotates the typing seat instead of voting commands.
  • From cloud gaming — thinness on the client side is fine when the server emits a renderable stream. ADR-163's TurnPacket is the text-IF equivalent of a frame.

ADR-177 picks the unusual middle-left position — IF's quiescence between turns makes per-turn durability essentially free, and that durability removes the failure mode that killed the Deno design.

Invariants

  • The process is HTTP. Governance, identity, saves, story listing, state reads, command submission — all request/response.
  • WebSocket is the live-broadcast rail. Chat + command-line sharing only.
  • The submitter is just another subscriber. POST /command returns {turnId}; the TurnPacket comes back over the same WS broadcast. Do not add an HTTP-body fallback — WS is not the authoritative copy. SQLite is. A flaky WS recovers via the same GET /state path a fresh browser uses. The Deno design lost state because state lived in a process; here state lives in the save blob, and WS is purely live delivery.
  • GET /state is for hydration, not live updates. Called on entry/reconnect; WS authoritative between hydrations.
  • Handle is the entire credential. No tokens, no hashes.
  • No engine state survives between turns in memory.
  • PH is sticky on the room row. Succession promotes a different participant to PH; the old PH's row is demoted.

Out of scope (named so they don't drift in)

  • DMs between arbitrary tiers (PH ↔ Co-Host only per ADR-153 D8)
  • User-uploaded story bundles
  • Federation across server instances
  • MPIF (each participant controls a separate PC) — separate product, separate engine
  • OAuth / 3rd-party auth
  • I18n of recording-notice / UI labels (operator can override the notice string)
