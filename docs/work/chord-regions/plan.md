# Session Plan: Chord Regions (ADR-236 implementation)

**Created**: 2026-07-17
**Overall scope**: Give Chord (`.story` author language) a first-class surface for the shipped ADR-149 region system — the `region` kind noun, `containing`-list membership (with nesting), region-owned every-turn daemons, both-sides crossing reactions (`entering`/`leaving`), and a new story-owned every-turn daemon with no presence gate. Zero platform (world-model) changes expected — this is a compiler-frontend (`packages/chord`) and loader (`packages/story-loader`) surface over an already-shipped substrate.
**Bounded contexts touched**: `packages/chord` (lexer/parser/analyzer/IR/catalog for the new kind noun, body line, event verb, and story-header daemon body), `packages/story-loader` (lowering to `world.createRegion`/`assignRoom`, scheduler daemon presence gating, event-verb-to-crossing-event binding), `docs/reference/chord-grammar.md` + `docs/reference/chord.ebnf` (grammar docs), `docs/architecture/chord-grammar-changes.md` (ratchet log). No changes expected to `packages/world-model` or `packages/stdlib/.../going` — their region API (`createRegion`, `assignRoom`, `isInRegion`, `getRegionCrossings`, `region_entered`/`region_exited` events) is shipped and verified in ADR-236's Context section.
**Key domain language**: region (kind noun), containing (membership body line), nesting (`parentRegionId` ancestry), region-owned daemon (presence = `isInRegion` transitive), crossing reaction (`entering`/`leaving`), story-owned daemon (no presence gate), ratchet entry (R1–R4).

This plan is the regions workstream that umbrella's Phase 5 flagged as queued follow-on work ("its implementation plan is the remaining regions workstream"). `chord-go-live/plan.md` remains the umbrella tracking G1–G4; this plan's completion closes the regions line of that umbrella's G1. ## Phases

### Phase 1: Region entity + `containing` membership + nesting (D1–D3)
- **Tier**: Large
- **Budget**: 400
- **Domain focus**: `region` kind noun and region-as-entity composition (D1); `containing <list>` as the sole membership form, additive across multiple lines (D2); nesting via regions-as-members setting `parentRegionId` (D3). Files: `packages/chord/src/catalog.ts` (`KIND_NOUNS`), `packages/chord/src/ast.ts`/`parser.ts`/`analyzer.ts`/`ir.ts` (new `containing` body-line AST/IR shape, entity-block composition rules for `region`), `packages/story-loader/src/loader.ts` (lowering to `world.createRegion` + `world.assignRoom`, never setting `RoomTrait.regionId` directly per the D1 one-wiring-path invariant).
- **Entry state**: ADR-236 ACCEPTED (done). No dependency on any other phase in this plan.
- **Deliverable**:
  - `region` added to `KIND_NOUNS`; loader kind-mapping case built onto the `createRegion`/`assignRoom` seam (never touches `RoomTrait.regionId` directly).
  - `containing <name list>` body line parses inside a region block, additive across multiple lines; grammar accepts `aka` aliases and description paragraphs on a region block (composable like any entity) and rejects placement lines (`in <place>`, `starts in`) as a load error mirroring ADR-234 D3.
  - Nesting: a `containing` member that resolves to another region sets that region's `parentRegionId`.
  - All D2/D3 never-guess load gates implemented, each with a named, testable diagnostic:
    1. unresolved `containing` member → standard unresolved-entity error
    2. member is neither room nor region → load error naming the entity and its kind
    3. room listed in two regions (direct membership) → load error naming both spans
    4. room listed in both an ancestor and a descendant region → same double-membership error class
    5. region with no `containing` line anywhere → hard load error (memberless region)
    6. region contained by two parent regions → load error (parent is single-valued)
    7. containment cycle → load error naming the cycle
    8. placement line (`in`/`starts in`) on a region block → load error
  - Ratchet entries R1 (`region` kind noun) and R2 (`containing` body line) logged as dated rows in `docs/architecture/chord-grammar-changes.md` with David's approval captured in the row.
  - Test suite (new `packages/chord/tests/region-membership.test.ts` and `packages/story-loader/tests/region-membership.test.ts`, following the existing `readFileSync` fixture pattern against a new `packages/chord/tests/fixtures/region-nesting.story`): one behavioral test per D10 AC-1 assertion (region entities exist, member rooms' `regionId` set, child regions' `parentRegionId` set — asserted on the loaded `WorldModel`, not on IR shape alone) and one rejection test per gate above (AC-3), each asserting the specific diagnostic code/message, not just "load failed."
- **Exit state**: A Chord story with a nested region tree loads cleanly (AC-1 satisfied); every gate in D2/D3 has a passing rejection test (AC-3 satisfied for all 8 D1–D3 gates); R1/R2 ratchet rows are committed. `pnpm --filter '@sharpee/chord' test` and `pnpm --filter '@sharpee/story-loader' test` both green.

### Phase 2: Region-owned daemons (D4)
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: region blocks become a legal home for the existing owner-attached `on every turn [while <cond>][, once]` clause, with presence redefined as `isInRegion(playerLocation, region)` (transitive through nesting) rather than single-room presence. Files: `packages/story-loader/src/runtime.ts` (`buildSchedulerDaemons()` — add the region-owner presence branch alongside the existing entity/trait branches), no chord-side grammar change (D4 is explicit: "No new clause grammar").
- **Entry state**: Phase 1 complete — region entities and membership must exist and be loadable before a region-owned daemon can be presence-gated against them.
- **Deliverable**: `buildSchedulerDaemons()` recognizes a region block's `on every turn` clause and lowers it to one scheduler daemon whose `condition(ctx)` calls `isInRegion(playerLocationId, regionId)`, satisfied transitively for a player in a nested child region. `it` binds the region inside the clause body. `while <condition>` and `, once` compose exactly as on any other owner (reuse the existing lowering path, do not fork it).
- **Test gate (rule 13a REAL-PATH)**: extend `packages/story-loader/tests/scheduler.test.ts` (or a new `region-daemon.test.ts`) with a fixture region tree (reuse or extend `region-nesting.story` from Phase 1) plus a daemon clause; tests drive real `go` commands through a Chord-loaded `WorldModel` + `SchedulerDaemon[]` tick loop (the existing `tick(turn)` pattern in `scheduler.test.ts`) moving the player between: a member room (daemon fires), a member room of a nested child region (daemon fires — transitivity), and a non-member room (daemon does not fire). Assert on emitted phrase/message-id events per turn, not on daemon registration count alone (AC-2's explicit requirement).
- **Exit state**: AC-2 satisfied — a region daemon fires only while the player is in a member room including nested-child rooms, verified against real movement, with `while`/`, once` composing. `pnpm --filter '@sharpee/story-loader' test` green.

### Phase 3: Crossing reactions, both sides (D6, ratchet R3)
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: `after entering it` becomes legal on region blocks (reusing the existing `entering` EVENT_VERBS binding, new host only); `after leaving it` is new catalog growth bound to `if.event.region_exited`. Files: `packages/chord/src/catalog.ts` (`EVENT_VERBS` gains `leaving`), `packages/chord/src/analyzer.ts` (accept region blocks as a legal home for both event clauses), `packages/story-loader/src/runtime.ts` (bind `entering`/`leaving` clauses on a region to the region's `region_entered`/`region_exited` crossing events, nesting-transitive via `getRegionCrossings` as the source of truth for which boundary actually crossed).
- **Entry state**: Phase 1 complete (region entities/membership must exist to have a crossing to react to). Independent of Phase 2 — may run before, after, or interleaved with it.
- **Deliverable**: Both `after entering it` and `after leaving it` clauses parse and lower inside region blocks; `while <condition>` composes on both. Ratchet R3 (the `leaving` event verb) logged as a dated row in `docs/architecture/chord-grammar-changes.md` with David's approval.
- **Test gate (REAL-PATH)**: new `packages/story-loader/tests/region-crossing.test.ts` driving real `go` commands: (a) crossing into a region fires its `entering` reaction, (b) crossing out fires its `leaving` reaction, asserted on emitted output/message ids, (c) a move between two rooms of the *same* region fires neither, (d) a move into a nested child region fires the *parent's* reaction only when the parent boundary is actually crossed (i.e., moving room-to-room within the same parent but across the child boundary does not spuriously fire the parent's clause) — this is AC-5's most failure-prone case and needs its own explicit assertion, not just an implied pass.
- **Exit state**: AC-5 fully satisfied with all four scenarios asserted on real movement-driven output; R3 ratchet row committed; `docs/reference/chord-grammar.md` and `chord.ebnf` updated for the region-block event-clause forms (partial AC-7, finished in Phase 5).

### Phase 4: Story-level daemon, no presence gate (D7, ratchet R4)
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: the story header block grows a new legal indented-body line — `on every turn [while <cond>][, once]` — alongside its existing `states`/`score … worth N` lines, story-owned (no presence gate, fires every turn regardless of player location), with `it` unbound and a compile-time diagnostic if referenced. Files: `packages/chord/src/parser.ts` (story header block body grammar), `packages/chord/src/analyzer.ts` (new unbound-referent diagnostic class for `it` in a story-owned clause — the case cannot arise in any existing clause host since every current host has an owner entity), `packages/story-loader/src/runtime.ts` (`buildSchedulerDaemons()` gains the story-owner branch: no `condition(ctx)` gate, fires unconditionally).
- **Entry state**: ADR-236 ACCEPTED. **Fully independent of Phases 1–3** — this daemon has nothing to do with regions; it can be implemented in any order relative to the rest of this plan, including first.
- **Deliverable**: A story header block with an `on every turn` clause in its indented body lowers to one scheduler daemon with no presence condition. `it` referenced inside a story-owned clause body produces a specific, tested compile diagnostic (new code in the analyzer's unbound-referent class). Ratchet R4 logged as a dated row in `docs/architecture/chord-grammar-changes.md` with David's approval.
- **Test gate (REAL-PATH)**: new `packages/story-loader/tests/story-daemon.test.ts` — a fixture story with rooms in at least two different regions (or no regions at all, proving the daemon doesn't care), driving real `go` commands across rooms, asserting the story-owned daemon fires every turn regardless of player location; a second test compiles a fixture with `it` referenced in a story-clause body and asserts the specific compile diagnostic (chord-side unit test, not story-loader).
- **Exit state**: AC-6 fully satisfied (both the every-turn-regardless-of-location behavior and the `it`-unbound compile diagnostic); R4 ratchet row committed.

- Phases 2 and 3 are independent of each other once Phase 1 is done — either order, or interleaved.
- Phase 5 is terminal — depends on all four prior phases being DONE.
- Total budget across all phases: 1,300 tool calls (Large 400 + Medium 250×3 + Small 150), consistent with this being a full ADR implementation spanning ~5 sessions.
