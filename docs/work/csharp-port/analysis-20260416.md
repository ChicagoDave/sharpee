# Porting Sharpee to C# — Detailed Analysis

**Date**: 2026-04-16
**Status**: Analysis / exploratory — no implementation decisions made
**Prerequisite reading**: `docs/spec/` (the platform-agnostic specification)

---

## 1. Overall Feasibility

**Verdict: highly feasible, with a well-understood shape.** The platform is already well-factored: 9 clearly bounded subsystems, contract-level interfaces, pure-data traits, message-ID protocol keeping English out of the core. There are no TypeScript-specific idioms (no JSX, no dynamic proxies, no Node-runtime dependencies) that would fight a C# port.

The work is **mostly mechanical translation** with **three genuinely hard spots**: the grammar DSL compiler, polymorphic JSON serialization of traits and events, and the extension/plugin model (including locale swap-in).

**Realistic effort** for one experienced .NET developer: **6–12 months** to reach feature parity with a single story (Cloak of Darkness) running end-to-end in a CLI client, plus a second locale proof. Dungeo-level parity is another 3–6 months.

---

## 2. Recommended Target & Project Layout

**Target**: .NET 8 LTS (or .NET 10 when stable). Enable nullable reference types globally — matches Sharpee's TypeScript strict-null discipline.

**Solution structure** (mirrors the spec documents):

| .NET Project             | Maps to TS package         | Notes |
|--------------------------|----------------------------|-------|
| `Sharpee.Core`           | `core`                     | Event envelope, types, data-store primitives |
| `Sharpee.WorldModel`     | `world-model`              | Entities, traits, behaviors, scope |
| `Sharpee.Parser`         | `if-domain` (parser core)  | Locale-neutral parser core + grammar DSL |
| `Sharpee.Parser.EnUs`    | `parser-en-us`             | English parser configuration |
| `Sharpee.Language`       | (abstract)                 | `ILanguageProvider` interface |
| `Sharpee.Language.EnUs`  | `lang-en-us`               | English templates/vocabulary |
| `Sharpee.Stdlib`         | `stdlib`                   | 49 actions, message IDs |
| `Sharpee.Engine`         | `engine`                   | Turn cycle, plugins, save/restore |
| `Sharpee.TextService`    | `text-service`             | Event→TextBlock pipeline |
| `Sharpee.Sdk`            | `sharpee` facade           | One-stop `using` for story authors |
| `Sharpee.Cli`            | `dist/cli/sharpee.js`      | Console client + transcript runner |

Stories ship as separate class libraries, loaded via reflection or an `IStory` contract — the NuGet analogue of "install a story package."

## 3. Subsystem-by-Subsystem Porting Notes

### 3.1 Data model (spec 01)

- **Entity** → `sealed record IFEntity(string Id, string Type, ...)` with mutable collections for traits/attributes, OR a `class` with init-only properties. Records make equality cheaper; mutable class is closer to the reference implementation.
- **Trait composition** → `Dictionary<Type, ITrait>` keyed by runtime type. Gives you compile-time safety on `entity.Get<OpenableTrait>()` that TS's string keys don't. Consider a parallel `Dictionary<string, ITrait>` for trait-type-ID lookup (wire format + ADR-090 capability dispatch both need it).
- **Spatial index** → two `Dictionary<string, HashSet<string>>` maps exactly like the spec. Trivial.
- **Event envelope** → `record SemanticEvent(...)`. The `data?: Any` field is the live question: `object?`, `JsonElement`, or strongly-typed hierarchy. Recommendation: **`object?` at the boundary, typed records per event type**. A `TakenEventData` record deserialises cleanly and the text service pattern-matches on it.
- **Save envelope** → `System.Text.Json` with a source-generated `JsonSerializerContext` for perf and AOT-friendliness. Polymorphic trait/event deserialisation needs `[JsonPolymorphic]` + `[JsonDerivedType]` attributes OR a custom converter — there are 37 traits and ~50 event types, so go with a convention-driven converter that reads the `type` discriminator string.
- **Seeded RNG** → port the reference implementation directly. Do **not** use `System.Random`; it's non-deterministic across .NET versions.

### 3.2 World model (spec 02)

- **Behaviors** → static classes with static methods. `public static class OpenableBehavior { public static bool Open(IFEntity e, IWorldModel w) {…} }`. Maps cleanly.
- **Capability dispatch (ADR-090)** → `Dictionary<(string traitTypeId, string actionId), ICapabilityBehavior>`. The trait's `capabilities` static array becomes a `[TraitCapabilities("if.action.lowering", "if.action.raising")]` attribute read at registration.
- **Scope system** → the scope levels and rule registry port straight. Watch for performance: `getVisible(actor)` is called a lot; pre-allocate lists, avoid LINQ chains in hot paths.
- **AuthorModel** → C# supports this cleanly via a separate class that wraps the same `IDataStore`. No language-specific friction.
- **Interface surface** → The `WorldModel` interface in the spec has ~80 methods. Split into `IWorldQuery`, `IWorldMutation`, `IScopeQuery`, `ICapabilityStore` interfaces; the concrete class implements all. C# callers appreciate this; TS callers didn't need it.

### 3.3 Parser & grammar DSL (specs 03, 04)

**This is the hardest part.** The TS grammar engine compiles `.define("put :item in :container")` patterns into a runtime matcher. You have to port:

- Token stream model with POS tags and vocabulary candidates
- The pattern compiler (slot extraction, alternates, optional words)
- Scope-constrained slot resolution with partial-match error analysis
- Pronoun context with gender inference
- Pattern priority resolution
- Story grammar extension

**Recommendations**:

1. **Do a spike early.** Port `.forAction()` and `.define()` for a dozen representative patterns from `parser-en-us/src/grammar.ts`. Validate the matching semantics match (side-by-side against TS output on the walkthrough transcripts) before writing any actions.
2. **The fluent builder translates naturally** — C# extension methods and init-only properties make it ergonomic. `grammar.ForAction("if.action.taking").Verbs("take", "get").Pattern(":item").Where("item", s => s.Visible().Portable()).Build();`
3. **The matcher can be `Regex`-driven or a hand-rolled state machine.** Reference uses the former. A compiled `Regex` per pattern is fine for ~200 patterns; don't premature-optimise.
4. **Slot constraint DSL** — `scope.Visible().Portable().Matching(new { Openable = true })` — C# anonymous types and `Func<IFEntity, bool>` handle this well; avoid `dynamic`.

### 3.4 Engine (spec 05)

- **13-phase turn cycle** — a `Pipeline<TurnContext>` with ordered phase handlers. Each phase is `async Task ExecuteAsync(TurnContext ctx)`. Phase ordering stays explicit (no DI-resolved magic).
- **Async** — `Promise<T>` → `Task<T>`. Save/restore hooks become `Func<SaveData, Task>`. Use `ValueTask` only if profiling demands it.
- **Plugin registry (ADR-120)** — `IPlugin` interface, `IPluginRegistry` registers by priority. The TS plugin contract ports one-to-one.
- **Undo** — snapshot before execute. `System.Text.Json` round-trip is the simplest snapshot mechanism; memory snapshots via deep-clone are an optimisation.
- **Meta-commands** — dispatch table `Dictionary<string, IMetaCommandHandler>`. Straight port.

### 3.5 Stdlib (spec 06)

- **Four-phase action contract** → `abstract class ActionBase : IAction` with optional virtual `Report` / `Blocked`. C#'s default-interface-methods (C# 8+) are tempting but `abstract class` is clearer for this pattern — story authors are used to inheriting.
- **`ValidationResult.data`** → `Dictionary<string, object?>` or a stronger `record ValidationData(…)`. The TS version is untyped; C# can tighten this per-action with generics (`ValidationResult<TData>`) if you want.
- **49 actions** — mechanical port. Budget ~1–2 days per action including tests. After the first 5, this goes faster.
- **Message-ID protocol** — trivial: action emits `new MessageEvent("if.action.taking.taken", new { item = target.DisplayName })`. The anonymous-object-as-params pattern is idiomatic.

### 3.6 Language layer (spec 07)

- **`ILanguageProvider`** — interface with ~25 methods. Per-locale implementation is a `record`-heavy data bundle.
- **Message templates** — `{placeholder}` substitution. Port the template engine directly; it's ~100 lines in the reference. **Do not use `string.Format`** — the placeholder syntax differs and the reference supports formatter pipelines (`{item:indefinite}` style).
- **Formatters** → `Dictionary<string, Func<object?, FormatContext, string>>`. Simple.
- **Narrative context** (`{You}`, `{take}` perspective) — port the conjugation tables.
- **i18n boundary** — the contract is airtight in TS; C# enforces it even more naturally via project references (stdlib cannot reference `Sharpee.Language.EnUs`).

### 3.7 Text service (spec 08)

- Pure function `ProcessTurn(events) → List<TextBlock>`. Straight port.
- **Decoration syntax `[type:content]`** → a parser method; ~50 lines.
- **Specialized handlers** (`command.failed`, `client.query`, etc.) — a strategy dictionary keyed by event type.

## 4. Key Cross-Cutting Decisions

### 4.1 Nominal vs structural typing

TypeScript's structural typing lets stories write `{ cannotTake: true }` on an entity and read it back with a cast. C# requires named types. **Consequence**: stories must define their custom data as traits or attribute dictionaries. This is actually more disciplined — the Sharpee CLAUDE.md already discourages `(axe as any).cannotTake`. The port will enforce what the style guide asks for.

### 4.2 Dynamic event payloads

The `data?: any` field on `SemanticEvent` is the chief place TS's permissiveness shows. Options:

- **Option A — `object?` everywhere**: matches TS semantics, forces casts at consumer sites. Simplest, most TS-like.
- **Option B — `JsonElement`**: save/restore is trivial; consumers call `.Deserialize<T>()`. Verbose in handler code.
- **Option C — typed per event**: `record TakenEventData(string ItemId)` etc. Most .NET-idiomatic; requires a registry mapping event type string → CLR type. Best long-term.

**Recommendation: C.** The spec already names 50+ event types; defining records up front is manageable and pays off everywhere downstream.

### 4.3 Extension model for stories

TypeScript stories import `@sharpee/sharpee`, call `createStory({ … })`, and export. C# stories should:

1. Implement `IStory` (metadata + `InitializeWorld(IWorldModel, IAuthorModel)` + `ExtendParser(IParser)` + `ExtendLanguage(ILanguageProvider)`).
2. Ship as a NuGet package or a side-loaded DLL.
3. Be discovered by the engine via an attribute-marked `[Story]` class or a `story.json` manifest.

Stories can reference custom traits/actions directly without reflection gymnastics — C#'s assembly model is simpler than npm packages for this.

### 4.4 Client layer

- **CLI**: plain `Console.WriteLine` + `Console.ReadLine`. Trivial.
- **Web**: Blazor WebAssembly — viable but expect ~5-10 MB payload vs TS's smaller bundle. Blazor Server is lighter but requires a host.
- **Desktop**: Avalonia is the best cross-platform option (matches Electron + TS in reach). .NET MAUI works but is Windows/macOS/mobile only.
- **Zifmia-equivalent (packaged single-story binary)**: `dotnet publish` with `PublishSingleFile=true` + AOT gives a ~30MB executable. Signable/notarisable on macOS.

### 4.5 Save-file compatibility

**Decide up front**: do you want C# saves to load in TS (and vice versa)?

- **Yes → hard**: requires pinning both to the *same* divergence (the list-vs-record shape in `WorldSerializer`, for instance). Worth it if you envision users migrating between runtimes.
- **No → easier**: C# picks the cleanest shape (records, maps not lists) and keeps its own save format. Stories are portable via source/recompile.

Recommendation: **No, at least initially.** The divergence list in the spec is already flagged; a C# port is a good occasion to clean up to the declared shape. Document the format version bump.

## 5. Risk & Effort Hotspots

| Area                        | Risk   | Why |
|-----------------------------|--------|-----|
| Grammar DSL compiler        | High   | ~1500 lines of subtle TS; regex semantics vary between engines; transcript tests are the only real validation |
| Polymorphic JSON            | Medium | 37 traits × 50 events; convention + source generator mitigates but needs upfront design |
| Scope system performance    | Medium | Hot path; TS's JIT forgives LINQ abuse, .NET does too but profile early |
| Transcript-test equivalence | High   | The walkthroughs are the real regression baseline; you need a C# transcript runner behaving identically to the TS one |
| Plugin model (ADR-120)      | Medium | Spec is still marked Proposed; pick a shape and commit |
| ADR-017 disambiguation      | Low    | Behaviour is required (never silent-pick), specific rule order is not |
| 49-action stdlib            | Low    | Volume, not complexity. Boring work, not risky work. |
| Client layer                | Low    | Solved problem in .NET |

## 6. Suggested Execution Strategy

**Phase 1 (weeks 1–4): Foundation spike.** Core + WorldModel + a handful of traits (Openable, Container, Room, Identity). Prove the serialisation round-trip and capability-dispatch plumbing work. Write xUnit tests mirroring the TS world-model test suite. **Milestone**: create a two-room world, open a door, move the player, save/restore round-trip.

**Phase 2 (weeks 5–10): Grammar spike + parser.** Port the grammar DSL, pattern compiler, and token stream. Port `parser-en-us`'s verb patterns. **Milestone**: parse a dozen reference sentences byte-identical to TS output.

**Phase 3 (weeks 11–18): Engine + first 10 stdlib actions + minimal language layer.** Turn cycle works end-to-end. TAKE / DROP / LOOK / EXAMINE / OPEN / CLOSE / GO / INVENTORY / WAIT / HELP. **Milestone**: play Cloak of Darkness in a CLI. Runs its walkthrough transcript identically to TS.

**Phase 4 (weeks 19–26): Remaining stdlib + scheduler + plugins + text service polish.** **Milestone**: all 49 actions passing; stdlib unit tests green.

**Phase 5 (weeks 27–36): Second locale proof-of-concept + story SDK + first non-trivial story.** Port a minimal `lang-de-de` or `lang-fr-fr` to prove the locale boundary holds. Ship a story authored natively in C#.

**Phase 6 (weeks 37+): Client work, build tooling, NuGet packaging, docs.** The visible user-facing polish.

## 7. Open Questions to Validate Before Starting

1. **Do you want the port to stay in lockstep with TS** (shared save format, synchronous roadmap) or **fork cleanly** (C# evolves independently, interop via source/recompile only)?
2. **What client matters most?** Desktop-packaged-Zifmia-equivalent is the richest C# story; Blazor web is possible; CLI is easy. Priority changes what Phase 6 looks like.
3. **Is this for you alone, a collaborator, or a separate team?** The effort estimate assumes one experienced .NET developer with full context. A team of 2–3 could compress to 4–6 months; a newcomer would take much longer because the spec does not substitute for reading the TS source.
4. **AOT or JIT?** AOT (Native AOT in .NET 8+) gives fast startup (matters for CLI / transcript runner) but forbids runtime reflection. The plugin model for stories would need source generators instead. Worth deciding before writing the extension APIs.
