## Phases

### Phase 1: R1 + R2 — fold execute-phase refusals into validate; key slot on locking/unlocking
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: stdlib — `taking_off` and `wearing` (execute-phase ad-hoc refusals become validate-phase, hook-visible refusals); `locking`/`unlocking` (a second consultable slot, the KEY, resolved via `instrument ?? indirectObject`).
- **Entry state**: ADR-228 fully complete (33/33 actions wired). `taking_off` (`packages/stdlib/src/actions/standard/taking_off/`) and `wearing` (`packages/stdlib/src/actions/standard/wearing/`) still run `checkRemovalBlockers`/`hasRemovalRestrictions`/`checkWearingConflicts` inside `execute()`, rendering failure from `report()`'s `sharedData.failed` path — these refusals never reach `onBlocked`/`postReport`. `locking`/`unlocking` descriptors (`packages/stdlib/src/actions/standard/locking/`, `.../unlocking/`) consult only the target slot; a key-side `on locking it` / `on unlocking it` Chord registration loads cleanly (target id IS consulted) and silently never fires.
- **Deliverable**:
  - R1: `checkRemovalBlockers`/`hasRemovalRestrictions` (taking_off) and `checkWearingConflicts` (wearing) moved into `validate()`, running after standard behavior checks and before the canonical `postValidate` hook placement. Failures become ordinary `ValidationResult`s flowing through `blocked()` → `runOnBlocked` — same messages, event types, message ids. The `cursed`-flag probe is kept and folded (author-set `(trait as any).cursed` still works, now interceptably). Execute's behavior-result failure branches stay as the defensive safety net only; `report()`'s `sharedData.failed` path shrinks to that safety net.
  - R2: Both descriptors gain a second slot `id: 'key'`, same action id (`if.action.locking` / `if.action.unlocking`), resolving `context.command.instrument?.entity ?? context.command.indirectObject?.entity` — explicit keys only, no auto-inferred key (same documented rule as attacking's inferred weapons). Published consultation order **target → key** (D3-B). Symmetric seedData: the key consultation receives the target id, mirroring putting/inserting's existing pattern.
  - Pinning tests: R1 — a folded refusal reaches `onBlocked`/`postReport` (state-asserted, not just "didn't throw"); the cursed-flag probe still fires through the new path. R2 — key hook fires on an explicit key; a key-side `preValidate` veto blocks the action (state asserted); target → key order pinned; no key consultation when the command names no key.
- **Exit state**: `taking_off`/`wearing` produce one hook-visible blocked-shape per refusal path; `locking`/`unlocking` consult both target and key when a key is named. Before landing R2, grep `stories/` to confirm no existing story registers a key-side `on locking/unlocking it` interceptor (ADR-229 Consequences flags this as unverified). `pnpm --filter '@sharpee/stdlib' test` green; `./repokit build dungeo` succeeds; relevant transcripts (any taking_off/wearing/locking/unlocking coverage) pass via `node dist/cli/sharpee.js --test`; walkthrough chain run once per the one-good-run rule.
- **Status**: COMPLETE (2026-07-16, session 2c43df) — R1 folds land with identical error codes/messageIds (prevents_removal, cant_remove, hands_full/already_wearing), execute keeps only the behavior-result defensive net; R2 key slots symmetric-seeded both directions per putting's convention. +9 pinning tests (all state-asserting). Grep: no story registers lock/unlock interceptors at all. stdlib 1469 green; build green; grating-key + tiny-room-puzzle 40/40; walkthrough chain 913/913 first run; mutation-verification clean (helpers confirmed pure reads — no mutation stranded behind a refusal).

### Phase 2: R5 — gated dispatch clause falls through to the miss
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: Chord story-loader — `buildDispatchAction.validate` (routing per `packages/chord/src/analyzer.ts:1055`, lowering in `packages/story-loader/src/runtime.ts`). "Sits out" now means as-if-never-declared: a behavior whose validate sets `chordSkip` (false `while` gate, or a consumed `, once`) does not claim the dispatch.
- **Entry state**: Phase 1 complete (independent of R5, but sequenced first since R1/R2 are the smaller, self-contained stdlib change). ADR-228 Phase 7 (D8) fixed gate-evaluation timing on entity/trait/capability lowering sites but left dispatch-verb capability behaviors claiming the dispatch even when gated out — a `chordSkip`'d behavior with no action body renders a blank turn today (ADR-229 Context item 5).
- **Deliverable**: In `buildDispatchAction.validate`, a `chordSkip`'d behavior is treated as not claiming the dispatch: selection falls through in order to the action body (if present), then another trait's behavior, then the `otherwise refuse` miss. No platform change — both sides of the contract are Chord-loader-owned (`buildDispatchAction`, `buildCapabilityBehavior`). The D8 evaluation-point ruling (once per firing, at validate time) is unchanged; this only changes what a `chordSkip` result does to dispatch *selection*, not when the gate is evaluated.
- **Pinning tests**: extend `packages/story-loader/tests/while-gate-d8.test.ts` — a gated-out dispatch clause with no body falls through to `otherwise refuse` (state asserted: the miss message/event fires, not a blank turn); a gated-out clause with an action body falls through to that body; a gated-out clause with another trait's live behavior falls through to that behavior; a `, once`-consumed clause on a dispatch verb falls through the same way on its second firing.
- **Exit state**: A while-false or once-consumed dispatch-verb behavior never produces a blank turn — the player always sees either the fallback behavior, the action body, or the miss. `pnpm --filter '@sharpee/story-loader' test` green; zoo gate (atomic + wt-chain, the project's dispatch-verb-heavy story) green; cloak-of-darkness transcripts unaffected (no dispatch-verb regression); `./repokit build dungeo` succeeds.
- **Status**: COMPLETE (2026-07-16, session 2c43df) — candidate behaviors probed with per-candidate sharedData (gated probes structurally side-effect-free: gate/once returns precede the counter write); real refusals still claim immediately; claiming candidate's sharedData is what execute/report see. while-gate-d8.test.ts reworked to R5 (2 Phase-7 pins updated per the ruling) + 2 new fall-through tests + occurrence-counter-never-written assertions on all three gated-probe tests (mutation-verification's one finding, closed). story-loader 156/156; zoo atomic + wt-chain green; cloak green; build green. Fixture note: trait-declared states are one namespace per entity, so the fall-through second trait is states-less.
