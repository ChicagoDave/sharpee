# Session Plan: VS Code Extension — Tier 1

**Created**: 2026-04-12
**Overall scope**: Build and publish the first usable version of the Sharpee VS Code extension. Tier 1 covers the three features that give authors immediate value with no new platform infrastructure: transcript file syntax highlighting, story authoring snippets, and a new-story wizard. The extension lives in `tools/vscode-ext/` (outside `packages/` — it is tooling, not a platform package).
**Bounded contexts touched**: N/A — tooling only, no engine/stdlib/world-model changes
**Key domain language**: N/A

---

## Scope Decisions

### Extension directory: `tools/vscode-ext/`

Not `packages/` because the extension is not a runtime package consumed by stories or the engine. It does not need to be registered in the platform build pipeline, pnpm workspace, or `packages/sharpee`. It builds with `vsce` independently.

### New Story Wizard: deferred scaffold, no `npx init`

`npx @sharpee/sharpee init <name>` does not exist yet. The wizard will generate a minimal story scaffold directly from extension templates (TypeScript files) without shelling out to a CLI command that doesn't exist. This is explicitly noted in Phase 3. If `npx init` is built later, the wizard can be updated to call it.

### Gutter button vs. CodeLens

The "Run Transcript" button is best implemented as a CodeLens (text link above matching lines) rather than a true gutter button. VS Code's gutter decoration API does not support clickable buttons — CodeLens is the idiomatic way to provide in-editor run actions.

### Problem matcher scope

The CLI outputs lines like `  > examine mailbox                                FAIL` and assertion details like `    ✗ Expected output to contain "mailbox"`. The problem matcher in Phase 1 will target the assertion failure lines and map them back to line numbers in the transcript file. This requires the CLI to emit file:line information, which the fast-cli does not currently emit. Phase 2 handles the CodeLens runner; Phase 1 produces the TextMate grammar and a task definition that uses the problem matcher in watch mode.

## Transcript Grammar Reference

The TextMate grammar (Phase 1) must cover all token types found in the corpus:

**Header block** (YAML front matter, separated by `---`):
- Keys: `title`, `story`, `author`, `description`

**Commands**: Lines starting with `> `

**Comments**: Lines starting with `#`

**Annotation commands**: Lines starting with `$`
- `$session start <name>`, `$session end`
- `$bookmark <name>`, `$review`, `$export`
- `$note <text>`, `$bug <text>`, `$confusing`, `$expected <text>`
- `$save <name>`, `$restore <name>`
- GDT/test helpers: `$teleport`, `$gdt`, `$tk`, `$ah`, `$kill`, etc.

**Block directives** (bracket-wrapped, open/close pairs):
- `[GOAL: <name>]` / `[END GOAL]`
- `[IF: <condition>]` / `[END IF]`
- `[WHILE: <condition>]` / `[END WHILE]`
- `[RETRY: max=N]` / `[END RETRY]`
- `[DO]` / `[UNTIL "<text>" OR "<text>"]`
- `[NAVIGATE TO: "<room>"]`

**Assertions** (follow a `>` command):
- `[OK: contains "text"]`
- `[OK: not contains "text"]`
- `[OK: contains_any "text" "text"]`
- `[OK: matches /regex/]`
- `[FAIL]`, `[SKIP]`, `[TODO]`
- `[ENSURES: <condition>]`
- `[REQUIRES: <condition>]`
- `[EVENT: true|false, type="<id>"]` with optional `messageId=`, `blocked=` attributes
- `[STATE: true|false, <expr>]`

**Condition sub-language** (inside WHILE/IF/ENSURES/REQUIRES):
- `location = "Room Name"`
- `not location = "Room Name"`
- `inventory contains "item"`
- `not inventory contains "item"`
- `not entity "name" alive`

## Phases

### Phase 1: Extension Scaffold and Transcript TextMate Grammar
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: N/A — tooling
- **Entry state**: `tools/vscode-ext/` directory does not exist; no VS Code extension for Sharpee exists
- **Deliverable**:
  - `tools/vscode-ext/` initialized as a VS Code extension project:
    - `package.json` with correct manifest (`contributes.languages`, `contributes.grammars`, `contributes.taskDefinitions`)
    - `tsconfig.json` targeting `vscode` engine typings
    - `esbuild` or `webpack` bundle config (keep it light — no framework)
    - `.vscodeignore` and `.gitignore`
  - Language registration: `.transcript` files associated with the `sharpee-transcript` language ID
  - `syntaxes/sharpee-transcript.tmLanguage.json` — TextMate grammar covering all token types in the corpus (header, commands, comments, annotations, block directives, assertions, condition sub-language). Full scope names listed so themes can colorize correctly.
  - Bracket matching config in `package.json`: `[GOAL]`/`[END GOAL]`, `[IF]`/`[END IF]`, `[WHILE]`/`[END WHILE]`, `[RETRY]`/`[END RETRY]`, `[DO]`/`[UNTIL...]`
  - Hover provider (TypeScript, registered in `extension.ts`): when cursor is on any `[OK: ...]`, `[ENSURES: ...]`, `[EVENT: ...]`, or `[STATE: ...]` assertion, shows a tooltip explaining the assertion syntax and accepted values
  - Task definition in `package.json`: `"type": "sharpee"` task with `"command": "test"`, wired to `node dist/cli/sharpee.js --test ${file}` — gives authors a "Run Transcript Test" task they can bind to a key
  - Problem matcher registered in `package.json` that parses failure output lines from the CLI and creates VS Code diagnostics pointing to the correct line in the open `.transcript` file
  - Extension builds to `out/extension.js` without error (`vsce package` produces a `.vsix`)
  - Opening any `.transcript` file from the dungeo corpus in VS Code shows correct syntax highlighting
- **Exit state**: The extension `.vsix` installs cleanly; transcript files are syntax-highlighted; hover tooltips work on assertion lines; the "Run Transcript Test" task runs the CLI against the open file

### Phase 2: CodeLens "Run Transcript" and Inline Test Diagnostics
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: N/A — tooling
- **Entry state**: Phase 1 complete; extension installs and highlights transcripts; no in-editor run button or inline pass/fail markers exist
- **Deliverable**:
  - `RunTranscriptCodeLensProvider` (TypeScript): places a `Run Transcript` CodeLens link above the first command line (`> `) in each transcript file. Clicking it runs `node dist/cli/sharpee.js --test <filepath>` in a VS Code output channel named "Sharpee Tests"
  - CLI output parsed in the extension (not just a task): failed assertions mapped to diagnostic `Range` objects on the correct line in the open file; VS Code Problems panel shows them with source `sharpee-transcript`
  - Pass/fail gutter decorations: green circle on lines whose assertions passed in the last run, red circle on lines with failures. Decorations cleared and re-applied on each run.
  - `[GOAL: ...]` CodeLens: shows `Run Goal` to run just the commands between `[GOAL]` and `[END GOAL]` that the cursor is inside. Implemented by slicing the file to that range and running with a temp file or using a `--from-line`/`--to-line` flag if available; otherwise run the full file and highlight only that goal's results.
  - Status bar item: shows "Sharpee: N passing / M failing" after a test run, cleared when the file changes
  - The `dist/cli/sharpee.js` path is resolved relative to the workspace root (configurable via `sharpee.cliBundlePath` setting, defaults to `dist/cli/sharpee.js`)
- **Exit state**: Author opens a `.transcript` file, clicks "Run Transcript" above the first command, sees inline pass/fail decorations and Problems panel entries for failures without leaving the editor

### Phase 3: Story Authoring Snippets
- **Tier**: Small
- **Budget**: 100
- **Domain focus**: N/A — tooling
- **Entry state**: Phase 1 complete (extension scaffold exists and builds); no snippets registered
- **Deliverable**:
  - `snippets/sharpee-story.json` with 12 snippets, all scoped to TypeScript files (`"scope": "typescript"`):
    - `room` — `world.helpers().room(...)` builder with identity, description, exits placeholder
    - `object` — entity with `IdentityTrait`, aliases, description
    - `container` — entity with `ContainerTrait` (openable, capacity)
    - `door` — entity with `OpenableTrait` + `LockableTrait`, connecting two rooms
    - `actor` — NPC entity with `NpcTrait` and behavior registration stub
    - `action` — four-phase action skeleton (`validate` / `execute` / `report` / `blocked`) with correct TypeScript types
    - `grammar` — `.forAction(...)` builder with `.verbs([...])`, `.pattern(...)`, `.where(...)`, `.build()`
    - `grammar-define` — `.define(...)` phrasal verb pattern with `.mapsTo(...)`, `.build()`
    - `message` — `language.addMessage(ID, () => ...)` registration
    - `daemon` — `scheduler.addDaemon(id, { interval, handler: (world) => ... })` skeleton
    - `fuse` — `scheduler.addFuse(id, { turns, handler: (world) => ... })` skeleton
    - `trait` — custom trait class skeleton with `static readonly type`, `static readonly capabilities`
    - `behavior` — `CapabilityBehavior` object with `validate`, `execute`, `report`, `blocked` stubs
  - Each snippet uses tab stops (`$1`, `$2`) at the most likely edit points (entity name, description, action ID)
  - Snippets registered in `package.json` under `contributes.snippets`
  - Extension `.vsix` rebuilt and installs cleanly
- **Exit state**: Typing `room`, `action`, `trait`, etc. in a TypeScript story file offers the Sharpee snippet via IntelliSense; tab stops move through the scaffold correctly

### Phase 4: New Story Wizard
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: N/A — tooling
- **Entry state**: Phases 1-3 complete; extension installs with grammar, CodeLens, and snippets; no new-story command exists
- **Deliverable**:
  - Command `sharpee.newStory` registered in `package.json` and `extension.ts`
  - Command palette entry: `Sharpee: New Story`
  - Multi-step quick-pick/input flow (VS Code `InputBox`/`QuickPick` APIs):
    1. Story ID (slug, e.g. `my-story`) — validates lowercase-alphanumeric-hyphen
    2. Story title (display name)
    3. Author name (defaults to git config `user.name` if detectable)
    4. Brief description (one line)
  - Wizard generates a new directory at `stories/<story-id>/` with:
    - `src/index.ts` — minimal `Story` implementation that compiles: imports, class stub, `initializeWorld()`, `extendParser()`, `getPlayerStart()`
    - `src/regions/start/index.ts` — one starter room ("Starting Room") with a placeholder description and no exits
    - `package.json` — correct name, version `0.1.0-beta`, pnpm workspace entry, dependencies on `@sharpee/sharpee`
    - `tsconfig.json` — extends the project root tsconfig pattern used by other stories
    - `tests/transcripts/hello-world.transcript` — a starter transcript: `> look` / `[OK: contains "Starting Room"]`
  - After generation, the wizard opens the new `src/index.ts` in the editor and shows an information message: "Story created. Build with: ./build.sh -s <id>"
  - If `stories/<story-id>/` already exists, shows an error message and aborts (no overwrite)
  - Note in code comment: when `npx @sharpee/sharpee init` exists, replace the file generation with a call to that command
- **Exit state**: Running "Sharpee: New Story" from the command palette produces a compilable story scaffold in `stories/<id>/`; the starter transcript passes when run against the built bundle

## Phase Dependencies

Phases 2, 3, and 4 each depend on Phase 1 (the extension scaffold must exist). Phases 2, 3, and 4 are independent of each other and can be done in any order after Phase 1.

The natural sequence is Phase 1 → Phase 3 (snippets are the simplest addition after the scaffold) → Phase 2 (CodeLens adds test integration) → Phase 4 (wizard is the most self-contained command feature).

---

## Notes on `dist/cli/sharpee.js` Dependency

Phases 2 and 4 both require the story to be built before running tests or verifying the wizard's output. The extension does not build the story — it delegates to the existing `./build.sh` workflow. The wizard's generated transcript will fail if the author has not run a build. The error message shown in the output channel should say "Run ./build.sh -s <story-id> first" if the CLI bundle is not found.
