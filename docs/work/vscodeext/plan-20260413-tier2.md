# Session Plan: VS Code Extension — Tier 2

**Created**: 2026-04-13
**Overall scope**: Add Build & Play integration, native Test Explorer support, a World Explorer sidebar panel, and Entity Autocomplete to the existing `tools/vscode-ext/` extension. Tier 1 (TextMate grammar, hover tooltips, CodeLens test runner, 13 snippets, New Story wizard) is already shipped as `sharpee-vscode-0.1.0.vsix`. Tier 2 makes the edit–build–test–play loop seamless inside VS Code, then adds world-aware intelligence that requires a new `--world-json` flag in the CLI bundle (a platform change, flagged below).

**Bounded contexts touched**: N/A — tooling only. One platform change required: adding `--world-json` output flag to `dist/cli/sharpee.js`.

**Key domain language**: N/A

---

## Platform Change Notice

Phases 3 and 4 both require extracting the live world model (rooms, entities, NPCs) from a built story. The cleanest mechanism is a new `--world-json` flag on the CLI bundle that runs `initializeWorld()`, then serializes the world to JSON on stdout and exits. This is a change to the story runner entry point in `packages/sharpee/` — a platform package. It **must be discussed and approved before implementation**. The flag is listed as an entry condition for Phase 3 so the dependency is explicit.

If `--world-json` is not approved, Phase 3 (World Explorer) falls back to static TypeScript source parsing, which is weaker but does not require a platform change. Phase 4 (Entity Autocomplete) similarly has a static-parsing fallback. Fallback approaches are noted in each phase.

## Phases

### Phase 1: Build & Play Integration
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: N/A — tooling
- **Entry state**: Tier 1 extension installed (`sharpee-vscode-0.1.0.vsix`); `package.json` has no `sharpee.buildStory`, `sharpee.playStory`, or `sharpee.playInBrowser` commands; no build task provider exists; status bar shows only transcript test results
- **Deliverable**:
  - New source file `src/build-provider.ts`:
    - `SharpeeTaskProvider` implementing `vscode.TaskProvider` — provides a "Build Story" task that runs `./build.sh -s <story-id>` in an integrated terminal. Story ID resolved from workspace settings (`sharpee.storyId`), falling back to detection from `stories/` subdirectories.
    - Build output parsed with a problem matcher that maps TypeScript compiler errors to source files: pattern `<file>(line,col): error TS<code>: <message>`.
    - Task registered as `"type": "sharpee", "command": "build"` with Ctrl+Shift+B as the default build keybinding.
  - New commands registered in `package.json` and `extension.ts`:
    - `sharpee.buildStory` — "Sharpee: Build Story" — runs `./build.sh -s <story-id>` via the task provider
    - `sharpee.playStory` — "Sharpee: Play Story" — opens an integrated terminal and runs `node dist/cli/sharpee.js --play`
    - `sharpee.playInBrowser` — "Sharpee: Play in Browser" — runs `./build.sh -s <story-id> -c browser`, then opens `dist/web/<story-id>/index.html` via `vscode.env.openExternal`
  - New configuration properties in `package.json`:
    - `sharpee.storyId` — story identifier (e.g., `dungeo`); used by build and play commands; if not set, a quick-pick shows detected stories from `stories/` subdirectories
    - `sharpee.buildScript` — path to build script relative to workspace root, default `./build.sh`
  - Status bar item extended: after a build run, shows `$(pass) Sharpee: Build OK` or `$(error) Sharpee: Build failed` with click-to-open-terminal. When a transcript test runs, reverts to test-result display. Status bar item is now persistent (not just shown when a transcript is open).
  - `package.json` `activationEvents` updated: add `onCommand:sharpee.buildStory`, `onCommand:sharpee.playStory`, `onCommand:sharpee.playInBrowser`
  - Extension rebuilt to `sharpee-vscode-0.2.0.vsix`
- **Exit state**: Author can build from Ctrl+Shift+B and see TypeScript errors as diagnostics; "Play Story" opens an interactive terminal; "Play in Browser" builds and opens the browser client; status bar shows build state

### Phase 2: Native Test Explorer Integration
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: N/A — tooling
- **Entry state**: Phase 1 complete; Tier 1 CodeLens runner works but transcripts do not appear in VS Code's native Test Explorer panel (Activity Bar beaker icon)
- **Deliverable**:
  - New source file `src/test-controller.ts`:
    - `SharpeeTestController` using `vscode.tests.createTestController('sharpee-transcripts', 'Sharpee Transcripts')`
    - File watcher discovers all `*.transcript` files in the workspace and creates a `TestItem` per file, grouped under a root item per directory (e.g., `tests/transcripts/`, `walkthroughs/`)
    - Each transcript file becomes a parent `TestItem`; each `[GOAL: ...]` block within it becomes a child `TestItem` (parsed by reading the file, not by running it)
    - `runHandler`: for a given set of `TestItem`s, runs `node dist/cli/sharpee.js --test <files>` (with `--chain` if multiple walkthrough files selected), parses output using the existing `parseCliOutput` logic from `test-runner.ts`, and reports pass/fail/duration back through the `TestRun` API
    - Failed `TestItem`s include a `vscode.TestMessage` with the failure text and a `Location` pointing to the correct line in the transcript file
    - Watch mode: a `FileSystemWatcher` on `**/*.transcript` re-runs the affected test item on save; re-run is debounced 500ms
  - Existing CodeLens provider (`src/codelens-provider.ts`) retained — the two surfaces are complementary. CodeLens gives in-file run buttons; Test Explorer gives the aggregate panel view and re-run-failures workflow.
  - `package.json` `activationEvents`: add `workspaceContains:**/*.transcript` so the controller activates as soon as a workspace with transcripts is opened (previously activated only when a `.transcript` file was opened)
  - Extension rebuilt to `sharpee-vscode-0.3.0.vsix`
- **Exit state**: All `.transcript` files in the workspace appear in the Test Explorer tree; author can run all, run failures, and see per-goal pass/fail; watch mode re-runs on save

### Phase 3: World Explorer Panel
- **Tier**: Large
- **Budget**: 400
- **Domain focus**: N/A — tooling (requires `--world-json` platform change, see notice above)
- **Entry state**:
  - Phase 1 complete (build command exists, story ID configured)
  - `--world-json` flag approved and implemented in CLI bundle, OR decision made to use static TypeScript parsing fallback
  - A built story bundle exists at `dist/cli/sharpee.js` with the target story loaded
- **Deliverable**:

  **Part A — World JSON extraction (one of two approaches):**

  _Approach A (preferred, requires platform change):_ New CLI flag `--world-json` added to the story runner. When passed, runs `initializeWorld()`, serializes the world model to a JSON structure with the following shape, then exits:
  ```json
  {
    "rooms": [{ "id": "r01", "name": "West of House", "exits": { "north": "r02" }, "sourceHint": "west-of-house.ts" }],
    "entities": [{ "id": "y08", "name": "Elvish sword", "location": "r04", "traits": ["portable", "weapon"] }],
    "npcs": [{ "id": "n01", "name": "Troll", "location": "r12", "behaviors": ["TrollBehavior"] }],
    "actions": [{ "id": "dungeo.action.say", "verbs": ["say"] }]
  }
  ```
  The extension runs `node dist/cli/sharpee.js --world-json --story stories/<id>` and parses stdout.

  _Approach B (fallback, no platform change):_ Static TypeScript source parser in the extension. Scans `stories/<id>/src/**/*.ts` for patterns like `world.helpers().room(...)`, `world.createEntity(...)`, `world.moveEntity(...)`. Extracts names and IDs via regex. Less reliable but requires zero platform work.

  **Part B — World Explorer tree view:**
  - New source file `src/world-explorer.ts`:
    - `WorldExplorerProvider` implementing `vscode.TreeDataProvider<WorldNode>`
    - Tree root has four categories: Rooms, Entities, NPCs, Actions
    - Rooms: each room as a tree item; child items for exits (showing destination room name → ID)
    - Entities: grouped by location (room name); each entity shows name and key traits
    - NPCs: each NPC with behavior names as children
    - Actions: each story-specific action with its registered verbs
    - Tree items have `resourceUri` set to the source file (when known from Approach A's `sourceHint`, or from Approach B's scan), enabling Go to Definition on click
    - Click on any room, entity, or NPC: calls `vscode.commands.executeCommand('vscode.open', uri, { selection: range })` to navigate to the creating code
  - Sidebar view container registered in `package.json`: `"viewsContainers": { "activitybar": [{ "id": "sharpee-explorer", "title": "Sharpee", "icon": "..." }] }`, with view `sharpee.worldExplorer`
  - New command `sharpee.refreshWorldExplorer` — "Sharpee: Refresh World Explorer" — re-runs the world JSON extraction and rebuilds the tree. Also triggered automatically after a successful build.
  - World data cached in memory; stale when the story bundle is newer than the cache timestamp.
  - Extension rebuilt to `sharpee-vscode-0.4.0.vsix`
- **Exit state**: After a successful build, the Sharpee sidebar shows a tree of rooms, entities, NPCs, and actions from the story; clicking a node navigates to the source file

### Phase 4: Entity Autocomplete
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: N/A — tooling (shares the world JSON infrastructure from Phase 3)
- **Entry state**: Phase 3 complete; world JSON (or static parse results) available as in-memory data from `WorldExplorerProvider`
- **Deliverable**:
  - New source file `src/entity-completions.ts`:
    - `SharpeeCompletionProvider` implementing `vscode.CompletionItemProvider`
    - Registered for TypeScript files (`{ language: 'typescript' }`) with trigger characters `'`, `"`, `` ` ``
    - Completion contexts detected by scanning backwards on the current line for known patterns:
      - `destination: ` / `destination: '` → suggest room IDs from world data
      - `world.helpers().room(` → suggest room IDs
      - `keyId: ` → suggest entity IDs where traits include `lockable` or `key`
      - `{ location: ` → suggest room IDs
      - `moveEntity(` → second argument position → suggest room or entity IDs
      - Entity name/alias fields: `'name': '` or similar → no autocomplete (freeform text)
    - Each `CompletionItem` has:
      - `label`: the entity ID (e.g., `r01`)
      - `detail`: the entity name (e.g., "West of House")
      - `documentation`: MarkdownString showing entity type, location, key traits
      - `kind`: `vscode.CompletionItemKind.Reference` for rooms, `.Value` for items, `.Class` for NPCs
    - World data sourced from the same cache used by `WorldExplorerProvider` (shared singleton); if cache is empty, completions silently produce nothing until a build and refresh occur
    - No language server process — completions run in the extension host process using in-memory world data. This keeps the extension lightweight (no LSP infrastructure).
  - New configuration property: `sharpee.enableEntityAutocomplete` (default `true`) — allows authors to disable completions if they find them intrusive
  - `package.json` `activationEvents`: add `onLanguage:typescript` so the completion provider activates when TypeScript files are opened in a Sharpee workspace
  - Extension rebuilt to `sharpee-vscode-0.5.0.vsix`
- **Exit state**: Typing `destination: '` or `moveEntity(entityId, '` in a story TypeScript file offers a completion list of known room and entity IDs with human-readable names and documentation; completions are accurate to the last build

## Phase Dependencies

```
Phase 1 (Build & Play)       — independent of phases 2/3/4; depends only on Tier 1 scaffold
Phase 2 (Test Explorer)      — independent of phases 1/3/4; depends only on Tier 1 scaffold
Phase 3 (World Explorer)     — depends on Phase 1 (build command + storyId config)
                               depends on platform decision re: --world-json flag
Phase 4 (Entity Autocomplete)— depends on Phase 3 (world data cache established)
```

Phases 1 and 2 are independent and can be done in either order, or concurrently in separate sessions. Phase 3 cannot start until Phase 1 exists (it needs the build integration and the `sharpee.storyId` config to know which story to analyze). Phase 4 cannot start until Phase 3 exists (it consumes the same world data cache).

The natural sequence for a single-threaded delivery is: **1 → 2 → (platform decision) → 3 → 4**.

If the `--world-json` platform change is rejected, Phases 3 and 4 can still proceed using Approach B (static TypeScript parsing), but the completions and tree view will be less accurate.
