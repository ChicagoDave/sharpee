## Design decisions pinned before implementation (read before starting Phase 1)

These two decisions were left open by the ADRs ("the implementation" decides); pinning them now so no phase invents silently mid-session:

1. **Where manifest data lives.** `@sharpee/chord` may depend on "nothing platform-runtime" (ADR-210 Packages table) — it cannot import `world-model`, `stdlib`, `plugin-npc`, `plugin-state-machine`, or `extensions/basic-combat`. So the **static manifest data itself lives inside `packages/chord`** (e.g. `packages/chord/src/manifests/combat.ts`, `npc.ts`, `state-machines.ts` — pure data: extension name, kind-noun/adjective words, per-adjective `with`-field names and value types, event/state words; zero platform imports). This is the "names" half of the names-vs-mappings split. The **"mappings" half — which adjective composes onto which actual trait/behavior — lives in `packages/story-loader`** (it already depends on world-model/stdlib/extensions), as a separate loader-internal table the manifest does NOT encode. The **manifest-conformance test therefore lives in `packages/story-loader/tests/`** (the only place that can import both the chord manifest data and the real platform trait/behavior registrations to assert they agree).
2. **Custom-channel declaration spelling is NOT decided.** ADR-216 explicitly left the exact form to the implementation. Phase 6 below opens with an explicit step to draft 2-3 candidate spellings and get David's ratchet sign-off *before* writing any parser code for it — do not invent a spelling silently.

Also pinned: the `client has <capability>` predicate binds to the REAL `ClientCapabilities` interface fields confirmed in code (`images`, `sound`, `music`, `animations`, `video`, `speech`, `splitPane`, `statusBar`, `sidebar`, `clickableText`, `clickableImage`, `dragDrop`, `transitions`, `customFonts`) — **not** the ADR's illustrative list, which named a `layers` flag that does not exist on the interface. Phase 6 confirms the final name set with David if any ambiguity remains rather than silently picking one.

## Dependency summary
- Phase 1 is the prerequisite for everything else — it builds the `use` mechanism, the trusted registry, and the manifest pattern every later phase extends.
- Phases 2 and 3 are independent of each other once Phase 1 is done — either order, or interleaved (this mirrors ADR-235 D3's S2 grouping NPC-auto-wire and state-machine-depth together, but they don't share files or gates).

Phase 6 depends on Phase 4 (channel produce-rule projection reuses the payload evaluator) and Phase 1 (the trusted registry's reserved channel-registration slot); Phase 5 helps but is not strictly required for Phase 6's own fixture if it reuses a built-in renderer.
