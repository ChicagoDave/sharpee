# Session Plan: npm regression suite — complete Sharpee authoring coverage

**Created**: 2026-03-25
**Overall scope**: Extend the existing npm regression story "The Maintenance Facility" to prove that the published `@sharpee/*` packages expose the complete Sharpee authoring surface. The current suite covers 17 features via 15 transcripts. This plan adds ~30 more transcripts across 4 phases, bringing coverage to roughly 45+ features. The story grows from ~780 lines and will be split into multiple TypeScript source files during Phase 1.
**Bounded contexts touched**: N/A — test infrastructure / regression fixture, no domain model changes
**Key domain language**: N/A

## Pre-work findings (verified before planning)

### Packages confirmed published on npm at 0.9.95
- `@sharpee/ext-basic-combat` — FOUND (combat service, CombatantTrait interceptor)
- `@sharpee/plugin-state-machine` — FOUND (StateMachinePlugin, StateMachineRegistry)
- `@sharpee/lang-en-us` — FOUND (already imported as a type-only import in the story)

### Traits confirmed exported from `@sharpee/world-model`
All these are in the `export { ... }` line in the world-model index:
`ClothingTrait`, `EdibleTrait`, `DoorTrait`, `ClimbableTrait`, `PullableTrait`, `AttachedTrait`, `PushableTrait`, `ButtonTrait`, `MoveableSceneryTrait`, `WeaponTrait`, `BreakableTrait`, `DestructibleTrait`, `CombatantTrait`, `EquippedTrait`, `VehicleTrait`, `EnterableTrait`

### Grammar coverage gaps (must add in extendParser)
The stdlib grammar (`packages/parser-en-us/src/grammar.ts`) does NOT register:
- `if.action.climbing` — no "climb X" pattern
- `if.action.smelling` — no "smell X" pattern
- `if.action.listening` — no "listen to X" pattern
- `if.action.talking` — no "talk to X" pattern

These four must be registered in the story's `extendParser()`.

### Annotations verdict
Annotations (ADR-124) are metadata only — `entity.annotate()` attaches data but there is no rendering assertion in transcripts (no text output to check). The test approach is to call `getAnnotations()` / `getActiveAnnotations()` from a custom action that dumps the annotation count as a message. One transcript can verify the round-trip.

### Save/restore verdict
`engine.save()` / `engine.restore()` return `Promise<boolean>` and are exposed through `GameEngine`. However, the transcript tester runs synchronous commands — it cannot interleave a `save` command mid-run, wait for the file I/O, and then restore into the same test. Testing save/restore requires a two-transcript `--chain` run, which the requirements explicitly prohibit ("each transcript tests one feature in isolation, no --chain"). Save/restore is therefore deferred and noted as out-of-scope for this plan.

### Pronoun resolution
Pronoun resolution ("take it", "drop it") is handled automatically by the parser. No story-side setup is needed. A single transcript can verify it.

## Source file split strategy

The story is currently one file (`npm-test/src/index.ts`, ~780 lines). By Phase 3 it would exceed ~2000 lines with all additions. Plan: split into modules during Phase 1, all imported and re-composed by `index.ts`.

Proposed layout:
```
npm-test/src/
  index.ts          — story class, imports from all modules
  traits.ts         — all custom trait classes (InspectableTrait + new ones)
  behaviors.ts      — all capability/combat behaviors
  actions.ts        — all custom actions (inspect, ping + new ones)
  npcs.ts           — NPC behavior definitions
  world-setup.ts    — initializeWorld() split into region helper functions
```

## Notes on deferred items

- **Save/restore**: Requires `--chain` across two transcript files (save state in first, restore in second). The constraints say no `--chain` between transcripts. Deferred until the transcript tester supports a within-transcript save/restore assertion.
- **EquippedTrait**: Relates to wielding weapons in combat. Can be added to Phase 3 combat test if the trait interacts cleanly with `registerBasicCombat()`, but it may just be a data trait with no parser verb. Added as a stretch goal in Phase 3.
- **Implicit take**: The `requireCarriedOrImplicitTake()` API on `ActionContext` handles this automatically. A transcript that says "unlock toolbox" when the key is nearby (not in inventory) would trigger an implicit take. Can be added as an extra transcript in Phase 1 (`21b`) if the implicit-take pathway is confirmed to produce detectable output.
