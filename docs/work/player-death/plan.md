# Session Plan: Implement ADR-224 (Conditional/Hazard Death — a First-Class Player-Death Model)

**Created**: 2026-07-15
**Overall scope**: Build the platform's single player-death primitive (`killPlayer` → canonical `if.event.player.died` event → engine game-over routing with story-policy veto ordering), layer the two reusable trigger shapes ADR-224 requires (deadly-room verb-allowlist, seeded-probabilistic), re-point ADR-215 combat's lethal path onto it, fold Dungeo's ten-plus hand-rolled dead-flag call sites into it (five named in the ADR, more found by grounding), and give the Chord Way (`kill the player when`, `deadly` room marker) the same surface — closing the platform/Chord parity gap the 2026-07-15 capability audit flagged (row #16).
**Bounded contexts touched**: `packages/world-model` (new `DeadlyRoomTrait`; reads `HealthTrait`/`HealthBehavior`, ADR-226, already landed), `packages/stdlib` (new `death/` module: `killPlayer`, seeded-probability helper, deadly-room interceptor; `channels/standard.ts` re-point; `actions/standard/attacking/attacking.ts` re-point), `packages/engine` (`game-engine.ts` game-over routing + veto ordering), `packages/story-loader` (Chord `kill the player when` + `deadly` room marker lowering), `stories/dungeo` (fold falls/gas/grue/balloon/cage + extra sites into `killPlayer`; re-point `death-penalty-machine.ts` reincarnation policy).
**Key domain language**: canonical death event (`if.event.player.died`), `killPlayer`, terminal vs. reincarnate intent, deadly-room verb-allowlist, seeded-probabilistic hazard, story-policy veto ordering ("first crack"), cause (`combat`/`gas`/`grue`/`fall`/`balloon`/`poison`).

## Phases

### Phase 1: The canonical death primitive — `killPlayer`, event, engine routing
- **Tier**: Medium
- **Budget**: 250 tool calls
- **Domain focus**: The platform primitive ADR-224 Decisions 1–2 require. Purely additive at this point — nothing calls it yet, so no existing death mechanism is disturbed.
- **Entry state**: David's explicit go-ahead to implement ADR-224 has been given (see Entry gate). `HealthTrait`/`HealthBehavior` exist and are exported (ADR-226, complete).
- **Deliverable**:
  - `packages/stdlib/src/death/kill-player.ts` — `killPlayer(world, playerEntity, {cause, messageId?, terminal=true})`: resolves the player's `HealthTrait` (lazily attaches one if absent — ADR-223 AC-1 caveat), calls `HealthBehavior.kill(healthTrait, cause)`, constructs and returns/emits a semantic event typed `if.event.player.died` with payload `{cause, messageId, terminal}`. Idempotent: if the player is already `dead`, no-op (no re-emission, no double game-over routing).
  - `packages/stdlib/src/channels/standard.ts:64-69,300-313` — retire `STANDARD_CHANNEL_EVENTS.PLAYER_DIED = 'combat.player_died'`; re-point `deathChannel` to scan for `if.event.player.died`. This is the hard cutover the ADR's Consequences section requires — no alias left behind.
  - `packages/engine/src/game-engine.ts` — after the event-processor dispatches `if.event.player.died` to all registered handlers for the turn (this is where a story `plugin-state-machine` listening on the canonical event gets "first crack" — ordinary handler dispatch, no new veto API needed), the engine re-checks `HealthBehavior.isAlive(player's HealthTrait)`. If still dead, route to the existing `game.lost` path (`createGameLostEvent`, the same call already at `:818`). If a handler reset `dead=false` during dispatch, `game.lost` is skipped — the *re-check of live derived state*, not a payload flag, is the engine's "final word" per ADR-224 Q-2's explicit text ("the `terminal` field... is the story policy's signal, not the engine's final word").
  - Behavior Statement (CLAUDE.md rule 12) for `killPlayer` and for the engine's post-dispatch routing check, before writing tests.
- **Behavior Statements**:
  - **`killPlayer`** — DOES: sets `HealthTrait.dead=true`+`causeOfDeath=cause` on the player (lazily attaching `HealthTrait` if absent) and emits `if.event.player.died` with `{cause, messageId, terminal}`. WHEN: called by any hazard/combat call site with a cause. BECAUSE: single canonical death primitive — no call site hand-mutates a dead flag or hand-emits an event. REJECTS WHEN: player already `dead` — no-op, does not re-emit.
  - **Engine death routing** — DOES: after dispatching `if.event.player.died` to registered handlers, re-checks the player's derived `isAlive`; if still dead, emits `game.lost` via the existing path. WHEN: any `if.event.player.died` event flows through a turn's event processing. BECAUSE: AC-3's ordering contract. REJECTS WHEN: a handler reset the player to alive during dispatch — `game.lost` does not fire, regardless of the event's `terminal` payload value.
- **Tests** (assert on actual `HealthTrait`/event/game-state mutation, not return values alone):
  - `killPlayer` attaches `HealthTrait` if missing, sets `dead`+`causeOfDeath`, emits the event with correct payload (ADR-223 AC-1 caveat).
  - `killPlayer` is idempotent when called twice on an already-dead player.
  - Engine integration (AC-1, synthetic): a fixture interceptor calls `killPlayer({cause:'gas', terminal:true})` mid-turn → assert the event's `cause:'gas'` payload and that the turn ends in `game.lost`.
  - Engine integration (AC-3, synthetic): register a fake handler on `if.event.player.died` that resets `dead=false` and repositions the player *before* the engine's post-dispatch check runs → assert `game.lost` does **not** fire and the player is alive and relocated.
- **Exit state**: `killPlayer` and engine routing exist, are exported/wired, and pass the above tests. `packages/stdlib`, `packages/world-model`, `packages/engine` build clean. `combat.player_died` no longer appears in any live emission path.

- **Budget**: 250 tool calls

A general `HazardTrait` is deferred, not built here, unless Phase 4 discovers a shape that doesn't fit (flag and discuss, don't force it).
- **RESOLVED SEAM (2026-07-15, grounded + David-approved).** The plan's original "interceptor registered via the ADR-208 interceptor registry" assumption is **wrong** and was corrected before coding: ADR-208 action interceptors resolve per *direct-object entity + action* (`getInterceptorForAction(entity, actionId)`), so they cannot fire on objectless verbs (WAIT, INVENTORY) that a deadly-room allowlist must catch. The idiomatic seam — the one the current falls handler already uses — is a **`ParsedCommandTransformer`** (`@sharpee/engine`, runs after parse / before validate) that redirects a non-safe verb to a **new generic platform death action `if.action.deadly_room_death`** whose `report()` calls `killPlayer`. The transformer is **auto-registered by the engine** (like `SceneEvaluationPlugin`) so every story — TS or Chord — gets it for free; it early-returns when the player's room has no `DeadlyRoomTrait`. The seeded RNG for the `chance` variant is the engine's `this.random`, injected at auto-registration. Grounding: falls uses exactly this pattern (`falls-death-handler.ts` transformer → `falls-death-action.ts`, which *already emits the canonical `if.event.player.died`*), so this folds falls verbatim at platform level. David approved Design A (transformer → generic death action) over a before-action-hook variant.
- **Entry state**: Phase 1 complete — `killPlayer` and engine routing exist and are tested.
- **Deliverable**:
  - `packages/world-model/src/traits/deadly-room/deadlyRoomTrait.ts` — data-only: `cause: string`, `messageId?: string`, `safeVerbs: string[]`, optional `chance?: number` (0–1, enables the probabilistic variant reuse for grue-shaped hazards). Wired through the full root-barrel discipline (`traits/deadly-room/index.ts` → `traits/index.ts` → `src/index.ts`) — the exact ADR-218 gap ADR-226 Phase 1 flagged; verify with a root-import smoke test before moving on.

WHEN: called by the interceptor on every parsed command while the player occupies a `DeadlyRoomTrait` room. BECAUSE: single reusable trigger shape folding both the falls verb-allowlist and (via `chance`) grue-style probability, per ADR-224 Decision 3. ## Notes for the implementer

- Phases 1–3 are platform-only (`world-model`, `stdlib`, `engine`); Phase 4 is Dungeo-only; Phase 5 is story-loader + a Chord fixture. No phase mixes platform and story code — matches the health-trait plan's isolation discipline for clean build/test cycles per phase.
- Phase 2's `DeadlyRoomTrait` vs. a general `HazardTrait` is this plan's resolution of ADR-224's deferred Q-4 — a narrow trait, matching the ADR's own AC-2 wording. Both are marked "ground at phase start" in their phase's Deliverable — resolve them with a quick read before design, not by guessing.
