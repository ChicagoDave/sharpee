# Session Plan: Chord Topic Conversation (ADR-239 implementation)

> **REVISED 2026-07-18 (same session, ADR-239 D3 amendment)**: David superseded the per-topic clause-header spelling with an explicit table construct — **`define topics for <entity> … end topics`**, a define-family block holding any number of `about` rows (topics + responses in one place; one-line `phrase` responses or indented statement bodies; no `from`/`it` anywhere; a second block for the same entity is a compile error). The ruled `about the <entity>` / `about "<text>"` forms survive as row keys. Phase 1 below is rewritten accordingly; the biggest simplification: **no on-clause grammar changes at all** — `OnClause` is untouched, `analysis.duplicate-clause` needs no relaxation (the unfiltered `on asking it` catch-all remains the only clause, exactly one per owner as today). Phase 2's platform-seam stop-and-ask and lookup/suppression semantics are unchanged.

**Created**: 2026-07-18
**Overall scope**: Give Chord (`.story` author language) a topic-conversation surface — closing the parity audit's last two ⚠️ rows (asking, telling) and the go-live gate's G1 line for topic surface. The surface is the **`define topics for <entity>` table block** (D3 as amended): any number of rows keyed `about the <entity>` (entity tier — the platform's quiet `topicEntityId` resolution) or `about "<text>" [or "<text>" …]` (free-text tier with declared aliases), each row a one-line `phrase` response or an indented statement body. The table is the closed, compile-visible set of what the entity can be asked/told about (D4 — lookup, never fuzzy); the existing unfiltered `on asking it` clause is the catch-all, firing only on a miss and fully suppressed on a hit (never appended — the ISSUE-074/rug lesson, decided explicitly this time). The same table serves `tell` (D1). This is a **Chord + story-loader surface**, not a new platform feature — the platform (parser topic grammar, `if.event.asked`/`if.event.told` payload) already ships (ADR-231 D4). Grounding for this plan found **one small platform seam ADR-239 did not anticipate**: see "Platform-seam note" below — it is surfaced as a stop-and-ask, not silently bridged.
**Bounded contexts touched**: `packages/chord` (parser: the `define topics for … end topics` block + `about` rows + alias lists; `ast.ts`/`ir.ts`: new DefineTopics/IR topic-table shapes lowered onto the owner (`IREntity.topics`, additive); analyzer: D4's table gates), `packages/story-loader` (runtime dispatch rework — per-owner topic-table lookup + catch-all suppression inside the existing interceptor arm; row bodies execute through the shared body machinery), `packages/stdlib` (the one platform-seam addition, discussed-first per CLAUDE.md — see below), `docs/reference/chord-grammar.md` + `chord.ebnf` + `docs/architecture/chord-grammar-changes.md` (ratchet row) + `docs/work/stdlib-reference/chord-availability-audit.md` (closure).
**Key domain language**: topic table (the entity's `define topics for …` block — a definition, not a pattern, D4), entity tier / free-text tier (the two header forms), table hit / table miss (the lookup outcome that selects catch-all vs. matched entry), catch-all (`on asking it` — fires only on a miss), alias (a declared alternate quoted phrasing, fixed at ratchet time, D4), override (the existing ADR-228 `postReport` mechanism that lets a matched entry fully own the response), normalized lookup (case-insensitive, article-stripped, whole-topic equality — never fuzzy).

## Platform-seam note (read first)

ADR-239's Context section states the platform half is complete and "no stdlib/parser change is expected; if implementation finds a platform seam missing, it is raised to David, never silently bridged." Grounding for this plan (2026-07-18) found exactly one:

- `askingAction.report()` (`packages/stdlib/src/actions/standard/asking/asking.ts`) already computes `topic` and `topicEntityId` and puts them on the emitted `if.event.asked` event's `data` — but `ActionInterceptor.postReport(target, world, actorId, data)` (`packages/world-model/src/capabilities/action-interceptor.ts:465`) never sees the emitted event. Its only channel is `data: InterceptorSharedData` (a free-form bag seeded at `resolveLifecycle` time via each slot's optional `seedData(context, entity, multiObjectItem)` hook — `packages/stdlib/src/actions/lifecycle/descriptor.ts:78`).
- `askingLifecycle` / `tellingLifecycle` (same files) declare a `target` slot with `resolve` but **no `seedData`** — so `context.command.topic` never reaches the sharedData bag, and a Chord on-clause's `postReport` hook has no way to read what topic was asked about. Without this, D4's table lookup cannot be implemented at all.
- The fix is small, additive, and uses the exact extension point ADR-228 designed for "command-dependent context" (the doc comment's own example is symmetric cross-entity seeding for putting/inserting) — not a new mechanism: `askingLifecycle`/`tellingLifecycle`'s `target` slot gains `seedData: (ctx) => ({ topic: ctx.command.topic?.text, topicEntityId: ctx.command.topic?.entity })`. No behavior changes for any existing consumer; the field is simply not populated today.
- **This is still a `packages/stdlib` source change**, and CLAUDE.md requires platform changes to be discussed before implementation. It is embedded as Phase 2's entry-state stop-and-ask below — confirm the fix with David before Phase 2's runtime dispatch work lands, rather than treating ADR-239's "zero platform change" framing as settled fact.
- **Good news found alongside it**: D5's "matched entry fully owns the response, catch-all suppressed, never appended" requirement needs **no other platform work**. `InterceptorReportResult.override` (`messageId`/`params`/`text`) already exists and already fully replaces the primary event's message (`applyInterceptorReportResult`, `packages/world-model/src/capabilities/action-interceptor.ts:181`) — it is the exact mechanism every other `on <verb> it` clause already uses to override its action's default message (documented contract, `on` overrides / `after` appends — `packages/story-loader/src/runtime.ts:406`). `askingAction.report()`'s unconditional `unknown_topic` primary messageId is already the correct fallback for "no clause matched at all" — nothing about the default-message path needs to change. The seam above (topic reaching `data`) is the only missing wire.

## Code grounding (verified this session, 2026-07-18)
- `askingAction.report()` / `tellingAction.report()` (`packages/stdlib/src/actions/standard/asking/asking.ts`, `.../telling/telling.ts`) already emit `if.event.asked`/`if.event.told` with `{ topic, topicEntityId }` on the event `data`, and set the primary `messageId` unconditionally to `${action.id}.unknown_topic` / `.not_interested` — confirming that path needs no change; only the interceptor sharedData wiring is missing (Platform-seam note above).
- `on asking it` / `on telling it` today run through the **ADR-228 interceptor path**, not the event-clause path (`EVENT_TRIGGERS`/`REGION_EVENT_TRIGGERS` in `packages/story-loader/src/event-contract.ts` cover only `entering`/`leaving` — asking/telling are absent). `isConsultedGerund` (`runtime.ts:253`) checks `interceptorConsultingActionIds` (`packages/stdlib/src/actions/lifecycle/registry.ts:135`), which does include `if.action.asking`/`if.action.telling` — confirming the clauses ARE wired today, just through interceptors, not events. This corrects an assumption in this plan's own grounding brief (the regions `crossingRegionId`-style payload-keyed event-clause filter is the WRONG precedent to extend; the right precedent is the `postReport`/`override` interceptor contract).
- `buildDispatchingInterceptor` (`runtime.ts:377-400`) and its caller (`runtime.ts:~160-180`) group clauses **globally by gerund action** (`byAction.get(clause.action)`) into ONE `registerActionInterceptor` call per gerund, and `armFor` (`runtime.ts:383`) picks the **first** clause whose entity matches the target — i.e., today's model is "at most one `on asking it` per owner" (enforced by `analysis.duplicate-clause`, `analyzer.ts:873`). **[Post-amendment note: this stays true.** Topic responses are table ROWS, not clauses — the arm consults the owner's table before falling back to its single catch-all clause; `analysis.duplicate-clause` needs no relaxation.**]**
- `OnClause` (`packages/chord/src/ast.ts:337-355`) / `IROnClause` (`ir.ts:207-232`) / `parseOnClause` (`parser.ts:1983-2052`) — **untouched post-amendment** (the original clause-header design would have grown them; the table block is a new top-level define construct instead, clean slate, no dormant `about` or `topics` reservation anywhere in chord/src — verified).
- Inside a clause body, `ExecContext.it` is already bound to the owner entity regardless of header form (`{ world, it: entity.id }`, set identically in every `buildInterceptor` hook) — confirms D3's "owner implicit, `it` inside filtered bodies still binds to the owner" needs no new mechanism, only the header parse.
- `analysis.duplicate-clause`'s existing message ("a second one would silently mask it") is the right shape to extend for D4's own two overlap gates (duplicate quoted entry per owner; quoted entry colliding with an entity's name/aka used in an entity-tier clause on the same owner) — same diagnostic family, new fingerprint dimension (topic identity, not just clauseKind+action).

## Phases

### Phase 1: Chord grammar — the `define topics` table block, compile gates (D2, D3 as amended, D4's syntax half, AC-3)
- **Tier**: Large
- **Budget**: 350
- **Domain focus**: `packages/chord` only — parser (the `define topics for <entity> … end topics` block; `about` rows in both tiers; `or`-joined alias lists; one-line vs indented-body responses), `ast.ts`/`ir.ts` (new `DefineTopics` declaration; IR lowered onto the owner — `IREntity.topics: IRTopicRow[]`, additive, e.g. `{ filter: { kind: 'entity'; id: string } | { kind: 'text'; primary: string; aliases: string[] }; body: IRStatement[]; span }`), analyzer (D4's table gates). **No on-clause grammar changes**: `OnClause` is untouched; `analysis.duplicate-clause` stays exactly as it is (the unfiltered `on asking it` catch-all remains the only clause form, one per owner as today).
- **Entry state**: ADR-239 ACCEPTED and D3-amended (2026-07-18). No dependency on Phase 2 — Chord-only, so it can proceed without waiting on the platform-seam confirmation.
- **Embedded stop-and-ask (D4 alias spelling)**: before landing the alias-list parser code, confirm the exact separator spelling with David (the amended D3 shows `or` between quoted forms — e.g. `about "treasure" or "the hoard":` — as the working example, "fixed at ratchet time"). Quick confirm, not a design question.
- **Deliverable**:
  - Block grammar: `define topics for <name>` opens the block; body rows are `about the <entity>: <response>` (entity tier — a `NameRef` resolved like any entity reference) and `about "<text>" [or "<text>" …]: <response>` (free-text tier, declared aliases). `<response>` is either a one-line statement (the common `phrase <key>` case) or an indented statement body on the following lines (the same statement kit clause bodies use; `it` inside a row body binds to the owner). `end topics` closes. Multiple rows per block is the structural point; parse an empty block to a named error.
  - Analyzer gates, each a named diagnostic with a rejection test (AC-3):
    1. `analysis.duplicate-topic` — the same entity, or the same normalized quoted text (primary or alias), declared twice within an owner's table (D4 "closed set" — no accidental double-answer).
    2. `analysis.topic-entity-collision` — a quoted entry whose normalized text matches the name/aka of an entity used in an entity-tier row of the same table (D4's cross-tier gate — collisions are compile errors, no runtime tie-break).
    3. `analysis.duplicate-topics-block` — a second `define topics` block for the same entity (the define-family duplicate discipline; the table lives in one place).
    4. `analysis.topics-host` — `define topics for <X>` where X is not a person-kind entity (asking/telling target actors; a table on a rock is declared-but-unanswerable, the region-memberless class). Unknown `for` names take the standard unresolved-entity error.
  - Ratchet row **the `define topics` block** (D2) — transcribed into `docs/architecture/chord-grammar-changes.md` this phase (approval = ADR-239's acceptance + D3 amendment, both David's, already recorded).
- **Test gate (rule 13a, mutation-verification convention)**: chord unit tests assert on the lowered `IREntity.topics` shape for both tiers incl. aliases and a body-form row (not just "parsed OK"); one rejection test per gate above asserting the specific diagnostic code; IR golden churn verified additive-only (`topics: []` on existing entities). `pnpm --filter '@sharpee/chord' test` green.
- **Exit state**: AC-3's diagnostic coverage fully in place. Chord parses, resolves, and lowers the topic table into IR; every table violation is a compile error with a specific code. No runtime behavior yet — story-loader does not consume `IREntity.topics` until Phase 2.

### Phase 2: story-loader runtime — table lookup, catch-all suppression, telling symmetry (D1, D4's runtime half, D5, AC-1, AC-2, AC-4, AC-5)
- **Tier**: Large
- **Budget**: 350
- **Domain focus**: `packages/stdlib` (the one platform-seam addition, discussed-first), `packages/story-loader` (`buildDispatchingInterceptor`/`armFor` rework to a per-owner table+catch-all dispatch, normalized lookup).
- **Entry state**: Phase 1 complete (IR carries `topicFilter`). **Platform-change stop-and-ask (CLAUDE.md)**: confirm with David before this phase's stdlib edit lands — `askingLifecycle`/`tellingLifecycle` gain a `seedData` hook on their `target` slot (see Platform-seam note above) so `context.command.topic`/`topicEntityId` reach `InterceptorSharedData`. This is the one platform touch this plan needs; nothing else in `packages/stdlib` changes. Confirmation is a go/no-go gate for this phase, not optional groundwork.
- **Deliverable**:
  - `askingLifecycle`/`tellingLifecycle` (`packages/stdlib/src/actions/standard/asking/asking.ts`, `.../telling/telling.ts`): `target` slot gains `seedData: (ctx) => ({ topic: ctx.command.topic?.text, topicEntityId: ctx.command.topic?.entity })`.
  - `packages/story-loader/src/runtime.ts`: for `asking`/`telling` specifically, the owner's arm grows a table: it holds the owner's `IREntity.topics` rows (entity-tier + free-text-tier, each with declared aliases) plus the owner's optional catch-all clause (`on asking it` — still at most one, the existing `analysis.duplicate-clause` contract untouched). Row bodies execute through the shared statement machinery (the `execMachineBody`-class seam), `it` bound to the owner. The arm's `postReport`:
    1. Normalizes the asked topic (case-insensitive, leading-article-stripped — reuse the existing entity-name normalization helper, not a new one) — D4's "same normalization entity-name resolution already applies."
    2. Looks up the normalized topic against the owner's declared table: entity tier first (topic resolved to `topicEntityId` matching a declared entity), then free-text tier (exact normalized-text or alias match) — whole-topic equality only, never substring/word-subset.
    3. **Hit**: runs the matched ROW's body, returns its `override` (or `emit`) as today's single-clause `on` behavior already does — the catch-all clause does not run at all (D5 — suppression, not append).
    4. **Miss**: if a catch-all (`on asking it`) is declared on the owner, runs it exactly as today's single-clause path does. If none is declared, returns `{}` — the primary event's already-unconditional `unknown_topic`/`not_interested` messageId stands untouched (no platform work needed for this branch, confirmed in the Platform-seam note).
  - `on telling about …` reuses the same dispatch code (D1 — gerund-parameterized, not asking-specific) — this phase proves it, it does not reimplement it a second time.
- **Test gate (REAL-PATH, rule 13a)**: new fixture(s) (e.g. `packages/chord/tests/fixtures/topic-basic.story` — an NPC with one entity-tier entry, one free-text entry with a declared alias, and a catch-all) loaded into a real `WorldModel`; `packages/story-loader/tests/topic-dispatch.test.ts` drives the real `askingAction` (and `tellingAction`) end to end:
  - AC-1: entity-tier hit and free-text-tier hit each produce the entry's own message id/params (asking AND telling).
  - AC-2: the declared alias reaches the same response as its primary; a plausible but undeclared paraphrase does NOT match (asserted both ways — lookup, not search).
  - AC-4: a topic matching no table entry fires the catch-all when one is declared; with no catch-all declared, the standard `unknown_topic` default fires; a table hit is proven to suppress the catch-all entirely (a fixture with both a matching entry and a catch-all — assert only the entry's message appears, never both, never neither).
  - AC-5: `on telling about …` proven symmetric — one test per tier, mirroring the asking assertions.
  - A rejection/regression test pinning the `seedData` addition's non-effect on any other consumer of `askingLifecycle`/`tellingLifecycle` (no existing test's sharedData assumptions change).
- **Exit state**: AC-1, AC-2, AC-4, AC-5 all green with REAL-PATH assertions. `pnpm --filter '@sharpee/stdlib' test` and `pnpm --filter '@sharpee/story-loader' test` green. `packages/chord` unaffected (no rerun needed beyond the standing gate).
