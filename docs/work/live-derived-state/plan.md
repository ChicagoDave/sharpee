# Session Plan: Live Derived State (ADR-240 implementation)

**Created**: 2026-07-18
**Overall scope**: Implement ADR-240 — replace the story-loader's cache-and-invalidate derived-property mechanism (`ChordRuntime.recomputeDerived`, stamped into `RoomTrait.blockedExits`/`requiresLight`, invalidated off an eleven-event `chainEvent` trigger list) with point-of-use evaluation through one generic per-world evaluator registry. This closes the staleness bug class found live during the Fernhill tutorial's Kettle guard beat: a Chord `change` statement outside the enumerated trigger list left a blocked exit stale forever. This is a **platform workstream** (`packages/world-model`, `packages/stdlib`, `packages/story-loader`) — a detour discovered by, and blocking, `docs/work/chord-tutorial-story/plan.md`'s Phase 4 (NPCs & conversation, CURRENT — the Kettle guard beat). The umbrella above both is `docs/work/chord-go-live/plan.md` (ADR-233); this workstream does not touch the umbrella directly.
**Bounded contexts touched**: `packages/world-model` (the registry itself — forced by dependency direction, ADR-240 D6; `VisibilityBehavior`'s darkness read points), `packages/stdlib` (`going`'s blocked-exit read points), `packages/story-loader` (registration replaces stamping; the deleted recompute mechanism). Chord and the IR do not change — the surface's existing promise ("the refusal applies only WHILE the condition holds") is finally kept structurally, not altered.
**Key domain language**: evaluator registry (`world.registerEvaluator`/`world.evaluate`, D6 — the one generic per-world seam, idempotent last-wins per the ADR-207/208 binding-map convention), key builder (each read point's module owns and exports its own key-string constructor — `exit.blocked.<roomId>.<direction>`, `exit.message.<roomId>.<direction>`, `dark.<roomId>`), point-of-use evaluation (D1 — no cache, no invalidation list, ever), derived vs. static (a `dark while`/`is blocked while` condition is derived and lives only in the registry; a bare `dark`/unconditional block is a plain trait fact, stamped once as today).

## Code grounding (verified this session, 2026-07-18)
- **The registry belongs on `WorldModel` itself**, mirroring `registerCapabilityBehavior`/`registerActionInterceptor` exactly: interface + doc comments at `packages/world-model/src/world/WorldModel.ts:225-332`, private-Map-backed implementation at `WorldModel.ts:670-770`, delegated forwarding at `packages/world-model/src/world/AuthorModel.ts:381-410`. `registerEvaluator(key, fn)`/`evaluate(key)` follow the identical idempotent-last-wins, per-instance-Map, "never throw on missing binding — caller's normal fallback handles it" shape.
- **`VisibilityBehavior.requiresLight` has THREE separate read sites**, not one, despite `isDark()`'s docstring calling itself "the single source of truth": `isDark()` (`packages/world-model/src/world/VisibilityBehavior.ts:39-45`), `canSee()` (`:123-142`), and `getVisible()` (`:171-177`) each independently read `roomTrait.requiresLight`. All three need the `dark.<roomId>` consult-first-then-fallback treatment (D3) — missing one leaves a stale read path alongside two fixed ones. Whether to also collapse the three into one shared consult helper is an implementation-time simplification opportunity (cohesive modularity), not a scope change.
- **`RoomTrait.requiresLight` is stamped ONLY for static dark today** (`packages/story-loader/src/loader.ts:930`: `requiresLight: irEntity.traits.some((t) => t.name === 'dark' && t.condition === null)`) — a `dark while` room's `requiresLight` starts `false` and was previously flipped live by the now-deleted `recomputeDerived` (`packages/story-loader/src/runtime.ts:1193-1199`, `room.requiresLight = this.evaluator.evalCondition(condition, { world })`). Post-D4, a `dark while` room's `requiresLight` trait fact stays permanently `false` — the room is dark **only** via the registry. The `undefined`-means-"fall through" contract in D3's read points must therefore mean "no evaluator registered → trust `requiresLight` exactly as it is stamped," which is correct for both the static-dark case (`true`, no registry entry) and the plain-lit case (`false`, no registry entry); a conditional room ALWAYS has a registry entry once Phase 2 lands, so its stamped `false` is never consulted.
- **`going`'s blocked-exit state has TWO independent read sites**, both of which currently call `RoomBehavior.isExitBlocked`/`getBlockedMessage` (the `RoomTrait.blockedExits` map) and BOTH must consult the registry first, or a Chord-authored block goes silently unblocked in one path while correctly blocked in the other: `resolveExitEntities()` inside `goingLifecycle`'s slot resolution (`packages/stdlib/src/actions/standard/going/going.ts:109-120`, feeds the interceptor door/destination slots) and `validate()`'s own blocked-exit check (`going.ts:245-252`, the actual refusal + message). Since D2 removes ALL `RoomTrait.blockedExits` stamping for Chord stories (conditional AND unconditional alike), after Phase 2 lands the trait-map fallback is **permanently empty for every Chord-authored story** — it only remains live for hand-written TS stories that call `RoomBehavior.blockExit` directly (per the ADR's explicit "leaving TS stories untouched" clause). Both call sites must be updated in the same phase or the interceptor slot machinery and the actual refusal will disagree.
- **The blocked-exit message is currently a load-time, single-variant snapshot**, not phrase-strategy-aware: `RoomBehavior.blockExit()` stores a literal string computed once via `loader.ts`'s `phraseText()` (`variants[0].text` only — no strategy selection, no per-attempt variety) for unconditional blocks, and via `runtime.ts`'s now-deleted `blockedPhraseText()` (identical single-variant read) for conditional ones. D6's `exit.message.<roomId>.<direction>` key is explicitly required to resolve **at refusal time** so phrase strategies vary per attempt (matching ADR-211/212 render-time semantics) — this is a real behavior fix, not just a mechanism swap: the registered evaluator closure must call the runtime's live phrase-resolution path (`ChordRuntime.resolvePhraseKey`, `runtime.ts:1524-1527`, entity-scoped-override-aware) at consult time, not a value frozen at load.
- **Deletion surface confirmed by direct grep** (not by the ADR's audit claim alone): `recomputeDerived`, `hasConditionalBlockedExits`, `derivedDarkRooms`, and `blockedPhraseText` live only in `packages/story-loader/src/runtime.ts` (`:1178-1228`, plus the `if (this.derivedDarkRooms().length > 0 || this.hasConditionalBlockedExits())` gate and the eleven-event `chainEvent` loop at `:240-259`); `packages/story-loader/src/loader.ts` references the deletion only via the unconditional-block stamping path (`:408-415`) that D2 replaces. Two test files reference the deleted symbols/behavior directly and need updating, not just the source: `packages/story-loader/tests/hatch-context.test.ts`, `packages/story-loader/tests/runtime.test.ts` — read them at Phase 2 start to determine update vs. replacement-with-a-live-path test.
- **`is deadly while` is confirmed unwired and out of scope**: `loader.ts:428-434` throws a load-time `LoadError` for any conditional `deadlyExits` entry today ("is not wired yet... post-scope") — nothing to touch; matches D4's exemption note verbatim.

- **Deliverable**:
  - `WorldModel.registerEvaluator(key: string, fn: (world: WorldModel) => unknown): void` / `WorldModel.evaluate(key: string): unknown | undefined` — a new per-instance `Map<string, (world: WorldModel) => unknown>`, idempotent last-registration-wins, doc-commented and delegated through `AuthorModel` exactly like `registerCapabilityBehavior`/`registerActionInterceptor` (`WorldModel.ts:225-340,670-770`; `AuthorModel.ts:381-410` are the precedent to mirror, not reinvent).
  - `VisibilityBehavior` exports its key builder (`darkKey(roomId: string): string` returning `dark.<roomId>`) from its own module (D6: "key conventions are owned by each read point's module"). All three `requiresLight` read sites (`isDark`, `canSee`, `getVisible` — `VisibilityBehavior.ts:39-45,123-142,171-177`) consult `world.evaluate(darkKey(room.id))` first; `undefined` (nothing registered) falls through to `roomTrait.requiresLight` exactly as today; a `boolean` result from the registry is authoritative.
  - `going` (or its `-data.ts` sibling, per the package's 4-file action convention) exports `exitBlockedKey(roomId, direction)` / `exitMessageKey(roomId, direction)` key builders (`exit.blocked.<roomId>.<direction>` / `exit.message.<roomId>.<direction>`). Both `resolveExitEntities()` (`going.ts:109-120`) and `validate()`'s blocked check (`going.ts:245-252`) consult `exitBlockedKey` first; `undefined` falls through to `RoomBehavior.isExitBlocked`/`getBlockedMessage` unchanged (the TS-story path, D2's explicit "leaving TS stories untouched"). A registry `true` result resolves its message via `exitMessageKey` before falling back to `RoomBehavior.getBlockedMessage`; the event shape (`GoingMessages.MOVEMENT_BLOCKED`, `params: { direction, message }`) does not change.
- **Test gate (rule 13a, mutation-verification convention)**: `packages/world-model` unit tests assert directly on `WorldModel`'s registry state (idempotent overwrite on re-registration with the same key; `evaluate()` returns `undefined` for an unregistered key, not a thrown error or `false`) and on `VisibilityBehavior`'s three call sites (a registered `dark.<roomId>` evaluator returning `true`/`false` overrides a room's stamped `requiresLight` in both directions; an unregistered room's stamped fact is used unchanged). `packages/stdlib` `going` tests assert the same shape for both call sites (a registered `exit.blocked.*` evaluator blocks/unblocks a room with NO `RoomTrait.blockedExits` entry at all — proving the registry path is truly independent of the trait map — and its message comes from `exit.message.*` when registered). Existing `world-model`/`stdlib` suites stay green unchanged (this phase adds a consult-first branch that is a no-op until Phase 2 registers anything — the regression gate for "nothing changed yet" is the existing suite passing byte-for-byte).
- **Exit state**: The registry exists on `WorldModel`/`AuthorModel`; both `VisibilityBehavior` and `going` consult it first and fall through correctly; zero existing story behavior has changed (nothing registers yet). `pnpm --filter '@sharpee/world-model' test` and `pnpm --filter '@sharpee/stdlib' test` green.
