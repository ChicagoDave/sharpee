# Phase 6 Plan — Web Client and Two-User End-to-End

**Parent**: `docs/work/zifmia/plan-20260511-adr-175.md` § Phase 6
**ADR-175** (parent product), **ADR-176** (multi-user component vocabulary — ACCEPTED 2026-05-12)
**Tier**: Large — 400 tool-call budget
**ACs satisfied**: AC-2 (two-user E2E), AC-3 (WS reconnect non-event), AC-10 (lock-on-typing)
**Entry state (2026-05-12)**: 260/260 tests; Phases 1–5 complete; admin HTTP surface live; framework decision RESOLVED — **no framework**, plain-TS managers + ADR-170/176 component vocabulary + native HTML5 + CSS-class theming, matching `packages/platform-browser/`.

## Decisions taken at session start

- **Implementation location**: `tools/zifmia/web/` — carve-out. Zifmia owns its multi-user DOM and managers; imports primitives + base CSS from `packages/platform-browser/` and `templates/browser/`. (ADR-176 § Implementation location alternative B.)
- **Base CSS authority**: ADR-176's 14 multi-user classes extend `templates/browser/base.css`; theme overlays live in `templates/browser/themes/` next to the existing single-player themes.
- **Build integration**: web build wired into `./build.sh -s {story} -c zifmia` so a single command produces server + client + bundle.
- **Wire contract**: WS message kinds already shipped by the server are the contract — `presence:roster/joined/left`, `chat:send/message`, `lock:acquire/release/state`, `turn:broadcast`, `room:subscribe/subscribed/unsubscribe/unsubscribed`, `room:restored`, `command:submit`. The client is a consumer, not a co-author.
- **Channel renderers**: reuse `packages/platform-browser/src/channels/` (main/text-content/status/notify/prompt/info/animation/audio/image/layout/lifecycle) verbatim. Multi-user concerns (presence, chat, lock) get NEW managers in `tools/zifmia/web/src/managers/` — they do not collide with channel rendering.

### 6a — Web scaffold + identity flow (Medium, ~60 calls)

**Deliverables**

- `tools/zifmia/web/` package: `package.json`, `tsconfig.json`, `vite.config.ts`, `src/`, `tests/`, `public/`. Build with vite (matches Sharpee web build elsewhere). Output to `tools/zifmia/dist/web/`.
- Fastify static-asset registration in `src/server/index.ts` (`@fastify/static` mounted under `/`); web build copied into the container image (`Dockerfile`).
- `index.html` ships the single-page shell: title, theme stylesheet links, root `<main>` element.
- `IdentityManager` (plain class): renders `.sharpee-identity-form` with login/register tabs; POST `/identity/register` and POST `/identity/login`; persists `{sessionToken, identityId, handle}` in `localStorage` under key `zifmia:session`.
- Session bootstrap: on page load, check `localStorage`; if no token → render `.sharpee-identity-form`; if token → fetch `GET /identity/me` (add this if missing) and proceed to lobby.
- Theme loader: read `?theme=` query param OR localStorage `zifmia:theme`, default DOS Classic. Loads `templates/browser/themes/{theme}.css`.
- Base CSS extensions for the 14 new ADR-176 classes added to `templates/browser/base.css` (structural rules only — `pointer-events: none` on `--hidden`, scroll-pin for `.sharpee-chat-history`, etc.).
- Build hook: `build.sh -c zifmia` builds the web assets and copies them into `tools/zifmia/dist/web/` before the container build.

**Behavior Statements**

- `IdentityManager.register(handle, passcode)`: posts to `/identity/register`; on 201 persists session to localStorage, returns identity. REJECTS on 4xx with surfaced detail code; does not persist.
- `IdentityManager.login(handle, passcode)`: posts to `/identity/login`; on 200 persists session; REJECTS on 401 `invalid_credentials` with form-level error.
- Session bootstrap: missing token → render form; present token + 401 from `/identity/me` → clear localStorage + render form; valid token → navigate to lobby.

**Tests (vitest jsdom)**

- IdentityManager register happy + 409 handle_taken + 422 validation
- IdentityManager login happy + 401 invalid_credentials
- Session bootstrap: no token → form; expired token → cleared + form; valid token → lobby
- Theme loader: query param > localStorage > default

**Exit state**: web bundle builds; unauthed visit shows identity form; auth flow round-trips against real server (integration test against a real Fastify instance).

### 6c — Room view core: channels + transcript + command input (Large, ~70 calls)

**Deliverables**

- Page layout (in `tools/zifmia/web/src/views/room.ts`):
  - Left: `.sharpee-presence-panel` (stub container; populated in 6d)
  - Center: `.sharpee-prose-pane` (ADR-170 main channel)
  - Right: `.sharpee-chat-panel` (stub container; populated in 6d)
  - Bottom: `.sharpee-input-bar` for command input
- `RoomManager`: on entry calls `GET /rooms/:id/state` (returns `{room, cmgt, transcript, presence, chat}`); applies CMGT to channel renderers; replays transcript into `.sharpee-prose-pane`.
- Channel renderer wiring: import the existing `packages/platform-browser/src/channels/` renderers; route channel packets to their renderer.
- `CommandInputManager`: focus/blur/submit handlers. Submit → `POST /rooms/:id/commands`; on success the server's `turn:broadcast` will arrive over WS and update the prose pane (no double-render).
- `WsClient`: opens `wss://.../rooms/:id`; sends `room:subscribe` on open; routes incoming packets by `kind` to registered handlers; basic JSON framing.

**Behavior Statements**

- `RoomManager.enter(roomId)`: persists `currentRoomId`; fetches state; renders main channel from CMGT; replays transcript; subscribes WS. REJECTS WHEN 404 → navigate to lobby with error toast; 403 → identity form.
- `CommandInputManager.submit(text)`: POSTs to `/rooms/:id/commands`; on 2xx clears input and yields to WS turn:broadcast for the render. REJECTS WHEN 409 lock_held → surface as banner update; 5xx → show error pip.

**Tests**

- RoomManager.enter happy + 404 + 403
- CommandInputManager.submit happy + 409 + 5xx
- Channel renderer routing: each known channel kind reaches its renderer
- Transcript replay deterministic across two state-fetch cycles

**Exit state**: signed-in user can enter a room, see the existing transcript, submit a command, and see the resulting narrative — single user only. Presence/chat panels are stub containers.

### 6d — Presence + chat panel (Medium, ~55 calls)

**Deliverables**

- `PresenceManager`: subscribes to WS `presence:roster` (initial) + `presence:joined`/`presence:left` (deltas). Renders `<ul class="sharpee-presence-list">` > `<li class="sharpee-presence-item">` per identity; toggles `--self` and `--admin` modifiers.
- `ChatManager`: subscribes to WS `chat:message`; renders `<li class="sharpee-chat-message">` rows with `.sharpee-chat-message-author` + `.sharpee-chat-message-text` slots; toggles `--self` (own messages) and `--system` (server-emitted joins/leaves). Send via `.sharpee-chat-input` → WS `chat:send`.
- Scroll-pin behavior in `.sharpee-chat-history`: pin to bottom unless user has scrolled up; resume pin on new message if at-or-near bottom.

**Behavior Statements**

- `PresenceManager.onJoin(packet)`: appends a `.sharpee-presence-item` `<li>` with id `presence-{identityId}`; reapplies `--self`/`--admin` from packet flags. REJECTS WHEN identityId already rendered → idempotent no-op.
- `PresenceManager.onLeave(packet)`: removes the matching `<li>`; idempotent on unknown id.
- `ChatManager.onMessage(packet)`: appends `<li class="sharpee-chat-message">`; renders author + text slots; sets `--self` if `packet.identityId === me`, `--system` if `packet.system === true`. Re-pins scroll if pinned.
- `ChatManager.send(text)`: emits WS `chat:send`; clears local input; renders nothing locally (server echoes via `chat:message` for the canonical render).

**Tests**

- Presence roster initial render + join + leave + idempotent re-join
- `--self` applied only to own row; `--admin` applied when flag set
- Chat: render outgoing-then-incoming round trip; `--system` on server-emitted join message; scroll-pin: stays pinned at bottom unless scrolled
- ADR-176 contract: every rendered class is in the 36-class vocabulary

**Exit state**: two browsers in the same room see each other's presence and chat in real time. Narrative still goes through the single-user path; lock not yet wired.

### 6e — Lock-on-typing + WS reconnect (Medium/Large, ~65 calls — AC-3 + AC-10)

**Deliverables**

- `LockManager`:
  - Focus on `.sharpee-input-bar` (or first keystroke if focus-without-typing should not lock — TBD inline; default: keystroke, not focus) → WS `lock:acquire`.
  - Blur with empty input OR successful submit → WS `lock:release`.
  - On WS `lock:state {holder, expiresAt}` from server: if `holder === me` → `--locked-by-me` on `.sharpee-input-bar`; else → `--locked` + populate `.sharpee-lock-banner` with `"{handle} is typing…"` and clear `--hidden`. On `{holder: null}` → remove both modifiers + `--hidden` on banner.
  - 200 ms acquire deadline: server responds within 200 ms (already validated in Phase 3); client just renders.
  - 1 s implicit release: server emits `lock:state {holder: null}` 1 s after holder disconnect; client renders.
- `WsClient` reconnect:
  - Exponential backoff: 500 ms, 1 s, 2 s, 4 s, 8 s (cap at 8 s), max 10 attempts.
  - On disconnect: render a `.sharpee-lock-banner`-style or status-channel "reconnecting…" pip (reuse existing `.sharpee-status-pane` from ADR-170 — no new class).
  - On reconnect: re-issue `GET /rooms/:id/state`, reapply CMGT + transcript (idempotent — RoomManager.enter is the seam), resubscribe WS.
  - `room:restored` handler: per ADR-176 OQ-1, log to console for v1; deferred to v2.

**Behavior Statements**

- `LockManager.onLockState({holder})`: if holder === me, `.sharpee-input-bar` gets `--locked-by-me`, `.sharpee-lock-banner` stays hidden. Else if holder !== null, `.sharpee-input-bar` gets `--locked`, banner unhides and reads `{handle} is typing…`. Else holder is null, both modifiers cleared and banner hidden. REJECTS WHEN no `.sharpee-input-bar` element present → no-op (room not entered).
- `WsClient.reconnect()`: closes existing socket if any; sleeps per backoff; opens new socket; on open, re-emits `room:subscribe`; calls `RoomManager.refresh()` to re-fetch state and re-apply. REJECTS WHEN max attempts hit → surfaces persistent banner; halts retry.

**Tests**

- LockManager: holder=me → `--locked-by-me` set, banner hidden; holder=other → `--locked` + banner unhidden with handle; holder=null → cleared
- LockManager: lock state arriving before room entry → no-op (no throw)
- WsClient: simulated drop + reconnect → state refetched + WS resubscribed + idempotent transcript (no duplicate prose rows)
- WsClient: 10 failed attempts → persistent banner + no further attempts

**Exit state**: AC-3 (reconnect) and AC-10 (lock-on-typing) wired in the client and unit-asserted. End-to-end timing validation deferred to 6g.

### 6f — Named saves + admin UI rendering (Medium, ~55 calls)

**Deliverables**

- `SavesManager` (in-room panel):
  - `.sharpee-saves-panel` with create-button + `<ul class="sharpee-saves-list">` > `<li class="sharpee-saves-item">` rows
  - GET `/rooms/:id/saves` on entry (populates list)
  - Create-save `<dialog>`: label input → POST `/rooms/:id/saves`
  - Update label via PUT `/rooms/:id/saves/:saveId`
  - Restore: row click → confirm `<dialog>` → POST `/rooms/:id/saves/:saveId/restore`; server emits `room:restored` to all subscribers
  - **No delete** (per `feedback_no_save_delete` — named saves persist with the room)
- Admin UI (deferred from Phase 5): a separate `#admin` route only shown when `identity.isAdmin === true`.
  - Story library tab: `.sharpee-lobby-list`-style list of `GET /admin/stories`; upload bundle via `<input type="file">` + POST `/admin/stories`
  - Active rooms tab: list `GET /rooms` with kill button → DELETE `/admin/rooms/:id`
  - Audit log tab: `GET /admin/audit` with `?sinceTs` paging
  - Identity reset: handle search + POST `/admin/identities/:id/passcode_reset`; render plaintext-once banner with copy button (memory: "plaintext lives exactly once in the response body")

**Behavior Statements**

- `SavesManager.create(label)`: POSTs; on 201 prepends `.sharpee-saves-item` row; emits no broadcast (server does that). REJECTS WHEN 422 invalid_label → error pip.
- `SavesManager.restore(saveId)`: posts to restore endpoint; on 200, awaits WS `room:restored` to re-render via `RoomManager.refresh()` (idempotent).
- Admin story upload: posts file as `application/octet-stream`; on 201 prepends row; on 422 surfaces detail code from ADR-175 validation pipeline.
- Admin room kill: DELETEs; on 204 removes row.
- Admin passcode reset: POSTs; renders plaintext exactly once in a `<dialog>` with copy button; does not persist to localStorage or repeat in DOM after dialog close.

**Tests**

- SavesManager create + list + update label + restore (mock WS for `room:restored`)
- Admin story upload happy + 422 bundle validation surfacing
- Admin room kill + idempotent re-kill
- Admin passcode reset: renders plaintext once; assert plaintext not present in DOM after dialog close
- Non-admin user: `#admin` hash returns 403-equivalent UX (redirect to lobby)

**Exit state**: Phase 5's HTTP surface has rendered UI. Author override hooks are documented for the saves panel (a story may want to suppress named saves entirely; this is the documented extension point).

### 6g — Playwright two-user E2E (AC-2 + AC-3 + AC-10) (Medium, ~50 calls)

**Deliverables**

- Playwright config in `tools/zifmia/tests/e2e/` (separate from vitest unit suites under `tools/zifmia/tests/`)
- Fixture: spawn the Zifmia server in a subprocess against a temp SQLite DB; install a known story bundle via the admin route; create two identities (Alice + Bob)
- **AC-2 (two-user E2E)**: open two browsers, both join the same room; assert: Alice sees Bob in `.sharpee-presence-list` and vice versa; Alice sends `chat:send` → Bob sees the `.sharpee-chat-message` row; Alice submits a command → both Alice and Bob see identical narrative in `.sharpee-prose-pane`
- **AC-3 (WS reconnect)**: Alice forcibly closes her WS; reconnect fires; Alice's `.sharpee-prose-pane` matches Bob's after replay; Alice's `.sharpee-presence-list` does not double-list anyone
- **AC-10 (lock-on-typing)**: Alice focuses + types in `.sharpee-input-bar`; within 200 ms Bob's `.sharpee-input-bar` has `--locked` and `.sharpee-lock-banner` is unhidden with "Alice is typing…"; Alice disconnects without submitting; within 1 s Bob's input is re-enabled

**Integration Reality Statement**

- **OWNED**: Zifmia server (Node subprocess), SQLite DB, web bundle (vite-built static assets), Playwright Chromium drivers
- **EXTERNAL**: none (no third-party services; story bundle is bundled)
- **REAL-PATH TEST**: all three AC tests drive a real browser against a real Fastify server against a real SQLite file. No mocks.
- **STUB JUSTIFICATION**: none — phase named after the integration must have a real-path E2E.

**Tests**

- 3 Playwright tests (one per AC); each runs in CI and locally via `pnpm test:e2e` in `tools/zifmia/`

**Exit state**: AC-2, AC-3, AC-10 GREEN via Playwright. Phase 6 complete.
