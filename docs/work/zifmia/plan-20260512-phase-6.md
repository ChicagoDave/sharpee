# Phase 6 Plan — Web Client and Two-User End-to-End

**Parent**: `docs/work/zifmia/plan-20260511-adr-175.md` § Phase 6
**ADR-175** (parent product), **ADR-176** (multi-user component vocabulary — ACCEPTED 2026-05-12)
**Tier**: Large — 400 tool-call budget
**ACs satisfied**: AC-2 (two-user E2E), AC-3 (WS reconnect non-event), AC-10 (lock-on-typing)
**Entry state (2026-05-12)**: 260/260 tests; Phases 1–5 complete; admin HTTP surface live; framework decision RESOLVED — **no framework**, plain-TS managers + ADR-170/176 component vocabulary + native HTML5 + CSS-class theming, matching `packages/platform-browser/`.

## Decisions taken at session start

- **Implementation location**: `tools/zifmia/web/` — carve-out. Zifmia owns its multi-user DOM and managers; imports primitives + base CSS from `packages/platform-browser/` and `templates/browser/`. (ADR-176 § Implementation location alternative B.)
- **Base CSS authority**: ADR-176's 14 multi-user classes extend `templates/browser/base.css`; theme overlays live in `templates/browser/themes/` next to the existing single-player themes.
- **Build integration**: web build wired into `./build.sh -s {story} -c zifmia` so a single command produces server + client + bundle.
- **Wire contract**: WS message kinds already shipped by the server are the contract — `presence:roster/joined/left`, `chat:send/message`, `lock:acquire/release/state`, `turn:broadcast`, `room:subscribe/subscribed/unsubscribe/unsubscribed`, `room:restored`, `command:submit`. The client is a consumer, not a co-author.
- **Channel renderers**: reuse `packages/platform-browser/src/channels/` (main/text-content/status/notify/prompt/info/animation/audio/image/layout/lifecycle) verbatim. Multi-user concerns (presence, chat, lock) get NEW managers in `tools/zifmia/web/src/managers/` — they do not collide with channel rendering.

### 6c-server — Server-side ChannelService adoption (Medium/Large, ~80 calls)

**Why this exists**: when 6c was sized at 70 calls, the assumption was "ChannelService is already wired into Zifmia; the client just attaches renderers." Discovery during 6c kickoff: Zifmia ships **raw** `{blocks, events}` over the wire and never instantiates ChannelService. The engine already emits `channel:manifest` and `channel:packet`, but Zifmia ignores them. To deliver the "Full 6c — wire CMGT end-to-end" path David chose, the server must capture and ship the typed packets first.

**Deliverables**

- `buildEngine` in `turn-executor.ts` captures `channel:manifest` (one-shot from `engine.start()`) and `channel:packet` (per turn) alongside the existing raw blocks/events.
- Wire `TurnPacket` (in `tools/zifmia/src/engine/types.ts`) gains a `channelPacket: ChannelTurnPacket` field carrying the `@sharpee/if-domain` channel-typed packet. Raw `blocks` + `events` are retained for back-compat with v1 transcript envelopes and as the canonical engine output.
- `TranscriptEntry` gains an optional `channelPacket?: ChannelTurnPacket` field. Save envelope version bumps `1 → 2`. Decoder reads both: v1 entries return `channelPacket: undefined`; v2 entries round-trip the field. This honors `feedback_save_format_versioning` (version-reader, no hard-break) and is consistent with `feedback_no_backcompat_server_lifecycle` (the *wire* between server and current client cuts over; the *envelope* on disk is forward-readable).
- `GET /rooms/:id/state` returns `cmgt: ChannelCmgtPacket` (built from the engine's per-room `ChannelService`) instead of `null`. Transcript entries on the wire carry `channelPacket` when present. `ClientCapabilities` query-param-negotiated; defaults to `BROWSER_CAPABILITIES`.
- WS `turn:broadcast` shape gains `channelPacket`. `command_echo` is unchanged (it carries only the command string).
- All server tests that assert on the wire shape updated: `command.test.ts`, `command-fanout.test.ts`, `rooms.test.ts`, `restore.test.ts`, `save-envelope.test.ts`, `transcript-truncation.test.ts`, `turn-executor.test.ts`, `crash-recovery.test.ts`, `ws.test.ts`.

**Wire-shape cutover**

Per `feedback_no_backcompat_server_lifecycle`: schema/wire changes are one-shot. The HTTP response from `POST /rooms/:id/command` and the WS `turn:broadcast` shape both change in the same commit. There is no two-phase rollout. Phase 6c-client (next sub-phase) consumes the new shape; no other clients exist.

**Tests**

- `turn-executor.test.ts`: assert `channelPacket` present on returned `TurnPacket`; assert `channelPacket.turn` matches save_blob turn.
- `save-envelope.test.ts`: encode v2, decode v2 round-trips channelPacket; decode v1 returns entries with channelPacket undefined.
- `rooms.test.ts`: `GET /rooms/:id/state` returns non-null cmgt for a room with at least one turn; cmgt's channel list matches the story's registered channels.
- `command.test.ts`: HTTP response carries channelPacket on success; engine-throw path returns 500 with no channelPacket (existing behavior preserved).
- `ws.test.ts`: `turn:broadcast` carries channelPacket.

**Exit state**: Zifmia server is channel-aware end to end. The HTTP/WS wire is the typed-packet shape ADR-163/165 prescribes. 6c-client can wire `Renderer` + `registerDefaultBrowserRenderers` against this without extra translation.

**Status**: COMPLETE 2026-05-12 — 332 → 335 tests (+3: channelPacket on wire-TurnPacket, channelPacket on turn:broadcast, save-envelope round-trip with/without channelPacket). Server build clean.

### 6e — Lock-on-typing + WS reconnect (Medium/Large, ~65 calls — AC-3 + AC-10)

**Deliverables**

- `LockManager`:
  - Focus on `.sharpee-input-bar` (or first keystroke if focus-without-typing should not lock — TBD inline; default: keystroke, not focus) → WS `lock:acquire`.
  - Blur with empty input OR successful submit → WS `lock:release`.
  - On WS `lock:state {holder, expiresAt}` from server: if `holder === me` → `--locked-by-me` on `.sharpee-input-bar`; else → `--locked` + populate `.sharpee-lock-banner` with `"{handle} is typing…"` and clear `--hidden`. On `{holder: null}` → remove both modifiers + `--hidden` on banner.
  - 200 ms acquire deadline: server responds within 200 ms (already validated in Phase 3); client just renders.
  - 1 s implicit release: server emits `lock:state {holder: null}` 1 s after holder disconnect; client renders.
- `WsClient` reconnect:
  - Exponential backoff: 500 ms, 1 s, 2 s, 4 s, 8 s (cap at 8 s), max 10 attempts.
  - On disconnect: render a `.sharpee-lock-banner`-style or status-channel "reconnecting…" pip (reuse existing `.sharpee-status-pane` from ADR-170 — no new class).
  - On reconnect: re-issue `GET /rooms/:id/state`, reapply CMGT + transcript (idempotent — RoomManager.enter is the seam), resubscribe WS.
  - `room:restored` handler: per ADR-176 OQ-1, log to console for v1; deferred to v2.

**Behavior Statements**

- `LockManager.onLockState({holder})`: if holder === me, `.sharpee-input-bar` gets `--locked-by-me`, `.sharpee-lock-banner` stays hidden. Else if holder !== null, `.sharpee-input-bar` gets `--locked`, banner unhides and reads `{handle} is typing…`. Else holder is null, both modifiers cleared and banner hidden. REJECTS WHEN no `.sharpee-input-bar` element present → no-op (room not entered).
- `WsClient.reconnect()`: closes existing socket if any; sleeps per backoff; opens new socket; on open, re-emits `room:subscribe`; calls `RoomManager.refresh()` to re-fetch state and re-apply. REJECTS WHEN max attempts hit → surfaces persistent banner; halts retry.

**Tests**

- LockManager: holder=me → `--locked-by-me` set, banner hidden; holder=other → `--locked` + banner unhidden with handle; holder=null → cleared
- LockManager: lock state arriving before room entry → no-op (no throw)
- WsClient: simulated drop + reconnect → state refetched + WS resubscribed + idempotent transcript (no duplicate prose rows)
- WsClient: 10 failed attempts → persistent banner + no further attempts

**Exit state**: AC-3 (reconnect) and AC-10 (lock-on-typing) wired in the client and unit-asserted. End-to-end timing validation deferred to 6g.
