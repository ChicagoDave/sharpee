# Session Plan: Zifmia Multi-User Product Rebuild (ADR-175)

**Created**: 2026-05-11
**ADR**: `docs/architecture/adrs/adr-175-zifmia-multiuser-product.md` — ACCEPTED
**Overall scope**: Greenfield implementation of the Zifmia multi-user web product at `tools/zifmia/`, from empty directory through all 14 acceptance criteria. Single Docker image, stateless-engine per ADR-164, HTTP command path, WebSocket for live coordination, SQLite default with Postgres adapter.
**Bounded contexts touched**: N/A — infrastructure/tooling (deployable application, not a library)
**Key domain language**: N/A (infrastructure framing); relevant contracts from consuming ADRs: TurnPacket, CmgtPacket, ChannelService, save_blobs, named_saves, room, identity, lock, presence

## Open Questions — Decide Before Each Affected Phase

These are ADR-175 "Open Questions for Implementation." They are surfaced under the phase where they become binding. A decision is needed **before that phase begins**, not during it.

| # | Question | ADR Recommendation | Binding In |
|---|---|---|---|
| OQ-1 | Story upload UX: single-step install vs. staging gate | ADR: staging gate (upload → validate → promote) | Phase 5 |
| OQ-2 | Worker pool sizing: `os.cpus().length - 1` default, tune via env | ADR: default formula; confirm env var name | Phase 3 |
| OQ-3 | Admin UI location: bundled (role-gated routes) vs. separate admin port | ADR: bundled for v1; separate port is v2 option | Phase 5 |
| OQ-4 | Web client framework: React (default) vs. vanilla web components | ADR: React default, but spike first; see Phase 2 | Phase 6 |
| OQ-5 | Save migration on bundle upgrade: admin "migrate" button vs. v2-only | ADR: recommend v2 (saves stay pinned to prior bundle) | Phase 4 |
| OQ-6 | Audit log scope: which events to capture beyond story install/remove | Obvious: story CRUD, room kill, identity reset; commands out | Phase 5 |

### Phase 3: Identity, Room, and Turn-Execution Core
- **Tier**: Large
- **Budget**: 400 tool calls
- **ACs satisfied**: AC-9 (transport split), AC-11 (identity rejection), AC-13 (engine-throw recovery), AC-14 (performance baseline)
- **Entry state**: Phase 1 complete; storage adapter passes contract tests; OQ-2 confirmed (worker pool env var name)
- **Deliverables**:
  - `POST /identity/login` — ADR-161 `{handle, passcode}` → `{id, handle, sessionToken}`; `401 { error: 'invalid_credentials' }` on bad creds (no handle/passcode distinction)
  - `POST /rooms` — room create with story selection; `GET /rooms` — lobby list
  - `GET /rooms/:id/state` — returns CMGT manifest + transcript replay from in-blob window; used for page-load and WS-reconnect recovery
  - `POST /rooms/:id/command` — full stateless turn lifecycle: acquire lease → load latest save_blob → instantiate WorldModel (ADR-164 invariant: no WorldModel held past this request) → run engine → snapshot → DB commit → release lease → return TurnPacket; engine-throw path: no DB commit, no turn:broadcast, lock:state {holder: null} broadcast, return 500 {error: 'turn_failed'}
  - Worker pool: `WorkerPool` class bounded by `os.cpus().length - 1` (default) or `ZIFMIA_WORKER_COUNT` env; each worker holds no per-room state between requests
  - WS server wired: `chat:send/message`, `lock:acquire/release/state`, `presence:update`, `turn:broadcast`, `command_echo`; `command:submit` over WS returns protocol-error frame (AC-9)
  - Session token validation middleware applied to all protected routes
  - Behavior tests:
    - `POST /identity/login` — valid creds return token; wrong passcode returns 401 same body; nonexistent handle returns 401 same body (AC-11)
    - `POST /rooms/:id/command` happy path — TurnPacket returned, save_blob row written, turn:broadcast sent to non-submitter WS, lock released (AC-9)
    - `POST /rooms/:id/command` engine-throw path — 500 returned, no save_blob written, no turn:broadcast, lock:state {holder: null} broadcast (AC-13)
    - WS command:submit rejected (AC-9)
    - Performance: `look` on Dungeo under 100 ms p95 on a 2-vCPU SQLite container (AC-14); record baseline number in `docs/work/zifmia/perf-baseline.md`
- **Exit state**: AC-9, AC-11, AC-13, AC-14 pass; turn execution is the definitive stateless path; no WorldModel reference outlives a request; all behavior tests green
- **Status**: PENDING

### Phase 4: Save Lifecycle and Transcript Truncation
- **Tier**: Medium
- **Budget**: 250 tool calls
- **ACs satisfied**: AC-4 (crash recovery), AC-6 (save lifecycle), AC-7 (transcript truncation)
- **Entry state**: Phase 3 complete; `save_blobs` and `named_saves` schema in place; turn execution writes save_blobs rows
- **Deliverables**:
  - `POST /rooms/:id/saves` — name a save slot: validates atTurn exists in save_blobs, writes named_saves pointer row
  - `GET /rooms/:id/saves` — list named saves
  - `POST /rooms/:id/restore` — load save_blob at atTurn, run a "restore" turn (world re-initialized to that blob), write new save_blob row, return TurnPacket; restore is a full turn lifecycle (load → execute → snapshot → commit)
  - Transcript truncation: per-turn blob append trims in-blob transcript to 1000 entries; absolute turn counter preserved separately (world scoring capability); mid-session join via `GET /rooms/:id/state` replays the 1000-entry window
  - Compaction job: periodic background task GCs save_blob rows with no named_saves reference and older than the transcript window; implemented but off-by-default in v1 (env flag to enable)
  - Save format: v3 read/write (ADR-164 Decision 7); v2 blobs load with empty transcript per ADR-164 AC-4
  - Crash recovery verification: force-kill mid-turn (before DB commit) → restart → reconnect → room loads at prior save; force-kill post-commit → restart → reconnect → committed turn replays in transcript (AC-4)
  - Behavior tests:
    - Named save round-trip: save → container restart → restore → correct world state (AC-6)
    - Save persists across restart (AC-6)
    - 1500-turn session: in-blob transcript caps at 1000; mid-session join sees 1000 entries (AC-7)
    - Crash pre-commit: room recoverable at prior baseline (AC-4)
    - Crash post-commit: committed turn visible in transcript after restart (AC-4)
- **Open question surfaced**: OQ-5 — does admin get a "migrate saves to new bundle version" button? Confirm v2-only before Phase 5 begins.
- **Exit state**: AC-4, AC-6, AC-7 pass; all behavior tests green; save_blobs and named_saves are the definitive persistence surface
- **Status**: PENDING

### Phase 5: Story Library, Admin UI, and Audit Log
- **Tier**: Large
- **Budget**: 400 tool calls
- **ACs satisfied**: AC-1 (full — admin installs story via UI), AC-5 (story install/upgrade/remove), AC-8 (partial — adapter swap documented), AC-12 (bundle validation)
- **Entry state**: Phases 1–4 complete; `/health`, identity, room, turn, and save routes all operational; OQ-1 (upload UX), OQ-3 (admin port), OQ-5 (save migration), OQ-6 (audit scope) resolved before this phase begins
- **Deliverables**:
  - `POST /admin/stories` — bundle upload: validation sequence (structure → signature → IFID → format version); first failure short-circuits; no partial library row on failure; staging gate if OQ-1 resolves to that (upload → staging row → promote to library); `422 { error: 'invalid_bundle', detail: '...' }` shapes (AC-12)
  - `PUT /admin/stories/:id` — upgrade: new bundle replaces active pointer; in-flight rooms finish on old bundle version (saves pinned to bundle_version); new rooms use new version
  - `DELETE /admin/stories/:id` — remove: blocks new room creation for this story; existing rooms continue until participants leave
  - Story bundles stored as DB blobs (Decision 3 invariant: not on container filesystem)
  - Story library metadata table: `story_id`, `version`, `title`, `ifid`, `installed_at`, `installed_by`
  - Bundle version pinning: save_blobs rows carry `bundle_version` FK; engine loads the bundle version that produced the save
  - Audit log: append on story install/upgrade/remove, room kill, identity passcode reset, room creation (if OQ-6 confirms); `GET /admin/audit` returns entries
  - Admin UI: role-gated routes (bundled, not separate port — per ADR recommendation for v1); pages: story library, active rooms, audit log, room kill action
  - Behavior tests:
    - Bundle validation: truncated bundle → 422 bad_structure; valid structure + missing IFID → 422 missing_ifid; v2-format → 422 unsupported_format; after each failure, library row count and blob count unchanged (AC-12)
    - Upgrade: new rooms get new bundle; in-flight rooms load saves from prior bundle version
    - Remove: new room creation blocked; existing room continues
    - `docker compose up` + admin installs bundle + bundle joinable in lobby (AC-1 full)
    - Audit log captures each admin action
- **Exit state**: AC-1 (full), AC-5, AC-12 pass; admin can manage stories via UI without touching the container filesystem; all behavior tests green
- **Status**: PENDING

### Phase 6: Web Client and Two-User End-to-End
- **Tier**: Large
- **Budget**: 400 tool calls
- **ACs satisfied**: AC-2 (two-user end-to-end), AC-3 (WS reconnect), AC-10 (lock-on-typing)
- **Entry state**: Phases 1–5 complete; framework choice from Phase 2 approved; all server routes and WS message kinds operational
- **Deliverables**:
  - Web client using the approved framework (React or vanilla web components), built and served as static assets from the same container and port
  - ADR-165 Renderer/ChannelRenderer contract wired to multi-user transport: HTTP for turn submission and state load; WS for chat, lock, presence, turn:broadcast, command_echo
  - Client pages: lobby (room list), room creation, room view (main channel, chat panel, presence roster, command input, save management)
  - Lock-on-typing: input focus → `lock:acquire`; other participants see read-only input + "A is typing…" within 200 ms; disconnect → implicit release → inputs re-enabled within 1 s; submit → HTTP POST, then WS turn:broadcast to others + lock release (AC-10)
  - WS reconnect: disconnect/reconnect triggers `GET /rooms/:id/state` (CMGT + transcript replay) then fresh WS subscription; no lost packets observable (AC-3)
  - Client holds no authoritative state: browser refresh recoverable entirely from `GET /rooms/:id/state` (ADR-175 invariant)
  - Author-override entry point: documented extension point for per-story UI customization (ADR principle: platform ships defaults authors can override)
  - Playwright end-to-end tests (patterns from `legacy/tools-server`):
    - Two browsers, same room: each sees the other in presence roster; commands echo via command_echo; chat messages delivered; identical narrative in main channel (AC-2)
    - Browser A disconnects mid-session; reconnects; CMGT + transcript replayed; no lost packets (AC-3)
    - Browser A focuses input → Browser B sees read-only + "A is typing…" within 200 ms; A disconnects without submitting → B sees input re-enabled within 1 s (AC-10)
- **Exit state**: AC-2, AC-3, AC-10 pass via Playwright; client is author-customizable; all behavior tests green
- **Status**: PENDING

## Sequencing Notes

- Phases 1 → 3 → 4 → 5 are a strict dependency chain (each phase's entry state requires the prior).
- Phase 2 (renderer spike) can run in parallel with Phase 3 if David approves it early — it is read-only and produces no production code. It must complete and produce a framework decision before Phase 6 begins.
- Phase 6 (web client) depends on Phases 1–5 but not Phase 7.
- Phase 7 (Postgres) depends on Phase 6 being green (no regression before hardening).
- The compaction job (Phase 4) is gated behind an env flag; it does not block any other phase.

## Test Strategy Summary

Tests are not deferred to a final "QA phase." Every phase ships its own behavior tests and integration tests as part of the phase deliverables. Playwright end-to-end tests are introduced in Phase 6 (they require a running client). The performance baseline (AC-14) is recorded as a regression-tracking artifact in `docs/work/zifmia/perf-baseline.md` starting in Phase 3.

The StorageAdapter contract suite introduced in Phase 1 is the single test suite that validates both adapters — it runs against SqliteAdapter in Phase 1 and against PostgresAdapter in Phase 7. No duplicate test code between adapter implementations.
