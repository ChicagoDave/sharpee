# ADR-174 Phase 2 — Migrate Wire-Production Consumers off `renderToString`

**Date**: 2026-05-10
**Branch**: TBD (suggest `adr-174-phase2-renderToString-migration`)
**ADR**: [ADR-174 — Decoration Architecture and Engine-Internal Prose Pipeline](../../architecture/adrs/adr-174-decoration-and-prose-pipeline.md) (status: ACCEPTED for Phase 1)
**Predecessor**: Phase 1 complete on `adr-174-phase1-prose-pipeline` (engine cutover); summary at `docs/context/session-20260509-2300-adr-174-phase1-prose-pipeline.md`
**Scope**: Phase 2 only. Phase 3 (`@sharpee/text-service` package deletion) gets its own plan once Phase 2 lands.

---

## Goal

Remove every first-party and downstream consumer's import of `@sharpee/text-service`, so after Phase 2 the only thing keeping the package alive is its own internal exports. Specifically:

- Replace every `renderToString(blocks)` call site with a `text-blocks`-hosted equivalent.
- Drop the re-exports of `renderToString` / `renderStatusLine` / `ITextService` / `createTextService` from `bridge`, `runtime`, and `sharpee`.
- Remove the `@sharpee/text-service` workspace dependency from every package outside text-service itself.

After Phase 2: `grep -r "@sharpee/text-service" packages/ stories/` returns zero matches outside `packages/text-service/` (its own files) — that's the AC-8 acceptance.

The text-service package itself stays compilable and on-disk; Phase 3 deletes the directory and the workspace entry.

## Why this is necessary

ADR-174 §Migration phasing makes Phase 2 the bridge between the Phase 1 engine cutover (engine → ProsePipeline) and the Phase 3 package deletion. Phase 1 left text-service compilable specifically so downstream wire-production callers could continue working until they were migrated. Phase 2 does that migration.

The work is mechanical:
- One callsite shape (`renderToString(blocks)`) and one return shape (joined string) cover every consumer. No consumer uses ANSI mode, story-defined colors, or `renderStatusLine` — the audit confirmed this.
- The replacement is a thin, dependency-free utility that can live alongside the existing `extractPlainText` helper in `text-blocks/src/guards.ts`.

The only question worth pausing on is OQ-1: where does the helper live. Resolved below.

## Naming discipline

The replacement function is named **`flattenBlocks`**. Not `renderToString`, not `flatten`, not `blocksToString`.

- **`flatten`** would collide with array `Array.prototype.flat()` semantics in reader expectations.
- **`renderToString`** is the retired name and reusing it muddies the migration ("did you import the new one or the old one?").
- **`blocksToString`** is technically clear but less Sharpee-vocabulary aligned.

`flattenBlocks` reads like a verb-on-noun, names the input shape (`blocks`), and avoids any "render" connotation that might confuse renderer-vs-projection readers. It also pairs naturally with the existing `flattenTextContent` in `platform-browser/channels/text-content.ts`.

## OQ-1 resolution — Where does the helper live?

**Resolved: extend `@sharpee/text-blocks` with `flattenBlocks`.**

The ADR's three candidates were:
- **`@sharpee/channel-service`** (ADR's recommendation)
- **New `@sharpee/wire-utils` package**
- **Per-consumer reimplementation** (rejected)

After auditing the actual call shapes, the ADR's recommendation doesn't fit:

- Every call site receives `ITextBlock[]` from `engine.on('text:output')`, not a `channel:packet`. `channel-service` operates on packets; passing it `ITextBlock[]` would be off-shape.
- A new package adds 6 registration points (per `feedback_new_package_config.md`) for one ~30-line function. Disproportionate.

`@sharpee/text-blocks` is the natural home:

- It already exports the input type (`ITextBlock`).
- It already exports a similar primitive (`extractPlainText` in `guards.ts`) — pure, no runtime dependencies, the exact shape of helper we need.
- Every consumer already imports from text-blocks for the type. Zero new import paths.
- text-blocks' description says "Pure interfaces ... type guards - no runtime dependencies" — `extractPlainText` is already runtime helper code, so the precedent is set. Adding `flattenBlocks` is incremental.

Plan revision retroactively recorded here: ADR-174 OQ-1's "channel-service" recommendation is reversed in favor of text-blocks.

### `flattenBlocks` shape

```ts
export interface FlattenBlocksOptions {
  /** Separator between consecutive same-key blocks. Default `'\n'`. */
  readonly sameKeySeparator?: string;
  /** Separator between blocks with different keys. Default `'\n\n'`. */
  readonly differentKeySeparator?: string;
  /** Include status.* blocks in output. Default `false`. */
  readonly includeStatus?: boolean;
}

/**
 * Project an `ITextBlock[]` to a single string.
 *
 * - Decorations are stripped via the same `extractPlainText` walk.
 * - Same-key consecutive blocks are joined with `sameKeySeparator`
 *   (default single newline) so multiple "Taken." lines stay tight.
 * - Different-key transitions use `differentKeySeparator` (default
 *   double newline) so "West of House\n\nDescription..." reads naturally.
 * - status.* blocks are filtered unless `includeStatus` is true.
 *
 * Mirrors `renderToString`'s plain-mode behavior bit-for-bit. ANSI
 * mode is not provided here; no current consumer uses it. Future
 * terminal renderer ANSI support can land in a separate helper if
 * demand surfaces.
 */
export function flattenBlocks(
  blocks: ReadonlyArray<ITextBlock>,
  options?: FlattenBlocksOptions,
): string;
```

ANSI mode is dropped. Audit confirmed zero consumers pass `{ ansi: true }`. If the eventual terminal-renderer-with-styling work (ADR-174 Implementation Modules row "Terminal renderer translation of `IDecoration`") needs ANSI, that's its own helper at that time.

`renderStatusLine` is also dropped from the migration target. Audit found zero call sites; only re-exports. The re-exports vanish in 2.5.

## Sub-phase 2.1 — `flattenBlocks` in text-blocks

**Files (modified)**:

- `packages/text-blocks/src/index.ts` — add `flattenBlocks` to the named exports.

**Files (new)**:

- `packages/text-blocks/tests/guards.test.ts` — text-blocks has no tests today; this is the first one. (Vitest config and `test` script need to be added if missing — sub-phase 2.1 includes that small package config update.)

### Behavior Statement — `flattenBlocks`

- **DOES**: Returns a single string built from the input `ITextBlock[]`. Each block's `content` is flattened with `extractPlainText` (decorations stripped, nested content concatenated). Empty/whitespace-only blocks are dropped. Status blocks (`isStatusBlock`) are dropped unless `options.includeStatus === true`. Surviving blocks are joined: same-key consecutive pairs by `sameKeySeparator` (default `'\n'`), different-key transitions by `differentKeySeparator` (default `'\n\n'`). Empty input or fully-filtered input returns `''`.
- **WHEN**: Called by transcript renderers, chat overlays, and dev scripts that need a single string for display or assertion.
- **BECAUSE**: ADR-174 deletes `@sharpee/text-service`'s wire-production exports. Consumers still need a way to flatten blocks to strings. Hosting the helper in text-blocks (the package that already owns `ITextBlock`) keeps the migration import-line-only for every consumer.
- **REJECTS WHEN**:
  - Empty input array → returns `''`.
  - All blocks whitespace-only or filtered → returns `''`.
  - `options` is omitted or partial → defaults apply (`'\n'`, `'\n\n'`, `includeStatus: false`).

### Tests (derived from Behavior Statement + parity check against `renderToString`)

| # | Asserts | Source line |
|---|---------|-------------|
| F1 | Empty input → `''` | DOES + REJECTS |
| F2 | Single plain block → its text content | DOES |
| F3 | Two same-key blocks → joined with `'\n'` (default) | DOES |
| F4 | Two different-key blocks → joined with `'\n\n'` (default) | DOES |
| F5 | Custom separators respected | DOES |
| F6 | Whitespace-only blocks dropped | DOES |

## Risks and rollback

### Risks

- **zifmia chat bubble formatting regression.** zifmia's path is `blocks → renderToString → formatText (HTML escape + newline-to-br) → dangerouslySetInnerHTML`. `flattenBlocks` matches `renderToString` plain output bit-for-bit, so `formatText` should see the same input. Mitigation: 2.3 manual smoke.
- **Bundle re-export miss.** `transcript-tester/fast-cli.ts` destructures from a pre-built bundle. If the bundle's exports change between 2.2 and 2.5, transcript-tester might pick up a stale bundle. Mitigation: rebuild bundle between sub-phases when needed; the bundle-deref path goes away once the source migrates.
- **Story-dev script breakage.** cloak-of-darkness scripts are CommonJS; the import-statement swap is straightforward but easy to typo. Mitigation: 2.4's runtime smoke test catches any.

### Rollback per sub-phase

| Sub-phase | Revert command | Effect |
|-----------|---------------|--------|
| 2.1 | `git revert <sha>` | `flattenBlocks` removed from text-blocks; consumers can't use it but they haven't migrated yet, so no breakage. |
| 2.2 | `git revert <sha>` | transcript-tester reverts to importing renderToString from text-service. Still works because text-service is still compilable. |
| 2.3 | `git revert <sha>` | zifmia same. |
| 2.4 | `git revert <sha>` | cloak-of-darkness scripts same. |
| 2.5 | `git revert <sha>` | re-exports return; nothing depends on them anymore (consumers already migrated), so no behavioral effect — just one extra commit on the branch. |
| 2.6 | n/a — verification only | n/a |

The cleanest cumulative rollback is `git revert` of every Phase 2 sub-phase commit in reverse order. Each is independently revertible.

## What this plan does NOT do

- Does not delete `@sharpee/text-service`. That's Phase 3, contingent on Phase 2 landing.
- Does not add ANSI rendering to `flattenBlocks`. No consumer uses ANSI. A future ADR adds it if a real terminal-with-styling consumer materializes.
- Does not migrate anything that calls `extractPlainText` directly — that helper stays exactly as it is.
- Does not change channel-service or the renderer architecture. zifmia and transcript-tester continue consuming `engine.on('text:output')` blocks as they do today.
- Does not touch the prose pipeline, decoration vocabulary, or any Phase 1 deliverable.
- Does not address any of the cloak-of-darkness `text-services` (plural) breakage — those scripts are already broken on main and are out of scope for this migration.
