### Phase 7: State-machine transitions must not fire on refused/blocked actions
- **Tier**: Medium
- **Budget**: 250 tool calls
- **Domain focus**: `plugin-state-machine`'s trigger evaluation — both the action-trigger and event-trigger paths currently lack a success signal.
- **Entry state**: `plugin-state-machine/src/state-machine-runtime.ts:124-133`'s action triggers match on `actionId` + `target` only, with no success signal in `EvaluationContext`, so a transition fires even when the underlying action was refused. Event triggers have the same hole from a different angle: refusals reuse the primary event type with `blocked: true`/`failed: true`, so an event trigger can false-fire on a blocked event too.
- **Deliverable**: the engine populates a `success` flag on `EvaluationContext` for action triggers, and action-trigger evaluation requires it to be true before firing; event-trigger evaluation is fixed independently to skip any event carrying `blocked: true` or `failed: true`. Both paths get a test: a refused action attempt against a state-machine-gated entity must NOT advance the machine's state; a successful one must.
- **Exit state**: a `when turning …` (or any action-triggered) transition fires only on a genuinely successful action; event-triggered transitions never false-fire on a blocked/failed event reusing the primary event type.
