### Phase 3: Browser host wiring — bundle-map resolver for compile-at-boot
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: `packages/devkit/src/standalone/build-browser.ts` (the build-time step that currently ships only `story.story`) and `packages/devkit/templates/browser/chord-browser-entry.ts.template` (the client-side script that calls `compile(source)` at boot).
- **Entry state**: Phase 2 complete — devkit has a working fs-backed resolver pattern to model this on. **This phase carries a genuine open design question the ADR did not anticipate needing an answer to** (its "no structural change" claim was wrong, per Grounding) — present the two options below to David as part of requesting this phase's go-ahead; do not pick one unilaterally mid-implementation.
- **Deliverable** (design options to confirm before coding, then implement the chosen one):
  1. **Discovery**: `build-browser.ts` must find which `.chord` files a project's `.story` file imports. Cheapest option: after Phase 1's parse succeeds, walk the (already-parsed) AST's `ImportDecl` nodes for their `.chord`-appended paths — this reuses the compile the build already runs (`build-browser.ts:274`) rather than re-parsing or hand-rolling a separate scan. Flat-only (D5) means this is a single pass, no recursion.
  2. **Shipping**: two shapes to choose between —
     - **(a) fetch-per-file**: copy each imported `.chord` file into `dist/web/` alongside `story.story` (mirroring the existing `fs.copyFileSync` at `build-browser.ts:293`), and have the browser entry template `fetch()` each by name before calling `compile()`, assembling a `Record<name, text>` map into an `importResolver` closure.
     - **(b) inline bundle**: emit a single generated JSON/JS asset (`dist/web/imports.json` or similar) mapping `name.chord → text`, fetched once, turned into the same closure. Fewer round-trips, one more generated-artifact type to keep in sync with the build.
     Recommendation to bring to David: (a) is simpler and consistent with the existing "ship the source, compile at boot" philosophy (`build-browser.ts:258-262`'s comment); (b) is fewer HTTP round-trips but adds a new artifact shape. Either is a small addition, not a redesign — the choice doesn't gate Phase 1/2, only this phase's exact shape.
  3. Update `chord-browser-entry.ts.template` to build the resolver from whichever shape is chosen and pass `{ importResolver }` to `compile(source, …)` (currently `compile(source)` with no options, `template:30`).
  4. At least one real browser-build test: a fixture project with a `.story` importing a `.chord`, run through `sharpee build --browser` (or the standalone build-browser entry point directly), asserting the shipped `dist/web/` output contains what the chosen shape requires and that a headless load of the generated entry script compiles successfully with the fragment spliced in. (Per Integration Reality: a stub bundle map is scaffolding only — the acceptance gate needs the real build pipeline exercised once.)
  5. `packages/devkit` suite green (extended), and a manual or scripted smoke check that an actual `--browser` build of a two-file fixture story boots clean (mirrors the existing `./repokit build dungeo --browser` clean-build bar from other plans, scoped down to the new fixture rather than the full dungeo build, which has zero imports today).
- **Exit state**: A browser build of a project whose `.story` imports a `.chord` fragment ships everything the client needs and compiles correctly at boot, using the design shape David confirmed. `chord-browser-entry.ts.template` passes a real `importResolver`, not none. No regression to today's single-file browser stories (zero imports = zero behavior change, verified by re-running the existing browser build path for a story with no imports).
- **Design decision (David, 2026-07-21)**: **inline bundle** chosen over fetch-per-file — the build emits one `dist/web/imports.json` (`<name>.chord` → text); the entry fetches it once and builds a sync resolver. Rationale: the compiler's resolver is synchronous and the page can't discover imports until after it compiles, so both shapes must pre-fetch — which erases fetch-per-file's simplicity edge while it keeps 1+N round-trips + manifest-drift + partial-fetch failure modes. Inline = one fetch, one artifact, atomic, JSON-escaped.
