# Session Plan: Chord Dynamic Channels (ADR-241 implementation)

**Created**: 2026-07-18
**Overall scope**: Give Chord (`.story` author language) the ability to reach the platform's dynamic-channel surface from pure IR — `play ambient <asset> [in <word>]` / `stop ambient [in <word>]` become real in the browser, `define ambient <word>` / `define layer <word>` declare named beds/layers, undeclared channel words are a load error, and any story-declared channel with no story renderer is visible by default through a platform generic panel rather than only the console JSON tree. This closes the gap Fernhill Phase 8 exposed: `play ambient` compiled, degraded correctly in text, and was silently inaudible in the browser. **Custom UI implementations (a story shipping its own panels/layouts beyond the generic box) are explicitly OUT of scope** — ADR-241's scope note records that as future work riding this same mechanism, not a stretch of this plan's D2 family list or D4 generic panel.
**Bounded contexts touched**: N/A — this is compiler/loader/renderer plumbing, not new domain modeling (no `docs/ddd/notation.yaml` exists in this project). Packages touched: `packages/chord` (lexer/parser/analyzer/IR/catalog), `packages/ide-protocol` (type re-export), `packages/story-loader` (channel registration), `packages/platform-browser` (renderer binding + new generic panel), `packages/stdlib` (verify-only — builders already exist), plus `docs/reference/chord-grammar.md`, `docs/architecture/chord-grammar-changes.md`, `packages/chord/chord.ebnf`.
**Key domain language**: dynamic channel (ADR-163 §7's term for any story-registered `IOChannel`), family channel (`ambient` / `layer` — named, parameterized, platform-rendered by default) vs. data channel (plain `define channel`, content-agnostic), default bed/layer (`main`; `background`/`main`/`overlay` implied for layers), declared-beyond-implied (the never-guess rule — channel words beyond the defaults must be declared or the load fails with `analysis.unknown-channel`), generic panel (the platform's default renderer for an unrendered dynamic channel — one labelled box per channel id, distinct from the console JSON-tree debug fallback).

## Phases

### Phase 1: Chord compiler surface — grammar, analyzer, IR, goldens, grammar docs (D1–D3, D6, AC-2 compile-side)
- **Tier**: Large
- **Budget**: 350
- **Domain focus**: `packages/chord` (`parser.ts`, `analyzer.ts`, `catalog.ts`, `ir.ts`) and `packages/ide-protocol` (type re-export) — the full compile-time surface, plus its grammar documentation.
- **Entry state**: ADR-241 ACCEPTED (all five open questions ruled, adr-review 14/14). `packages/chord` builds and tests green on `chord-foundations`. No prior phase.
- **Deliverable**:
  - Parser: `in <word>` optional tail on both `play ambient <asset>` and `stop ambient` (mirrors the shipped `show image <asset> [in <layer>]` spelling — one ratchet row per D3). `define ambient <word>` and `define layer <word>` one-liner declarations, parallel to existing asset declarations.
  - Analyzer: channel-word resolution against the declared set plus the implied defaults (`main` bed; `background`/`main`/`overlay` layers need no declaration). A channel word used but neither declared nor implied is `analysis.unknown-channel`, a load error carrying a nearest-match suggestion. Family channels (declared or implied-and-used) are collected into `ir.channels: IRChannelDef[]` alongside existing `define channel` entries.
  - IR: `IRChannelDef` gains `family: 'data' | 'ambient' | 'layer'` (additive; every existing entry reads as `data`). Lowering stamps `channel` onto `media.ambient.play` / `media.ambient.stop` payload `data` (the field the platform's `createAmbientChannel` filter already expects).
  - Catalog: any new reserved words for the two declaration keywords (`ambient`, `layer` in declaration position), verified against the current reserved-word set rather than assumed clean.
  - `packages/ide-protocol`: re-export the extended `IRChannelDef` (mechanical; gates Phase 2's TypeScript compile against the new field).
  - Docs: `docs/reference/chord-grammar.md` (Extension surface + define sections), `docs/architecture/chord-grammar-changes.md` (two ratchet rows — the ambient channel word on both `play` and `stop` forms; the two declaration one-liners), `packages/chord/chord.ebnf`.
- **Test gate (REAL-PATH, rule 13a)**: new fixtures exercising (a) a declared ambient bed (`define ambient wind`; `play ambient X in wind`; `stop ambient in wind`), (b) a declared layer beyond the implied three, (c) an undeclared channel word (`play ambient X in wnd`) asserting the specific `analysis.unknown-channel` code and nearest-match suggestion text — not just "load failed" (AC-2), (d) a bare `play`/`stop ambient` resolving to the implied `main` bed with no declaration required. IR-shape assertions: `family` discriminator present and correct per case; `channel` stamped on the `media.ambient.play`/`stop` payload. Golden diff reviewed and confirmed additive-only (every existing golden gains only the `family` field — AC-6's compile-side clause). `pnpm --filter '@sharpee/chord' test` green.
- **Exit state**: `packages/chord` compiles ambient/layer channel words end to end, rejects undeclared ones with a specific diagnostic (AC-2 satisfied at the compile boundary), and emits `ir.channels` carrying the family discriminator plus stamped payload data. `packages/ide-protocol` re-exports the extended type. Both ratchet rows landed. Phase 2 can consume `ir.channels[].family` through the re-exported type.

### Phase 3: platform-browser family binding + generic panel (D4's client leg, AC-4, AC-5 precursor)
- **Tier**: Medium
- **Budget**: 250
- **Domain focus**: `packages/platform-browser` (`registerDefaultBrowserRenderers` in `src/channels/index.ts`; new generic-panel renderer module under `src/channels/`). The family renderer functions themselves already exist (`createAmbientChannelRenderer` in `audio.ts`, `createImageChannelRenderer` in `image.ts`) — this phase wires manifest-driven binding to them, it does not write new media renderers.
- **Entry state**: Phase 2 complete — a loaded story's manifest carries family-tagged channel ids with no exact-id renderer registered.
- **Deliverable**:
  - `registerDefaultBrowserRenderers` binds family renderers for any manifest channel id in a known family (`ambient:*` → `createAmbientChannelRenderer`, `image:*` → `createImageChannelRenderer`) that has no exact-id renderer already registered — extends today's hard-coded three image layers to any declared/implied layer id.
  - New generic-panel renderer module under `src/channels/` (e.g. `panel.ts`) implementing D4's pinned contract exactly: one labelled box per channel id, hidden until its first value; `replace` mode overwrites the box's key/value rows; `append` mode appends rows; `event` mode shows the latest value. Registered as the platform default for any dynamic channel id with no exact-id renderer and no family match.
  - Confirm ordering respects ADR-165's last-write-wins invariant: platform defaults (family + generic panel) register first, story overrides (`registerRenderer`) replace them — no change to that existing precedence rule, only new default registrations feeding into it.
  - Console JSON-tree fallback (ADR-165) is left untouched — the generic panel is the new player-facing default; the JSON tree remains the debug view underneath, per ADR-241 D4's explicit text.
- **Test gate**: new tests under `packages/platform-browser/tests/channels/` — (a) a manifest channel id in a known family with no exact-id renderer binds to that family's renderer automatically; (b) a dynamic channel with no exact-id renderer and no family match renders through the generic panel, and its box is hidden until the first value arrives, with `replace`/`append`/`event` mode behavior each asserted; (c) an exact-id story renderer registered after the defaults wins — only its output appears, the generic panel does not double-render the same channel (package-level precursor to AC-5, which gets its full real-browser proof against Fernhill's `clock` renderer in the closure phase). `pnpm --filter '@sharpee/platform-browser' test` green.

## Dependency summary
- This workstream is an inherently sequential pipeline — compile (Phase 1) → load/register (Phase 2) → render (Phase 3) → real-browser proof (Phase 4) — each phase's IR/manifest/registration output is the next phase's direct input. There is no independent or parallelizable phase to call out here, unlike sibling plans with an orthogonal ratchet item; every decision (D1–D5) was already ruled in the ADR interview, so there are no embedded stop-and-asks either, beyond the closure phase's flag-not-edit note above.
- Total budget across all four phases: 1,000 tool calls (Large 350 + Medium 250×2 + Small 150) — smaller than the door-loading (1,250) and author-pipeline workstreams, consistent with ADR-241's narrower, single-mechanism scope (custom UI explicitly excluded).
