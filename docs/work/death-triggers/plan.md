# Session Plan: Implement ADR-227 (Death Triggers Use the Prescribed Logic Surfaces)

**Created**: 2026-07-16
**Overall scope**: Re-home the ~15 Dungeo death sites (currently bespoke `killPlayer` re-points from the ADR-224 Phase-4 migration) onto their ADR-090-correct prescribed logic surface; build the three Chord authoring constructs (`kill the player`, `<direction> is deadly:`, `deadly:`) that lower to those same surfaces with no new platform mechanism; and fix the one real platform gap (`basicNpcResolver` still emitting legacy `if.event.death` instead of `killPlayer`). No new primitive is introduced anywhere — `killPlayer` (ADR-224) and `DeadlyRoomTrait` (ADR-224 Phase 2) are already built; every phase below is either a re-home onto an *existing* surface or new Chord grammar lowering to one.
**Bounded contexts touched**: `stories/dungeo` (falls, grue, cake, melt-glacier re-homes — story-level), `packages/extensions/basic-combat` (`basicNpcResolver` fix — platform), `packages/chord` (`parser.ts`/`catalog.ts`/`ast.ts`/`ir.ts` — new grammar, platform), `packages/story-loader` (`runtime.ts`/`loader.ts` — new lowering, platform). No `packages/world-model` or `packages/stdlib` changes are authorized by this ADR (killPlayer + DeadlyRoomTrait already exist).
**Key domain language**: `killPlayer` (the single mortality sink, ADR-224), `DeadlyRoomTrait` (verb-allowlist no-escape room), deadly *exit* (a lethal direction, new concept this ADR names but does not newly implement as a mechanism — it is an ADR-118/208 action interceptor on `going`), Capability Dispatch (ADR-090 — entity owns a verb's lethal meaning), Action Interceptor (ADR-118/208 — block/redirect a standard action), the ADR-090 decision tree (the rule that picks a site's surface).

- **Deliverable**:

Register a direction-specific `ActionInterceptor` on `'if.action.going'` for the Aragain Falls room + south direction (the ADR-118/208 registration API, used exactly as `gas-room-entry-interceptor.ts` already uses it — no new API) that calls `killPlayer({cause:'falls', messageId, terminal:true})`.   - **Grue** (`stories/dungeo/src/handlers/grue-handler.ts`, `stories/dungeo/src/actions/grue-death/grue-death-action.ts`): convert the current `ParsedCommandTransformer` (`createGrueDeathTransformer`, `grue-handler.ts:169-264`) to a proper `ActionInterceptor` on the movement action, per ADR-227's Decision 2 table ("gas-room entry, sphere/cage, grue-on-move → Action Interceptor"). **Also fix a live RNG policy violation found during this planning session's grounding**: `grue-handler.ts:147-149` calls raw `Math.random()`, not the engine's seeded RNG — re-point to the seeded pattern (matching `troll-daemon.ts`'s `ctx.random`/`world`'s RNG usage) while preserving the exact 75%/25% odds.   - Behavior Statements (CLAUDE.md rule 12) for the new falls-exit interceptor and the grue interceptor, before writing tests.
- **Behavior Statements**:
  - **Falls deadly-exit interceptor** — DOES: calls `killPlayer({cause:'falls', terminal:true})` when the player goes south from Aragain Falls. WHEN: the `going` action targets south from that specific room. BECAUSE: AC-3's deadly-exit semantics — only the fatal direction kills, all other verbs and directions are unaffected. REJECTS WHEN: any other direction or verb in that room — no interception, normal `going`/`waiting`/`taking`/`examining` behavior.
  - **Grue interceptor** — DOES: calls `killPlayer({cause:'grue', terminal:true})` on a seeded 75% roll when the player moves in a dark room with no light source. WHEN: a movement action succeeds into or within darkness without light. BECAUSE: ADR-227 Decision 2 classifies grue-on-move as Action Interceptor; the platform ships no darkness-death rule (ADR-227 Consequences, Q-3) so this is entirely Dungeo-authored logic living correctly at the interceptor seam. REJECTS WHEN: the room is lit or the player holds a light source — no roll, no interception.
- **Tests** (assert on actual world/event state, not return values):
  - Falls: a walkthrough/transcript confirms `wait`, `take`, `inventory`, `look`, `examine` all survive in the falls room (proves the over-implementation is gone); going south kills via `if.event.player.died{cause:'falls'}`; going north (or any other exit) does not.
  - Grue: existing grue transcripts (simple + mechanics) produce identical narrative outcomes; a seeded-RNG determinism test confirms the same seed reproduces the same live/die sequence; grep-gate confirms zero `Math.random()` remains in `grue-handler.ts` or its interceptor.

### Phase 4: Chord constructs — `kill the player`, `<direction> is deadly:`, `deadly:`
- **Tier**: Large
- **Budget**: 400 tool calls
- **Domain focus**: Platform (`packages/chord`, `packages/story-loader`). Three new authoring constructs, all lowering to already-built or already-registered-pattern platform surfaces — no new Chord mechanism, no new world-model/stdlib API.
- **Entry state**: `killPlayer` and `DeadlyRoomTrait` are built and exported. Phase 1's falls interceptor is a live worked example of "direction-specific `going` interceptor calling `killPlayer`" that this phase's `<direction> is deadly:` lowering can mirror directly (soft dependency — helpful precedent, not a hard blocker; this phase can proceed before Phase 1 lands using the same ADR-118/208 registration pattern gas-room-entry-interceptor.ts already demonstrates).
- **Deliverable**:
  - **`kill the player [when <cond>]`** — a bare statement, peer to `win`/`lose`:
    - `packages/chord/src/parser.ts` — add `'kill'` to `STATEMENT_OPENERS` (~line 154); a new case in the statement switch (near the `win`/`lose` cases, ~lines 1900-1912) parsing `kill the player`, an optional preceding `phrase <key>` reference (never an inline string, per the ADR's locked idiom), and `parseStatementWhen` for the optional `when <cond>` clause — reusing the existing helper, not a new one.
    - `packages/chord/src/ast.ts` — a `KillStmt` node, structurally a peer to `WinStmt`/`LoseStmt` (~lines 589-603): `phraseKey`, `stmtWhen`, `span`. No `cause` field — the ADR is explicit that cause is derived/defaulted, not authored.
    - `packages/chord/src/ir.ts` — an IR statement variant mirroring the `win`/`lose` IR shapes, carrying the (already-real, reused-not-rebuilt) `IRCondition`.
    - `packages/story-loader/src/runtime.ts` — a sibling case to the `'win'`/`'lose'` lowering (~lines 919-926): instead of `this.host.triggerEnding(...)`, import `killPlayer` from `@sharpee/stdlib` directly (already a `story-loader` package dependency) and call it with a derived cause/messageId from the phrase key. `triggerEnding`/`loader.ts:539-548` is untouched — a distinct lowering target, not routed through win/lose.
  - **`<direction> is deadly: <phrase>`** (deadly exit):
    - `packages/chord/src/parser.ts` — a sibling branch next to the existing `<direction> is blocked:` parse (`parseCreate`'s body loop, ~lines 518-521, delegating to `parseBlockedExit` ~lines 558-582): detect `isWord('deadly', 2)` and parse a mirroring `parseDeadlyExit` (direction, optional `while <cond>`, mandatory `: <phrase-key>`).
    - `packages/chord/src/ast.ts` — a `DeadlyExitDecl` node, peer to `BlockedExitDecl` (~lines 176-187).
    - `packages/chord/src/ir.ts` — an IR shape peer to `IRBlockedExit` (~lines 125-131).
    - `packages/story-loader/src/loader.ts`/`runtime.ts` — **this does not reuse the blocked-exit lowering path.** Grounded this planning session: blocked-exit lowers to a native `RoomTrait.blockedExits` field (`RoomBehavior.blockExit`, `loader.ts:278-285`) that stdlib's own `going.ts` reads directly (`going.ts:183-193`) — there is no interceptor involved. A deadly exit needs an actual `killPlayer` call, which that native field-read cannot produce. New lowering code registers a real `world.registerActionInterceptor(..., 'if.action.going', ...)` (the same ADR-118/208 API Phase 1's falls interceptor uses) for the declared direction, calling `killPlayer` with the phrase-derived cause/messageId.
  - **`deadly: <phrase>`** (no-escape room marker):
    - `packages/chord/src/catalog.ts` — a new `TRAIT_ADJECTIVES` entry `'deadly'` (~near `'dark'`, line 39), since this is a trait-adjective composition on `create`, not a statement — no new AST node needed (composes into the existing `CompositionItem`, ~lines 137-147).
    - `packages/story-loader/src/loader.ts` — a new `case 'deadly':` in `applyTraitAdjectives` (~lines 621-698), mirroring `case 'dark':` (~676-681): reads `safe:`/`chance <p>` config via `configValue` (the same helper `readable`/`lockable` already use) and attaches a `DeadlyRoomTrait{cause, messageId, safeVerbs, chance}` to the room entity. **Fail-fast**: a `deadly` room with no `safe:` clause is a load-time validation error naming the missing clause (ADR-226 AC-7 precedent), not a silent default.
  - A Chord `.story` fixture exercising all three constructs, alongside the ADR's idiom rules: message always a `phrase <key>` (never inline), conditions positive (no `not` — absence via `without`/`no <cond>`/antonym states), probability as `one chance in <n>`, no authored `cause`.
- **Behavior Statements**:
  - **`kill the player` lowering** — DOES: calls `killPlayer` with a derived cause/messageId when its (optional) condition evaluates true, at the same per-turn/clause-evaluation point `lose when` already uses. WHEN: the enclosing clause fires (`on`/`after <action>` → capability-dispatch/interceptor timing; `on every turn` → daemon timing). BECAUSE: reuses Chord's entire existing condition-evaluation seam (`evalCondition`, `story-loader/src/evaluator.ts:95`) — the only new thing is the sink. REJECTS WHEN: the condition is false — no call, no event.
  - **`<direction> is deadly:` lowering** — DOES: registers an action interceptor on `going` for the declared direction that calls `killPlayer` with the phrase-derived cause/messageId. WHEN: the player attempts to go that direction (optionally gated by `while <cond>`). BECAUSE: mirrors `<direction> is blocked:` syntactically but needs a real interceptor, not the native blocked-exit field, since it must call `killPlayer`. REJECTS WHEN: any other direction, or the optional `while` condition is false — normal `going` behavior.
  - **`deadly:` room-marker lowering** — DOES: attaches `DeadlyRoomTrait` with `safeVerbs` from `safe:` and optional `chance` from `chance <p>` (translated from `one chance in <n>`). WHEN: the story-loader parses a room composed with `deadly`. BECAUSE: AC-3 requires Chord and Sharpee-Way to be identical. REJECTS WHEN: no `safe:` clause — load-time validation error, not a silent default.
- **Tests**:
  - Parser unit tests: `kill the player [when <cond>]`, `<direction> is deadly: <phrase>`, and `deadly: <phrase>` (with and without `safe:`, expecting a load-time error in the latter case) all parse to the expected AST/IR shapes.
  - Runtime/loader integration: a `.story` fixture exercising all three constructs produces `if.event.player.died` with the correct cause/messageId in each case, and routes to `game.lost` when unhandled.
  - Parity fixture (AC-1, AC-3): the Chord fixture's `deadly:` room and `<direction> is deadly:` exit produce behavior identical to Phase 1's Sharpee-Way falls interceptor and to a `DeadlyRoomTrait` fixture — the literal "both Ways identical" check.
  - Regression: existing `.story` fixtures using `win`/`lose`/`<direction> is blocked:`/`dark` compile and run unchanged — confirms no shared code path broke.
- **Exit state**: `.story` authors can express all three ADR-227 death constructs. The AC-1/AC-3 parity fixtures pass. Existing Chord fixtures (cloak, zoo) unaffected. `packages/chord` and `packages/story-loader` build and test green.
- **Status**: PENDING
