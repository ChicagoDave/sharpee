## Notes

### Augmentation Pattern (from `packages/helpers/src/augment.ts`)

The helpers package augments `WorldModel` (the class), NOT `IWorldModel` (the interface), to avoid forcing `AuthorModel` to implement the added methods. However, ADR-150 explicitly states the augmentation should target `IWorldModel` because the entry points are read-only and safe for `AuthorModel`. Follow the ADR — use `IWorldModel` declaration merging, patch `WorldModel.prototype`.

```typescript
declare module '@sharpee/world-model' {
  interface IWorldModel {
    readonly entities: EntityQuery;
    // ...
  }
}
(WorldModel.prototype as any).entities = get function() { ... };
```

### Build Position in build.sh

`@sharpee/queries` depends only on `@sharpee/world-model`. Insert after `@sharpee/helpers` in the PACKAGES array so it builds immediately after its only dependency.

### ADR-149 Forward Declarations

Until ADR-149 is implemented, `EntityType.REGION` and `EntityType.SCENE` do not exist in `@sharpee/if-domain`. The queries package must either:
- Define local stub constants that match the values ADR-149 will use, or
- Guard the `w.scenes` and `w.regions` getters to return `EntityQuery([])` with a safe fallback when the entity types are absent

The ADR states: "Until ADR-149 is implemented, `w.scenes` and `w.regions` return empty queries." The forward declarations are "just enough for type checking and filtering." Design the stubs accordingly — do not introduce values that will conflict when ADR-149 ships.
