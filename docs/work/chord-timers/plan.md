# Session Plan: Implement ADR-217 (Chord Timer & Scheduler Controls)

**Created**: 2026-07-14
**Overall scope**: Expose the scheduler's *already-implemented* imperative timer API (`packages/plugin-scheduler`) to Chord authors — first-class fuses (`in N turns:`, `after N turns as <name>:`, `cancel`/`reschedule`/`pause`/`resume`), period timers (`on every N turns …`), data-driven delays over Chord's existing value-expression grammar, named `early`/`normal`/`late` priority bands, `cancel sequence <name>`, and entity-bound auto-cleanup (`entityId` + `cleanupEntity`). This is **compiler + loader work only** — no new scheduler-service methods, no platform scheduler change. The one platform-adjacent change is widening the story-loader's *reduced* `SchedulerDaemon` struct (`runtime.ts:101-106`) to the real `Daemon` type (`plugin-scheduler/src/types.ts:37-61`).
**Bounded contexts touched**: Chord composable vocabulary and IR (`packages/chord/src/{catalog,parser,analyzer,ir}.ts`), story loading / scheduler-construction (`packages/story-loader/src/{runtime,loader}.ts`), and the ADR-210 grammar ratchet (`docs/architecture/chord-grammar-changes.md`). No `packages/plugin-scheduler` change — the scheduler's `Daemon`/`Fuse` types and `SchedulerService` controls are consumed as-is.
**Key domain language**: fuse (one-shot countdown), daemon (recurring), named-timer-handle (lexically-scoped `<name>` binding from `as <name>`), entity-bound timer (cross-action control idiom via `entityId` + `cleanupEntity`), priority band (`early`/`normal`/`late` → reserved scheduler integers `+10`/`0`/`-10`), grammar ratchet (ADR-210).

## Grounding (confirmed by direct source read, 2026-07-14)

- `Daemon` interface, full shape: `{ id, name, condition?, run, priority?, runOnce?, getRunnerState?, restoreRunnerState? }` (`packages/plugin-scheduler/src/types.ts:37-61`).
- `Fuse` interface, full shape: `{ id, name, turns, trigger, entityId?, tickCondition?, onCancel?, priority?, repeat?, originalTurns? }` (`types.ts:66-96`).
- `SchedulerService` controls: `registerDaemon`/`removeDaemon`/`pauseDaemon`/`resumeDaemon`/`hasDaemon` (`scheduler-service.ts:89-123`); `setFuse`/`cancelFuse`/`getFuseRemaining`/`adjustFuse`/`pauseFuse`/`resumeFuse`/`hasFuse` (`scheduler-service.ts:127-197`).
- Story-loader's reduced struct today: `SchedulerDaemon { id, name, condition?, run }` and `SchedulerTick { world, turn, playerLocation? }` (`runtime.ts:94-106`) — no `priority`/`runOnce`/serialization hooks; a "structural mirror... registration-compatible" comment marks it as the drift-prone parallel type ADR-217 retires.
- `buildSchedulerDaemons()` (`runtime.ts:656-780ish`) constructs three daemon families today: one world-state-pointer daemon per `define sequence` (progression key + fired-turn key in world state, no runner-state plumbing, per its own header comment — "ALL progression state is namespaced world state... with no getRunnerState plumbing (design.md §6)"), one daemon per entity `on every turn` clause, and one per trait `on every turn` clause. Registered via `loader.ts:429,434` (`registerSchedulerConstructs` calling `scheduler.registerDaemon(daemon)` for each). Fuses are **never constructed** by the loader today — `setFuse`/`cancelFuse`/etc. are entirely unreached from Chord.
- Statement grammar precedent: the closed statement-keyword list lives at `parser.ts:154` (`'refuse', 'phrase', 'emit', 'set', 'change', 'move', 'remove', 'award', 'win', 'lose'`), each with a matching discriminated-union member in `ir.ts` (e.g. `{ kind: 'move'; entity: IRValue; place: IRValue; stmtWhen?; span }` at `ir.ts:344`, `{ kind: 'remove'; ...}` at `ir.ts:346`) and a parse case in `parser.ts` (`case 'move':` / `case 'remove':` ~1858-1881) plus an analyze case in `analyzer.ts` (~1099-1233). New timer statements (`in`, `after ... as`, `cancel`, `reschedule`, `pause`, `resume`) join this exact keyword/IR-variant/parse-case/analyze-case pattern — no new mechanism, six more members of an established closed set.
- `EVENT_VERBS` (`catalog.ts:78`) is today `Set(['entering'])` — unrelated to timers, but confirms the catalog's closed-set-with-single-owning-array shape that priority-band and other new tokens will extend similarly (a new `PRIORITY_BANDS` or equivalent closed table, not a parallel ad hoc check).
- Sequence step timing today: `IRSequenceStep.timing: 'at-turn' | 'later' | 'becomes'` with NUMBER-only turn counts (`ir.ts:324-325`, `parser.ts:1619-1632`) — the data-driven-delay work (Phase 3) generalizes the turn-count slot to a value-expr without touching this timing-kind union.
- The old top-level `once`/`every N turns` forms are already removed from the parser (confirmed absent; ADR-214 §4 cites `parser.ts:237,245` as their prior, now-removed location) — Phase 3's `on every N turns` is a **new, owner-attached** clause form, not a revival of the removed top-level rule (consistent with Given 9 stickiness, ratchet entry 2026-07-11).

## Phases

### Phase 1: Widen the loader's Daemon + build fuse-construction infrastructure
- **Tier**: Medium
- **Budget**: 250 tool calls
- **Domain focus**: Loader-internal plumbing — no new author-facing syntax yet. This phase is the prerequisite infra every later phase's imperative controls sit on: replacing the reduced `SchedulerDaemon` with the real `Daemon`, and adding the loader-side machinery to construct and register `Fuse` objects (which the loader does not do at all today).
- **Entry state**: David's explicit go-ahead to implement ADR-217 has been given (see Entry gate above). W1 (ADR-218 foundations) is BUILT. ADR-217 is ACCEPTED.
- **Deliverable**:
  - No grammar-ratchet entries required (pure loader-internal + IR-schema-that-doesn't-alter-surface-syntax work, explicitly exempt per the ratchet log's own scope rule).
  - `packages/story-loader/src/runtime.ts`: delete the reduced `SchedulerDaemon`/`SchedulerTick` structural-mirror types (`runtime.ts:94-106`); `buildSchedulerDaemons()` returns real `plugin-scheduler` `Daemon[]` imported from `@sharpee/plugin-scheduler` instead. Each of the three existing daemon-construction sites (sequence, entity every-turn, trait every-turn) gets `priority: 0` (normal band, today's behavior — see Phase 3 for band wiring) and leaves `runOnce`/`getRunnerState`/`restoreRunnerState` undefined (unchanged behavior; these constructs keep using world-state pointers per the existing design note, not the new serialization hooks — ADR-123 constraint above).
  - `SchedulerContext` (the real one, `plugin-scheduler/src/types.ts:17-32`) reconciled against the loader's calling convention where the widened fields need it (e.g. `playerLocation`/`playerId` non-optional vs. the loader's optional `SchedulerTick.playerLocation`) — a mechanical adapter at the one call site, not a new context type.
  - New loader-side fuse-construction path: a `buildSchedulerFuses(): Fuse[]`-shaped method (or equivalent registered alongside daemons) that the story's engine-ready hook calls to `setFuse` any IR-declared fuses — empty/no-op in this phase (no IR fuse nodes exist yet), but the registration wiring (analogous to `loader.ts:429-434`'s `registerDaemon` loop) is built and unit-tested against a hand-constructed `Fuse` object so Phase 2 has a proven call path to compile into.
  - A unit test (not a `.story` fixture — no new syntax exists yet) asserting: `buildSchedulerDaemons()` returns objects satisfying the real `Daemon` interface (TypeScript structural check plus a runtime assertion that `priority`/`runOnce` keys are present, even if `undefined`/`0`), and a hand-built `Fuse` registered through the new fuse-registration path is retrievable via `scheduler.hasFuse(id)` / fires via `scheduler.getFuseRemaining(id)` after simulated ticks.
- **Exit state**: `buildSchedulerDaemons()` emits the real `Daemon` type (verified by the unit test); the reduced `SchedulerDaemon` struct no longer exists in `runtime.ts`. The fuse-registration call path exists and is proven against a hand-built `Fuse`, ready for Phase 2 to target. `cloak.story` and `zoo.story` still compile and their walkthroughs still pass unchanged (pure internal refactor — run once via `node dist/cli/sharpee.js --test --chain` against both stories' existing transcripts as the regression check for this phase).
- **Status**: CURRENT

### Phase 2: Named fuses + imperative controls (cancel/reschedule/pause/resume) + entity-binding
- **Tier**: Large
- **Budget**: 400 tool calls
- **Domain focus**: The core author-facing surface — fuses as first-class Chord constructs with a lexically-scoped named-timer-handle mechanism, the four imperative controls, and entity-bound auto-cleanup. This is the largest phase: six new statement forms across parser/IR/analyzer/loader, plus scope-checking that has no precedent elsewhere in Chord (every existing name reference — entity, trait, condition, phrase — is checked against a flat catalog or declaration list, never a lexical block scope).
- **Entry state**: Phase 1 complete (real `Daemon` in place; fuse-registration path proven).
- **Deliverable**:
  - Six dated ratchet entries, each written before its code, following the existing log's `| Date | Form | Rationale | Example | Decision |` row format:
    1. **Anonymous one-shot** — `in N turns: <statement>` (fire-and-forget; compiles to an unnamed `setFuse` call with no handle).
    2. **Named fuse** — `after N turns as <name>: <statement>` — binds the timer-handle `<name>`, lexically scoped to the enclosing block/clause (not story-global — the ADR-217 resolution).
    3. **Cancel** — `cancel <name>` (`cancelFuse`; runs the fuse's `onCancel` if declared).
    4. **Reschedule** — `reschedule <name> by <delta>` (`adjustFuse`; `<delta>` may be negative).
    5. **Pause/resume** — `pause <name>` / `resume <name>` (`pauseFuse`/`resumeFuse`).
    6. **Entity-bound timers** — `after N turns as <name> on <entity>: <statement>` (or the ADR-217-designed equivalent binding form — confirm exact spelling against ADR-217's entity-binding note before writing the ratchet entry; the ADR resolves the *mechanism* — `entityId` + `cleanupEntity` — but the exact declaration spelling for "bind this fuse to an entity" is one of the "spellings under design" items ADR-217 left to this implementation phase, so pin it in the ratchet entry, not assume it).
  - `packages/chord/src/parser.ts:154`: extend the closed statement-keyword list with `in`, `cancel`, `reschedule`, `pause`, `resume` (the `after ... as ...` form reuses the existing `after` keyword with a new trailing grammar, disambiguated from the D3 `after <verb> it` clause-header form by position — a clause header is a block opener, `after N turns as <name>:` is a body statement; the parser must distinguish these by what follows `after` — a NUMBER vs. a verb-gerund — not by a new keyword).
  - `packages/chord/src/ir.ts`: new discriminated-union members alongside `move`/`remove` (`ir.ts:344-346`) for each of the six forms — `{ kind: 'fuse'; turns: IRValue; name?: string; entityId?: IRValue; body: IRStatement[]; span }` (anonymous vs. named collapses to `name` present/absent), `{ kind: 'cancel-fuse'; name: string; span }`, `{ kind: 'reschedule-fuse'; name: string; delta: IRValue; span }`, `{ kind: 'pause-fuse' | 'resume-fuse'; name: string; span }`.
  - `packages/chord/src/analyzer.ts`: the **lexical-scope table** — a new analysis pass tracking, per block/clause, which `<name>` handles are in scope (declared by an enclosing `after N turns as <name>:`) and rejecting (a) `cancel`/`reschedule`/`pause`/`resume` referencing a name not in scope (`analysis.timer-handle-out-of-scope` or equivalent, with a fix-it suggesting entity-binding for cross-action control — this is the ADR-217 consequence made concrete as a load error, not just documentation) and (b) a duplicate handle name shadowing an outer one in the same scope chain (never-guess: ambiguous which fuse `cancel <name>` targets). This scope table is new infrastructure — no existing Chord analysis pass tracks lexical scope; base it on the block-nesting the parser already produces for clause bodies, but write it as its own pass, not bolted onto the existing flat-catalog name resolver.
  - `packages/story-loader/src/loader.ts` / `runtime.ts`: compile each IR fuse-statement kind into a `scheduler.setFuse(...)`/`cancelFuse`/`adjustFuse`/`pauseFuse`/`resumeFuse` call at the statement's execution point (reusing the existing statement-execution dispatch `execStatements` uses for `move`/`remove`); entity-bound fuses set `Fuse.entityId` and rely on the scheduler's existing `cleanupEntity` (auto-called wherever the loader already handles entity removal — confirm this is the same choke point Z6's `remove <entity>` statement uses, `runtime.ts` per the Z6 ratchet entry, so `cleanupEntity` is invoked from that one place, not duplicated).
  - Four fixture `.story` files (AC-1, AC-3): (a) a named fuse (`after N turns as bomb: lose …`) that fires and asserted via the countdown; (b) a separate path that arms then `cancel`s the same fuse before it fires, asserted by the loss NOT occurring; (c) a fixture exercising `reschedule … by` (arm at N, reschedule by +2, assert it fires 2 turns later than the un-rescheduled baseline) and `pause`/`resume` (arm, pause, tick several turns, assert no countdown progress, resume, assert it then fires on schedule); (d) an entity-bound timer fixture where the entity is removed via `remove <entity>` before the fuse fires, asserting the trigger never runs (`cleanupEntity`, AC-3).
  - A rejection-test fixture: a `cancel <name>` (or `reschedule`/`pause`/`resume`) referencing a handle out of lexical scope, asserting the load error and its fix-it.
- **Exit state**: All four fixtures pass via the bundle (`./repokit build` then `node dist/cli/sharpee.js --test <fixture>.story` for each); the rejection fixture produces the expected load error. The `chord-availability-audit.md` "Daemons/fuses" row gains a note that fuse controls (cancel/reschedule/pause/resume) and entity-bound auto-cleanup are now reachable (full flip deferred to Phase 4's sweep, once Phase 3's remaining items land too). `cloak`/`zoo` still compile and their walkthroughs still pass unchanged.
- **Status**: PENDING

### Phase 3: Period timers, data-driven delays, priority bands, `cancel sequence`
- **Tier**: Medium
- **Budget**: 250 tool calls
- **Domain focus**: The remaining ADR-217 surface — all additive over Phase 1/2's infrastructure, none requiring new lexical-scope machinery. Bundled together because each is a comparatively small, independent grammar extension (a new clause form, a value-expr substitution in an existing NUMBER slot, an optional trailing modifier, and one more statement reusing Phase 2's `cancel` precedent) rather than because they are tightly coupled to each other.
- **Entry state**: Phase 2 complete (fuse statements, IR kinds, and the scheduler-call compilation path all exist; period timers and bands are additive alongside them, not novel infrastructure).
- **Deliverable**:
  - Four dated ratchet entries, written before their code:
    1. **`on every N turns …`** — a period-timer clause form (owner-attached, per Given 9 stickiness — joins the existing `on every turn` clause family on entities/traits, `runtime.ts:705-729`/`735-767`, generalizing the period from 1 to N; NOT a revival of the removed top-level `every N turns` rule per ADR-214 §4's grounding).
    2. **Data-driven delays** — the turn-count slot in `in N turns:`, `after N turns as <name>:`, and sequence-step timings (`at turn N` / `N turns later`, `ir.ts:324-325`) accepts Chord's existing value-expression grammar in place of a bare NUMBER literal (e.g. `in (fuse-length of the bomb) turns: …`) — no bespoke delay grammar, per ADR-217's explicit resolution; confirm the exact value-expr production Chord already uses elsewhere (condition/`with`-field values) and reuse its parser rule, not a parallel numeric-expression parser.
    3. **Priority bands** — an optional trailing `early`/`normal`/`late` token on a timer declaration (`on every turn, late: …`, `after 3 turns as bomb, early: …`, and — confirm against ADR-217 — likely also on `on every N turns`); a closed lookup table (`early: +10, normal: 0, late: -10`) feeding the `Daemon.priority`/`Fuse.priority` field Phase 1 wired through; the raw integer is never author-facing (analyzer rejects a bare number in band position with a fix-it naming the three words).
    4. **`cancel sequence <name>`** — reuses Phase 2's `cancel` statement shape but targets a `define sequence` timeline instead of a fuse handle (the sequence's own IR name, not a lexically-scoped handle — sequences are story-owned/top-level per Given 9, so no scope-check needed here, unlike Phase 2's fuse handles); compiles to `scheduler.removeDaemon(id)` on the sequence's daemon id (`chord.sequence.<slug>`, `runtime.ts:685`) plus clearing its world-state progression pointer so a cancelled sequence does not silently "resume" if re-triggered.
  - Parser/IR/analyzer/loader changes for each of the four, following Phase 2's established six-form pattern (keyword admission, IR discriminated-union member, parse case, analyze case, loader compile step).
  - Four fixture `.story` files (AC-2, AC-4 partial): (a) `on every N turns` firing on the Nth turn and not on intermediate turns; (b) a data-driven delay (`in (value-expr) turns: …`) where the value-expr reads world/entity state, asserting the fuse fires at the world-state-determined turn, not a fixed one; (c) two competing `on every turn` clauses on different owners in different bands (`early`/`late`), asserting the early one's effects are observable before the late one's within the same turn's event order; (d) `cancel sequence <name>` on a running `define sequence`, asserting no further steps fire after cancellation.
- **Exit state**: All four fixtures pass via the bundle. The band-ordering fixture's assertion is on **event order within one turn**, not just "both fired" (per the project profile's Event Sourcing mutation-assertion guidance — assert the actual dispatch order, not merely that dispatch occurred). `cloak`/`zoo` still compile and their walkthroughs still pass unchanged.
- **Status**: PENDING
