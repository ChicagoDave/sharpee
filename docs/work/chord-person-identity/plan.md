# Session Plan: Chord Person Identity (ADR-242 implementation)

**Created**: 2026-07-19
**Overall scope**: Close two folded gaps in how Chord persons render: (1) every Chord NPC currently loads as a common noun ("You can see a Tobias here.") because the loader pins `article: undefined` on `IdentityTrait` and never sets `properName`; (2) the ADR-197 pronoun seam is fully built but never fed from entities, so every singular NPC renders as "it". ADR-242 resolves both with one person-identity gate: an explicit, person-scoped `proper` trait adjective; a `pronouns <word>` person body line (standard four sets, by-number fallback when absent — never guessed); and a story-definable named pronoun set (`define pronouns <name> ... end pronouns`, five case rows) for non-standard gender identity. This workstream is the same shape as ADR-241 (chord grammar → loader → closure) but narrower — no client-rendering leg, since this is pure text-rendering plumbing, not a new channel/renderer surface.
**Bounded contexts touched**: N/A — this is compiler/loader/language-layer plumbing, not new domain modeling (no `docs/ddd/notation.yaml` exists in this project). Packages touched: `packages/chord` (parser/analyzer/catalog/IR/docs), `packages/ide-protocol` (type re-export), `packages/world-model` (`IdentityTrait.pronounSet`, additive), `packages/stdlib` (`nounPhraseFor` copies the field), `packages/lang-en-us` (pronoun-set registry consulted by `genderOf`/`renderPronoun`), `packages/story-loader` (person branch: `proper` mapping, pin removal, `pronouns` mapping, defined-set registration through `extendLanguage`), `stories/fernhill` (Tobias + Mrs Kettle migration).
**Key domain language**: proper-named person (`properName: true, article: ''` — bare "Tobias" everywhere), person-only trait adjective (`proper`, the first kind-scoped entry in `TRAIT_ADJECTIVES` — a new analyzer capability, not existing infrastructure), pronoun set (the four standard rows `he`/`she`/`it`/`they`, or a story-defined named set), by-number fallback (no `pronouns` line → no `pronounSet` → assembler's existing default, "it" for singular — never-guess rule, no injected default per ruled Q-2), declared-beyond-standard (an unknown `pronouns` word is a compile diagnostic with a nearest-match suggestion, mirroring `analysis.unknown-channel`), define-block-with-named-rows (the ADR-241 `define channel ... end channel` precedent D7 reuses for `define pronouns <name>` with exactly five case rows).

## Phases

### Phase 1: Chord compiler surface — `proper` adjective, `pronouns` line, named pronoun-set block (D1, D4 confirmation, D5, D7 compile-side, AC-2, AC-6 compile-side)
- **Tier**: Large
- **Budget**: 350
- **Domain focus**: `packages/chord` (`catalog.ts`, `parser.ts`, `analyzer.ts`) and `packages/ide-protocol` (type re-export) — the full compile-time surface for person identity, plus its grammar documentation.
- **Entry state**: ADR-242 ACCEPTED (both open questions ruled via interview; adr-review 13/13 post-interview). `packages/chord` builds and tests green on `chord-foundations` (370/370 confirmed by the prior session's closure run). No prior phase.
- **Deliverable**:
  - Catalog (`catalog.ts`): add `proper` to `TRAIT_ADJECTIVES`. Confirm `KIND_NOUNS` already includes `'person'` (verified live — it does). Since `TRAIT_ADJECTIVES` today is a flat `ReadonlySet<string>` with **no existing kind-scoped entry**, the analyzer needs a small new capability — a kind-scope table (or inline check) keyed to `proper` only, per D1's explicit note that this is "the first kind-scoped entry." Do not generalize this into a full kind-scope framework beyond what `proper` needs; the ADR scopes the door narrowly.
  - Analyzer (`analyzer.ts`): (a) reject `proper` composed on any kind other than `person` with a person-only diagnostic message; (b) reject `proper while <cond>` with its own legible diagnostic (D1 — identity isn't conditional; distinct from the generic conditional-composition-legality error so authors read the specific reason); (c) `pronouns <word>` resolution against the standard four (`he`/`she`/`it`/`they`) plus any story-defined set collected from `define pronouns` blocks in the same file/pass; an unrecognized word is `analysis.unknown-pronoun` (or the ADR's chosen code — confirm exact diagnostic-code naming against `analysis.unknown-channel`'s precedent) with a nearest-match suggestion; a duplicate `pronouns` line on the same person is a diagnostic; (d) `define pronouns <name> ... end pronouns` block parsing — five named rows (`subject`, `object`, `possessive`, `possessive-pronoun`, `reflexive`), each row dispatched by first-word match exactly like `parseDefineChannel`'s row loop; missing row, duplicate row, and a defined-set name that shadows a standard word (`he`/`she`/`it`/`they`) are each diagnostics with rejection tests.
  - Parser (`parser.ts`): `pronouns <word>` as a person body line parallel to the existing `aka` parsing (`parser.ts:506-509` — comma/word list on the entity's own line, not a block); `define pronouns <name>` block-opener parsing mirroring `parseDefineChannel` (`parser.ts:2298-2380`) structurally, closed by `end pronouns`.
  - IR (`ir.ts` / wherever `IREntity` is defined): `IREntity` gains an additive optional `pronouns?: string` field (the word or defined-set name); a new IR manifest entry for defined pronoun sets (analyzer collects `{ name, forms: {subject, object, possessive, possessivePronoun, reflexive} }` per set, IR-manifest-style per the ADR-241 precedent) so the loader (Phase 2) can register them.
  - `packages/ide-protocol`: re-export the extended `IREntity` and the new pronoun-set manifest type (mechanical; gates Phase 2's TypeScript compile against the new fields, same as ADR-241 Phase 1's re-export).
  - Docs: `docs/reference/chord-grammar.md` (People section — document `proper` and `pronouns`, plus the `define pronouns` block), `docs/architecture/chord-grammar-changes.md` (three ratchet rows: `proper` adjective, `pronouns` person line, `define pronouns` block-definition), `packages/chord/chord.ebnf`.
- **Test gate (REAL-PATH, rule 13a)**: new fixtures exercising — a `proper` person compiles with `IREntity.pronouns` absent (no pronoun line); a `proper` person with `pronouns she`; `proper` on a non-person kind rejected with the person-only diagnostic (AC-2); `proper while <cond>` rejected with the unconditional diagnostic (AC-2); an unknown `pronouns` word rejected with a specific diagnostic code and nearest-match suggestion text asserted, not just "load failed" (AC-6); a duplicate `pronouns` line on one person rejected (AC-6); a `define pronouns ze` block with all five rows present compiles and collects into the IR manifest; a block missing one row, one with a duplicate row, and one whose name shadows `he`/`she`/`it`/`they` are each rejected with specific diagnostics (AC-6). Golden diff reviewed and confirmed additive-only — every existing golden gains, at most, the `pronouns` field as absent/undefined on entities that don't use it (AC-8's compile-side clause). `pnpm --filter '@sharpee/chord' test` green.
- **Exit state**: `packages/chord` compiles `proper` and `pronouns` end to end, rejects every malformed/out-of-scope form with a specific diagnostic (AC-2, AC-6 satisfied at the compile boundary), and emits `IREntity.pronouns` plus an IR manifest of defined pronoun sets. `packages/ide-protocol` re-exports the extended types. All three ratchet rows landed. Phase 2 can consume the new IR shapes through the re-exported types.

- **Entry state**: Phase 1 complete — `IREntity.pronouns` and the defined-pronoun-set IR manifest exist and are re-exported through `ide-protocol`.
- **Deliverable**:

  - `packages/stdlib/src/utils/noun-phrase.ts`: `nounPhraseFor` copies `entity.pronounSet` onto `NounPhrase.pronounSet` alongside its existing `nounType`/`grammaticalNumber`/`properName`/`article` reads (confirmed the field is read nowhere today — this is the missing wire).
  - `packages/lang-en-us`: add a pronoun-set registry — `registerPronounSet(name, forms)` writing into a `Map` alongside `LanguageProvider`'s existing `this.messages` map (the `addMessage` precedent, confirmed structurally identical — `packages/lang-en-us/src/language-provider.ts:639-641`). `genderOf`/`renderPronoun` (`packages/lang-en-us/src/assembler/english-assembler.ts:415-437`) consult the registry **before** the standard-row lookup and the by-number fallback — `genderOf` already branches on `ref.pronounSet`, so this is inserting one more lookup tier ahead of the existing `he`/`she`/`it`/`they` string match, not restructuring the function.
  - `packages/story-loader/src/loader.ts`: person branch (currently `case 'person':` around line 961) — (a) when `proper` is present in the composed traits, construct `IdentityTrait` with `properName: true, article: ''` instead of today's `article: undefined`; (b) **delete** the `article: undefined` pin and its FLAG comment (D3 — after this phase, no code path in the loader constructs `IdentityTrait` with `article: undefined`; a non-`proper` person gets the plain default `article: 'a'`); (c) map `IREntity.pronouns` (a standard word or a defined-set name) onto `IdentityTrait.pronounSet`; (d) `applyTraitAdjectives` (`loader.ts:1136`, flat `switch (trait.name)`) gains `case 'proper': break;` — the first `case` in that switch to key behavior off `kind === 'person'` at all, since without it the adjective falls through to the authored-trait default path and mints a spurious `ChordDataTrait` (confirmed live: no case in the switch currently branches on kind); (e) defined pronoun sets from Phase 1's IR manifest are registered through the existing `extendLanguage(language)` seam (`loader.ts:585-597`, the same structural probe `addMessage` uses today — checking for a `registerPronounSet` method on the `LanguageProvider` and throwing a legible `LoadError` if absent, exactly like the existing `addMessage` check) — keeps the loader locale-neutral, no lang-en-us import in story-loader.
- **Test gate (REAL-PATH, rule 13a)**: no stubs of `IdentityTrait`, `LanguageProvider`, or the chord loader's real registration path.
  - AC-1: load a fixture with `a person, proper` and assert on the **real loaded world** — `IdentityTrait.properName === true`, `article === ''` — then run it through the real `nounPhraseFor` → real assembler render and assert the rendered listing string is bare ("Tobias", not "a Tobias").
  - AC-3: load a fixture with a plain (non-`proper`) person and assert on the real loaded world — `article === 'a'`, `properName === false`; additionally assert **no** `IdentityTrait` in the loaded world carries `article: undefined` (grep-style structural assertion across all loaded entities, not just the one fixture person).

  - **D6 no-injected-default check**: load a fixture with a `proper` (or plain) person that has **no** `pronouns` line and assert on the real loaded world that `IdentityTrait.pronounSet` is `undefined` — then drive it through the real pipeline and assert the rendered pronoun atom falls through to the existing by-number default ("it" for singular). This is a distinct assertion from AC-1/AC-3 (which cover the article, not the pronoun axis) and is the direct test of D6's explicit ruling that no default is injected.
  - AC-5: load a fixture with a `define pronouns ze` block (five declared forms) referenced by a person's `pronouns ze` line; assert the loader's `extendLanguage` call reaches a **real** `registerPronounSet` on a real `LanguageProvider` instance, and the real assembler renders the declared forms (not the by-number fallback) for that person's pronoun atoms.
  - `pnpm --filter '@sharpee/world-model' test`, `pnpm --filter '@sharpee/stdlib' test`, `pnpm --filter '@sharpee/lang-en-us' test`, `pnpm --filter '@sharpee/story-loader' test` all green.
