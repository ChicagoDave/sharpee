# Session Plan: Implement ADR-219 (Liquids & Pouring)

**Created**: 2026-07-14
**Overall scope**: Build the full liquids subsystem ADR-219 designs — transfer (`pouring`/`filling`/`emptying`) and mixing (reactions producing new substances) — on top of the marker `LiquidTrait` that W1 (ADR-218) introduces. World-model gains `LiquidTrait` intrinsics (type identity, reaction rules) and a reaction behavior, with the container remaining the single source of truth for amount; stdlib gains four new actions; parser-en-us gains their grammar; the Chord surface gains the four verbs, a `define reaction` recipe table, and an `on mixing with` on-clause hook.
**Bounded contexts touched**: World model liquid substance (`packages/world-model` — `LiquidTrait` intrinsics + reaction behavior; `ContainerTrait.liquidType`/`liquidAmount`, unchanged shape, read/written by transfer), standard actions (`packages/stdlib` — four new actions under the four-file convention), parsing (`packages/parser-en-us` — pour/fill/empty/mix grammar), Chord composable vocabulary + `EVENT_VERBS` (`packages/chord`, `packages/story-loader`), and the ADR-210 grammar ratchet.
**Key domain language**: transfer vs. mixing, substance identity vs. holder amount (container-is-source-of-truth), reaction (recipe-declared vs. on-clause-declared), holder, `drinkable` implies `liquid` (ADR-219, refining ADR-218 §1b).

## Phases

### Phase 1: `LiquidTrait` intrinsics and the reaction model (world-model)
- **Tier**: Medium
- **Budget**: 250 tool calls
- **Domain focus**: World-model liquid-substance identity. `LiquidTrait` today (post-W1) is marker-only (no fields beyond trait type). This phase gives it the substance's **intrinsic** state — the part that never varies by holder — while `ContainerTrait.liquidType`/`liquidAmount` (unchanged, already exist) stay the sole record of *how much is where*.
- **Entry state**: W1 Phase 2 complete — marker `LiquidTrait` exists at `packages/world-model/src/traits/liquid/liquidTrait.ts`, composed onto `drinkable` and `liquid`-adjective entities. David's go-ahead covers this world-model extension specifically.
- **Deliverable**:
  - Extend `LiquidTrait` (same file, same directory — no new trait, no new adjective) with intrinsic-only fields: a `liquidType: string` identity (the substance's type name — e.g. `'acid'`, `'water'`; this is the value that flows into `ContainerTrait.liquidType` when the substance is poured into a holder) and a `reactions?: ILiquidReaction[]` list (each entry: `{ with: string; result: string }`, the in-code shape the recipe table and reaction behavior both read). **No `amount`/`quantity` field on `LiquidTrait` — that would violate the container-is-source-of-truth invariant ADR-219 fixes explicitly.** State this invariant as a guard: any future field proposed for `LiquidTrait` that varies by holder (amount, temperature-in-a-specific-container, etc.) is rejected at review, not added.
  - A new `LiquidReactionBehavior` (`packages/world-model/src/behaviors/liquid/` or beside the trait, matching the sibling-behavior convention, e.g. `edibleBehavior.ts`'s placement relative to `edibleTrait.ts`) exposing `combine(holder: IFEntity, incomingType: string, incomingAmount: number): { resultType: string; resultAmount: number } | null` — pure combination logic (recipe lookup by `(holder.liquidType, incomingType)` pair, either order), independent of the on-clause hook (that's Phase 3's job; this phase only builds the data-driven recipe lookup path, not the on-clause escape hatch).
  - Unit tests (Vitest) for `LiquidReactionBehavior.combine`: two known types with a defined reaction return the result type with combined amount; two types with no defined reaction return `null` (mixing incompatible/undefined liquids is a caller-level "no reaction" case, not an error — Phase 2/3 actions decide what "no reaction" means at the action level); a reaction defined in either operand order (`A+B` vs `B+A`) resolves the same, since a Chord `define reaction: acid + water -> diluted acid` is unordered.
  - Update `LiquidTrait`'s bare-trait serialization note (ADR-218 §1b) — the intrinsic fields are the entity's own properties, so this stays "serializes as its own enumerable properties, no migration" (existing marker-only saves gain no new required fields since `reactions` is optional).
- **Exit state**: `LiquidReactionBehavior.combine` unit tests pass (asserting on the returned `{resultType, resultAmount}` state, not just "didn't throw" — per CLAUDE.md rule 12/13). `LiquidTrait` intrinsics compile and are importable by stdlib (Phase 2/3 consume them). No transient amount field exists anywhere on `LiquidTrait` — confirmed by a grep-based check as part of this phase's own review, not deferred to Phase 4.
- **Status**: PENDING (write-ahead; not yet active — see Entry gate)

### Phase 2: Transfer — `pouring`, `filling`, `emptying` actions + grammar
- **Tier**: Large
- **Budget**: 400 tool calls
- **Domain focus**: Liquid transfer between holders (containers, or a source that already holds liquid). Closes ADR-219 AC-1. Composes with the existing `drinking` container-liquid branch (`drinking.ts:101`) — transfer and drinking are two ways to deplete a holder's `liquidAmount`, and this phase's actions must leave that branch's assumptions (`containsLiquid`/`liquidType`/`liquidAmount` on `ContainerTrait`) intact.
- **Entry state**: Phase 1 complete (`LiquidTrait` intrinsics + `LiquidReactionBehavior` importable). W1's `ContainerTrait.liquidType`/`liquidAmount` fields are pre-existing and unchanged (`containerTrait.ts:41-47`).
- **Deliverable**:
  - Three new stdlib actions under `packages/stdlib/src/actions/standard/{pouring,filling,emptying}/`, each following the existing 4-file convention (`<name>.ts`, `<name>-data.ts`, `<name>-events.ts`, `<name>-messages.ts`, `<name>-types.ts`) and the validate/execute/report/blocked pattern (ADR-051), modeled on `putting`'s two-object shape (`putting.ts`) for the object+destination case:
    - `pouring` — `pour :liquid into :container` (or `pour from :source into :container`): moves an amount from a source holder (an entity carrying `LiquidTrait` directly, or a `ContainerTrait` holder) into a target `ContainerTrait` holder, decrementing the source's `liquidAmount` and incrementing/setting the target's `liquidType`/`liquidAmount`. If the target already holds a *different* `liquidType`, this action calls into `LiquidReactionBehavior.combine` (Phase 1) — see note below; if no reaction is defined, refuse (blocked) rather than silently overwrite, since silent overwrite would violate "container is source of truth" (an unexplained type change is a hidden mutation).
    - `filling` — `fill :container from :source`: same transfer direction as pouring but grammar/verb-first framed from the empty container's point of view (fill the bottle from the well) — validate the source has liquid to give, execute the same amount-transfer mutation `pouring` uses (share the transfer logic in a small helper, not duplicated across two action files — CLAUDE.md rule 7, one reason to change).
    - `emptying` — `empty :container`: sets `liquidAmount` to 0 and `containsLiquid` to `false` on the target (no destination — the liquid is discarded, not moved to a holder; note this explicitly since it's the one transfer verb that doesn't need a second object).
  - Parser-en-us grammar (`packages/parser-en-us/src/grammar.ts`, alongside the existing `insert`/`put`/`hang` block ~line 183-199): `pour :liquid into :container`, `fill :container from :source`, `empty :container`, each `.hasTrait(...)` gated appropriately (liquid slot needs `LiquidTrait` or a liquid-holding `ContainerTrait`; container/source slots need `ContainerTrait`) and `.mapsTo('if.action.pouring'|'if.action.filling'|'if.action.emptying')`.
  - Three dated `chord-grammar-changes.md` ratchet entries (verb grammar for pour/fill/empty), written before the grammar code.
  - Chord surface: admit `pour`/`fill`/`empty` as ordinary verbs (no special construct — ADR-219 "Verbs are standard actions") wherever the catalog/loader wires action-verb reachability for other standard actions (mirror however `drinking`'s verb wiring works today, since no new adjective or `EVENT_VERBS` entry is needed for these three — they're plain actions).
  - Three fixture `.story` files (new, `docs/work/chord-liquids/fixtures/`): pour from one liquid-holding container to another (assert both containers' `liquidType`/`liquidAmount` after); fill an empty container from a source; empty a container (assert `liquidAmount === 0`, `containsLiquid === false`).
- **Exit state**: All three fixtures compile and pass via `node dist/cli/sharpee.js --test <fixture>.story` after `./repokit build`, each asserting on `ContainerTrait.liquidType`/`liquidAmount` post-transfer (state assertion, not "didn't throw" — CLAUDE.md rule 12/13). `pouring`/`filling`/`emptying` gain new rows in `chord-availability-audit.md` (added, not flipped — these actions didn't exist pre-ADR-219). `cloak`/`zoo` still compile unchanged (additive). `drinking`'s existing container-liquid branch (`drinking.ts:101`) still passes its existing tests unmodified.
- **Status**: PENDING (write-ahead; not yet active — see Entry gate)

### Phase 3: Mixing — recipe table + `on mixing with` on-clause + `mixing` action
- **Tier**: Large
- **Budget**: 400 tool calls
- **Domain focus**: Reactions — combining two liquids in a holder into a result substance. Closes ADR-219 AC-2 and AC-3. Reuses W2's `EVENT_VERBS`-admission pattern and the pre-existing standard-verb on-clause (ActionInterceptor) mechanism `entering` already uses — not ADR-221's dispatch-specific clause synthesis.
- **Entry state**: Phase 1 (`LiquidReactionBehavior.combine`) and Phase 2 (transfer actions, so there's a way to get two liquids into one holder) complete. W2 (ADR-221 fallback wiring) complete — it establishes the `EVENT_VERBS` admission precedent and confirms the verb-agnostic arg-binding mechanism this phase's `<other>` slot uses; the clause-execution wiring itself follows `entering`'s standard-verb path (ratchet D3), not ADR-221's dispatch-verb path.
- **Deliverable**:
  - New `mixing` stdlib action (`packages/stdlib/src/actions/standard/mixing/`, 4-file convention): `mix :a and :b` — validates both operands carry `LiquidTrait` (directly or via a holder's `liquidType`), executes by (a) checking for a substance-level `on mixing with <other>` on-clause hook first (on-clause wins per ADR-219 precedence) and, only if none is registered/compiled for that pair, (b) falling back to `LiquidReactionBehavior.combine`'s recipe-table lookup (Phase 1); reports the result substance as the holder's new `liquidType`, or blocks with a "nothing happens" message if neither path defines a reaction for that pair.
  - Chord surface — **recipe table** (`define reaction: acid + water -> diluted acid`): a new top-level Chord construct, parsed/compiled by `packages/story-loader` into `LiquidTrait.reactions` entries (Phase 1's shape) attached to the named substance's definition at load time — data-driven, no code. One dated ratchet entry for the `define reaction: <A> + <B> -> <C>` syntax, written before the loader change.
  - Chord surface — **on-clause hook** (`on mixing with <other>: … become <result>`): admit `mixing` to the catalog's `EVENT_VERBS` (`packages/chord/src/catalog.ts:78` — whatever W2 left the set as, e.g. `{entering, lowering, raising}`, plus `mixing`). **`mixing` is a standard verb (real `mixingAction` with its own validate/execute/report, per ADR-219 "ordinary stdlib actions... no special construct"), not a capability-dispatch verb** — so its on-clause wiring follows the **standard-verb path** the ratchet log's D3 entry (`chord-grammar-changes.md`, 2026-07-11) reserves for verbs like `entering`: the compiled `on mixing with <other>` clause is spliced in via the **ActionInterceptor** hook (ADR-118 — `postValidate`, since the "on" form may `refuse` and its phrase is the primary message, per D3), not via ADR-221's synthesized-`CapabilityBehavior`/dispatch-report-phase mechanism — that path is reserved for verbs with *no* registered TS behavior at all (`lowering`/`raising`), and the D3 entry notes interceptor hooks empirically do not fire on that dispatch path, so cloning it here would be architecturally mismatched. `<other>` is still a general arg-binding slot (ADR-221's arg-binding mechanism, which is verb-agnostic) bound to the second operand's resolved entity.
  - Precedence rule as an explicit test (ADR-219: "on-clause wins when a recipe also matches"): a fixture substance with **both** a `define reaction` entry and an `on mixing with` clause for the same pair exercises the on-clause path, asserted by checking which effect ran (the on-clause's distinguishing side effect, e.g. an emitted event the recipe path wouldn't produce), not just the end `liquidType`.
  - Two dated ratchet entries: `define reaction: … -> …` syntax, and `mixing`'s `EVENT_VERBS` admission (`on mixing with <other>` parses).
  - Two fixture `.story` files: (i) `define reaction: acid + water -> diluted acid`, pour acid into a water-holding container, assert the container's `liquidType` becomes `diluted acid` (AC-2); (ii) a substance with `on mixing with <other>: emit "fizzes"; become <result>` and **also** a matching recipe entry for the same pair, asserting the on-clause's emitted effect ran and the recipe path did not override it (AC-3, precedence).
- **Exit state**: Both fixtures pass via the bundle. The precedence test passes (on-clause path exercised over recipe when both match). `mixing` gains a new row in `chord-availability-audit.md`. `mixing` in `EVENT_VERBS` doesn't regress `entering`'s existing on-clause behavior (a regression check, since this phase edits a shared `Set`). `cloak`/`zoo` still compile unchanged.
- **Status**: PENDING (write-ahead; not yet active — see Entry gate)
