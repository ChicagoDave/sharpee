## Part A — Recent Projects menu

### Behavior

- **Storage:** `[URL]` JSON-encoded in `UserDefaults` under key `SharpeeRecentProjects`. Mirrors the existing `SessionStateStore` pattern in `tools/ide/SharpeeIDE/Persistence/SessionState.swift`.
- **Capacity:** 10 entries max. LRU semantics — most recent first, duplicates removed on push.
- **Population:** push on every successful `loadProject(_:expandedFolderURLs:)`.
- **Menu:** File → Open Recent submenu. Built dynamically via `NSMenuDelegate.menuNeedsUpdate(_:)` so adds/removes are reflected without app restart.
- **Missing folders:** filtered out of the rebuild via `FileManager.default.fileExists(atPath:isDirectory:)` (with `isDirectory: true` check). If the folder vanishes between rebuild and click, present an `NSAlert`, remove the entry, no project load.
- **Empty state:** when no entries (or all filtered out), submenu contains a single disabled "No Recent Projects" item.
- **Clear:** "Clear Menu" item at the bottom of the submenu (matches macOS convention). Calls `RecentProjectsStore.clear()`.

### Files

| File | Action |
|---|---|
| `tools/ide/SharpeeIDE/Persistence/RecentProjectsStore.swift` | new — `enum RecentProjectsStore` with `load() -> [URL]`, `push(_:)`, `remove(_:)`, `clear()` |
| `tools/ide/SharpeeIDE/Menus/MenuBuilder.swift` | add Open Recent submenu under File; install `NSMenuDelegate` |
| `tools/ide/SharpeeIDE/AppDelegate.swift` | implement `menuNeedsUpdate(_:)` to rebuild submenu items; add `openRecentProject(_:)` and `clearRecentProjects(_:)` actions |
| `tools/ide/SharpeeIDE/MainWindow.swift` | call `RecentProjectsStore.push(url)` inside `MainSplitViewController.loadProject` success path |

### Out of scope

- Persisting tab state per recent project (each recent project relaunches with empty tabs unless it was the most-recent project, which is already covered by `SessionState`).
- Pinning / reordering recent entries beyond LRU.
- A separate Welcome / Start screen — Open Recent on the File menu is the only entry point for now.

## Part B — Test target

### Why now

P1's `Document`, `SessionState`, recent-projects logic, and the disambiguation algorithm all have shapes that test cleanly without an `NSWindow` harness. Scaffolding the target now means subsequent IDE phases get tests for free; postponing it means cumulative untested code grows.

### Target shape

- **Bundle:** `SharpeeIDETests`, type `bundle.unit-test`, host app `SharpeeIDE`.
- **Scheme:** auto-generated by xcodegen; bound to ⌘U via the default Test action.
- **Sources path:** `tools/ide/SharpeeIDETests/`.
- **xcodegen config:** add a `targets:` entry in `tools/ide/project.yml`, then regenerate `SharpeeIDE.xcodeproj/`.

### Initial test files

| File | What it covers |
|---|---|
| `DisambiguationTests.swift` | `EditorViewController.makeDisplayTitles(for:)` — no collisions, single collision pair, multi-way collision, identical-suffix-at-depth-2 forcing depth 3, empty input |
| `SessionStateTests.swift` | Codable roundtrip; iteration-1 forward-compat (decode payload with no `expandedFolderURLs` defaults to `[]`); `SessionStateStore` load/save/clear against an injectable `UserDefaults(suiteName:)` |
| `RecentProjectsStoreTests.swift` | push dedupes; push past 10 trims to 10; remove; clear; load on empty key returns `[]` |
| `DocumentTests.swift` | `load(from:)` reads file content + url binding; `save()` persists content + clears `isDirty`; `isDirty` flips `false → true` on content change; `save()` failure path preserves `isDirty` |

### Out of scope for the initial test pass

- `FileNode` — simple value type, low test value.
- `EditorViewController` integration tests (tab switching, close-confirmation alert flow) — need NSWindow + delegate plumbing; defer until there's a real reason.
- `ProjectTreeViewController` expansion-tracking — depends on `NSOutlineView` UI lifecycle; defer.

## Boundary Statements

Two modules in this plan are cross-boundary enough to warrant the rule-8a check.

**`RecentProjectsStore`**
- OWNER: IDE persistence layer; same context as `SessionStateStore`.
- SHARED?: no — single-user, single-process, no two consumers in the same context can legitimately disagree.
- PROMISE: none yet (new file). The promise being established: "URL list returned in most-recent-first order; never contains duplicates; capped at 10."
- ALTERNATIVES: could live inside `SessionState` as another field, but recent projects outlive any one workspace and should not be tied to per-workspace persistence.

**`SessionState` (modified only if needed for recent-projects integration)**
- Not modifying it in this plan. Recent projects live in their own store. Flagged here so the boundary stays clean.

## Risks / open questions

- xcodegen's exact incantation for an XCTest bundle target. Resolved at step 1 by checking xcodegen docs or an example project; not blocking the plan.
