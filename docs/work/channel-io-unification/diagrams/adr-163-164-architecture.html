3Session continuity — three repaint policies

The platform guarantees one thing: re-emission identity — a captured TurnPacket fed back through the renderer produces output identical to its original live emission. Each consumer chooses which restore policy to use based on whether its renderer state survives a restore.

Policy Used by Why What happens on restore
No-repaint CLI Terminal owns scrollback; renderer state survives the process boundary Resume live turns. No packet re-emission. Persistence may be world state only.
Local repaint zifmia, platform-browser DOM is empty on app reopen / page refresh; renderer state lost Consumer persists per-turn artifacts locally; on restore, replays them through its own renderer before resuming live.
Network repaint multi-user (ADR-164) Each connecting client has a fresh DOM; renderer state never lived on the server Server persists packets in the world-model transcript capability; replays to client on every connect (first join, reconnect, refresh, new joiner).

Why it works without a replay flag:
• Event-mode channels (death, score_notify, sound, animation) are not persisted per ADR-163 §4 → don't refire.
• Replace-mode channels converge to the final value through any sequence.
• Append-mode channels accumulate identically, including clear events at original positions.
Persistence-unit sub-choice (local repaint only): A local-repaint consumer may persist TurnPacket values and replay them through the renderer (renderer-version-independent, slower on long sessions), or persist the renderer's output directly — markup, DOM fragments, captured XAML — and re-inject on restore (faster, but tied to the renderer that produced it). Both produce the same visual result. Hybrid is also valid. The Secret Letter (OpenSilver) project shipped using rendered-output persistence; zifmia and platform-browser may pick either. Network repaint cannot use rendered output because rendered output isn't portable across clients with different renderers.

4The platform API & how multi-user adds its three channels

ADR-163 — produceTurnPacket

function produceTurnPacket(input: {
  textBlocks: ITextBlock[];
  events:     DomainEvent[];
  world:      WorldModel;
  prevValues?: Record<string, unknown>;
}): {
  turn_id: string;
  payload: Record<string, unknown>;
};

Plain inputs in, plain payload out. No knowledge of chat / presence / command_echo. No knowledge of transport or persistence.

ADR-164 — merge-after pattern

const platformPacket = produceTurnPacket({
  textBlocks, events, world, prevValues
});

const turnPacket = {
  ...platformPacket,
  payload: {
    ...platformPacket.payload,
    chat:         newChatEntries,
    presence:     currentRoster,
    command_echo: commandsThisTurn,
  },
};

Multi-user server populates its own three channels by merging into the payload after the platform produces it. No platform extension needed.

Single fact: the platform API knows nothing about multi-user. Server-sourced channels are a downstream-consumer concern, populated by the consumer with a one-line merge.
