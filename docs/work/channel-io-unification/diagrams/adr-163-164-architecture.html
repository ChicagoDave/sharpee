1Channels are IOChannel instances

An IOChannel is a typed object with id, mode, emit policy, content type, optional capability gating, and a produce closure. The type lives in @sharpee/if-domain. Platform code creates IOChannels; authors create IOChannels; same construction. They go into stdlib's registry through the same registry.add(channel) call.

stdlib defines a standard IOChannel

// packages/stdlib/src/channels/score.ts
import type { IOChannel } from '@sharpee/if-domain';

export const scoreChannel: IOChannel<{
  current: number; max: number | null
}> = {
  id: 'score',
  contentType: 'json',
  mode: 'replace',
  emit: 'always',
  produce: ({ world }) => {
    const cap = world.getCapability('scoring');
    return cap
      ? { current: cap.scoreValue, max: cap.maxScore ?? null }
      : undefined;
  },
};

Co-located with the scoring capability. Reads world directly; no intermediary block. Returns undefined on idle turns; always emit re-emits the prev value.

A story registers its own IOChannel

// stories/the-alderman/src/channels.ts
import type { IOChannel } from '@sharpee/if-domain';

const evidenceChannel: IOChannel = {
  id: 'evidence',
  contentType: 'json',
  mode: 'append',
  emit: 'sparse',
  produce: ({ events }) => {
    const found = events
      .filter((e) => e.type === 'alderman.evidence_found')
      .map((e) => e.data);
    return found.length ? found : undefined;
  },
};

story.registerChannels = (registry) => {
  registry.add(evidenceChannel);
};

Same type, same registry, same add call. The story's hook receives stdlib's IChannelRegistry and adds an IOChannel to it.

Single fact: there is no "platform-privileged" registration path. Standard IOChannels and story IOChannels go into the same registry through the same call. Last-write-wins on id lets stories override any standard channel.

Single fact: the wire is identical. The producer side (engine + stdlib + channel-service) is identical. The only difference is whether channel:packet events fire in-process or get serialized over a transport. The renderer doesn't know or care which deployment it's in.

3Bootstrap — engine has-a ChannelService

Standard composition. The engine constructs a ChannelService instance during engine.start(), passing it stdlib's registry and the consumer's capabilities. Per turn, the engine calls channelService.build(ctx) and emits the packet.

Inside the engine

// @sharpee/engine/src/game-engine.ts
import { ChannelService } from '@sharpee/channel-service';
import { channelRegistry } from '@sharpee/stdlib';

class GameEngine {
  private channelService!: ChannelService;

  async start(opts: { capabilities: ClientCapabilities }) {
    this.world = …;        // build world
    await this.story.init?.();

    // stdlib's registry already holds standard IOChannels.
    // Story extends it through registerChannels?:
    this.story.registerChannels?.(channelRegistry);

    // Compose ChannelService. Captures the registry's contents
    // as the channel set for this session.
    this.channelService = new ChannelService(
      channelRegistry, opts.capabilities,
    );

    this.emit('channel:manifest', this.channelService.buildManifest());
  }

  async executeTurn(input: string) {
    const events = …;       // existing turn execution
    const blocks = this.textService.processTurn(events);

    const packet = this.channelService.build({
      world: this.world, events, blocks,
      turn: this.context.currentTurn,
    });
    this.emit('channel:packet', packet);
    this.emit('text:output', blocks, turn); // legacy
  }
}

Consumer (BrowserClient) just listens

// platform-browser BrowserClient.connectEngine()
const renderer = new Renderer();
mountDefaultBrowserLayout(renderer);
registerDefaultBrowserRenderers(renderer);

engine.on('channel:manifest', (cmgt) => {
  renderer.applyCmgt(cmgt);
});

engine.on('channel:packet', (packet) => {
  renderer.applyTurnPacket(packet);
});

renderer.onCommand((cmd) => {
  engine.executeTurn(cmd.text);
});

await engine.start({
  capabilities: BROWSER_CAPABILITIES,
});

The consumer hands capabilities to start(), listens to two events, routes commands. It does not import channel-service or instantiate ChannelService itself.

Single fact: consumers are passive. The same listener pattern works for platform-browser (in-process events) and the multi-user web-client (WebSocket-decoded events). The multi-user server defers engine.start({capabilities}) until the client's hello packet arrives over the wire and supplies the capabilities.

5Session continuity — three repaint policies

The platform guarantees one thing: re-emission identity — a captured TurnPacket fed back through the renderer produces output identical to the original live emission. Each consumer chooses a restore policy based on whether its renderer state survives a process / page boundary.

Policy Used by Why What happens on restore
No-repaint CLI Terminal owns scrollback; renderer state survives the process boundary Resume live turns. No packet re-emission. Persistence may be world-state only.
Local-repaint platform-browser, zifmia DOM is empty on page refresh; renderer state lost Phase 4 keeps BrowserSaveEnvelope + transcript-HTML autosave (the existing path). A future enhancement can persist packets and replay them through the renderer.
Network-repaint multi-user (ADR-164) Each connecting client has a fresh DOM; renderer state never lived on the server Server persists packets in the world-model transcript capability; replays to client on every connect (first join, reconnect, refresh, new joiner).

Why it works without a replay flag:
• Closures returning undefined on idle turns — sparse channels stay quiet; always channels re-emit prevValue from the registry's per-session state.
• Replace-mode channels converge to the final value through any sequence of stored packets.
• Append-mode channels accumulate in the renderer's state store, including clear events at original positions truncating the buffer.

6The platform API & how multi-user adds its three channels

ADR-163 — ChannelService class

class ChannelService {
  constructor(
    registry: IChannelRegistry,
    capabilities: ClientCapabilities,
  );

  buildManifest(): CmgtPacket;

  build(ctx: {
    world:  IWorldModel;
    events: readonly ISemanticEvent[];
    blocks: readonly ITextBlock[];
    turn:   number;
  }): TurnPacket;
}

Engine has-a ChannelService. Engine instantiates one in start(), calls buildManifest() once, calls build(ctx) per turn. Walks the registry, calls each IOChannel.produce(ctx), applies mode + emit policy.

ADR-164 — merge-after pattern

// server-side, after the engine emits channel:packet
engine.on('channel:packet', (packet) => {
  const turnPacket = {
    ...packet,
    payload: {
      ...packet.payload,
      chat:         newChatEntries,
      presence:     currentRoster,
      command_echo: commandsThisTurn,
    },
  };
  ws.send(JSON.stringify(turnPacket));
  transcript.append(turnPacket);
});

Multi-user server populates its own three channels by merging into the payload after the engine's listener fires. ChannelService has no idea about chat; the channel registry stays domain-pure.

Single fact: the platform API knows nothing about multi-user. Server-sourced channels are a downstream-consumer concern, populated by a server-side listener with a few lines of merge code before the packet hits the wire.
