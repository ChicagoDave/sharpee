1Three lifecycle phases — what the renderer does, and when

The renderer is driven entirely by the four packet kinds. It has three distinct phases.

Phase 1 — Boot
CmgtPacket arrives
For each ChannelDefinition:
look up renderer by channel.id
Instantiate channel renderer; reserve its slot in the layout
Set up empty state stores
(for replace + append channels)

Happens once per session at boot. Sets up everything before the first TurnPacket arrives.

Phase 2 — Live (per turn)
TurnPacket arrives
For each key in payload:
route to that channel's renderer
Channel renderer applies mode-specific update
(replace / append / event — see Panel 3)
Visual output committed to the screen

Repeats once per turn for the lifetime of the session. Re-used identically during repaint (Panel 5 of adr-163-164-architecture.html).

Phase 3 — UI input
User gesture: keypress, click, drag, widget
Renderer maps gesture to canonical command string
(hotspot's command field, story-defined verb, free-text input)
CommandPacket emitted upward
Engine processes, next turn fires

Fires whenever the user interacts. Gesture → command is the renderer's responsibility; the engine sees only the command string.

Single fact: the renderer is stateless across phases except for the per-channel state stores populated by replace/append channels. There is no "engine state mirror" — the renderer sees only what the producer emits.

2Channel renderer registry — three sources, in precedence order

For each channel id, the renderer needs a function/component that knows how to draw values of that channel's type. The registry consults three sources in order.

Precedence Source Examples When it applies
1 story override Story bundle ships a renderer for a channel id that the platform also has a default for. The Alderman restyles main as Playfair Display italic on cream paper. Story takes full control of how a standard channel renders. Platform default is bypassed.
2 story custom Story bundle ships a renderer for a channel id the story itself registered. The Alderman's evidence, assertions, notebook renderers. Channel only exists because the story registered it. No platform default could exist.
3 platform default Platform-shipped renderer for one of the standard or media channels. main → scrolling prose. location/score/turn → status line. image:* → <img> layers. sound → Web Audio. No story override or custom registered for this channel. Platform default applies.
4 generic JSON-tree Last-resort fallback. CMGT advertised a channel but neither story nor platform supplied a renderer. An unknown json channel renders as a collapsible JSON tree. Degraded but functional. Logged as a renderer-warning.
Open question — registration API. How does a story bundle register its renderers? Inline TS module export? Manifest file? Global window-attached registry? The asset-pipeline ADR (forthcoming, deferred from ADR-163 §13) is the natural home for this decision — it's coupled to how renderer assets physically ship inside .sharpee bundles.

3Mode-specific behavior — what a channel renderer does with each value

A channel's mode tells the renderer how to handle each new value. Three modes, three update rules.

replace

Newest value supersedes prior. Renderer keeps a single "current" value per channel.

value₁
store = value₁ render
value₂
store = value₂ render

Examples: location, score, turn, prompt, info, ifid, presence, image:<layer>, music, layout.
Repaint: sequence of values produces same final state regardless of intermediate steps.

append

Each value added to a chronological list. Renderer accumulates.

value₁
list = [value₁] render entry
value₂
list = [v₁, v₂] render entry

Examples: main (prose), chat, command_echo, story-defined transcripts.
Clear: a clear event with matching target resets the accumulated list.
Repaint: accumulates identically including clears at original positions.

event

Transient signal. Render once and discard. No state stored.

fire
trigger side effect (toast / sound / animation)
(silent turns)
nothing happens

Examples: death, endgame, score_notify, sound, animation, transition, clear.
Persistence: not persisted (per ADR-163 §4) — they don't refire on repaint or join.

Single fact: mode lives on the channel definition, not on the renderer. A channel always behaves the same way regardless of which renderer happens to draw it. The renderer reads mode from the channel definition (received via CMGT) and applies the matching update rule.

4Layout & composition — how channel renderers compose into a screen

The renderer's screen is a layout of slots. Each slot hosts one or more channel renderers. Standard slots are platform-shipped; story-defined channels can declare new slots or target existing ones.

5UI input → CommandPacket — the only path back to the engine

Every UI gesture that affects game state goes through one path: synthesize a command string, emit a CommandPacket, let the parser process it. The engine cannot tell the difference between a typed command and a UI-synthesized one.

Sources of synthesized commands

  • Hotspot click — image:* payload's hotspots[].command field carries the canonical string.
  • Action button — story renderer attaches a click handler that emits a fixed command.
  • Drag-drop — renderer maps source/target pair to a story-defined verb (e.g., drag evidence card onto suspect → accuse doyle with footprint).
  • Right-click context menu — pre-defined verb list per entity.
  • Free-text input — the standard parser path; user types directly.

What the engine sees

{ kind: 'command', text: 'turn dial 1' }
{ kind: 'command', text: 'identify gap doyle alibi' }
{ kind: 'command', text: 'north' }

Indistinguishable. The parser tokenises and grammar-matches the same way regardless of how the string was produced. Stories add custom verbs by extending the parser's grammar (per ADR-087); the renderer just emits matching strings.

Single fact: there is no separate "callback" or "event-from-UI" packet kind. UI input is collapsed onto the same CommandPacket shape used for typed commands. This drops ADR-101's onComplete animation callback entirely (per ADR-163 §10).

6Repaint reuses the live render path — no replay mode

The renderer has no notion of "live" vs "historical" packets. Re-emission identity (ADR-163 invariant) means feeding stored packets through Phase 2 produces the same screen as feeding live packets.

Phase 2 dispatcher (Panel 1)

↑ fed by live TurnPackets from the producer

↑ also fed by stored TurnPackets during local repaint or network repaint

Why this works:
  • Replace channels: any sequence converges to the final value. Replaying 1000 status updates = displaying the last one.
  • Append channels: accumulate identically. Replaying yields the same scrollback.
  • Event channels: not persisted (ADR-163 §4) → can't replay → can't refire wrongly.
  • Renderer state stores rebuild from packet stream — no special "import historical state" entry point.
Open question — local-state-store ownership. Are the per-channel state stores (current replace value, accumulated append list) part of the renderer itself, or held in a separate consumer-side cache that the renderer reads from? For local-repaint consumers using rendered-output persistence (Secret Letter pattern), the answer affects whether the cache survives a render-skip optimisation. Resolve when implementing platform-browser's renderer.
