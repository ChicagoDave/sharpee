# ADR-101 Capability Audit (for ADR-164)

**Date:** 2026-04-29
**Purpose:** Enumerate every capability ADR-101 (Graphical Client Architecture, Proposed) specifies, map each one to the ADR-163 channel model, and surface design tensions before drafting ADR-164 (Channel I/O Everywhere + Folded ADR-101).

**Implementation status check:** ADR-101 is not implemented. `grep -r 'media.image.\|ClientCapabilities\|media.sound.'` returns zero hits across `packages/`, `tools/`, and `stories/`. Any decision in this audit changes a paper design only.

**Audit method:** Walk ADR-101 section by section. For each capability, record (1) the ADR-101 specification (verbatim or paraphrased), (2) the proposed channel mapping under ADR-163, (3) a status flag, (4) notes / questions.

Status flags:
- ✅ **Clean** — maps onto an ADR-163 channel without design tension.
- ⚠️ **Design choice** — fits, but a deliberate decision is needed (no obvious right answer).
- 🔄 **Wrong direction** — ADR-101 specifies a server→client flow but the underlying need is client→server; needs reframing.
- 📦 **Out of scope** — the concern is real but doesn't belong on the wire ADR (asset pipeline, etc.).

---

## 1. Author-controlled rendering principle

**ADR-101 (line 39 onward):**

> "We will implement Option A: Author-controlled media events. The story author has full control over multimedia presentation. The engine emits explicit media directives, and the client renders them."
>
> "Simpler client — Client is a renderer, not an interpreter of game semantics."

**ADR-163 D6 (paraphrased):**

> "Rendering is author-overridable; the platform ships defaults. The wire shape is independent of the renderer. Channels carry data; rendering is presentation."

**Mapping:** Identical principle, two-time confirmation across ~3 months. ADR-164 inherits both ADRs' framing without modification.

**Status:** ✅ Clean. Carries forward as the central principle of ADR-164.

## 2. Capability negotiation

**ADR-101 (lines 70-104, 287-296):** At session start the client declares a `ClientCapabilities` object (boolean flags for text/images/animations/video/sound/music/speech/splitPane/statusBar/sidebar/clickableText/clickableImage/dragDrop/transitions/layers/customFonts plus optional `screenWidth` / `screenHeight`). Stories check `context.client.supports('images')` before emitting media events.

**Channel mapping:** The CMGT manifest (ADR-163 D8) is the natural carrier. Two changes from ADR-163's current spec:

1. **Direction reverses for the capability portion of CMGT.** ADR-163 today: server emits CMGT to client (channel registry). For capability-aware behavior we add a *pre-CMGT* client→server packet:
   ```ts
   { kind: 'hello', capabilities: ClientCapabilities }
   ```
   The server's CMGT response is then *conditional on capabilities* — channels for media surfaces the client doesn't support are not registered, so the engine's story init can `if (capabilities.images) channelService.registerChannel({ id: 'image:main', ... })`.

2. **Story-side capability access.** `context.client.supports('images')` becomes `context.capabilities.images` or similar — the engine receives the `hello` payload via the host (server in multi-user, the local runtime in zifmia/CLI) before story init runs.

**Status:** ⚠️ Design choice — three real choices to make:
- (a) Add a `hello` packet kind to the wire, or fold capabilities into a query string / HTTP header on the connection that the server hands to the engine.
- (b) Whether the engine receives capabilities via `context.capabilities` or via the channel-service runtime.
- (c) Whether *all* channels are capability-gated, or only the media channels (probably only media — `main` always exists).

## 3. Image events

**ADR-101 (lines 110-136):**
```ts
{ type: 'media.image.show', src, layer, position?, size?, transition?, duration? }
{ type: 'media.image.hide', layer, transition? }
{ type: 'media.image.preload', src }
```

**Layer convention (line 117):** `'background' | 'main' | 'overlay' | string` — three named layers plus author-extensible.

**Channel mapping:**

| ADR-101 event              | Channel                    | contentType | mode      |
| -------------------------- | -------------------------- | ----------- | --------- |
| `media.image.show` / `.hide` | `image:<layer>`            | `json`      | `replace` |
| `media.image.preload`      | `image:preload`            | `json`      | `event`   |

Show payload: `{ src, position, size, transition, duration }`. Hide payload: `null` (or `{ transition }` to specify the dissolve-out transition). The channel id encodes the layer; the renderer maintains z-order from the layer name (background < main < overlay; story-defined layers slot in by registration order or explicit `layer.zIndex` field on the `ChannelDefinition`).

**Layer ordering tension:** ADR-101 says background < main < overlay. Channel-service has no z-order semantics today. Two options:
- (a) Add `zIndex?: number` to `ChannelDefinition` for media channels — ordering carried as channel metadata.
- (b) Convention only: the renderer hard-codes background < main < overlay, and story-defined layers carry an explicit `z` in their show-event payload.

(b) is leaner; (a) makes layer-stacking behavior queryable from the manifest. Resolve in ADR-164.

**Preload semantics:** `event` mode is wrong if preload is a one-shot signal — but `event` is ADR-163's "transient signal." Actually fits. The renderer downloads the asset on receipt and discards the event.

**Status:** ⚠️ Design choice on layer ordering carrier (channel metadata vs payload field).

## 4. Audio events

**ADR-101 (lines 140-178):**
```ts
{ type: 'media.sound.play', src, channel?, volume?, loop? }
{ type: 'media.music.play', src, fadeIn?, loop?, volume? }
{ type: 'media.music.stop', fadeOut? }
{ type: 'media.ambient.play', src, channel, volume?, loop? }
{ type: 'media.ambient.stop', channel, fadeOut? }
```

**Channel mapping:**

| ADR-101 event            | Channel             | contentType | mode      | Notes                              |
| ------------------------ | ------------------- | ----------- | --------- | ---------------------------------- |
| `media.sound.play`       | `sound`             | `json`      | `event`   | Transient SFX                      |
| `media.music.play` / `.stop` | `music`         | `json`      | `replace` | Stop = `null` payload              |
| `media.ambient.play` / `.stop` | `ambient:<id>` | `json`      | `replace` | Per-channel ambient layer          |

**Note on ADR-101's `channel?` field on `media.sound.play`:** That `channel` is a *mixer-channel* concept ("for managing multiple sounds"), not the wire channel. Renaming inside ADR-164 to avoid conflation — call it `bus` or `slot`. The wire channel is always `sound` for SFX.

**Note on stop = null:** ADR-163 needs to allow `null` as a `replace`-mode value to mean "clear." Already implicit (the `replace` semantic is "newest value" — `null` is a valid newest value), but worth stating as an invariant in ADR-164.

**Status:** ✅ Clean (modulo the renaming and the null-clear invariant statement).

## 5. Transitions

**ADR-101 (lines 182-195):** `TransitionSpec` is an interface (`type`, `direction`, `duration`, `easing`) attached to several events plus a standalone `media.transition` for screen-level transitions.

**Channel mapping:**

| ------------------- | ------------- | ----------- | ------- |
| `media.transition`  | `transition`  | `json`      | `event` |

`TransitionSpec` as an attachable field on image-show / image-hide / music-play payloads is unchanged — those payloads already carry transition metadata.

**Status:** ✅ Clean.

## 6. Layout

**ADR-101 (lines 199-221):**
```ts
{ type: 'media.layout.configure', layout: { main: {...}, graphics: {...}, status: {...} } }
{ type: 'media.status.update', content }
{ type: 'media.clear', target? }
```

**Channel mapping:**

| ADR-101 event              | Channel    | contentType | mode      | Notes                                        |
| -------------------------- | ---------- | ----------- | --------- | -------------------------------------------- |
| `media.layout.configure`   | `layout`   | `json`      | `replace` | Whole layout description; renderer re-renders |
| `media.status.update`      | (existing channels) | —    | —         | **Already covered by `location`/`score`/`turn`/etc.** ADR-163 D5 status channels supersede this — `media.status.update` doesn't need its own channel. |
| `media.clear`              | `clear`    | `json`      | `event`   | Payload `{ target: 'all' \| 'main' \| ... }` |

**Tension on `media.clear`:** ADR-101 has `target?: 'all' | 'main' | 'graphics' | string` — clearing addresses surfaces, not channels. Under ADR-163 the renderer interprets `clear` and resets its local state for the named surface. Channel-service emits the event and forgets it — but consumers (renderer) treat it as a directive to drop accumulated `append`-mode state.

This is an interesting case: a `clear` event is server-side authority over what the renderer has accumulated. It means `append` mode isn't strictly monotonic — the server can issue a clear and the renderer truncates. Worth stating in ADR-164's invariants.

**`media.status.update` is a duplicate.** ADR-101 was written before ADR-163 split out `location` / `score` / `turn` as first-class channels. Folding these together: the platform's status channels *are* the status update mechanism. Stories that want richer status content emit to a story-defined `json` channel.

**Status:** ⚠️ Design choice on the `append`-truncation semantic; ✅ resolved on `media.status.update` (subsumed by status channels).

## 7. Animation

**ADR-101 (lines 225-250):**
```ts
{ type: 'media.animation.play', src, layer, position?, loop?, onComplete? }
{ type: 'media.animate', target, properties: {opacity, x, y, scale, rotation}, duration, easing? }
```

**Channel mapping:**

| ADR-101 event             | Channel       | contentType | mode    |
| ------------------------- | ------------- | ----------- | ------- |
| `media.animation.play`    | `animation`   | `json`      | `event` |
| `media.animate`           | `animate`     | `json`      | `event` |

**`onComplete: string` is a callback.** This is the first 🔄 wrong-direction signal in the audit: ADR-101 defines an animation completion hook as a string event-name the engine emits when the animation finishes. But the engine doesn't *know* when the animation finishes — the *client* does. The `onComplete` is a client→server message.

**Two ways to model it under channel I/O:**
- (a) **Synthesized command.** The client treats `onComplete` as instructions for "when the animation finishes locally, send a command back to the server with this name." Same shape as a player command — the server can't tell author-issued commands from animation-completion commands at the wire level. Engine routes by command name to a story-defined event handler.
- (b) **Dedicated client→server callback packet.** New packet kind `{ kind: 'callback', id: string, payload?: any }`. Channel-service receives the callback and emits a domain event the engine processes. Cleaner semantically but adds a new packet kind.

(a) is the smaller change; (b) is more honest about the data flow. The same decision applies to hotspots (next section). Resolve once, apply to both.

**Status:** 🔄 Wrong direction — needs the callback / synthesized-command decision in ADR-164.

## 8. Hotspots / clickable regions

**ADR-101 (lines 363-385):** Image hotspots are defined inside a `media.image.show` event:
```ts
hotspots: [
  { id: 'dial1', bounds: { x, y, w, h }, action: 'turn dial 1' },
  ...
]
```

The `action` is a parser command string. When the player clicks the hotspot, the client issues that command as if the player had typed it.

**Channel mapping:** Hotspot definitions ride along on the `image:<layer>` channel payload (server→client, no change). Hotspot *clicks* are client→server — same direction as the player's typed input. The cleanest model:

- The hotspot's `action` field carries either (a) a canned command string the client emits verbatim ("turn dial 1") through the existing input pipeline, or (b) a callback id that the client emits via the same callback mechanism as animation `onComplete`.

(a) keeps hotspots as pure command shortcuts. (b) decouples render-side identifiers from grammar. Recommend (a) for v1 because it works with no protocol additions — clicking a hotspot is exactly the same as typing the command. Authors who need richer click semantics emit a story-defined `clickable` channel and handle clicks via story-specific commands.

**Status:** ⚠️ Design choice — same decision as animation `onComplete`. Recommend (a) canned-command for v1.

## 9. Asset management

**ADR-101 (lines 254-285):** Story bundle layout:
```
story/
├── assets/
│   ├── images/{rooms,objects,characters}/...
│   ├── audio/{music,ambient,sfx}/...
│   └── animations/...
```

Asset paths in events are relative to `assets/`.

**Channel mapping:** None. This is the asset-pipeline question already deferred from ADR-163 to a separate ADR. Author-shipped renderer JS lives somewhere in the bundle too; a single asset-pipeline ADR should cover both.

**Status:** 📦 Out of scope for ADR-164. Defer to the asset-pipeline ADR (a third doc, alongside ADR-163 and ADR-164). ADR-164 references the convention but doesn't specify it.

## 10. Client implementation requirements

**ADR-101 (lines 287-303):** "Clients MUST: declare capabilities honestly, ignore unsupported events gracefully, handle missing assets, respect layer ordering, implement transitions (instant + fade min), manage audio channels."

**Channel mapping:** Most of this carries forward into ADR-164's invariants:
- "Declare capabilities honestly" — invariant on the `hello` packet contract.
- "Ignore unsupported events gracefully" — invariant on unknown channel ids: client logs and discards (mirrors ADR-163's "malformed snapshot is logged and discarded" pattern).
- "Handle missing assets" — renderer concern, not wire concern. Out of scope.
- "Respect layer ordering" — depends on §3 layer-ordering decision.
- "Implement transitions (instant + fade min)" — renderer concern, not wire concern. Out of scope.
- "Manage audio channels" — renderer concern; the wire just carries the events.

**Status:** ✅ Mostly clean; folds into ADR-164 invariants section.

## 11. Integration with engine pipeline

**ADR-101 (lines 305-318):**

> "Media events flow through the same event pipeline as semantic events:
>   Action.report() → ISemanticEvent[] (includes media events) → EventProcessor → Client receives all events
> The language layer does NOT process media events — they pass through unchanged."

**Channel mapping under ADR-163:**

> Action.report() → ISemanticEvent[] → EventProcessor → channel-service rules match events to channels → CMGT/turn packets emitted → Client renders per channel.

The "language layer doesn't process media events" claim translates cleanly: channel-service's default rule set has no rule that routes media events through the language layer. Stories can register rules that *do* (e.g., a story that wants to wrap image captions in a localized prose blob) but the default is pass-through.

**Status:** ✅ Clean. The pipeline shape is preserved; channel-service is the new producer of the wire shape.

## 12. Capability-conditional channel registration (new concern from this audit)

**Surfaced by the §2 capability-negotiation discussion:** under ADR-101, a story emits `media.image.show` regardless of whether the client supports it; the client ignores it. Under channel I/O the more efficient pattern is for the story to *not register* media channels the client doesn't support — saving wire bandwidth and CMGT manifest size.

But this couples story init to client capabilities, which means **story init must run after the `hello` packet arrives**. Today's sandbox / engine init is synchronous at story-load time; there's no equivalent of "wait for client capabilities before finalizing channel registry."

Two paths:
- (a) Story registers all media channels unconditionally; the channel-service (or CMGT producer) filters out channels the connecting client doesn't support before emitting the manifest. Per-client CMGT, single story registration.
- (b) Story re-runs init per client connection, with capabilities visible. More expensive, more flexible.

(a) is much cleaner under stateless multi-user (story init is per-turn anyway; capabilities are an input to the turn). (b) is overkill.

**Status:** ⚠️ Design choice in ADR-164 — recommend (a).

## 13. Things ADR-101 says that don't survive translation

A short list of ADR-101 specifics that ADR-164 explicitly drops or supersedes:

- **`media.status.update`** — subsumed by ADR-163's `location` / `score` / `turn` / `score_notify` channels.
- **`onComplete: string` as a free-form event-name** — replaced by the callback decision in §7.
- **Hotspot `action: string` as a parser command** — same callback decision in §8 (but the recommendation is to keep ADR-101's behavior as the v1 default).
- **Implicit "events flow through the EventProcessor"** — preserved in shape but the *consumer* of the event stream is now `channel-service`, not the client directly.
- **The phrase "Client is a renderer, not an interpreter of game semantics"** — preserved verbatim in ADR-164's Decision 1 framing, since it's still the load-bearing principle.

## 14. Summary: design choices ADR-164 must make

The audit surfaces six concrete decisions ADR-164 needs to land. Recommendations are first-pass instincts; revisit during the actual ADR draft.

| # | Decision                                                              | Recommendation                                                                                              |
| - | --------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| 1 | Capability handshake mechanism                                        | New `hello` packet, kind `'hello'`, payload `ClientCapabilities`. Server emits CMGT after receiving it.     |
| 2 | Capability scope: gates all channels, or only media channels?         | Only media channels. `main` / `prompt` / `location` / `score` / `turn` / `chat` / `presence` / `command_echo` always exist. |
| 3 | Layer z-ordering carrier                                              | Convention only — renderer hard-codes background < main < overlay; story layers carry explicit `z` in show-event payload. |
| 4 | Callback path (animation `onComplete`, hotspot click, etc.)           | Synthesized commands — client emits the callback name through the existing input pipeline. No new packet kind. |
| 5 | `append`-mode truncation by `media.clear`                             | Allowed. Adds an invariant: `clear` events with `target` matching an `append` channel's domain reset its accumulated state on the renderer. Server-issued. |
| 6 | Story init timing relative to client capabilities                     | Story registers channels unconditionally; CMGT producer filters by per-client capabilities at emission time. |

## 15. Risks / lossy translations

- **The `bus` rename for `media.sound.play`'s mixer channel** is a tiny but real ergonomic break vs ADR-101 — anyone who internalized "media.sound.play with channel" will need to relearn. Worth calling out in ADR-164's "Replaces" section.
- **`media.image.preload` as `event` mode** assumes the renderer downloads on receipt. If the renderer is a thin shell that delegates asset fetch to a worker, the `event`-mode-once-and-discard semantic might mean the worker misses the preload. Probably fine — but a renderer-side concern for the renderer ADR.
- **Capability negotiation in single-user (CLI, zifmia)** is "free" in the sense that the runtime knows its own capabilities — but the protocol still requires the `hello` exchange as a formality. ADR-164 should specify that single-user runtimes synthesize a `hello` rather than actually transmit one.
- **The Alderman exemplar uses no media events.** It's all text-and-structured-data. Worth explicitly verifying ADR-164 doesn't accidentally make text-only renderers depend on a media handshake (the answer is in §14 #2 — capability gates only media channels).

---

## 16. Conclusion

ADR-101 is a clean fold into ADR-164. Six design choices to make (§14), one design tension worth thinking carefully about (§7 / §8 callback path — affects two distinct ADR-101 features), one section explicitly out of scope (§9 asset pipeline → its own ADR). Nothing in ADR-101 is irreconcilable with the channel model, and several ADR-101 concerns (status bar, mixer channels) get *cleaner* under channels.

**Ready to draft ADR-164.** Suggested working title: *Channel I/O Everywhere: Single-User Adoption + Author-Controlled Media (supersedes ADR-101)*.
