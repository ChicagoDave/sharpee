## Pre-plan research findings that change scope (read before Phase 1)

A research pass against the live codebase (not just the ADR text) surfaced facts that revise three of the ADR's assumptions. These are folded into the phases below but are worth stating up front so David can veto before Phase 1 locks them in:

1. **D1's "new export" is smaller than the ADR implies.** `GrammarRule` already carries an `action` field and the base `GrammarEngine` (`packages/if-domain/src/grammar/grammar-engine.ts`) already has public `getRules()`/`getRulesForAction()`. The gap is only that `EnglishParser` (`packages/parser-en-us/src/english-parser.ts`) never re-exposes them. The minimal fix is one new method on `EnglishParser`, not new engine machinery.
2. **D5's "resolution path" is already fully wired.** `story-loader`'s `extendLanguage()` already calls `registry.addMessage(key, ...)` into the same flat message map every other template renders from, and bootstrap/bridge/runtime all extend language *after* loading platform defaults, so a story override already wins once it registers under the right key. D5 reduces to the parser-key-truncation fix (`packages/chord/src/parser.ts:1018`) plus regression coverage — there is no separate "wire the resolution path" sub-task to build.
3. **Defect (a) (loader `lockable with key` drop) is three bugs, not one.** Beyond the container-branch guard skip the ADR names, the trait-config value that reaches `LockableTrait` is an unresolved display-name string (no name→world-id resolution exists for Chord trait config anywhere), and three TS-side builder call sites (`packages/helpers/src/builders/container.ts`, `.../door.ts`, `packages/world-model/src/world/WorldModel.ts` `createDoor`) pass a `requiredKey` field that `ILockableData` does not have (only `keyId`/`keyIds`) — a silently-dropped typo, independent of Chord. Phase 8 scopes all three.
4. **D4's verb list has three constants with no backing action at all** (`if.action.turning`, `if.action.using`, `if.action.answering` — `using` is explicitly commented out of stdlib's `constants.ts` as "not idiomatic IF"; `turning` and `answering` have no `standard/` directory). These are not in the ADR's example list and need an explicit build-vs-drop call in Phase 1, not a mechanical alias.
5. **The ADR's dead-id example `too_heavy` is confirmed live** (5 call sites, golden-tested) — `feels_cold`/`feels_rough` are the ones confirmed genuinely unreachable. Phase 8 should drop `too_heavy` from the cleanup list.
