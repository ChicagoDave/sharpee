## Scope

Phase 6 closes the ADR-172 loop by wiring `propagate()` into the turn cycle.
Three deliverables:

1. `ActionContext.emitSound` — authoring API for registering a sound emission
   from inside an action's report phase.
2. Per-turn audibility dispatcher — iterates all `ListenerTrait` entities, calls
   `propagate()` for each buffered sound, emits `sound.audibility.heard` semantic
   events.
3. End-to-end integration test — the tapestry puzzle scenario from ADR-172
   §End-to-end scenario.
4. ADR-172 status flip to ACCEPTED (last step, no code).

Phase 7 (browser audio rendering) is explicitly out of scope.

## Turn-Cycle Integration Point Analysis

Reading `game-engine.ts` `executeTurn()` reveals the following ordering (simplified):

```
createUndoSnapshot()
commandExecutor.execute()           ← action's validate / execute / report
  eventProcessor.processEvents()    ← entity event handlers (ADR-052)
                                    ← this is where allEvents is returned
processEvent() enrichment loop
pluginRegistry tick (NPC phase)
processPlatformOperations()
textService.processTurn()           ← text rendering
emitChannelPacket()
turn counter increment
```

The sound dispatcher must run **after** `commandExecutor.execute()` has produced
`allEvents` (so all sounds from the action's report phase are buffered) and
**before** `textService.processTurn()` (so the `sound.audibility.heard` events
are included in the text turn and the `audibility` channel packet). This maps
to the gap between the existing enrichment loop and `textService.processTurn()`.

The plugin-registry tick (NPC phase) runs before the platform operations check.
NPCs that emit sounds during their tick also need those sounds dispatched — the
dispatcher must also fire after `pluginRegistry` if NPC-emitted sounds are a
requirement in Phase 6. For now, Phase 6 only requires player-action sounds (the
tapestry test scenario uses NPC speeches pre-existing in the world, not sounds
emitted by NPC plugin ticks). The dispatcher therefore fires once per turn, after
enrichment and after the plugin tick, before `textService.processTurn()`. This
single-dispatch point is sufficient for L1.

**Precise insertion point in `game-engine.ts`**:

After line `const existingEvents = this.turnEvents.get(turn) || [];` /
`this.turnEvents.set(turn, [...existingEvents, ...semanticEvents]);` block, and
after the plugin tick loop that produces `pluginEvents`, but before the
`processPlatformOperations` call. The dispatcher reads the per-turn sound buffer,
runs propagation, and pushes `sound.audibility.heard` events into `turnEvents.get(turn)`.

## Open Design Questions (must be settled before coding starts)

**Q1 — Where does the per-turn sound buffer live?**

Options:
- A) Private field on `GameEngine` (`private soundBuffer: ISound[] = []`), cleared
  at the start of each `executeTurn()`.
- B) A new file `packages/engine/src/sound/dispatcher.ts` that receives the buffer
  as a parameter from the engine, keeping `GameEngine` as a thin coordinator.

Recommendation: B. The dispatcher is a distinct responsibility from the engine's
orchestration; the same pattern as `TurnEventProcessor` (extracted service). The
engine constructs a `SoundDispatcher` in its constructor (parallel to how it
constructs `TurnEventProcessor`), passes the world and the buffer, and calls
`dispatcher.dispatch(listeners, timestamp)` at the appropriate point.

The buffer itself can live as a field on `SoundDispatcher` (or passed in from the
engine). Either way it must be cleared at the start of `executeTurn()` — not at
the end — so that sounds from the current turn's report phase are available when
the dispatcher runs.

**Q2 — Who calls `propagate()` from `ActionContext.emitSound`?**

The ADR says `emitSound` buffers the sound for later dispatch (not immediate
propagation). This means `emitSound` appends to the per-turn sound buffer only.
Propagation runs once at the end of the turn by the dispatcher. The buffer is the
contract.

Two sub-questions:
- The buffer must be accessible to both `createActionContext` (which calls
  `emitSound`) and the `SoundDispatcher`. The clean path: the engine passes a
  reference to the mutable buffer array when constructing the action context.
  `createActionContext` gains an optional `soundBuffer?: ISound[]` parameter.
- `emitSound` fills in `emitterId` and `roomId` from `player.id` and
  `world.getLocation(player.id)`. For NPC actions, the actor is not always the
  player. `ActionContext` already has `player` — but for NPC action contexts the
  "actor" might differ. In Phase 6, the `emitterId` field should be the action's
  actor (already accessible via `context.player` in the current architecture,
  since NPC actions use the NPC as the player for context). Verify this is safe.

**Q3 — `ISound` field name alignment**

The `ISound` type in `if-domain` uses `sourceEntity` and `sourceLocation` as field
names (checked in `propagation.ts`: `sound.sourceEntity`, `sound.sourceLocation`).
The ADR's authoring example uses `source_entity` and `source_location` (snake_case).
The TypeScript types ship camelCase. The `emitSound` parameter type must use the
TypeScript field names. Verify by reading `packages/if-domain/src/sound/types.ts`
before writing code.

**Q4 — Listener enumeration determinism**

The constraint says listeners must be processed in entity-id sort order for
snapshot stability. `world.findByTrait(TraitType.LISTENER)` returns an array;
its ordering is not guaranteed stable across mutations. The dispatcher must sort
by `entity.id` before iterating. Confirm `findByTrait` exists on `WorldModel`
(it is used in `game-engine.ts` at line ~362 for `STORY_INFO`, so it does).

**Q5 — Test world isolation**

The integration test (tapestry scenario) creates a small custom world: parlor,
library, hall, wall entities, two NPCs, player, tapestry. The two options:
- Unit-style test in `packages/engine/tests/sound/` using `WorldModel` directly
  (no story) — consistent with existing `propagation.test.ts`.
- Story-level integration using a minimal test story.

Unit-style is cleaner for pinning the two-turn state transition (tapestry taken →
effective cost changes). It requires wiring the engine's full turn cycle, which
means using `GameEngine` in a test, not just `WorldModel`. `propagation.test.ts`
does not use `GameEngine`. The integration test must use `GameEngine` because it
exercises the dispatcher (which is inside the engine). The prior `listener-wiring.test.ts`
in `packages/engine/tests/sound/` already constructs a full engine to test player
auto-attachment, so that pattern is available.

Decision: Unit-style in `packages/engine/tests/sound/integration.test.ts`.
Construct a real `GameEngine`, define a minimal story (parlor + library + hall
topology per ADR-172 §End-to-end scenario), emit sounds via a test action that
calls `context.emitSound`, verify `sound.audibility.heard` events on the turn
result. Two-turn test: turn 1 (tapestry present, expect `presence-only`), take
tapestry (move it to player inventory, no longer in hall), turn 2 (expect tier
upgrade). This pinning AC-2, AC-4, and AC-7's "removal causes cost to drop."

**Q6 — `emitSound` in `ActionContext` vs capability-dispatch contexts**

`ActionContext` is the interface. `createActionContext` is the implementation.
The implementation already gets `GameContext` and `WorldModel`. Extending the
factory function to accept a `soundBuffer` reference and populate `emitSound` is
a contained change. Capability-dispatch contexts are also created via
`createActionContext`, so capability behaviors automatically gain `emitSound` as well.

## Steps

### Step 6.1 — `emitSound` on `ActionContext` (interface + factory)

**Tier**: Small — ~100 tool calls
**Files touched**:
- `packages/stdlib/src/actions/enhanced-types.ts` — add `emitSound` method
  signature to `ActionContext` interface
- `packages/engine/src/action-context-factory.ts` — add `soundBuffer` parameter,
  implement `emitSound` closure
- `packages/engine/src/types.ts` — no changes (sound buffer is engine-internal)

**Behavior Statement — `ActionContext.emitSound`**:
- DOES: Appends an `ISound` to the per-turn sound buffer, filling in
  `sourceEntity` from `context.player.id` and `sourceLocation` from
  `world.getLocation(context.player.id)` (the actor's current room).
- WHEN: Called from any action's `report()` phase (or `execute()` if needed),
  with a partial `ISound` omitting `sourceEntity` and `sourceLocation`.
- BECAUSE: Actions know what sound to emit but should not resolve their own
  location; the context provides the actor's room automatically (DRY, correct
  for both player and NPC callers).
- REJECTS WHEN:
  - `sourceLocation` cannot be resolved (actor not in any room): emitted to
    buffer with `sourceLocation = ''`; the dispatcher's `propagate()` call will
    return `null` for that sound and it will be silently dropped. No throw (silent
    sounds in unlocated actors are explicitly permitted in the ADR's rejection
    rules — "detached entities cannot emit propagating sound, emission dropped
    silently with a debug warning").
  - `volumeTier` is not a valid `VolumeTier`: caught at TypeScript compile time;
    runtime guard logs a debug warning and drops the emission (same ADR rule as
    unknown volume tier).

**Test surface** (`packages/stdlib/tests/actions/emit-sound.test.ts` OR
`packages/engine/tests/sound/emit-sound.test.ts`):
- `emitSound` appends to the buffer with correct `sourceLocation` auto-fill.
- `emitSound` appends to the buffer with correct `sourceEntity` auto-fill.
- Calling `emitSound` multiple times in one report phase produces multiple buffer entries.
- After the action context is created with no buffer (meta-command path), `emitSound` is
  a no-op (the meta-command path does not create a sound buffer).

**Cross-package impact**: `@sharpee/stdlib` (interface change), `@sharpee/engine`
(factory change). Stories that import `ActionContext` will see the new method.
Because it is additive, no existing action is broken.

### Step 6.2 — `SoundDispatcher` (new file in engine/src/sound/)

**Tier**: Small — ~100 tool calls
**Files touched**:
- `packages/engine/src/sound/dispatcher.ts` (new)
- `packages/engine/src/sound/index.ts` — re-export `SoundDispatcher`

**Behavior Statement — `SoundDispatcher.dispatch`**:
- DOES: For each `ISound` in the buffer, calls `propagate(sound, listenerId,
  world, timestamp)` for every `ListenerTrait` entity (sorted by entity id for
  determinism); for non-null results emits one `sound.audibility.heard`
  `ISemanticEvent` with the `IAudibilityEvent` as its `data` field and the
  `listenerId` in `entities.listener`.
- WHEN: Called once per turn by `GameEngine.executeTurn()`, after action events
  are collected and after the plugin tick, before `textService.processTurn()`.
- BECAUSE: The dispatcher closes the loop between `emitSound` (authoring API) and
  the channel system (ADR-163 `audibility` channel), implementing AC-2 and AC-4.
- REJECTS WHEN (does not throw; silently skips):
  - Buffer is empty (no sounds emitted this turn).
  - `propagate()` returns `null` for a (sound × listener) pair (below silence
    threshold or no path).
  - A `ListenerTrait` entity has no resolvable room.

**Method signature**:
```typescript
dispatch(
  buffer: readonly ISound[],
  world: WorldModel,
  timestamp: number,
): ISemanticEvent[]
```
Returns the array of `sound.audibility.heard` events (may be empty). The engine
merges these into `turnEvents` before rendering.

**Test surface** (`packages/engine/tests/sound/dispatcher.test.ts`):
- Empty buffer → empty result, `propagate` not called.
- One sound, one listener in same room → one `sound.audibility.heard` event with
  `audibilityTier: 'full'`.
- One sound, one listener, no path → empty result.
- One sound, two listeners at different ranges → two events with correct
  per-listener tiers.
- Two sounds, one listener → two events.
- Listener order is entity-id sorted (snapshot-stable).
- Returned event `type === 'sound.audibility.heard'` and `data` is an
  `IAudibilityEvent`.

**Cross-package impact**: Engine only. `SoundDispatcher` is engine-internal.

### Step 6.3 — Wire dispatcher and buffer into `GameEngine`

**Tier**: Small — ~100 tool calls
**Files touched**:
- `packages/engine/src/game-engine.ts`
  - Add private `soundBuffer: ISound[]` field.
  - Add private `soundDispatcher: SoundDispatcher` field, constructed in
    constructor.
  - In `executeTurn()`: clear `soundBuffer` at turn start; pass buffer ref
    through `createActionContext`; after plugin tick, call
    `this.soundDispatcher.dispatch(this.soundBuffer, this.world, turn)`;
    merge returned events into `turnEvents` before `textService.processTurn()`.
  - In `restartGame()`: clear `soundBuffer` (same as other per-turn bookkeeping).
- `packages/engine/src/action-context-factory.ts`
  - Add optional `soundBuffer: ISound[]` parameter to `createActionContext`.
  - Wire into the `emitSound` closure.

**Turn-cycle ordering (precise)**:

```
turn start — clear soundBuffer
commandExecutor.execute()
  → action report phase may call context.emitSound()
  → emitSound appends to soundBuffer
processEvent() enrichment loop
pluginRegistry tick
soundDispatcher.dispatch(soundBuffer, world, turn)
  → returns sound.audibility.heard events
merge dispatcher events into turnEvents
processPlatformOperations()
textService.processTurn(turnEvents)   ← sound events included here
emitChannelPacket()                   ← audibility channel sees them
```

**Behavior Statement — `GameEngine` sound buffer lifecycle**:
- DOES: Clears `soundBuffer` at the start of every turn; populates it via
  action contexts during command execution; dispatches all buffered sounds via
  `SoundDispatcher.dispatch()` after the plugin tick; merges resulting
  `sound.audibility.heard` events into `turnEvents`; clears buffer state for
  the next turn (implicitly, via the clear at turn start).
- WHEN: Every call to `executeTurn()`. Meta-commands use a separate code path
  (`executeMetaCommand`) that never touches `soundBuffer`; sounds are not
  emitted during meta-commands.
- BECAUSE: Sounds must be accumulated for the whole turn (multi-action scenarios
  are handled in one turn), dispatched once, and expired afterward so they don't
  persist into the next turn.
- REJECTS WHEN: Not applicable — buffer clear + dispatch is unconditional.

**Test surface** (`packages/engine/tests/sound/engine-wiring.test.ts`):
- After `executeTurn()`, `soundBuffer` is empty (cleared for next turn).
- `sound.audibility.heard` events appear in `result.events` when a sound was
  emitted and at least one listener is in range.
- `sound.audibility.heard` events do NOT appear when no sound was emitted
  (quiet turn).
- The `audibility` channel packet entry is present in `channel:packet` emission
  when sound events exist.

Note: These tests construct a real `GameEngine` with a minimal test story (one
room, player, a simple action that calls `emitSound`). They exercise the full
pipeline end-to-end without a tapestry or wall topology. The tapestry scenario
belongs in Step 6.4.

**Cross-package impact**: Engine only (internal wiring). `createActionContext`
signature change is additive (optional parameter); all existing callers pass no
`soundBuffer` and get the no-op `emitSound` behavior.

### Step 6.4 — End-to-end integration test (tapestry puzzle scenario)

**Tier**: Medium — ~250 tool calls
**Files touched**:
- `packages/engine/tests/sound/integration.test.ts` (new)

**Test world topology** (per ADR-172 §End-to-end scenario, adapted for unit test):

Rooms: Parlor, Library, Hall.
- Parlor ↔ Hall: open exit (cost 1, no door).
- Hall ↔ Library: wall with `AcousticTrait('thin')` (cost 2). Tapestry on Hall
  side with `AcousticDampenerTrait({ contribution: +3 })`.
- Parlor ↔ Library: wall with `AcousticTrait('thick')` (cost 6).
- Player starts in Parlor with `ListenerTrait` (auto-attached).
- Hervé (NPC entity, no `ListenerTrait` needed for this scenario) is in Library.
- Tapestry: portable entity in Hall, referenced as obstructor on Hall-side of the
  Hall ↔ Library wall, carrying `AcousticDampenerTrait({ contribution: +3 })`.

**Sound emission mechanism**: Define a minimal test action
(`sound.test.action.herve_speaks`) that calls `context.emitSound({volumeTier:
'raised', kind: 'speech', content: { messageId: 'test.herve.speech' }})`. The
action's `sourceLocation` is auto-filled from Hervé's location (Library). The
test calls `engine.executeTurn('herve speak')` (requires grammar registration in
a tiny test story).

Alternatively — and this is simpler — directly call the dispatcher with a
pre-built `ISound` in the test (bypassing the action layer) to test the
propagation outcomes. This is acceptable for the AC-4 tier-verification part but
does not exercise the `emitSound` API. **Both sub-tests are needed:**
- Integration sub-test A: exercises `emitSound` via an action + engine turn.
- Integration sub-test B: exercises multi-turn state transition (tapestry removal
  changes tier).

**Behavior Statements for integration test**:

**Turn 1 — tapestry present**:
- Sound path: Library → (thin wall + tapestry contribution = 2+3=5) → Hall (room
  hop = 1) → (open door = 1) → Parlor. Total = 7. Clarity = 7 − 7 = 0 →
  `silent`.
- Path 2: Library → (thick wall = 6) → Parlor. Clarity = 7 − 6 = 1 →
  `presence-only`.
- Best path wins: `presence-only`. One `sound.audibility.heard` event delivered
  to player with `audibilityTier: 'presence-only'`.

**After tapestry taken from hall**:
- Tapestry moves to player inventory (no longer in Hall).
- Effective cost of Hall ↔ Library wall: 2 + 0 = 2 (tapestry contribution gone).
- Sound path 1: Library → (thin wall = 2) → Hall (1) → (open door = 1) →
  Parlor. Total = 4. Clarity = 7 − 4 = 3 → `muffled`.
- Path 2: Library → (thick wall = 6) → Parlor. Clarity = 1 → `presence-only`.
- Best path: `muffled`.

**Turn 2 — tapestry gone**:
- Player hears `sound.audibility.heard` with `audibilityTier: 'muffled'`.

**Acceptance criteria pinned by this test**:
- AC-2 (content-bearing sound with correct tier delivered to listener).
- AC-4 (multiple sounds, single listener, per-turn correct tier).
- AC-7 (obstructor removal causes cost to drop; subsequent emissions reflect change).

**Cross-package impact**: Engine tests only. No production code changes.

## Tradeoffs and Flags

**Sound buffer as a `GameEngine` field vs. passed through `GameContext`**: The
buffer is engine-internal state (cleared each turn, not part of the serializable
game state). Placing it on `GameEngine` as a private field is correct; it must
not appear in save/restore snapshots. `GameContext` is serialized for save/restore
(indirectly, through the world model); do not add the sound buffer there.

**`emitSound` in `ActionContext` interface vs. optional method**: Making it
required in the interface means all existing `ActionContext` implementations
(including tests that construct minimal mock contexts) must add the method. Because
test mocks typically only implement the methods they use, a missing `emitSound`
would cause TypeScript errors in test files across the codebase. Options:
- Make it optional (`emitSound?`) on the interface — callers that use it
  do a null check.
- Provide a default no-op implementation in a base/factory so mocks inherit it.

Recommended: Add `emitSound` as a required method on `ActionContext` and update
the `createActionContext` factory to always provide it (passing an empty dummy
buffer when no `soundBuffer` is given). This means zero existing callers break
because the factory always returns a complete context. Tests that construct mock
`ActionContext` objects directly will need to add `emitSound: () => {}` — but
searching for manual mock constructions is a bounded task.

**NPC turn sounds**: The plugin tick runs before the dispatcher call. NPC actions
in the plugin tick would need to call `emitSound` via their own `ActionContext`.
The plugin tick today does not use `ActionContext` — it uses `TurnPluginContext`.
NPC sound emission in the plugin tick is therefore deferred (it is an L2 concern);
Phase 6 only wires sounds from player-turn actions. This is acceptable because
Phase 6's test scenario places Hervé's speech in a test action (player-turn),
not in an NPC plugin tick.

**What the test story needs**: The integration test requires:
1. A real `GameEngine` constructed with a `WorldModel`, player, parser, language
   provider. The existing `listener-wiring.test.ts` shows the minimal
   construction pattern.
2. A test action registered with the engine that calls `context.emitSound`.
3. Grammar pattern registered so the parser can match a command to that action.
4. Wall entities created with ADR-173's `createWall` API (already tested
   extensively; re-use the same pattern).
5. A tapestry entity with `AcousticDampenerTrait` placed in Hall and referenced
   as the Hall-side obstructor of the Hall ↔ Library wall.

The test must not use any real story (Dungeo). It is a self-contained engine test
with synthetic world content. This keeps it isolated from story changes.
