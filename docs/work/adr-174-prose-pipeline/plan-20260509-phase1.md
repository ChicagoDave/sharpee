**Date**: 2026-05-09
**Branch**: main

**Predecessor**: session `session-20260509-2352-main` (ADR drafted, audibility handler shipped under text-service)
**Scope**: Phase 1 only. Phase 2 (downstream consumer migration) and Phase 3 (`@sharpee/text-service` deletion) get their own plans.

---

## Goal

Stand up `packages/engine/src/prose-pipeline/` as the event-to-block translator, replace `@sharpee/engine`'s import of `@sharpee/text-service` with the new internal pipeline, and ship the bracket decoration model end-to-end so an `[em:emphasized]` template renders italic in a default-theme browser build.

After Phase 1:

- `@sharpee/engine` does not import `@sharpee/text-service`.
- `IDecoration` carries `className` (resolved final-form), not `type`.
- Bracket markup `[name:content]` parses into structured `IDecoration` nodes; platform names get the `sharpee-` prefix; author names pass through bare.
- The full Dungeo walkthrough chain passes with zero regressions.
- AC-1 through AC-7 from ADR-174 are satisfied.

`@sharpee/text-service` remains compilable and on disk for downstream wire-production consumers (zifmia, transcript-tester, etc.) — those migrations are Phase 2.

## Why this is necessary

ADR-174 §Migration phasing makes Phase 1 the load-bearing one: it introduces the new mechanism (decorations) and the new home (engine-internal pipeline) in a single hard-cutover for engine. Phases 2 and 3 are mechanical removals once Phase 1 has shipped a working replacement.

The Phase 7a audibility handler from the prior session lives under text-service today. Phase 1 ports it — and every other handler — to the new location with no shape change. No code is rewritten; almost all of it is moved and rewired.

The new code in Phase 1 is concentrated in two surfaces:

1. **Decoration parser/resolver** — the bracket syntax itself is new. ADR-091's `decoration-parser.ts` exists in text-service and parses `[type:content]` plus asterisk emphasis, but the output shape (`{ type, content }`) and the asterisk pathway both go away. The parser is rewritten to the ADR-174 contract: brackets only, `{ className, content }` output, platform-vocabulary resolution against a frozen `Set`.
2. **`ProsePipeline` class** — wraps the same per-turn contract `TextService` had. Most of the work is composing the ported stages; the class itself is small.

Everything else in Phase 1 is a port: same code, new package, updated import paths.

## Naming discipline

Three concepts coexist and must not be confused:

| Concept | Where it lives | Identifier examples |
|---------|----------------|---------------------|
| **`@sharpee/text-service` package** | retiring; deprecated end of Phase 1 | `TextService`, `createTextService`, `renderToString` |
| **`@sharpee/engine` prose pipeline** | new, engine-internal | `ProsePipeline`, `processTurn`, `prose-pipeline/` |
| **`@sharpee/text-blocks` block types** | preserved package, modified shape | `ITextBlock`, `IDecoration`, `TextContent` |

The new pipeline is named for **prose** — never "text" (collides with the retiring package) or "decoration" (one of three responsibilities, alongside event filtering and message resolution).

### Behavior Statement — `parseDecorations`

- **DOES**: Parses a template string containing zero or more bracketed expressions of the form `[name:content]` into a `TextContent[]` tree. Each well-formed bracket becomes an `IDecoration { className, content }` node where `className` is the result of `resolveClassName(name)`. Plain runs of text become string entries. Escape sequences `\[`, `\]`, `\\` produce literal characters. Nested brackets recurse.
- **WHEN**: Called by the prose pipeline once per resolved template, before the template is wrapped in an `ITextBlock`.
- **BECAUSE**: ADR-174 §Markup syntax requires bracket-form decorations as the sole markup mechanism. Without parsing, brackets in templates would render literally to the player, defeating the decoration model.
- **REJECTS WHEN**:
  - Unclosed bracket (`"hello [em:world"`) → emits the unclosed segment as literal text. No `IDecoration` created. AC-10.
  - Bracket without colon (`"[em world]"`) → emits literal text. No `IDecoration` created. AC-11.
  - Empty class name (`"[:world]"`) → emits inner content as plain text without decoration wrapping. No `IDecoration` created. AC-12.

### Behavior Statement — `resolveClassName`

- **DOES**: Returns `"sharpee-{name}"` if `name` is in the platform vocabulary `Set`; otherwise returns `name` verbatim.
- **WHEN**: Called by the parser for every well-formed bracket name encountered.
- **BECAUSE**: ADR-174 §Platform vs author classes locks the resolution rule; collapsing the platform-vs-author distinction at parse time makes the wire shape renderer-ready.
- **REJECTS WHEN**: Empty string input — returns empty string (caller is responsible for the AC-12 no-op-wrapper behavior; resolver is pure).

### Behavior Statement — `ProsePipeline.processTurn`

- **DOES**: Receives a `ReadonlyArray<ISemanticEvent>` for one turn; runs the filter stage to drop `system.*` and `platform.*` events; runs the sort stage applying ADR-094 chain metadata; dispatches each remaining event to its handler family by event-type prefix; collects per-handler `ITextBlock[]` outputs; returns the flat array of blocks in turn order. Each block's `content` is a `TextContent[]` produced by `parseDecorations` against the resolved template, so platform-vocabulary brackets resolve to `sharpee-`-prefixed `IDecoration` nodes and author-class brackets to bare `IDecoration` nodes.
- **WHEN**: Called by `GameEngine` from `executeTurn`, `restartGame`, and the meta-command path — the same three call sites the retiring `TextService.processTurn` had.
- **BECAUSE**: ADR-174 §Engine-internal prose pipeline locks the per-turn contract; ADR-097, ADR-094, and ADR-133 establish the event filter, ordering, and block-shape responsibilities that this class composes.
- **REJECTS WHEN**:
  - `events` is empty → returns empty `ITextBlock[]`. No throw.
  - A handler throws → log once at `debug` level (matching today's text-service behavior); skip the offending event; continue with remaining events. The pipeline is non-throwing per the existing `ITextService` contract.
  - `languageProvider` is missing at construction time → constructor throws (programming error; not runtime-recoverable).

### Behavior Statement — browser renderer `IDecoration` translation

- **DOES**: Walks a `TextContent` tree; for each `IDecoration` node, emits a `<span class="${className}">` wrapping the recursively-rendered children; for each string node, emits its text content (HTML-escaped).
- **WHEN**: Called by the browser renderer when projecting an `ITextBlock` to the DOM.
- **BECAUSE**: ADR-174 §Wire shape mandates structured decorations on the wire and translation in renderers; without this, decorations would either render as JSON or as plain text — losing the styling intent.
- **REJECTS WHEN**: `className` empty → renders inner children without the wrapping span (defense-in-depth; the parser already filters empty class names per AC-12, but the renderer should not crash if one slips through).

## Risks and rollback

### Risks

- **Sub-phase 1.6 cutover regression.** If a handler is mis-ported or a stage's import path is wrong, walkthroughs may fail in non-obvious ways (silent prose-drop in a single event type). Mitigation: per-handler tests in 1.4 + the walkthrough chain in 1.6 catch most cases. The original handler files in text-service are deleted only after the engine ones pass tests.
- **OQ-2 underestimation.** If shipping the platform CSS turns out to require a new `@sharpee/prose-css` package (third OQ-2 candidate), sub-phase 1.7 grows from "ship one CSS file" to "create a workspace package" (per `feedback_new_package_config.md`, that is six registration points). Mitigation: discuss OQ-2 explicitly before starting 1.7. If the new-package path is needed, treat it as its own sub-phase 1.7a.
- **`IDecoration` shape change cascades.** `CORE_DECORATION_TYPES` removal may break a downstream package outside the surveyed import list. Mitigation: sub-phase 1.2's `pnpm -w typecheck` step is the canary.
- **Ported tests rot.** Tests written against text-service paths and helpers may use idioms that don't fit engine's test infrastructure (e.g., different mock-language-provider helper). Mitigation: port one handler's tests in isolation, run, and only proceed if green.

### Rollback per sub-phase

| Sub-phase | Revert command | Effect |
|-----------|---------------|--------|
| 1.1 | `git revert <sha>` | New decoration files vanish; nothing else affected; engine continues using text-service. |
| 1.2 | `git revert <sha>` | `IDecoration` shape reverts; text-service decoration-parser restored. |
| 1.3 | `git revert <sha>` | Stages unported; text-service stages remain canonical. |
| 1.4 | `git revert <sha>` | Handlers unported; text-service handlers remain canonical. |
| 1.5 | `git revert <sha>` | `ProsePipeline` class removed; engine still using text-service. |
| 1.6 | `git revert <sha>` | Engine reverts to importing text-service. **This is the riskiest revert** because intervening fixes may cluster around the cutover commit; revert only after verifying the chain. |
| 1.7 | `git revert <sha>` | Platform CSS unshipped; renderer falls back to no-styling decorations (still rendered as spans, just no platform-default styling). |
| 1.8 | n/a — verification only | n/a |

The cleanest cumulative rollback is `git revert` of every sub-phase commit in reverse order. Each sub-phase is sized to be independently revertible.

## What this plan does NOT do

- Does not start Phase 2 (downstream consumer migration off `renderToString`). That has its own plan.
- Does not delete `@sharpee/text-service` (Phase 3).
- Does not change channel-service, channel definitions, renderer architecture, language-provider interface, or action authoring. ADR-174 §What this doesn't touch enumerates the unchanged surfaces.
- Does not introduce multi-class span syntax. Single class per `IDecoration`; nesting handles multi-style cases.
- Does not add new platform decoration names beyond the ADR-174 §Closed platform vocabulary list. Adding `[blink:...]` is a future ADR-and-CSS change.
- Does not add author-vocabulary discovery, inheritance, or conflict warnings (e.g., warn if a story declares a class name that collides with a platform name). Out of scope.
- Does not address terminal-renderer translation of `IDecoration` (Implementation Modules row "Terminal renderer translation of `IDecoration`"). That is OQ-1 work and lands in Phase 2 alongside the `renderToString` replacement helper.
- Does not test mainChannel's downstream rendering of decorations (mainChannel reads `ctx.blocks` and projects them; whether the projection preserves `IDecoration` nodes through the channel boundary is an integration concern that the walkthrough chain implicitly covers).
