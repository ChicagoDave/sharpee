# ADR-174 Phase 2 — Migrate Wire-Production Consumers off `@sharpee/text-service`

**Date**: 2026-05-10
**Branch**: TBD (recommended: `adr-174-phase2-text-service-consumers`)

**Scope**: Phase 2 only. Phase 3 (`@sharpee/text-service` directory deletion) gets its own plan.

## Goal

Move every non-text-service, non-zifmia `import` of `@sharpee/text-service` to its real home. After Phase 2:

- `renderToString` and `renderStatusLine` ship from `@sharpee/channel-service` (per OQ-1 resolution).
- Every active-release consumer (transcript-tester, cloak-of-darkness scripts, bridge/runtime/sharpee re-exports) imports those helpers from `@sharpee/channel-service`.
- The dead block-production re-exports (`ITextService`, `createTextService`) are dropped from bridge/runtime/sharpee public surfaces — engine has its own internal `ITextService` (engine-private) and no first-party consumer instantiates a text-service.
- Outside `packages/text-service/` and `packages/interpreter/`, **zero** `*.ts` / `*.tsx` / `*.js` source files import from `@sharpee/text-service`.
- Dungeo walkthrough chain remains regression-clean.

`@sharpee/text-service` itself stays on disk and compilable through Phase 2 — and through Phase 3 as long as zifmia is deferred. Phase 3 deletion is gated on either zifmia migration or zifmia decoupling from the workspace.

### Zifmia and cloak-of-darkness carve-outs

**Zifmia is hard-deferred.** `@sharpee/platform-browser` is the primary release mechanism. Zifmia is parked, will not see active development, and if demand returns it will be redesigned from scratch — its current text-service coupling is not a constraint on platform evolution.

**Cloak-of-darkness is also deferred.** Discovered during sub-phase 2.2: cloak is not in `pnpm-workspace.yaml`, its `@sharpee/*` deps are manually maintained symlinks rather than pnpm-managed workspace links, and its build was already failing before Phase 2 with pre-existing errors:
- `src/browser-entry.ts` — 11 errors: `Cannot find name 'document'` (tsconfig missing DOM lib)
- `src/debug-runner.ts:43` and `src/test-runner.ts:52` — `'textService' does not exist in EngineConfig` (API drift; engine removed `textService` from config)

Cloak is in maintenance limbo similar to zifmia. The four cloak edits initially staged in 2.2 (test-runner.ts, run-platform.js, test-parser-events.js, package.json) were reverted; cloak retains its `@sharpee/text-service` symlink/dep entry through Phase 2.

**Practical implications for Phase 2 and Phase 3:**

- `packages/interpreter/**`, `packages/interpreter/package.json`, `stories/cloak-of-darkness/**`, and `stories/cloak-of-darkness/package.json` are **untouched** in Phase 2.
- AC-8's grep verification is **scoped to exclude `packages/interpreter/` and `stories/cloak-of-darkness/`**.
- **Phase 3 still deletes `@sharpee/text-service` outright.** Both zifmia and cloak fall out of build at that point — accepted collateral. Any future revival of either is a redesign, not a port.

This is a deliberate choice: parked products' couplings to a deprecated package would otherwise hold the entire platform hostage. Cutting the cord at Phase 3 frees the platform to evolve and pushes future revivals to start from a clean modern foundation.

## OQ-1 resolution recap

`renderToString` and `renderStatusLine` ship from **`@sharpee/channel-service`**. Rationale lives in ADR-174 §Open Questions OQ-1; summary:

- Channel-service sits downstream of engine, upstream of clients — the right dependency position for a block-flattening helper consumed by renderers, chat overlays, and transcripts.
- Channel-service already owns the `channel:packet` shape that wraps the `ITextBlock[]` consumers want to flatten.
- Engine is rejected (would force chat overlays / transcript tooling to depend on the engine for display flattening — wrong direction).
- A new `@sharpee/wire-utils` package is rejected (premature for two functions; revisitable later).

## Naming discipline

| Concept | Where | Identifier |
|---|---|---|
| Block-flattening helpers (Phase 2 target) | `@sharpee/channel-service/src/render-to-string.ts` | `renderToString`, `renderStatusLine`, `CLIRenderOptions` |
| Engine-internal ITextService contract | `@sharpee/engine/src/prose-pipeline/types.ts` | `ITextService` (engine-private after Phase 2) |
| Retired text-service block-production | `@sharpee/text-service` (whole package, deleted Phase 3) | `TextService`, `createTextService`, `ITextService` |

Note the file name **`render-to-string.ts`** rather than the legacy `cli-renderer.ts`. The function isn't CLI-specific — zifmia uses it for browser chat bubbles. Renaming on move corrects a misleading name.

The `CLIRenderOptions` type name stays, because consumer call sites already use it and renaming creates churn that doesn't belong in this phase. A future cleanup ADR can rename if desired.

### 2.3 — Re-route bridge/runtime/sharpee re-exports; drop dead block-production exports

**Why drop `ITextService` / `createTextService`:**
After Phase 1, the engine's prose pipeline has its own engine-private `ITextService` interface (`packages/engine/src/prose-pipeline/types.ts`). No first-party consumer instantiates a text-service. The re-exports are dead — they survive only because nobody pruned them. Phase 3 deletes the source package; the re-exports must be dropped at some point. Doing it now keeps Phase 3 as a pure directory removal.

If a hypothetical downstream story has hand-coded against `createTextService`, that's an accepted break per project policy ("we currently don't care about backward compatibility" — CLAUDE.md). The story would migrate to engine's `ProsePipeline` or call `engine.setTextService(myCustomImpl)` against the engine-private interface.

## Risks and gotchas

- **fast-cli.ts indirect consumer.** It destructures `renderToString` from the pre-bundled `dist/cli/sharpee.js`. Sub-phase 2.3 must re-route the sharpee re-export so the bundled symbol stays valid. The smoke check `node -e "typeof require('./dist/cli/sharpee.js').renderToString"` is the load-bearing test.
- **`scripts/bundle-entry.js` and `scripts/test-bundle-template.js`** spread text-service's exports into the bundle directly (relative-path `require`, not `@sharpee/text-service` form). The original AC-8 grep pattern missed these; the broader pattern (also matching relative-path requires) caught them in 2.3. Both were swapped to channel-service alongside the package re-exports. The Phase 2.4 grep verification must include `scripts/` in its scope.
- **Stale `dist-esm/` artifacts from Phase 1** can mask source-level migrations. Found and rebuilt in 2.3:
   - `packages/sharpee/dist-esm/` — was still re-exporting text-service
   - `packages/transcript-tester/dist-esm/` — was still importing text-service
   - `packages/platform-browser/dist-esm/BrowserClient.js` — still has old text-service import (timestamp Feb 19; predates Phase 1). Does not affect the CLI bundle (which aliases platform-browser to `dist/`, not `dist-esm/`). Leaving as-is for Phase 2 — flag for Phase 3 cleanup or earlier if a consumer of `dist-esm/` surfaces.
- **Pre-existing `runtime/src/bridge.ts` build errors.** 5 TypeScript errors on Story-narrowed-to-`never`. Confirmed pre-existing via git-stash test (errors identical with or without my Phase 2 edits). Joins zifmia/cloak in pre-existing-maintenance-debt category. Phase 2 does not fix; runtime is allowed to remain in this state.
- **Public API breakage on `ITextService`/`createTextService`.** Drop is intentional; project policy is no-backcompat. Document in Phase 2 plan outcome and ADR Phase 2 acceptance section. If David requests a deprecation period instead, swap 2.3's drop-entirely for keep-as-deprecated-re-exports-pointing-to-engine-internals — that's a 5-line change but inverts the intent. Get explicit confirmation in 2.3 entry.
- **Cloak-of-darkness CommonJS scripts.** `run-platform.js` and `test-parser-events.js` use `require()`, not ES `import`. Sub-phase 2.2 has to update both forms. Easy to miss the JS files when only grep-ing `*.ts*`.
- **OQ-1 stance on `flattenContent` overlap.** `@sharpee/channel-service/utils/flatten.ts` already exists with a `flattenContent` function that strips decorations entirely. `renderToString` is a different concern (translates decorations to ANSI; preserves block separation; handles status filtering). Two helpers, two roles, deliberate. Plan does not consolidate them.
- **Zifmia stays compilable through Phase 2 only.** Phase 2 leaves zifmia importing text-service, so text-service must keep building and exporting `renderToString` / `renderStatusLine` for the duration of Phase 2. Phase 2 explicitly does NOT remove anything from text-service's `src/index.ts`. Phase 3 deletes text-service outright; zifmia breakage at that point is accepted collateral per the hard-defer policy.
- **Phase 3 ordering.** Phase 2 leaves `packages/text-service/` on disk and fully compilable. Phase 3 deletes the directory and the workspace entry. Zifmia falls out of build at Phase 3 — that is intentional. Do not add zifmia migration to Phase 3 scope; if zifmia revives, it gets a fresh redesign rather than a port.
