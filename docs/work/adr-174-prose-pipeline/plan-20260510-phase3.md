## Goal

Delete `packages/text-service/` from disk. Remove every workspace, build-script, and project-reference entry that points at it. Accept that Zifmia and Cloak-of-Darkness fall out of build — both were formally deferred as accepted collateral in Phase 2. After Phase 3:

- `packages/text-service/` does not exist on disk.
- `pnpm install` succeeds without errors.
- The CLI bundle, platform-browser bundle, engine, channel-service, and Dungeo walkthrough chain all pass at their Phase 2 baselines.
- A repo-wide grep for `@sharpee/text-service` returns zero matches outside the two accepted-collateral surfaces.

This is a deletion phase. Nothing new is built. The only migration items are cleanup of stale build artifacts and stale configuration entries.

## Order of Operations

Phase 3 executes in four sub-phases: pre-flight grep (3.1), workspace and configuration cleanup (3.2), package directory deletion (3.3), final regression (3.4).

**The package directory is deleted in 3.3, after config cleanup in 3.2.** Doing config cleanup first means `tsc --build` in 3.3 immediately reflects the updated references; doing them in the reverse order would cause `tsc` to try to resolve the now-deleted directory against the old project reference list.

The dist-esm artifact cleanup for `platform-browser` is part of 3.2 (before the delete), so that `platform-browser` is in a clean state when the full regression runs in 3.4.

**Rollback unit**: a single git commit encompassing sub-phases 3.1–3.3. If 3.4 regression fails, revert the commit; `pnpm install` restores the workspace.

## Rollback

Phase 3 is a single-commit operation. Rollback is:

```bash
git revert HEAD   # or git reset --hard HEAD~1 if on a local branch
pnpm install
```

After revert, `pnpm install` restores the workspace link to the reinstated directory. The `packages/text-service/` directory comes back as git restores it. `pnpm install` re-links it into the workspace.

**What rollback does NOT fix:** `packages/platform-browser/dist-esm/` will be in a freshly rebuilt state (stale artifact is gone). The fresh build is correct and correct to keep; there is no reason to revert the dist-esm rebuild even if the rest of Phase 3 is rolled back.

**Rollback is expected to be unnecessary.** The configuration-first ordering in 3.2 ensures all alias paths are removed from the build scripts before the directory is deleted. If anything fails in 3.3 or 3.4, it is because of an audit miss in 3.1 — fix forward rather than rolling back unless there is a structural reason to abort the phase.

## Open Questions

### OQ-3 — engine/tsconfig.json project reference: is it live after Phase 1?

**Status**: Open.

Phase 1 (engine cutover) removed the engine's runtime import of `@sharpee/text-service`, but `packages/engine/tsconfig.json:21` still lists `{ "path": "../text-service" }` as a TypeScript project reference. This was not removed in Phase 1 or Phase 2. It needs to be removed in Phase 3.

The question is whether removing it causes any `tsc --build` cascades that need attention. The engine has no source-code imports from text-service post-Phase-1, so removing the project reference should be a no-op at the `tsc` type-checking level — but verify in 3.2 that `tsc --build packages/engine` still exits clean after the removal.

If it does not exit clean, record the error and discuss before proceeding with 3.3.

### OQ-4 — Zifmia workspace status: does pnpm resolve it as a dep of anything active?

**Status**: Open.

`packages/interpreter/package.json:25` lists `"@sharpee/text-service": "workspace:*"`. After `packages/text-service/` is deleted, pnpm may complain about this unresolvable workspace dep when processing zifmia's lockfile entry — even if zifmia itself is only listed in `package.json:22` (`"packages/interpreter"`) as a workspace member.

If pnpm errors during 3.3's `pnpm install`, the fix is to update `packages/interpreter/package.json` to remove the text-service dep entry, or to remove zifmia from the workspace. Neither of these decisions has been made; they need explicit user direction if the error surfaces. This is the most likely non-trivial failure mode for Phase 3.

**Resolution options (decide if OQ-4 fires):**

A. Remove `"@sharpee/text-service": "workspace:*"` from `packages/interpreter/package.json`. Minimal change; Zifmia stays in the workspace but loses the dep. Its source code still imports from the package, so Zifmia still breaks at build time — but pnpm install passes.

B. Remove `"packages/interpreter"` from `package.json workspaces[]`. Removes Zifmia from pnpm's resolution graph entirely. Cleaner. Consistent with the "parked, not actively developed" classification.

Preferred: option B is the more consistent signal that Zifmia is parked, but this is a workspace-membership decision that warrants explicit user confirmation, not a unilateral Phase 3 action.
