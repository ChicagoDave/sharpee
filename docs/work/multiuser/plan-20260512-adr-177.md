# Plan — ADR-177 Multi-User (Corrected)

**Date**: 2026-05-12
**ADR**: [`adr-177-multiuser-corrected.md`](../../architecture/adrs/adr-177-multiuser-corrected.md) (ACCEPTED)
**Location**: `tools/zifmia/` (reclaimed; `tools/shite/` retained as parts bin)
**Build flag**: `-c zifmia` (reserved in `build.sh`; not yet wired)

## Scope

Implement the corrected multi-user web product. HTTP carries the
process; WS carries chat + command-line sharing (with awareness
signals); engine runs in-process per turn; SQLite holds durability.

Reclaims the room-governance product surface from the prior working
`tools/server/` (PH / tiers / succession / pin / delete / mute / DMs
/ recording notice / session-events) without inheriting the
Deno-subprocess engine model that killed it.

This plan sequences the eight phases the ADR specifies. Each phase
gets its own session-scoped sub-plan when it starts; this document is
the master spine.

## Phase 1 — Identity

**ADR refs**: §5 (identity), §6 (`identities` table), AC-1, AC-9.

**Deliverables**:
- `identities` SQLite table + migration (greenfield boot).
- `POST /api/identities { handle }` → 201 `{id, handle, is_admin}` | 409 `handle_taken`.
- `POST /api/identities/erase { handle }` → 200; hard-delete + CASCADE.
- Handle validation per ADR-161 amended (3–12 alphabetic; case-insensitive unique; case preserved for display).
- Client localStorage helper (`sharpee:identity` key).
- Unidentified lobby view (read-only `GET /api/rooms` allowed).

**Behavior Statements** required for: `createIdentity`, `eraseIdentity`. Each must DOES against the `identities` row, WHEN on route hit, REJECTS WHEN on duplicate / nonexistent handle.

**Tests** (REAL-PATH against a temp SQLite file):
- AC-1 round-trip: claim → second claim → 409 → erase → re-claim succeeds with new id.
- AC-9 erase frees handle.
- Handle validation rejects: empty, too-short, too-long, non-alpha, mixed-case-collision.

**Exit gate**: AC-1 + AC-9 pass against an actual SQLite file (not mocked).

**Estimated**: 1 session.

- `StoryScanner` minimal (lists `stories/` directory entries; defer `StoryHealth` to Phase 5 or 7).
- `GET /api/stories` returns scanned list.

**Tests**:
- AC-3 PH creation: creating identity creates room → row exists with `primary_host_id = id`, `tier = primary_host` participant row created.
- AC-4 tier-gate: `participant` calling `/pin` returns 403; PH succeeds.
- Join-code uniqueness across N rooms.
- Tier enforcement table — every governance route × every tier.

**Exit gate**: AC-3 + AC-4 pass; `./build.sh -c zifmia` produces a runnable server bundle.

**Estimated**: 1–2 sessions.

## Phase 3 — WS Handshake + Chat + Lock + Turn Broadcast

**ADR refs**: §3 (WS), AC-6 (transport split), AC-10 (lock heartbeat), AC-11 (reconnect).

**Deliverables**:
- WS endpoint `/ws/rooms/:id` (or chosen path).
- Hello handshake — first-frame requirement; close codes 4001/4003/4004/4005/4006.
- Per-room broadcast group; presence tracking.
- `chat:send` → `chat:message` broadcast.
- Lock-on-typing per AC-10 (400ms expiry / 200ms heartbeat / explicit release).
- `HTTP POST /api/rooms/:id/command { handle, text }` → engine.executeTurn → persist save + session_event → broadcast `turn { turnId, submitter, packet }` over WS to every participant (submitter included).
- Reconnect lookup: `hello → identity → (room_id, identity_id) → participant_id` (AC-11).

**Behavior Statements** required for: `submitCommand`, `acquireLock`, `releaseLock`, `broadcastTurn`, `closeOnEraseOrTeardown`.

**Tests** (Playwright or ws-driver integration, not mocked):
- AC-6 transport split: capture WS frames during a session, assert allowed frames only.
- AC-10 lock: A acquires → B blocked → A heartbeats → A stops → 400ms later auto-release → B acquires.
- AC-11 reconnect: same `participant_id` after disconnect/reconnect cycle.
- POST /command returns `{ turnId }` only; no `TurnPacket` in body.

**Exit gate**: AC-6, AC-10, AC-11 pass against a real Fastify+WS server.

**Estimated**: 2 sessions.

## Phase 4 — Succession + Grace + Recycle

**ADR refs**: §4 (governance), §Invariants (PH stickiness), AC-5.

**Deliverables**:
- Successor nomination (`POST /api/rooms/:id/nominate-successor`).
- PH disconnect → grace timer → auto-promote chain.
- `role_change` WS broadcast on every promotion / demotion.
- 14-day idle recycle sweeper (interval-driven; `config.recycle_interval`).
- `presence { graceDeadline }` populated when current PH disconnects.

**Tests**:
- AC-5 cascading succession: PH disconnect → grace elapses → successor promoted → demoted PH row drops → next-in-line inherits successor flag.
- Recycle sweeper: room with `last_activity_at` past threshold gets marked `deleted_at`.
- Grace cancellation: PH reconnects before deadline → no promotion → `presence { connected: true }`.

**Exit gate**: AC-5 passes; grace timer is deterministic in test mode (injectable clock).

**Estimated**: 1–2 sessions.

## Phase 5 — Saves + Restore + Session-Event Log + Transcript Backlog

**ADR refs**: §4 (saves), §6 (`saves`, `session_events`), AC-6 (transcript_backlog), AC-8 carries here.

**Deliverables**:

- `session_events` append on every turn / chat / role-change / lifecycle event.
- Welcome backlog: `GET /api/rooms/:id/state` populates `transcript_backlog` from session_events (capped 1000 per ADR-164).
- `room_restored` WS broadcast post-restore; clients re-fetch state.
- `StoryHealth.validate()` (per-story engine boot check).
- Add per-save 14-day-room-tied lifecycle (no DELETE endpoint per memory `feedback_no_save_delete`).

**Tests**:
- Save → restore → world state matches.
- Transcript backlog cap (1000): turn 1001 evicts turn 1.
- `room_restored` triggers state re-fetch.
- StoryHealth flags broken stories out of `GET /api/stories`.

**Exit gate**: A 50-turn session save → restore → next-turn produces correct world state.

**Estimated**: 2 sessions.

## Cross-Phase Concerns

- **No backwards compatibility**: per project memory + ADR-177 §6 greenfield policy. Schema changes after v1 are one-shot cutovers; ADRs do not include migration plans.
- **Boundary Statements**: required when touching `state/`, `stores/`, `projections/`, `reducers/`, `selectors/` (CLAUDE.md rule 8a). Zifmia's room manager + participant store will trigger this from Phase 2 onward.
- **REAL-PATH tests**: anything named `integration`, `engine`, `subprocess`, `database`, `migration` must run against production code paths, not stubs (rule 13a). Phase 1 onward — every test that touches SQLite or the engine uses a real ephemeral instance.
- **Capability dispatch**: any new in-engine behavior story-side uses ADR-090 (existing pattern). Out-of-scope for the server phases.
- **Channel-io is the wire**: never invent new packet kinds at the server level; everything renderable rides existing channels per ADR-163.
