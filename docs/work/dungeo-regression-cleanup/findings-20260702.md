# Dungeo Regression Cleanup — Findings (2026-07-02)

**Context.** After ADR-207 Phases 1–4 landed (per-`WorldModel` capability registry), the
dungeo unit-transcript suite was driven from ~100 failing assertions plus 12 non-running
transcripts to **deterministically green** (106 transcripts, ~1,690 assertions, "All tests
passed" on consecutive runs). The walkthrough chain passes under the **one-good-run rule**
(user-confirmed baseline: a single fully-passing chain run counts; residual per-run flakes
are canonical MDL RNG — see §4).

A pre-ADR-207 baseline (commit `aa99440d` built in a throwaway worktree) confirmed the
unit-suite failures **predated** ADR-207 — they were Phrase Algebra migration debt
(ADRs 192–206), documented as deferred follow-ups in `session-20260630-2339-main.md`.

This document records (1) platform findings surfaced during the cleanup — **none fixed**,
per the platform-change gate; (2) story bugs found and fixed in `stories/dungeo/`;
(3) test-harness findings; (4) the RNG analysis behind the one-good-run rule.

## 1. Platform findings (packages/** — untouched, need discussion)

### P1. Engine universal dispatch does not prefix short capability-effect messageIds
- **Where:** `packages/engine/src/capability-dispatch-helper.ts` (`effectsToEvents`) vs
  `packages/stdlib/src/actions/capability-dispatch.ts` (`effectsToEvents`).
- **Symptom:** Story capability behaviors that emit short messageIds (`'lowered'`)
  expecting the stdlib dispatch factory's auto-prefix (`if.action.lowering.lowered`)
  render **blank** — the engine's universal dispatch intercepts first (it always wins when
  a binding exists) and forwards the payload unchanged, so the short ID matches no
  registered template.
- **Evidence:** `basket-elevator.transcript` failed 7 assertions with blank output —
  byte-identical in the pre-ADR-207 baseline. The behavior file's own comment documented
  the prefix expectation.
- **Consequence:** the stdlib factory's prefix convention is effectively dead code for any
  entity with a registered binding; every story behavior must use fully-qualified IDs.
- **DECISION (user, 2026-07-02): fully-qualified messageIds are the requirement.**
  Capability-behavior effects must always carry fully-qualified, story-registered
  messageIds; the stdlib factory's short-key auto-prefix is legacy and must not be relied
  on. Documented in `packages/stdlib/CLAUDE.md`.
- **Sweep (done):** all in-repo capability behaviors audited and conformant — dungeo
  basket (`dungeo.basket.*`), egg (`dungeo.egg.no_expertise`, `dungeo.egg.opened` — the
  last short key, fixed in this pass), troll (`dungeo.troll.*`), friendly-zoo petting
  (`zoo.petting.*`), familyzoo v2.0.0 petting (same pattern).
- **Follow-up (platform, when convenient):** remove the now-legacy prefix logic from
  `createCapabilityDispatchAction`'s `effectsToEvents` and the short-key example messages
  in `packages/lang-en-us/src/actions/lowering.ts`/`raising.ts` (the `'lowered'` /
  `'already_down'` keys are unreachable via capability dispatch under the requirement);
  update the factory's doc comment.

### P5. ADR-206/ADR-097 params-binding contract gap — **FIXED (user-directed, 2026-07-02, option 1)**

**Fix applied:** `tryProcessDomainEventMessage` now binds `data.params ?? data` — the
nested-preferred/flat-fallback rule `handleGameMessage` always had. One binding rule
everywhere: "params nest under `data.params`; flat event fields are the fallback; nested
wins when both exist." ADR-206 amended (its original rejection of this option is
superseded — see the ADR's 2026-07-02 Amendment section for the reasoning). Six unit
tests added (`domain-message.test.ts`: nested binds, flat fallback binds, nested-wins,
no-messageId fall-through, blocked block-key routing, inline fallback).
**Verified:** zero `[phrase] renderMessage failed` stderr lines across the full dungeo
unit suite; engine 461 green; dungeo units passed 4× consecutively; chain full pass.
(A pre-existing troll-combat knockout-RNG flake surfaced during verification —
unrelated to the binding change, output rendered correctly — hardened with the standard
sword-recovery guard + RETRY max=10; 8/8 isolated runs after.)

Original analysis (for the record):

**The mechanism.** The pipeline renders every event via a two-stage lookup:
1. `tryProcessDomainEventMessage` (ADR-097, `handlers/domain-message.ts`) claims **any**
   event whose `data.messageId` is set — regardless of event type — and binds template
   params **only from the nested `data.params` object** (`data.params ?? {}`). Flat
   fields on `data` are invisible to it.
2. Only if that returns null does the type switch run; the catch-all
   `handleGenericEvent` (`handlers/generic.ts`) binds **flat** — the whole `data` object
   is the params.

So the binding convention an event gets is selected by whether it carries a `messageId`
— not by anything the author would think of as a params-shape decision. ADR-206's
recorded rule ("`game.message` binds nested; `if.event.*` binds flat") only holds for
`if.event.*` **without** a messageId; adding one silently flips the event into the
nested-only path. (`handleGameMessage` is a third convention: `data.params ?? data`,
nested-preferred with flat fallback.)

**Failure sequence (the `box_rotates` case).** Push-panel emitted
`if.event.pushed { messageId: 'dungeo.mirror.box_rotates', direction: 'north' }` (flat).
Per event: domain path claims it → binds `{}` → `{verbatim:direction}` unbound → phrase
render fails with the stderr `[phrase] renderMessage failed: param 'direction' is not
bound` → no inline `data.message`/`data.text` fallback → returns null → generic handler
binds flat and renders **correctly**. The player sees the right text; the cost is a
failed render attempt + misleading stderr per event — which is why the 2026-06-30
session chased "box_rotates residual ×4" as a rendering bug. Two worse variants:
(a) if the event *also* carries an inline `data.message` string, the domain path consumes
it as the fallback and the correct flat-bound render **never runs** (stale inline text
wins over the template); (b) with a failing template and no fallback anywhere, output is
blank.

**Why it's an authoring trap.** The effective rule is: "params go flat — unless you set
a messageId, then they must nest — unless the type is game.message, which takes either."
Three conventions selected by incidental emit features. The story-side fix applied
(nesting push-panel's params) is whack-a-mole; nothing prevents the next emitter from
repeating it, and the only symptom is unrelated-looking stderr noise.

**Fix options:**
1. **(Recommended)** Make the domain path bind `data.params ?? data` — the
  nested-preferred/flat-fallback rule `handleGameMessage` already uses. One-line change;
  unifies the contract to "params nest under `data.params`; flat fields are the
  fallback"; kills the stderr class; nested still wins when both exist.
2. Keep nested-only, amend ADR-206 to "nest params whenever you emit a messageId," and
  write the planned ADR-206 guard test (Vitest fence: zero `[phrase] renderMessage
  failed` lines in a dungeo transcript run) to enforce it. No behavior change; trap
  remains for anyone who misses the rule.
3. Silence the stderr and rely on the fall-through — keeps the double render attempt and
  the divergent conventions; hides the evidence. Not recommended.

### P6. `registerCapability` semantics — copy + register-once traps — **FIXED (user-directed, 2026-07-02)**

**Fixes applied (all three recommended parts):**
1. **`registerCapability` returns the live stored data object** (`ICapabilityData`) —
   the existing live object on re-registration — so register-then-mutate is correct by
   construction in both branches. `IWorldModel` interface + `WorldModel` impl +
   `AuthorModel` delegate updated; the three rules are documented on the interface
   methods (register-once; deep-copied initialData that must be plain serializable data;
   mutate via the returned/`getCapability` live object or `updateCapability`).
2. **`initialData` is now DEEP-copied** (`structuredClone`) — Trap A′'s
   nesting-depth-dependent semantics are gone; function-valued fields throw (the honest
   failure for data that could never serialize). Note: a class instance clones to a
   plain data husk (methods lost) rather than throwing — one more reason services don't
   belong here.
3. **`dc.ts` misuse reworked**: the GDT scheduler handle moved out of the capability
   store into a module-level `WeakMap<world, ISchedulerService>` — per-world lookup, no
   serializable-state pollution, no dead `try/catch`; the `dungeo.scheduler.ref`
   capability key and `SCHEDULER_GDT_KEY` export are gone.
- **Tests:** 5 new (`world-model/tests/unit/world/capability-data.test.ts`) — live-object
  return, re-register-returns-existing, deep-copy detachment at nested depth, schema
  defaults + overrides, function-valued rejection.
- **Verified:** world-model 1306 / stdlib 1294 / engine 461 green; GDT `dc` verified
  live (22 daemons, 7 fuses through the WeakMap); dungeo units green ×2; friendly-zoo
  green; chain full pass.

Original analysis (for the record):

**The contract as implemented** (`packages/world-model/src/world/WorldModel.ts:462-524`,
ADR-129 data capabilities):
- **Register-once.** If the name is already registered, `registerCapability` silently
  no-ops (non-strict mode) or throws (strict). Re-registration never resets or merges
  data.
- **`initialData` is shallow-copied.** A fresh data object is built (schema defaults
  first, then `Object.assign(fresh, initialData)`). Top-level fields are detached from
  the caller's object; **nested objects/arrays are shared by reference**.
- **`getCapability` returns the live stored object** (not a copy) — mutating it directly
  works and persists.
- **`updateCapability` merges** (`Object.assign(stored, updates)`); silently no-ops when
  the capability isn't registered (non-strict).
- **Capability data serializes with the world** — everything in it lands in save blobs.

**The three traps:**
- **Trap A — register-then-mutate-local** (the flooding bug, S4): register with a local
  object, keep mutating the *local* object's top-level fields → the world's copy never
  changes. Dungeo's flooding daemon read `floodingLevel: 0` forever.
- **Trap A′ — shallow-copy inconsistency**: the *same buggy pattern accidentally works*
  when the mutated field lives one level deeper (nested objects are shared references).
  Whether register-then-mutate-local is correct depends on the nesting depth of the
  field — the worst kind of semantics to reason about.
- **Trap B — silent register-once**: callers who re-register expecting a reset (or an
  error) get neither. Bit twice this session: a test seeding data via re-registration
  (silently ignored), and `dc.ts`'s `try/catch` fallback that only ever runs in strict
  mode — in normal mode the catch branch is dead code and re-registration does nothing
  at all.

**Audit — all 15 `registerCapability` call sites (2026-07-02):**

| Site | Pattern | Verdict |
|------|---------|---------|
| engine `game-engine.ts:220` COMMAND_HISTORY | schema only, writes via updateCapability | SAFE |
| engine `game-engine.ts:226` TEXT_STATE | writes via updateCapability (`render-context.ts:157`) | SAFE |
| engine `game-engine.ts:420` storyInfo | writes via updateCapability (`refreshStoryInfoCapability`) | SAFE |
| engine `setStory` StoryInfo (P3 fix) | entity trait, not capability data | n/a |
| stdlib `capabilities/index.ts:75` (registerStandardCapabilities) | schema only | SAFE |
| stdlib `help.ts:130` GAME_META (P3 residual fix) | register-if-absent + updateCapability | SAFE |
| platform-browser `SaveManager.ts:363` sharedData | hasCapability branch + updateCapability | SAFE |
| world-model `AuthorModel.ts:352` | pass-through delegate | n/a |
| cloak-of-darkness `index.ts:102` debug | schema only | SAFE |
| dungeo `index.ts:235` SCORING | writers mutate the live `getCapability` object (death-penalty) | SAFE |
| dungeo `dam-state.ts:26` DAM_STATE | setters mutate the live `getCapability` object | SAFE |
| dungeo `maintenance-room-fuse.ts:97` FLOODING_STATE | was Trap A — **FIXED this session** (re-fetch after register) | FIXED |
| dungeo `gdt/commands/dc.ts:127` `dungeo.scheduler.ref` | **MISUSE** — see below | FLAGGED |

**The one bad citizen — `dc.ts` `setSchedulerForGDT`:** stores a live
`ISchedulerService` **service instance** as `initialData`. Three problems: (1) the
shallow copy takes only *own enumerable properties* — class prototype methods are
dropped, so the stored "scheduler" works only by luck of the service's implementation
shape; (2) a runtime service reference sits in **serializable capability data** (save
blobs); (3) the `catch` fallback expects the strict-mode throw and is dead code in
normal mode (Trap B). A runtime service handle doesn't belong in the capability store at
all — a module-level reference (like `setSchedulerForGDT` already is) or an engine
accessor is the right home.

**Fix options:**
1. **(Recommended)** `registerCapability` returns the live stored data object
   (`ICapabilityData`) — register-then-mutate becomes correct by construction; existing
   void-context callers unaffected. Pair with doc comments on
   `IWorldModel.registerCapability`/`updateCapability`/`getCapability` stating the three
   rules ("data is copied; re-register is a silent no-op; mutate via the getCapability
   object or updateCapability").
2. Additionally deep-copy `initialData` so Trap A′'s accidental-works becomes an honest
   break (consistent semantics at any nesting depth).
3. Story cleanup: rework `dc.ts` to keep the scheduler in a module-level ref instead of
   the capability store.

  3. Push-wall re-describe (`stories/dungeo/src/messages/puzzle-messages.ts`):
     `'{text}'` → `'{verbatim:text}'` — `text` is a generated room description.
- A repo-wide sweep for other bare full-sentence single-param templates
  (`'{text}'`/`'{reason}'`/`'{message}'`) found no further instances.
- **Verified:** `read frame` (mounted canvas) and `push south wall` (royal puzzle) both

## 4. Why the walkthrough chain cannot be deterministic (one-good-run rule)

The chain's residual flakes (~25–40% of runs) are **canonical MDL behavior**:

- **Thief steals the torch from your possession.** MDL `ROB-ADV`
  (`original_source/util.16`) takes any object with `OTVAL > 0` lacking `SACREDBIT`; the
  torch (`OTVAL 6`, dung.355:5778) is not sacred. When it happens mid-route, every
  subsequent dark-room assertion fails before any scripted guard could react — a scripted
  transcript cannot express "notice the theft and switch to the lantern on the next move".
- **Combat RNG** — disarm/stagger outcomes (mitigated with WHILE-take-weapon guards where
  scripted fights exist: wt-01, wt-13, and the combat unit transcripts).
- **Carousel rooms** — random exits (`PROB 50`/8-way), handled by documented WHILE/NAVIGATE
  retry loops.

Full determinism would require seeded/injectable RNG (explicitly deferred per
session-20260701-0046 user decision) or MDL-deviating story changes (e.g. a SACREDBIT-style
exemption for the torch). **Adopted standard (user, 2026-07-02): one fully-passing chain
run is the baseline; the unit suite is the deterministic gate.** Recorded in Claude memory
(`one-good-run-rule`).

Chain-robustness improvements made: lantern battery is now banked at lit Living Room
checkpoints (wt-13/15/16 turn it off, wt-16/17 turn it on before descending) — the wt-17
grue death was battery exhaustion after the lantern burned across four chained
walkthroughs.
